> [!info] HTTP | MDN  
> 超文本传输协议（HTTP）是一个用于传输超媒体文档（例如 HTML）的应用层协议。它是为 Web 浏览器与 Web 服务器之间的通信而设计的，但也可以用于其他目的。HTTP 遵循经典的客户端—服务端模型，客户端打开一个连接以发出请求，然后等待直到收到服务器端响应。HTTP 是无状态协议，这意味着服务器不会在两个请求之间保留任何数据（状态）。  
> [https://developer.mozilla.org/zh-CN/docs/Web/HTTP](https://developer.mozilla.org/zh-CN/docs/Web/HTTP)  
# 1. **HTTP 基本概念**

HTTP 是**超文本传输协议**，也就是HyperText Transfer Protocol

1. HTTP 是一个**在**==**两点之间**==**，**==**传输**==**文字、图片、音频、视频等**==**超文本数据**==**的**==**约定和规范**==
1. **主要是为** ==**Web 浏览器与 Web 服务器之间**==**的通信而设计的**
1. **HTTP 协是**==**基于 TCP 协议**==**，**发送 HTTP 请求之前首先要建立 TCP 连接也就是要经历 3 次握手
    
    1. 目前使用的 HTTP 协议大部分都是 1.1
    
    1. 在 1.1 的协议里面，默认是开启了 `Keep-Alive` 的，这样的话建立的连接就可以在多次请求中被复用了。
    
1. **HTTP 协议是”**==**无状态**==**”的协议，它无法记录客户端用户的状态**，一般我们都是通过 Session 来记录客户端用户的状态
# 2. HTTP 协议通信过程
1. HTTP 是应用层协议，它以 TCP（传输层）作为底层协议，默认端口为 80
1. 通信过程主要如下：
    
    1. 服务器在 80 端口等待客户的请求
    
    1. 浏览器发起到服务器的 TCP 连接（创建套接字 Socket）
    
    1. 服务器接收来自浏览器的 TCP 连接
    
    1. 浏览器（HTTP 客户端）与 Web 服务器（HTTP 服务器）交换 HTTP 消息
    
    1. 关闭 TCP 连接
    
# 3. **HTTP 状态码**
1. **`1xx`**：属于==提示信息==，是协议处理中的一种中间状态，实际用到的比较少
1. **`2xx`**：表示服务器==成功处理==了客户端的请求，也是我们最愿意看到的状态
    
    1. `200 OK`：是最常见的成功状态码，表示一切正常。如果是**非 HEAD 请求**，服务器返回的响应头都会有 body 数据
    
    1. `204 No Content`：也是常见的成功状态码，与 200 OK 基本相同，但响应头没有 body 数据
    
    1. `206 Partial Content`：是应用于 HTTP 分块下载或断点续传，表示响应返回的 body 数据并不是资源的全部，而是其中的一部分，也是服务器处理成功的状态
    
1. **`3xx`** ：表示客户端请求的资源发送了变动，==需要客户端用新的 URL 重新发送请求获取资源==，也就是==重定向==
    
    1. `301 Moved Permanently`：表示永久重定向，说明请求的**资源已经不存在了**，**需改用新的 URL 再次访问**
    
    1. `302 Found`：表示临时重定向，说明请求的**资源还在**，但**暂时需要用另一个 URL 来访问**
        
        - 301 和 302 都会在响应头里使用字段 `Location` ，指明后续要 跳转的 URL，浏览器会自动重定向新的URL
        
        - 301 和 302 在用户看来并没有区别，都会重定向到新地址，[[4.1 反向代理]]
        
    
    1. `304 Not Modified`：不具有跳转的含义，表示资源未修改，重定向已存在的缓冲文件，也称缓存重定向，用于缓存控制
    
1. **`4xx`**：表示客户端发送的==报文有误，服务器无法处理==，也就是错误码的含义
    
    1. `400 Bad Request`：表示客户端请求的报文有错误，但只是个笼统的错误
    
    1. `403 Forbidden`：表示**服务器禁止访问资源**，并不是客户端的请求出错
    
    1. `404 Not Found`：表示请求的**资源在服务器上不存在或未找到**，所以无法提供给客户端
    
1. **`5xx`**：表示客户端请求报文正确，但是==服务器处理时内部发生了错误，属于服务器端的错误码==
    
    1. `500 Internal Server Error`：与 400 类型，是个笼统通用的错误码，服务器发生了什么错误，我们并不知道
    
    1. `501 Not Implemented`：表示客户端**请求的功能还不支持**，类似“即将开业，敬请期待”的意思
    
    1. `502 Bad Gateway`：通常是服务器作为网关或代理时返回的错误码，表示**服务器自身工作正常，访问后端服务器发生了错误**
    
    1. `503 Service Unavailable`：表示服务器**当前很忙，暂时无法响应服务器**，类似“网络服务正忙，请稍后重试”的意思

## 4.1 请求报文头中
1. `**Host**` **字段**
    
    1. 客户端发送请求时，用来**指定服务器的域名**
        
        ```Plain
         Host: www.A.com
        ```
        
    
    1. 有了 Host 字段，就可以将请求发往「同一台」服务器上的不同网站
        

        ![[IMG-20260405035539246.png|Untitled 2 251.png]]

        
    
1. `**Connection**` **字段**
    
    1. 常用于客户端要求服务器使用 TCP **持久连接**，以便其他请求复用（请求完成后，是断开还是保持连接）
    
    1. [[7.3 HTTP 的版本优化]]，但为了兼容老版本的 HTTP，需要指定 Connection 首部字段的值为 `Keep-Alive`
        
        ```Plain
         Connection: keep-alive
        ```
        
    
    1. 这样一个可以复用的 TCP 连接就建立了，**直到客户端或服务器主动关闭连接**。但是注意：这不是标准字段
    
1. `**Accept**` **字段**
    
    1. 客户端请求的时候，可以使用 Accept字段声明自己**可以接受哪些数据格式**
    
    1. 示例
        
        ```Plain
         Accept: */*
         //客户端声明自己可以接受任何格式的数据
        ```
        
    
1. `**Accept-Encoding**` **字段**
    
    1. 客户端在请求时，用 Accept-Encoding 字段说明自己**可以接受哪些压缩方法**
    
    1. 示例
        
        ```Plain
         Accept-Encoding: gzip, deflate
        ```
        
    

    ![[IMG-20260405035544098.png|Untitled 3 189.png]]

    
## 4.2 响应报文头中
1. `**Content-Length**` **字段**
    
    1. 服务器在返回数据时，会有 Content-Length 字段，表明**本次回应的数据长度**
    
    1. 示例
        
        ```Plain
         Content-Length: 1000
         //本次服务器回应的数据长度是 1000 个字节，后面的字节就属于下一个回应了
        ```
        
    
1. `**Content-Type**` **字段**
    
    1. 用于服务器回应客户端时，**告诉客户端本次数据是什么格式**
    
    1. 示例
        
        ```Plain
         Content-Type: text/html; charset=utf-8
         //表明发送的是网页，而且编码是UTF-8
        ```
        
    
1. `**Content-Encoding**` **字段**
    
    1. Content-Encoding 字段说明数据的压缩方法。表示**服务器返回的数据使用了什么压缩格式**
    
    1. 示例
        
        ```YAML
         Content-Encoding: gzip
             //表示服务器返回的数据采用了 gzip 方式压缩，告知客户端需要用此方式解压
        ```

# 5. **HTTP 报文结构 ⭐**
1. HTTP是面向文本的，因此报文中的每个字段都是一些ASCI码串，并且每个字段的长度都是不确定的
1. ==整体分为四部分：====**请求行、请求报文头、空行、正文**==
1. 有两类 HTTP 报文
    
    1. **请求报文：**从客户向服务器发送的请求报文
        
        - 请求行包括：**请求方法、URL、HTTP 版本**
        
        - 请求报文头中的字段包括：`**Host**` 、`**accept**`、`**Connection**` 、`**Accept-Encoding**`
        

        ![[IMG-20260405035535212.png|Untitled 1 109.png]]

        
    
    1. **响应报文：**从服务器到客户的回答
        
        - 请求行包括：**HTTP 版本、状态码、短语**
        
        - 请求报文头中的字段包括：`**Content-Length**` 、`**Content-Type**`、`**Content-Encoding**`
        

        ![[IMG-20260405035547248.png|Untitled 2 91.png]]
