> [!important] **POST 与 GET 的区别**
> 
> ❓
> 
> 1. 从**作用**的角度，GET 方法用来**请求访问已被 URI 识别的资源，**POST 方法用来**传输实体**，与 GET 相反，它**向 URI 指定的资源提交数据**
> 
> 1. 从**参数**的角度，GET 一般放在 URL 中，因此不安全，POST 放在请求体中，更适合传输敏感信息
> 
> 1. 从**幂等性**的角度，GET是幂等的，而POST不是
> 
> 1. **GET产生一个TCP数据包；POST产生两个TCP数据包**
>     
>     1. 对于GET方式的请求，浏览器会把 http header 和 data 一并发送出去，服务器响应 `200` (返回数据)
>     
>     1. 而对于POST，浏览器先发送 header，服务器响应 `100` continue，浏览器再发送data，服务器响应 `200` ok (返回数据)
>     
> 
> 1. 从**缓存**的角度，GET 请求会被浏览器主动缓存下来，留下历史记录，而 POST 默认不会
# 1. GET 获取资源
1. GET 方法==用来====**请求访问已被 URI 识别的资源**==，这个资源可以是静态的文本、页面、图片视频等
1. 指定的资源经服务器端解析后返回响应内容
    
    1. 如果请求的资源是文本，那就保持原样返回
    
    1. 如果是像 CGI（Common Gateway Interface，通用网关接口）那样的程序，则返回经过执行后的输出结果
    
1. 浏览器发送 GET 请求的情况：
    
    - 直接在浏览器地址栏输入某个地址
    
    - 点击链接
    
    - 表单默认的提交方式
    
1. GET 方法就是**安全且幂等**的，因为它是「只读」操作，无论操作多少次，服务器上的数据都是安全的，且每次的结果都是相同的
1. GET 能够携带的**参数比较少**，参数会添加到请求资源路径的后面，大小有限制，会在浏览器的 URL 栏显示数据内容，不安全，但高效

# 2. POST 传输实体主体
1. POST 方法==用来====**传输实体的主体**====，与 GET 相反，它====**向 URI 指定的资源提交数据**==，数据就放在报文的 body 里
1. 当敲入了表单后点击提交
    
    1. 浏览器就会执行一次 POST 请求，把留言文字放进了报文 body 里
    
    1. 然后拼接好 POST 请求头，通过 TCP 协议发送给服务器
    
1. 浏览器只有在表单 `method = "post"` 的情况下才会发送 POST 请求，如果要测试需要借助 postman 工具
1. POST 因为是「新增或提交数据」的操作，会修改服务器上的资源，所以是**不安全**的，且多次提交数据就会创建多个资源，所以**不是幂等**的
1. POST **能够携带的参数没有限制**，不会在浏览器的 URL 栏显示数据内容，相对安全，但不高效

# 3. PUT 传输文件
1. PUT 方法用来传输文件。就像 FTP 协议的文件上传一样，要求在请求报文的主体中包含文件内容，然后保存到请求 URI 指定的位置
1. 但是，鉴于 ==**HTTP/1.1 的 PUT 方法自身不带验证机制**==，任何人都可以上传文件，存在安全性问题，因此**一般的 Web 网站不使用该方法**
1. 若配合 Web 应用程序的验证机制，或架构设计采用 **REST**（`REpresentational State Transfer`，表征状态转移）[[Restful 请求风格]]标准的同类Web 网站，就可能会开放使用 PUT 方法

# 5. DELETE 删除文件
1. DELETE 方法用来删除文件，是与 PUT 相反的方法
1. DELETE 方法按请求 URI删除指定的资源
1. 但是，HTTP/1.1的 DELETE 方法本身和 PUT 方法一样**不带验证机制，所以一般的 Web 网站也不使用 DELETE 方法**
1. 当配合 Web 应用程序的验证机制，或遵守 `REST` 标准时还是有可能会开放使用的
