# 一、I/O
1. Linux系统为了提高IO效率，会在用户空间和内核空间都加入缓冲区
2. 在操作系统中，输入输出操作通常都会包括以下两个阶段：
    1. **准备数据**：内核缓冲区准备数据，等待其准备好
    2. **数据拷贝**：从内核缓冲区向用户缓冲区复制数据

    ![[IMG-20260405035540073.png|471]]

3. ⭐因此整个过程中**影响 IO 效率的因素**主要有：
    1. **等待数据就绪的时间**
    2. 从内核空间的缓冲区和用户缓冲区之间，**等待数据拷贝完成的时间**
4. 以网络通信即 Socket 上的输入操作为例

    ![[IMG-20260405035544548.png|800]]

# 二、Linux 的五种 I/O 模型
> ❗ **操作系统系统管理输入和输出的方式就是 I/O 模型**
## 1. BIO
1. Blocking I/O 对应阻塞的系统调用 `recvfrom`

    ![[IMG-20260405035546530.png|668]]

    1. 第一阶段，**准备数据**：
        1. 当用户进程通过系统调用 `recvfrom` 进行数据读取，操作系统就开始了 I/O 的第一个阶段-准备数据
        2. 这个过程需要等待，也就是说数据被拷贝到操作系统内核的缓冲区中是需要一个过程的
        3. 而在用户进程这边，整个进程会被阻塞住
    2. 第二阶段，**数据拷贝**：
        1. 当内核一直等到数据准备好了，它就会将数据从内核空间中拷贝到用户空间
        2. 然后系统调用 `recvfrom` 返回结果，用户进程才解除阻塞的状态，重新运行起来
2. **从调用** **`recvfrom`** **开始，到它返回数据的整段时间，用户进程都是被阻塞住的**
## 2. NIO
> ❗
> 
> - 此 NIO 并不是 Java NIO，**Java NIO 是 Linux 中 NIO 和 多路复用的结合**
> 
> - NIO 的就是**用户进程在第一阶段，从 BIO 中被阻塞啥都干不了，变为非阻塞，可以做其他的，再轮询获得数据准备结果**
1. 如果某个用户进程进行系统调用 `recvform` 尝试获取数据，但这时候数据还没准备好：
    - 如果操作系统把这个进程挂起，那就是 Blocking I/O
    - ==如果操作系统选择====**立即给用户进程返回错误信息**====，那就是 Non-Blocking I/O==

    ![[IMG-20260405035549498.png|674]]

2. 第一阶段，**准备数据**：
    1. 非阻塞的 `recvform` 系统调用之后，如果数据还没准备好，应用进程不会被阻塞住，`recvfrom` 立即返回一个 `EWOULDBLOCK` 错误
    2. 用户进程在收到 `recvfrom` 调用的返回信息之后，可以干点别的事情，然后再发起 recvform 系统调用
    3. 用户需要==一直轮询==，直到数据准备好，再进入拷贝阶段
3. 第二阶段，**数据拷贝**：
    1. 同 BIO
    2. 需要注意的是，当 `recvfrom` 系统调用进行拷贝数据的时候，用户进程同样是被阻塞住的
4. **==虽然是非阻塞，但性能并没有得到提高，只是在不停的轮询再试==**
## 3. IO 多路复用
1. 由于后台可能有多个任务，NIO 需要不断主动轮询，轮询会消耗大量的 CPU 时间
2. **IO 多路复用**
    1. 引入了新的系统调用 `select/poll/epoll`（也成为多路复用器）
    2. 具体来说， IO 多路复用就是==将多个应用进程的 Socket 注册到一个多路复用器（select/poll/epoll）上==
    3. 然后使用==一个进程来监听该多路复用器==，==多路复用器会不断的====**轮询**====所有注册进来的 Socket，只要有一个 Socket 的数据准备好，就会返回该 Socket==
    4. 再由应用进程发起真正的 IO 系统调用（也就是 `recvfrom`，和 **Blocking I/O** 一样），来完成数据读取

        ![[IMG-20260405035552034.png|800]]

3. 性能
    1. 比较上图和 Blocking I/O，你会发现 I/O Multiplexing 的 I/O 操作和 Blocking I/O 似乎差不多，事实上，IO 多路复用还更差一些，因为这里需要使用两个系统调用 (`select` 和 `recvfrom`)，而 Blocking IO 只需要一个系统调用 (`recvfrom`)
    2. 但是，**IO 多路复用的优势并不是对单个连接能处理得更快，而是只需要一个进程就可以同时处理多个 I/O，能同时处理更多的连接**
4. ==⭐️====**IO 多路复用详解**==🙋‍♂️
    - 阅读全文
        1. **用户进程如何知道内核中数据是否就绪呢❓**
            1. 使用**文件描述符**，简称`FD`，是一个从0 开始的无符号整数，用来关联Linux中的一个文件
            2. 在Linux中，一切皆文件，例如常规文件、视频、硬件设备等，当然也包括网络套接字（Socket）
        2. 在Linux中，多路复用指的是一种实现**同时监控多个文件描述符 `FD`**（包括Socket，文件和标准输入输出等）的技术
        3. 总结：
            - **多路（多个客户端）**
            - **复用（复用一次系统调用或复用一个线程来监听多个客户端 `FD` ，处理客户端的链接请求和文件 IO 操作）**
            - **并在某个 `FD` 可读、==可写时得到通知==，从而避免无效的等待，充分利用CPU资源，提高程序的效率和响应速度**
        4. **监听-通知 方式⭐**
            - IO多路复用在监听 FD 的方式、通知的方式又有多种实现，常见的有：
                1. `select`
                    - 需要传入要监视的文件描述符数量，以及文件描述符集合
                    - 但最大支持传入`1024`个文件描述符
                    - 会阻塞整个进程
                2. `poll`
                    - 同样需要传入要监视的文件描述符数量，以及文件描述符集合
                    - 但不限制文件描述符数量（利用链表实现），这样意味着更少的系统调用
                    - 可以只阻塞等待的文件描述符。
                3. `epoll` ：
                    - **上面两种方式的缺陷**
                        1. 只会通知用户进程有 `FD` 就绪，但不确定具体是哪个 `FD`，需要用户进程逐个遍历 `FD` 来确认
                        2. 大量的 `FD` 需要在用户态和内核态互相拷贝
                    - **`epoll` 优化**
                        - 使用红黑树保存要监听的 `FD`，理论上大小无上限，而且增删改查效率都非常高
                        - 使用事件通知机制（`ep_poll_callback`），直接可以拿到已经就绪的 `FD`，无需遍历所有 `FD`，监听性能不会随监听的`FD` 数量增多而下降
                        - 无需从用户空间重复拷贝`FD`到内核空间

## 4. 信号驱动 IO
1. Signal Blocking I/O 就是当用户进程发起 I/O 操作的时候，首先通过系统调用 `sigaction` 向内核注册一个信号处理函数，这个系统调用会立即返回不会阻塞用户进程
2. **当内核数据准备好了就会发送一个** **`SIGIO`** **信号给用户进程**，这样用户进程就知道内核数据准备好了，可以开始执行 I/O 系统调用了
![[IMG-20260405035554715.png|800]]
3. 和 Non-Blocking I/O 一样，信号驱动 IO 的用户进程在 I/O 的第一阶段准备数据是非阻塞的，在第二阶段数据拷贝是阻塞的
4. 不过信号驱动 IO 基于回调机制，其实现和开发应用难度大，因此在实际中并不常用
5. 使用很少，并发场景下存在问题
## 5. 异步 IO
1. 异步IO的整个过程都是非阻塞的，用户进程调用完异步API后就可以去做其它事情，内核等待数据就绪并拷贝到用户空间后才会递交信号，通知用户进程
2. 领导发下任务就不管了，员工做完通知我
3. 内存图解：

    ![[IMG-20260405035556300.png|800]]

    1. 阶段一：
        1. 用户进程调用aio_read，创建信号回调函数
        2. 内核等待数据就绪
        3. 用户进程无需阻塞，可以做任何事情
    2. 阶段二：
        1. 内核数据就绪
        2. 内核数据拷贝到用户缓冲区
        3. 拷贝完成，内核递交信号触发aio_read中的回调函数
        4. 用户进程处理数据
