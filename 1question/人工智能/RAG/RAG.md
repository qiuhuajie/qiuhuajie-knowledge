# RAG 简介
1. 通过 RAG 我们可以把一些企业的知识还有一些我们自己所拥有的一些私有化的知识库跟大模型可以融合在一起，来解决一些用户的问题
2. RAG 的整体流程：
    ![[IMG-20260404032004922.png|image 5.png|512]]
    1. RAG 的核心就是知识库，知识库由大量的文本构成。
        1. 文档加载是 RAG 的一个重要功能，它最大的优势在于不需要将文档内容转换成某种固定的格式。在技术人员写索引时，通常需要将文档内容转换成某种结构化数据或特定语义的索引结构。
        2. 但是使用 RAG 后，我们可以直接将文档内容输入，大模型就能理解。它能够处理结构化数据、非结构化数据或其他语义性数据，并基于内容实现内部检索方式，从而检索到用户期望的答案。
    2. 这些文本存放在向量数据库，因为后续需要针对知识库进行一些检索。即从这个库里面找到跟这个问题文本类似的其他的文本，使用向量数据库可以加速检索的效率
    3. 放到向量数据库之前，先要把这个文本做一个切分，即把整个的文章分成若干个段落（chunks）
        > 具体如何去把它切分成段落呢当然这里面有很多的一些方法论，详见。。。
    4. 每个段落在存放到向量数据库之前，需要将每个段落转换成一个向量化的形式，即 embedding 向量化
    5. 接着，用户问了一个问题
        1. 这个问题首先要转换（embedding）成一个向量，接下来在向量数据库里面去检索，有可能包含此问题答案的那个段落
        2. 向量数据库返回有可能包含这个问题答案的一些段落（比如说前十个段落）
        3. 再把 topk 段落放在一起，作为检索结果 Context，这个 Context 里面有可能包含这个问题的答案，或者说至少有一定的线索在 Context 里面的
        4. 接着要把用户问题和 Context 合在一个 prompt 里面。让大模型基于这个 prompt 去回答问题
3. RAG 实战中的常见问题 ⚠️
    ![[IMG-20260404032005008.png|image 1 3.png|536]]
    1. **文档分割：**
        1. 不同的文档（PDF、PPT、Excel）实际上格式可能也是不一样的，所以在文档的读取和处理阶段，它本身是一个比较难的问题
        2. 还要考虑，文档读取完之后，如何把文档能合理的把它分割成不同的 chunks
    2. **Embedding 技术：**如何对文档 chunks 合理的做 embedding。这里面有很多不同的 embedding 的技术，另外当我们得到向量之后，到底使用什么样的向量数据库，但这个本质上也是一个工程化的问题
    3. **用户问题增强：**用户的问题有可能是需要进一步处理的，比如用户问问题的时候，这个问题本身是一个可能跟上下文关系不大或者是很简短的问题，这时大概率需要对问题做一个类似于扩充或者是重新去改写问题的一个过程
    4. **检索出 Context：**拿到问题之后，如何高效且合理的检索得到 Context，这里会大量的会使用到跟传统的搜索引擎相关的一些技术：==召回（粗排）+ 精排==
        1. 检索完之后我们会得到一些 context，但这里的context有可能会包含很多无关的内容
        2. 因此需要加入一个排序过程，即先检索再排序：检索的过程可以大致认为是一个粗排，也把它叫召回的过程
        3. 召回完之后，再通过一些比较精细化的手段和算法，把相应的信息做一个精排
    5. **Prompt 设计：**好的prompt可以让大模型激发更多潜能
    6. **模型选择：**通用的大模型、开源的大模型 + 微调
    7. **Response 的检查机制 + 二次加工：**得到 LLM 的 Response 后，这个回复可能还需要经过第二次处理，因为有些回复确实没有办法满足要求，或者甚至违背了我们的一些规范，所以这里面通常会加入一个检查的机制，然后检查完之后再返回给用户，如果没有通过，再返回到其中的一个路径重新来进行刚才的这些流程。
4. 为什么需要向量数据库❓
    1. 向量检索的复杂度：`O(N) * O(d)`
        ![[IMG-20260404032005086.png|image 2 3.png|547]]
        1. 向量的检索，本质就是使用问题 embedding 后的向量，与存储的向量依次做相似度计算，最终取 TopK 个向量返回，这一过程的复杂度是 `O(N)`，N 是向量的数量。
        2. 相似度计算一般使用的是余弦相似度，即在一个空间中，计算两个向量的相似度，本质是一个分数，在空间上表现为两个向量夹角越近，则认为这两个向量语义上越接近。这一过程的复杂度是 `O(d)`，d 是向量空间的维度。
    2. 但在实际场景中，我们希望系统可以对用户的问题进行快速的回复，所以使用向量数据库，本质上是希望降低检索的复杂度

# RAG 与 微调如何选型？
![[IMG-20260404032005141.png|image 3 3.png|409]]
1. 动态数据：适合使用 RAG，如果使用微调，每次数据的改动都需要去重新微调模型。
2. 模型能力的定制：希望模型拥有特殊的能力，基座模型不具备的能力，如研读金融研报、以某种口吻回复用户问题。
3. 幻觉：RAG 和 微调都可以降低模型幻觉。
4. 可解释性：我们希望模型告诉我们为什么？RAG 这方面大于微调，对于微调来说，模型内部工作更黑盒，很难追溯到某个单词为什么生成了。
5. 成本：RAG 首选，只需要工程化搭建，微调需要数据收集成本 + 训练成本 + 测试后的迭代。
6. 依赖通用能力：微调可能造成模型的遗忘。
7. 延迟：低延迟场景下，微调是首选。RAG 包含了检索 + 精排的流程，相对耗时
8. 智能设备：需要在某一方面比较强，选择微调。
