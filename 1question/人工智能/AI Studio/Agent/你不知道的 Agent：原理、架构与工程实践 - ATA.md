# 你不知道的 Agent：原理、架构与工程实践
> [!info] 原文信息
> 作者：汤威（侑夕）
> 来源：ATA / 飞猪 CTO 线
> 发表：3月23日，3月23日更新
> 链接：[原文](https://ata.atatech.org/articles/11020572828)
> 说明：原始抓取文本里没有明确年份，这里保留页面原始日期写法。
> [!abstract] 核心摘要
> 这篇文章的重点不是“模型换强一点就能把 Agent 做好”，而是把 Agent 当成一套完整的软件工程系统来设计。
> 真正影响落地效果的，往往不是主循环本身，而是 Harness、上下文工程、工具设计、记忆、评测、追踪与安全边界这些外围基础设施。
> 从实现角度看，Agent 的主循环通常很稳定，复杂度主要长在循环外部：状态如何外化、知识如何按需加载、工具如何设计、任务如何评测、错误如何追踪、风险如何隔离。

## 我的整理
### 一句话结论
能落地的 Agent，本质上是“稳定循环 + 外部状态 + 可验证边界”。
### 全文主线
1. 先解释 Agent Loop 和常见控制模式。
2. 再说明为什么 Harness、上下文工程、工具设计会决定系统稳定性。
3. 然后拆解记忆、自主度、多 Agent、评测、追踪与安全。
4. 最后用 OpenClaw 说明这些原则如何变成一套可运行的工程系统。

### 最值得记住的判断
- Agent 的主循环通常不是瓶颈，复杂度主要来自循环外部的工程系统。
- Workflow 和 Agent 的本质区别，在于控制权是否交给 LLM 动态决策。
- 高质量 Harness 往往比单纯升级模型更能提升成功率。
- 上下文问题常常不是窗口不够长，而是信号密度太低。
- Skills 应该像路由条件，而不是冗长的功能说明书。
- 工具应该面向 Agent 的目标设计，而不是直接映射底层 API。
- 记忆必须分层，并且整合过程要支持回退。
- 长任务要把进度写到文件或数据库里，不要依赖上下文来“记住现场”。
- 多 Agent 先解决协议与隔离，再谈并行收益。
- Agent 退化时，先修评测系统，再改 Agent 本体。

## 一、Agent 的底层运行模型
### 1.1 Agent Loop 是稳定的最小内核
文章强调，Agent Loop 的核心其实很简单，绝大多数实现都可以抽象成下面这类循环：
```typescript
const messages = [{ role: "user", content: userInput }];
while (true) {
  const response = await client.messages.create({
    model,
    tools,
    messages,
  });
  if (response.stop_reason === "tool_use") {
    const toolResults = await runTools(response.content);
    messages.push({ role: "assistant", content: response.content });
    messages.push({ role: "user", content: toolResults });
    continue;
  }
  return response.text;
}
```
可以把这个循环理解成四个阶段：
- 感知：读取当前输入和上下文。
- 决策：由模型决定下一步是回答还是调用工具。
- 行动：执行工具或发起外部操作。
- 反馈：把结果写回上下文，进入下一轮。

文章的关键判断是：主循环本身很稳定，新能力通常不是改 loop，而是加在 loop 外面。
- 工具扩展：让 Agent 能做更多事情。
- 提示结构调整：让 Agent 更稳地做对事情。
- 状态外化：把长期状态放到文件、数据库或记忆层里。

### 1.2 Workflow 和 Agent 的区别
Anthropic 给出的区分很直接：
- Workflow：执行路径由代码预先写死。
- Agent：下一步由 LLM 在运行时动态决定。

| 维度 | Workflow | Agent |
| --- | --- | --- |
| 控制权 | 代码预定义 | LLM 动态决策 |
| 执行方式 | 工具顺序固定 | 工具按需选择 |
| 状态管理 | 显式状态机 | 隐式累积在上下文里 |
| 可观测性 | 节点和路径更清晰 | 需要完整 Trace 理解决策 |
| 适用场景 | 边界清晰、流程固定 | 需要中间推理和灵活判断 |
这里没有绝对高下。文章更强调的是：别为了“看起来高级”把所有系统都做成 Agent。
### 1.3 五种常见控制模式
很多所谓 Agent 系统，本质上都是下面几种模式的组合：
1. `Prompt Chaining`：线性拆步骤，适合先大纲后正文、先生成后翻译。
2. `Routing`：按问题类型分流，适合简单问题走轻量模型、复杂问题走强模型。
3. `Parallelization`：拆任务并发执行，或者多次运行后投票取共识。
4. `Orchestrator-Workers`：主 Agent 分解任务，子 Agent 执行并返回结果。
5. `Evaluator-Optimizer`：生成结果后持续评估并优化，直到达标。

工程上真正重要的不是“是不是 Agent”，而是任务适合哪种控制模式。

## 二、为什么 Harness 比模型更关键
### 2.1 Harness 是什么
文章把 Harness 理解为围绕 Agent 构建的一整套“验证与约束基础设施”，至少包括四部分：
- 验收基线：什么算完成，什么算失败。
- 执行边界：哪些动作允许做，哪些动作必须拦截。
- 反馈信号：系统如何知道这一步有没有做对。
- 回退手段：做错之后如何恢复，而不是越错越远。

### 2.2 为什么它往往比模型更重要
作者的核心判断很明确：
- 在代码编写、配置修改、排障这类“可验证任务”中，Harness 对成功率的影响往往大于模型升级。
- 在研究、开放式探索、多轮协商这类“弱验证任务”中，模型上限仍然更关键。

也就是说，Harness 的价值与任务可验证性强相关。
> [!tip] 一个好用的判断标准
> 最适合 Agent 的任务，通常同时满足两件事：
> 目标清晰，且结果可以自动验证。

### 2.3 OpenAI 的 Agent-first 工程实践
文章引用 OpenAI 的实践，总结出几条很重要的工程原则：
- Agent 看不到的内容等于不存在，知识必须进代码库或可读取的文档，而不是散落在人的脑子里。
- 约束要编码化，不要只写在文档里，真正有效的规则应该进入 Linter、类型系统和 CI。
- 任务要尽量端到端闭环，让 Agent 能自己查日志、查指标、验证效果。
- 合并阻力要尽量小，在高吞吐环境里，过多人工阻塞往往比小错误更贵。

### 2.4 Harness 的关键结论
文章实际上给了一个很实用的四象限判断：
- 目标明确 + 可自动验证：最适合 Agent 发挥。
- 目标明确 + 需要人工验收：吞吐量受人审速度限制。
- 目标模糊 + 可自动反馈：系统会高效地朝错误方向跑。
- 目标模糊 + 不能自动验证：Agent 很难稳定工作。

Harness 的作用，就是把任务尽可能往“目标清晰、自动验证”的区域推进。

## 三、上下文工程为什么决定稳定性
### 3.1 问题通常不是窗口不够长，而是信息密度不对
文章指出，很多看起来像“模型变笨了”的问题，其实是 `Context Rot`：
- 无关内容越来越多。
- 高频信息和低频信息混在一起。
- 稳定规则和动态状态没有分层。
- 真正重要的信号被噪声稀释。

所以，上下文工程的目标不是单纯“塞更多内容”，而是“让模型更容易注意到重要内容”。
### 3.2 上下文分层的推荐做法
作者建议按稳定性和使用频率分层：
- 常驻层：身份、硬约束、禁止项，必须短且稳定。
- 按需加载层：Skills 和领域知识，只保留索引，需要时再读取全文。
- 运行时注入层：当前时间、用户信息、渠道信息等动态数据。
- 记忆层：跨会话事实，不直接常驻，需要时检索。
- 系统层：能用 Hook、规则或代码表达的逻辑，就不要塞给模型反复读。

这背后的原则是：
- 稳定内容要短。
- 易变内容要后置。
- 确定性逻辑尽量下沉到系统而不是写进 Prompt。

### 3.3 三种常见压缩策略
| 策略 | 优点 | 代价 | 适用场景 |
| --- | --- | --- | --- |
| 滑动窗口 | 实现简单，成本低 | 早期背景容易丢失 | 短对话 |
| LLM 摘要 | 能保留决策主线 | 细节会损失 | 长任务 |
| 工具结果替换 | 极省 token | 原始输出被裁剪 | 工具密集型任务 |
文章特别提醒，压缩时最容易误删的不是冗余文字，而是这些关键信息：
- 架构决策。
- 已修改文件和关键变更。
- 验证结果。
- 未完成事项和回滚笔记。
- 标识符原文，例如 UUID、端口、URL、文件名、PR 编号。

### 3.4 Prompt Caching 与上下文分层是一件事
Prompt Caching 的前提是“精确前缀一致”，所以缓存友好的上下文通常具备这些特征：
- 系统提示稳定。
- 工具定义尽量稳定。
- 长文档位于稳定前缀。
- 动态信息追加在后面。

这也是为什么文章强调“常驻层短而稳定”。它不只是节省 token，也是在保护缓存命中率。
### 3.5 为什么 Skills 要按需加载
作者把 Skills 总结成一句话：
> 系统提示里只保留索引，完整知识在需要时再加载。

真正有效的 Skill 描述，不是“我能做什么”，而是“什么时候该用我”。
高质量 Skill 描述一般应包含：
- `Use when`：什么情况下用。
- `Don't use when`：什么情况下不要用。
- 反例：避免路由边界模糊。
- 产出物：加载后希望模型得到什么。

文章还给了两个很值得记住的经验：
- 描述要短，不要把工作手册全塞进 description。
- 反例很重要，没有反例时，Skill 路由准确率会明显下降。

### 3.6 文件系统为什么适合做上下文接口
对于长 JSON、检索结果、历史会话，文件系统比直接塞上下文更适合作为中间层：
- 工具把结果写文件。
- Agent 按需 `rg`、读取或脚本过滤。
- 开发者也能直接查看同一份状态。

这样做的好处是：
- 让上下文更干净。
- 让压缩更可回溯。
- 让状态更容易调试和审计。

## 四、工具设计决定 Agent 能做什么
### 4.1 工具质量比工具数量更重要
文章提醒，工具定义本身就会吃掉大量上下文预算，所以问题通常不是“工具不够多”，而是：
- 工具边界不清晰。
- 工具描述写得不像路由条件。
- 返回值和下一步决策无关。
- 错误信息无法指导修复。

### 4.2 好工具和差工具的差别
| 维度 | 好工具 | 差工具 |
| --- | --- | --- |
| 设计对象 | 面向 Agent 目标 | 面向底层 API 操作 |
| 粒度 | 一次完成一个明确任务 | 需要模型自己拼装多个 API |
| 返回值 | 只给后续决策需要的信息 | 原样返回大块冗余数据 |
| 错误处理 | 结构化且有修正建议 | 只有 `"Error"` 之类的字符串 |
| 描述 | 说明何时用、何时不用 | 只讲功能，不讲边界 |
### 4.3 工具设计的三个阶段
作者把工具设计演进概括成三代：
1. API 封装：每个接口暴露成一个工具。
2. ACI：面向 Agent 目标重新定义工具边界。
3. Advanced Tool Use：支持动态搜索、代码编排、多示例增强调用准确率。

这背后的变化，本质上是“从工程师视角的接口设计”转向“从 Agent 视角的任务设计”。
### 4.4 ACI 设计原则
符合 ACI 思路的工具，一般会满足这些要求：
- 工具名直接对应任务目标。
- 参数约束明确，尽量提前防错。
- 定义与实现绑定，减少漂移。
- 错误信息结构化，并带修正建议。
- 尽量提供真实调用示例，而不只给 JSON Schema。

### 4.5 为什么工具消息也要隔离
框架内部事件不应该直接进入 LLM 上下文，例如：
- 某次压缩是否发生。
- 某个通知是否已发送。
- 某个工具是否被跳过。

更稳妥的做法是把消息分层：
- `AgentMessage`：框架内部完整状态。
- `Message`：真正发给 LLM 的标准消息。

这样才能保留框架信息，又不污染模型上下文。

## 五、记忆系统如何设计
### 5.1 记忆不是“存不存”，而是“怎么分层”
文章把记忆分成四类：
| 类型 | 作用 | 典型载体 |
| --- | --- | --- |
| 工作记忆 | 当前任务所需的最小信息 | 上下文窗口 |
| 程序性记忆 | “怎么做”某件事 | Skills |
| 情景记忆 | 历史发生过什么 | JSONL 会话历史 |
| 语义记忆 | 跨会话保留的重要事实 | `MEMORY.md` |
这个划分非常实用，因为它不是按存储介质分，而是按问题类型分。
### 5.2 MEMORY.md 和 Skills 的协作关系
两者的角色不同：
- `MEMORY.md`：沉淀稳定事实，例如偏好、长期结论、关键背景。
- Skills：沉淀做事方法，例如流程、规范、操作指南。

这意味着：
- 事实写进记忆。
- 方法写进 Skill。
- 两者都不应该无限膨胀成默认常驻大 Prompt。

### 5.3 为什么 Markdown 记忆库在很多场景已经够用
文章用 OpenClaw 举例，说明很多 Agent 并不需要一开始就上向量数据库：
- 每日记忆写成 `memory/YYYY-MM-DD.md`。
- 精炼事实写入 `MEMORY.md`。
- 检索时先用关键词和简单排序。

这种方式的优点很工程化：
- 可读。
- 可改。
- 可追溯。
- 成本低。

只有在规模明显变大、确实需要语义相似度检索时，再考虑向量检索更合适。
### 5.4 记忆整合必须可回退
文章非常强调的一点是：
> 真正关键的不是“摘要写得多漂亮”，而是整合过程要可回退。

推荐的做法是：
- token 使用率达到阈值时触发整合。
- 先做摘要。
- 把摘要写入 `MEMORY.md` 或等价位置。
- 只移动整合指针，不直接删除原始消息。
- 整合失败时，把原始消息归档到 `archive/`。

这能保证记忆整合不是一次不可恢复的硬截断。

## 六、如何逐步放开 Agent 自主度
### 6.1 长任务要能跨 Session 继续
文章提出一种很实用的分工方式：
- `Initializer Agent`：只运行一次，负责初始化任务状态。
- `Coding Agent`：多轮反复执行，每次恢复现场、完成一项、验证、提交。

核心思想是：把任务状态写进文件系统，而不是指望模型自己记住。
### 6.2 为什么任务状态必须显式写出来
长任务最怕两件事：
- 任务还没做完，但 Agent 误判为完成。
- Session 结束后，下次无法准确恢复现场。

作者建议用外部状态对象约束当前进度，例如：
```json
{
  "tasks": [
    { "id": "1", "desc": "读取现有配置", "status": "completed" },
    { "id": "2", "desc": "修改数据库 schema", "status": "in_progress" },
    { "id": "3", "desc": "更新 API 接口", "status": "pending" }
  ]
}
```
这里最重要的约束只有一个：
- 同一时间只能有一个 `in_progress`。

### 6.3 慢速 I/O 应该后台化
提高自主度后，真正拖慢主循环的往往不是 LLM 推理，而是：
- 文件操作。
- 网络请求。
- 长耗时命令。

更稳妥的做法是：
- 把慢 I/O 放到后台线程或异步任务里。
- 在每轮开头检查是否有新结果。
- 主循环只决定继续执行、等待还是调整计划。

这比把整个 Agent Loop 改成复杂并发系统更容易维护。

## 七、多 Agent 如何组织
### 7.1 先分清两种协作模式
文章把多 Agent 的工作方式分成两类：
- 指挥者模式：人与单个 Agent 高频同步协作。
- 统筹者模式：主 Agent 异步委派多个子 Agent 并行执行。

多 Agent 的真正价值并不是“同时多开几个模型”，而是把中间过程变成可以持久化的工件，例如分支、PR、任务结果和消息记录。
### 7.2 协作必须先协议化，再并行化
如果多个 Agent 只是靠自然语言互相对齐，很快就会出问题。
文章建议优先建立三样东西：
- 协议：消息结构、状态字段、收发路径。
- 任务图：谁依赖谁，哪些任务可并行。
- 隔离边界：每个子 Agent 的工作目录、权限和上下文范围。

一个典型实现是：
- `.team/inbox/{agentId}.jsonl`：结构化消息收发。
- `.tasks/`：任务图与依赖关系。
- `.worktrees/`：文件修改隔离。

### 7.3 为什么子 Agent 只该回传摘要
子 Agent 适合承担搜索、试错和调试，但这些细节不应该污染主 Agent 上下文。
更好的方式是：
- 子 Agent 保留完整探索历史。
- 主 Agent 只接收结论和必要结果。

这和单 Agent 场景里的“工具结果替换”逻辑是一致的，本质上都是在控制上下文密度。
### 7.4 多 Agent 下，幻觉会互相放大
文章特别提醒，多 Agent 不是天然更可靠。
常见风险是：
- Agent A 先带偏。
- Agent B 顺着这个偏差继续强化。
- Agent C 再把错误总结成“共识”。

所以在多 Agent 场景下，更需要：
- 独立交叉验证。
- 可持久化任务图。
- 结构化协议。
- 外部反馈，例如测试、编译器或人工审查。

### 7.5 子 Agent 的最小提示和深度限制
为了保证隔离，子 Agent 的系统提示应尽量精简，只给它完成任务所需的最小运行信息，不要默认继承 Skills 和 Memory。
另外还要设深度上限，防止无限递归地产生孙 Agent。

## 八、Agent 评测如何做
### 8.1 为什么 Agent 评测比单轮问答复杂很多
单轮问答只需要判断输出文本，而 Agent 评测要同时考虑：
- 任务定义。
- 工具可用性。
- 运行环境。
- 执行过程。
- 环境最终状态。

因此，Agent 评测不能只看“说了什么”，还要看“环境最后真的变成了什么样”。
### 8.2 需要记住的几个评测对象
文章给出了一组很有用的概念：
- `task`：测什么任务。
- `trial`：一次具体运行。
- `grader`：怎么评分。
- `transcript`：执行过程记录。
- `outcome`：环境最终结果。
- `evaluation harness`：负责把任务跑起来、打分并汇总结果的系统。

### 8.3 Pass@k 和 Pass^k 不要混用
| 指标 | 含义 | 更适合回答的问题 |
| --- | --- | --- |
| Pass@k | 跑 k 次，至少一次成功 | 这个 Agent 理论上能不能做到 |
| Pass^k | 跑 k 次，必须每次都成功 | 改动后有没有引入回归 |
文章的建议很明确：
- 开发阶段看 Pass@k，评估能力上限。
- 上线阶段看 Pass^k，评估稳定性和回归风险。

### 8.4 三类评分器
| 类型 | 优点 | 局限 | 适用场景 |
| --- | --- | --- | --- |
| 代码评分器 | 最稳定、噪声最小 | 只适合强约束任务 | 单测、字符串匹配、结构校验 |
| 模型评分器 | 能评估语义质量 | 评分可能漂移 | 风格、质量、推理过程 |
| 人工评分器 | 最可靠 | 慢且贵 | 建基准、校准自动评分器 |
有明确答案时，优先使用代码评分器，这是作者反复强调的一条原则。
### 8.5 如何从零开始搭评测
文章建议非常务实：
- 从 20 到 50 个真实失败案例开始，不要等体系完整再动手。
- 每个测试都从干净环境启动，避免任务间互相污染。
- 正例和反例都要覆盖，不要只测“应该做什么”。
- 定期阅读完整 Trace，不要只盯聚合分数。

### 8.6 先修评测，再改 Agent
这是全文我最认同的一条判断之一：
- 基础设施错误率上升，看起来像模型退化。
- 评分器 bug，看起来像 Agent 失败。
- 任务集与生产脱节，看起来像提示词有问题。

所以，当分数下滑时，第一步不是改 Prompt，而是排查：
- 环境是否脏了。
- 资源是否不足。
- 评分器是否漂移。
- 任务集是否饱和或失真。

## 九、如何追踪 Agent 的执行过程
### 9.1 Trace 是排查问题的前提
传统 APM 只能告诉你“接口慢了、报错了”，但 Agent 的失败常常发生在更细的层面：
- 某一轮决策错了。
- 工具选错了。
- 参数填错了。
- 反馈没有被正确吸收。

所以文章建议至少记录这些信息：
- 完整 Prompt。
- 多轮 `messages[]`。
- 每次工具调用的参数与返回值。
- 最终输出。
- token 消耗与延迟。

### 9.2 两层可观测性更现实
作者推荐“人工抽样 + LLM 自动评估”两层结构：
- 第一层：人工抽样，理解失败模式、建立标注基准。
- 第二层：LLM 自动评分，覆盖更大规模的流量。

这两层不能互相替代：
- 只有人工，规模不够。
- 只有 LLM 评分，标准容易漂移。

### 9.3 在线评测要规则采样，不要纯随机
更值得优先评估的 Trace 包括：
- 用户明确负反馈的对话。
- token 消耗异常高的对话。
- 模型或 Prompt 刚变更后的对话。
- 固定时间窗口内的正常样本。

### 9.4 用事件流做可观测性的底座
文章提倡在 Agent Loop 的关键节点发事件，例如：
- `tool_start`
- `tool_end`
- `turn_end`

然后把同一份事件分发到多个下游：
- 日志系统。
- UI 更新。
- 在线评测。
- 人工审查队列。

这样主循环不需要为了某个下游专门改代码。

## 十、用 OpenClaw 看 Agent 如何落地
### 10.1 OpenClaw 的五层结构
OpenClaw 在文章里主要作为一个“把原则变成工程”的例子，整体可以拆成五层：
| 层 | 主要职责 | 关键价值 |
| --- | --- | --- |
| Gateway | 接住外部连接并统一路由消息 | 入口集中、控制统一 |
| Channel Adapter | 适配 Telegram、Discord 等渠道 | 渠道差异被隔离 |
| Pi Agent | 维护主循环、会话和调度 | Agent 核心与渠道解耦 |
| Tooling | 提供 shell、fs、web、browser 等能力 | 能力边界可控 |
| Context + Memory | 管理 Skills、记忆与系统提示 | 上下文密度更可控 |
### 10.2 MessageBus 的意义
MessageBus 让渠道和 Agent 解耦：
- Channel 只管收发。
- AgentLoop 只管消费和处理。
- Session 状态不下沉到渠道层。

这能保证：
- 换渠道不动 Agent 核心。
- 不同 session 可以并发处理。
- 同一 session 可以单独串行保护，避免竞态。

### 10.3 系统提示应该按层叠加
OpenClaw 的提示设计给了一个很实用的参照：
- `SOUL.md`：身份、行为约束、完成标准。
- `MEMORY.md`：长期事实。
- Skills 索引：只给按需加载的入口。
- 运行时注入：时间、渠道、用户、会话态。

这和前文的上下文分层是一致的，说明好的 Prompt 设计并不是“写一篇长文”，而是“把不同性质的信息放到正确层级”。
### 10.4 长任务恢复要依赖外部状态
OpenClaw 的长任务恢复方法很直接：
- 每走一步保存进度到磁盘。
- 重启后从任务文件恢复。
- 超过半小时的任务默认必须支持恢复。

这再次说明一个核心原则：
> 长任务的可靠性来自状态外化，而不是上下文延长。

### 10.5 安全边界必须先于功能
文章在落地部分把安全讲得很实在，核心有三层：
1. 谁能用：白名单授权。
2. 能在哪用：工作空间隔离与路径检查。
3. 做了什么：完整审计日志。

在这之上，还要补两类兜底：
- Prompt Injection 防护。
- Provider 故障切换。

### 10.6 Prompt Injection 的正确处理方式
作者不建议指望单纯输入过滤，而是强调 `source-sink` 分离思路：
- 外部内容是 source。
- 危险动作是 sink。

防御重点包括：
- 最小权限。
- 敏感操作显式确认。
- 明确标注不可信内容边界。
- 关键路径交给独立 LLM 或外部机制复核。

### 10.7 Provider 故障切换是必需项
模型服务故障是常态，不是例外。
所以生产 Agent 应至少具备：
- Provider fallback。
- 限速处理。
- 超时控制。
- 失败后的重试与切换策略。

## 十一、常见反模式
| 反模式 | 问题 | 改法 |
| --- | --- | --- |
| 把系统提示当知识库 | Prompt 越来越长，规则被噪声淹没 | 系统提示只留约束，知识放到 Skills |
| 工具数量失控 | Agent 经常选错工具 | 合并重叠工具，明确命名空间 |
| 没有验证机制 | Agent 说完成，但没人知道真假 | 每类任务绑定可执行验收标准 |
| 多 Agent 没边界 | 状态漂移，难以归因 | 先建协议、任务图和隔离目录 |
| 记忆不整合 | 长对话后质量明显下降 | 监控 token，占阈值自动整合 |
| 没有评测 | 改动后是否回归完全不清楚 | 把真实失败案例立刻转成测试 |
| 过早上多 Agent | 协调成本大于收益 | 先验证单 Agent 上限，再决定扩展 |
| 约束只写文档 | 规则容易被忽略 | 用 Linter、Hook、工具校验落地 |

## 十二、可直接拿走的工程清单
如果把这篇文章压成一套落地顺序，我会记成下面 10 条：
1. 先把最小 Agent Loop 跑通，不要一开始就把循环写成大状态机。
2. 先补 Harness，再谈模型升级，尤其是可自动验证的任务。
3. 给上下文分层，稳定信息前置，动态信息后置，知识按需加载。
4. Skill 描述写成路由条件，不要写成功能宣传页。
5. 工具按 Agent 目标设计，错误返回必须结构化且可修复。
6. 把记忆分层，并确保整合可回退、原始历史可追溯。
7. 长任务进度写入文件或数据库，不要依赖上下文窗口。
8. 多 Agent 先做协议、任务图、隔离，再做并行。
9. 评测优先使用代码评分器，并区分 Pass@k 和 Pass^k。
10. Trace、审计和权限边界必须尽早接入，不要等出事故后补。
