# 一、背景
## 1. 签证办理
1. 签证是一国政府授权机关依照本国法律为外国人颁发的出入境许可证明，通常附载于护照或其他旅行证件。

政府机关有哪些？

1. 国家驻华办签机构，整体分为两类，一个是移民局，另外一个是使领馆。
2. 签证中心，是使领馆的一个外包机构，如申根。一般有签证中心的，使领馆自己不接受申请，全权由签证中心代理。
3. 签证办理角色职责与流程

![[IMG-20260404032001237.gif|800]]

3. 签证办理并非简单的资料传递，而是一个涉及**申请人、中介服务商（商家）与官方机构**三方深度协作的复杂流程。

## 2. 行业痛点
1. 签证办理这一环，长期受困于**“强依赖人工”**的传统模式。根据行业调研与全链路时效分析，传统签证办理流程通常需要**3-5天**，这种低效不仅折磨着申请人，也成为了服务商规模化增长的瓶颈。
2. 传统办签链路的四大核心痛点

3. 这种低效模式造成了B端（商家）与C端（用户）的双向痛苦：

- **B端（商家）面临高昂的人力杠杆**
    为了维持运转，商家不得不组建庞大的团队（如10人以上的咨询团队、5-10人的审核团队）。规模效应极低，业务量一旦爆发，人力成本便呈指数级上升，且服务质量难以标准化。
![[IMG-20260404032001318.png|800]]
- **C端（用户）面临繁琐的服务体验**
    用户在整个链路上感受到的是“秒级需求、天级响应”。99%的时间都消耗在等待人工处理的过程中，这种极高的时效损耗已无法满足现代用户对即时反馈的心理预期。
## 3. AI 驱动的全链路时效重塑
1. 传统的人工驱动模式签证办理流程中存在的诸多痛点，在此背景下，签证行业亟需引入AI Agent技术，推动全链路服务的智能化重塑。通过AI托管，构建覆盖“咨询—准备—审核—送签”全流程的自动化服务体系。

# 二、签证 Agent 整体设计
## 1. 产品形态
### 售前AI办签咨询
- 通过构建签证售前办签Agent，聚焦三大核心场景，在CHAT中支持按用户诉求推荐商品、回答各国实时办签政策和所需办签材料，同时可支持用户实时追踪办签进度，提升办签体验及效率。
1. **指令办签**：用户以自然语言表达需求（如“我要去韩国办签”），AI通过多轮对话智能收集关键信息（如常住地、出行时间、职业等），并精准推荐签证商品，同步提供所需材料、办理时长及费用，助力高效决策。
2. **政策咨询**：针对用户关于材料或政策的疑问（如“办韩签需要什么材料？”），AI结合上下文和多轮对话，从使领馆官网、知识库及合作商家数据中匹配权威信息，提供准确解答。
3. **进度查询**：支持用户通过自然语言查询签证状态（如“我的签证办好了吗？”），以时间轴形式清晰展示订单履约进度；若存在多笔订单或多申请人，可灵活选择对应记录进行查询。

![[IMG-20260404032001548.gif|800]]

### 售后AI材料预审
- **材料AI智能上传：**用户通过“一键上传”提交照片或文件（PDF、Word等）后，AI基于文件名及内容自动匹配材料清单中的对应项，完成智能归类，系统根据匹配结果动态反馈。提升用户办签便捷性与效率，降低重复操作成本，强化平台服务体验与用户粘性。
- **材料AI审核：**系统在上传环节同步执行智能校验，自动审核护照、证件照、户口本、经济证明等材料的合规性，提升材料一次性通过率与用户体验。有效降低因材料不规范导致的反复沟通与返工，商家可聚焦于关键决策环节，实现审核效率提升与人力成本优化。
- **出签结果AI校验**：对商家上传的电子签文件进行真实性核验，通过与平台标准模板或历史有效结果进行内容比对，识别异常或伪造文件。保障出签结果真实可信，强化平台对商家服务质量的监管能力，提升用户对在线预审服务的信任度。

# 三、RAG在签证业务中的落地实践
- **下面主要介绍我们通过“三孤岛”这一清晰的框架，在签证助手的开发中不再盲目试错。****它像一张系统性的地图，帮我们把看似杂乱的问题归因到三个关键断层：****数据是否可信、指令是否明确、泛化性是否足够。**
- **接下来，我们将结合办签助手从0到1的落地过程，逐一拆解****如何跨越这三座孤岛之间的鸿沟，并根据我们在实际开发过程中遇到真实问题，为大家提供一些解决问题的思路。**

## 1. RAG简介
1. **RAG（Retrieval-Augmented Generation，检索增强生成）**是一种结合信息检索与大语言模型（LLM）生成能力的技术框架，为的是提升模型回答的准确性、时效性和可解释性。其核心思想是：在生成答案前，先从外部知识库中检索与用户问题最相关的信息，再将这些信息作为上下文输入给大语言模型，从而引导其生成更可靠、基于事实的回答。

![[IMG-20260405035359306.png|800]]

2. RAG 主要由三个关键步骤组成：

3. **索引（Indexing）：**将原始文档清洗、提取并转换为统一的纯文本格式；随后按一定策略（如固定长度、语义边界等）分割成较小的文本块（chunks）；接着使用嵌入模型（如 BERT、text-embedding-ada-002 等）将每个 chunk 编码为向量，并存入向量数据库，为后续高效检索做准备。

4. **检索（Retrieval）：**当用户提出Query时，系统使用与索引阶段相同的嵌入模型将查询转化为向量，并在向量数据库中计算其与所有 chunk 向量的相似度；然后返回相似度最高的 Top-k 个 chunks 作为相关上下文。

5. **生成（Generation）：**将原始查询与检索到的 Top-k chunks 拼接成一个结构化的提示词，输入给大语言模型，由模型综合上下文信息生成最终答案。该过程可支持多轮对话，将历史对话也纳入 prompt 中以保持上下文连贯性。

## 2. 从理论到上线：三孤岛框架下的办签助手开发实战
### 2.1. 思考框架：三孤岛理论
1. “LLM 开发中的三重鸿沟”（The Three Gulfs of LLM Development）是由 Shreya Shankar 提出的一种框架，用于识别和解决 AI 应用中的关键故障模式。该框架认为大多数 LLM 应用的失败并非源于模型本身的局限性，而是系统中三个核心实体——开发者（Developer）、LLM 管道（LLM Pipeline）和数据（Data）之间存在的“鸿沟”。

![[IMG-20260405035400009.png|800]]

2. 上图中出现了三座孤岛，分别是开发者，AI 应用和数据，在 AI 应用研发中，每两座孤岛之间都存在着一条鸿沟。

3. **理解鸿沟：**在开发者与数据之间，这是指开发者难以完全、深入地理解 AI 应用所接收的海量数据，处理这个鸿沟需要我们能实现数据的观测，并对数据进行理解和清洗；

4. **具象鸿沟：**这座鸿沟在开发者与AI 应用之间，它是指开发者很难将自己的抽象意图，转化为LLM能够精确理解和执行的具体指令，这就要求我们在 Prompt 工程上有一些技巧和思考；

5. **泛化鸿沟：**存在于AI 应用和数据之间，指的是LLM模型在处理多样化和未知的输入时，其表现不稳定，难以保证其行为的一致性和可靠性（即泛化能力不足），解决方案在于工程挑战（拆分任务，fine-tuning，RAG）；

6. 通过该思维框架，我们能很清楚的知道，第一步要清洗数据，第二步要优化 Prompt，第三步可能需要引入更复杂的工程手段（如问题拆解、上下文管理、知识召回、甚至微调模型等）来提升泛化能力。

### 2.2. 构建高质量的签证行业知识库
1. 首先排查开发者和数据之间的理解鸿沟：
2. 用户每天和商家客服在旺旺里沟通签证办理相关的问题，这些聊天记录里到底有什么信息？用户在实际过程中哪些问题会普遍遇到？哪些问题会频繁提问？Top15 国家的使领馆数据，形态各异，其中哪些数据是我们能获取到的，哪些是我们可以利用的？这些我们都是不知道的。
3. 鸿沟体现：
- 数据价值不透明：海量存量数据（如旺旺客服聊天记录、使领馆官网信息、第三方攻略等）中蕴含哪些有效信息？哪些是高频痛点？哪些可直接使用，哪些需要处理？这些问题缺乏系统性洞察。
- 数据质量不可控：原始数据中存在大量噪声、冗余、错误甚至误导性内容。若未经清洗直接给大模型使用，将严重影响问答准确率与用户体验。
- 数据形态异构复杂：使领馆信息分散于网页、PDF、图片等多种格式；用户问题隐含在非结构化对话中；第三方攻略以移动端界面形式呈现——这些都增加了统一处理的难度。
2. 围绕着 RAG 我们进行了知识库建设，我们希望建设全面的、可管理可迭代的、满足实时性要求的签证行业知识库系统，为办签助手提供优质数据语料，更好的支撑端到端的RAG能力。

![[IMG-20260405035400713.gif|800]]

3. 数据收集和清洗数据

4. 【使领馆数据】基于Python开源的AI自动化工具 Browser Use（[https://browser-use.com/](https://browser-use.com/)），构建了一个使领馆官方信息的自动化知识处理工具。核心目标是从网页、PDF和图片等多种来源高效抓取签证相关信息，并通过阿里云通义千问大模型（如qwen3-max、qwen-vl-max等）智能提炼、改写为结构化的FAQ问答对。

5. 【客服旺旺数据】我们对大量离线旺旺客服聊天记录进行深度清洗与语义分析，基于业务场景抽象出七大问题类别：包括材料准备、时间进度、费用价格、物流快递、拒签担忧、特殊情况及其他咨询。并从中聚类提炼出高频用户问题；随后通过自动总结与人工审核相结合的方式，将非结构化对话转化为清晰、准确、覆盖核心办签痛点的标准化问答语料，为RAG系统提供源自真实用户需求的高质量知识输入。

6. 【第三方签证攻略数据】对于一些外部通过ADB控制安卓设备模拟用户操作，结合OCR文字识别、HSV色彩分析与智能交互逻辑，自动遍历目标标签页、弹窗及长页面内容，精准提取多国办签攻略；随后对采集数据进行去噪、去重、表格结构化及Markdown格式转换，最终生成清晰、规范、可直接用于RAG知识库的结构化签证攻略文档。

### 2.3. 把模糊意图变成办签助手能执行的指令
1. 接着排查开发者和办签助手之间的具象鸿沟，数据清理后，发现办签助手还是爱幻觉。
2. 检查 Prompt 发现，我们只给了它一句简单的“请提供的信息回答”。鸿沟体现：我们脑子里的意图是「你要严格基于检索到的片段信息，不要编造，不知道就说不知道」，但我们没能把这个复杂的约束转化为模型能听懂的、精确的 Prompt 指令。
3. 下面是一些常见的 prompt 工程框架和手段，这部分有很多详细介绍的文章，不再赘述。
4. CRISPE 框架：通过设定角色、请求、信息、步骤、人格和示例，构建全面而结构化的提示。
5. RTF 框架：明确 AI 的角色（Role）、要执行的任务（Task）和期望的输出格式（Format）。
6. ICIO 框架：包含清晰的指令（Instruction）、上下文（Context）、具体输入（Input）和预期输出（Output）。
7. 在办签助手的核心模块——意图识别（NLU）的开发中，我们遭遇了一个棘手的问题。
8. 我们为NLU设计的Prompt常常陷入“按下葫芦浮起瓢”的窘境：优化一个场景的识别准确率，却可能导致另一个场景出现误判。
9. 签证咨询场景繁多（指令办签、政策咨询、申根签证、落地签场景、入台证、港澳通行证...），用户表述各异。另外，签证很多场景是需要澄清一些信息的（签证类型、常住地等），多轮对话后，模型在意图判断上的准确率，会随上下文信息增加被严重干扰。

```java
{
  "output": "{\"index\": \"INSTRUCTED_VISA\", \"mind\": \"用户在最近的历史咨询中先询问了‘如何提交签证材料？’，随后指定了国家为‘日本’，这表明用户正在就日本签证的办理流程进行咨询。最新的补充信息明确了签证类型为‘日本三年多次旅游签证’，这是对签证办理意图的进一步细化，而非单纯询问政策或材料要求。结合用户此前已表达出办理日本签证的明确动作（如‘如何提交签证材料’隐含办理意图），且最新信息是对国家和签证类型的确认，符合INSTRUCTED_VISA场景中‘用户表达了要办理签证’并补充关键信息的情形，因此最终意图为INSTRUCTED_VISA。\"}"
}
```
3. 为了提升模型的理解力和稳定性，我们自然而然地想到了引入Few-Shot示例。然而，一个简单的静态Few-Shot策略很快暴露了其局限性：
4. 上下文长度爆炸：如果将所有典型场景的示例不加选择地全部塞进Prompt，很容易超出模型的上下文窗口限制（Token Limit）。
5. “多即是乱”：过多的示例不再是“Few-Shot”，反而可能成为噪音，干扰模型对当前核心任务的专注度，甚至诱发新的幻觉或错误。
6. 维护成本高昂：静态的示例库难以应对业务的快速迭代和新问题的涌现。
7. 为了解决上述问题，我们设计并实现了一套**动态Few-Shot机制**。
8. 构建高质量Bad Case样本库：我们将评测期间和线上真实交互中产生的典型Bad Case（错误识别案例）进行归纳、标注，并录入到一个专门的Few-Shot样本集中。这些经过实战检验的样本是提升模型能力的宝贵财富。
9. 用户Query预处理与相关性召回：在进行正式的意图识别之前，我们先对用户当前的Query进行改写或embedding化处理，然后用它在Bad Case样本库中进行一次快速的相关性召回（Relevance Retrieval）。
10. 动态加载：召回系统会找出与当前用户问题最相似、最相关的几个Bad Case示例。然后，我们将这几个高度相关的示例动态地加载到NLU Prompt的指定位置。
11. 通过这种方式，每次请求的Prompt都是为当前用户问题“量身定制”的，既保持了Prompt的简洁高效，又为模型提供了最关键的决策参考，极大地提升了复杂场景下意图识别的准确率。

```java
[system]  
你是一个具备复杂场景对话能力的飞猪签证办理专家，你的任务是根据提供的用户的历史咨询和用户新的咨询，理解用户的意图，并按要求返回结果。  
[user]
```

### 2.4. 让办签助手在未知问题面前不失控
- 最后排查办签助手和数据之间的泛化鸿沟：
- Prompt 写好了，评测也过了。但上线后，用户问了一个超级复杂的问题：陈先生和陈太太，两个人要办韩国签证。陈太太是广州户籍，陈先生不是。房产证上是陈太太的名字，但是是夫妻共同财产，那么陈先生是否可以通过结婚证、房产证来办理韩国签证呢？办签助手直接懵逼了，只会复读文档片段，甚至提取不到准确的用户诉求，收集的信息都是错误的，导致答非所问，乱答的情况。
- 鸿沟体现： 模型面对真实的、复杂的、从未见过的用户输入数据时，泛化能力不足，无法像处理简单测试题那样稳定地进行推理和回答。

#### 2.4.1. 上下文管理
1. 在签证办理这一复杂业务场景中，用户意图往往涉及多个关键维度的信息提取，包括但不限于：

|   |   |   |
|---|---|---|
|**维度**|**说明**|**示例**|
|**国家**|不同国家政策独立，无统一标准；部分国家（如申根区）涉及多国协同规则|日本、韩国、越南、美国、法国、申根国家|
|**签证类型 & 常住地**|不同国家有各类签证类型，“常住地”在不同国家语境下含义不同：可能指居住地、指纹录入地或面签地点|美国：常住地 = 面签领区；申根国：常住地 = 录入生物信息的使领馆所在地|
|**申请人身份**|身份直接影响所需材料清单、收入证明形式、是否需监护人等|在职人员、自由职业者、在校学生、退休人员、学龄前儿童、已婚人士|
|**领区归属**|部分国家严格限制送签领区（强领区），跨区可能导致拒收|北京领区、上海领区、广州领区、武汉面签点|
|**送签模式**|模式多样：强领区、线下录指纹、线下面签|强领区国家（如日本）、需线下录指纹国家（如英国）、非强领区国家（如部分东南亚国家）|
|**出行时间**|影响签证有效期选择、加急服务推荐|出行日期在3天内 → 推荐加急通道|

2. 在早期版本中，我们采用了一种“无状态”的上下文处理策略：每次用户发起新请求时，系统将当前 query 与历史对话拼接后输入信息抽取模型，从中识别上述关键字段，并基于该次抽取结果驱动后续的流程编排与回答生成。

3. 然而，这种方案在实际验收中暴露出严重问题，这在签证办理这类高复杂度、低容错的场景下是决不允许的：

4. **问题一：**由于每轮对话独立进行信息提取，缺乏对上下文状态的持续追踪，极易出现语义断层。例如，上一轮对话明确讨论的是“日本三年多次旅游签证”，但用户在下一轮基于此前上下文追加提问时，若当前 query 表述模糊或信息缺失，模型可能错误提取出“日本单次旅游签”，导致系统跳转至完全不相关的业务流程，最终产生答非所问甚至流程错乱的体验。

5. **问题二：**简单地将全部历史对话拼接后输入模型，又容易引入大量冗余或过时信息，导致上下文过长、噪声增多。这不仅会分散模型的注意力焦点，还可能因无关内容的干扰而引发关键字段的误识别或遗漏。

6. 根本原因在于：**传统无状态设计无法有效维护多轮对话中的语义一致性。尽管用户交互始终通过文本（输入框或澄清卡片）完成，但真实意图往往依赖于累积的上下文信息，而非孤立的当前query和有限的历史对话原文。**

7. 为此，我们重构了上下文管理机制，引入有状态会话上下文模型：

- 以 `session_id` 作为唯一标识，为每个用户会话建立持久化的上下文存储；
- 每轮对话中，信息抽取模块不仅解析当前 query，还结合已有会话状态进行增量更新；
- 通过规则引擎对各类上下文字段（如国家、签证类型、领区等）定义生命周期策略——例如，“出行日期”仅在显式提及或修改时更新，而“目标国家”一旦确认则长期锁定，除非用户有明确主动变更意图；
- 所有后续的流程决策、知识召回与回答生成，均基于这份动态维护的会话状态，而非单次 query 的瞬时解析结果。
6. 显著提升了多轮交互的连贯性与准确性，使签证 Agent 能够像人类客服一样“记住”用户之前的诉求，并在上下文演进中做出合理响应。

#### 2.4.2. 语义增强
1. 所谓语义增强，就是将某个文本片段（chunk）及其所属的完整文档（例如整篇论文）一同输入给大语言模型（LLM），由 LLM 结合全文上下文为该片段生成一段简洁概述，并将此概述附加到原始 chunk 中。通过这种方式，在后续进行语义检索时，即使用户查询与 chunk 原文措辞不一致，只要语义相关，也能更准确地匹配和召回该 chunk，从而提升检索精度。
2. 下面就是一个语义增强的prompt示例：

要求模型基于整个文档，为其中的一个特定片段（chunk）生成一句简洁的上下文说明，说明这个片段在全文中扮演什么角色、处于什么位置或讨论什么主题。

```java
DOCUMENT_CONTEXT_PROMPT = """
<document>
{doc_content}
</document>
"""
CHUNK_CONTEXT_PROMPT = """
Here is the chunk we want to situate within the whole document
<chunk>
{chunk_content}
</chunk>
Please give a short succinct context to situate this chunk within the overall document for the purposes of improving search retrieval of the chunk.
Answer only with the succinct context and nothing else.
"""
```
3. 在构建签证材料知识库时，由于材料内容繁杂，通常包含具体的受理范围（如适用地区、户籍要求、出行目的、拒签限制等）以及特殊办理类型（如有拒签经历、加急办理、简化办理、家庭办理等），**我们会执行一个摘要增强（summary）操作：将这些关键约束与办理条件提炼并附加到对应材料的知识条目中。**这样，即使用户查询中的办签诉求或表述与原始文本不完全一致，只要语义相关，系统也能借助该上下文更准确地召回对应的材料片段。

#### 2.4.3. Query改写
1. 在评测过程中有一个典型的例子，我们观察到一类典型的上下文漂移（Context Drift）问题。

![[IMG-20260405035402565.gif|800]]

1. 此时，理想情况下 Agent 应继续围绕韩国签证判断澳大利亚签证记录是否可作为辅助材料（例如用于简化流程或提升过签率）。但实际系统却可能回复：好的，办理澳大利亚签证需要……
2. 这一错误看似微小，实则暴露了深层问题：模型在解析当前 query 时，将用户举例中的“澳大利亚”误判为目标办签国家，并覆盖了此前已确认的“韩国”上下文。
3. 由于签证业务中，“目标国家”是核心锚点——签证类型、领区规则、常住地定义、材料清单等均强依赖于此——一旦该字段被错误替换，整个对话状态即发生偏移，后续所有流程编排（如材料推荐、办签指引、商品推荐）都将基于错误前提展开，导致用户体验断裂甚至误导。
4. 根本原因在于：传统端到端的信息抽取模型缺乏对“提及”与“意图”的区分能力。当用户以举例、对比或补充说明的方式引入其他国家时，模型容易将这些“背景信息”误识别为新的主目标。
5. 为避免此类问题，我们在信息抽取前引入**轻量级query重写机制**：
6. 限定上下文窗口：仅截取最近 3 轮交互（基于用户query与Agent的澄清/追问轮次上线），避免远期噪声干扰；
7. 融合对话意图生成显式重写 query：将当前 query 与近期上下文，通过规则+小模型方式，将模糊或含干扰项的原始问题转化为语义明确、无歧义的标准化表述。
8. 以上述案例为例，原始 query：

“我持有的其他国家签证记录（如澳大利亚三年多次往返）是否符合特定签证的要求？”

经重写后变为：

“我持有澳大利亚三年多次往返签证，这能否作为办理韩国签证的辅助材料？”

重写后的 query 显式保留了已确认的目标国家（韩国），同时将“澳大利亚”降级为辅助信息，从而确保下游信息抽取模块能准确维持上下文一致性。

#### 2.4.4. Query拆解 + 多路召回
1. 在签证咨询场景中，用户常在一个自然语句中隐含多个子问题。例如：“我计划带5岁的孩子一起申请日本单次旅游签证，可以用我的白金信用卡作为财力证明吗？另外，小孩需要单独准备哪些材料？”，这句话实际包含三个信息需求：
2. 办理日本单次签证需要准备哪些材料？
3. 信用卡作为财力证明需要满足哪些条件？
4. 5岁儿童申请日本单次签证需提供哪些特定材料？
5. 若直接使用原始 query 进行 QA 知识库的向量检索，往往因语义过载或关键词稀疏，导致关键信息漏召或噪声干扰。
6. 为此我们引入了一个query拆解 + 多路召回模块：
- 按语义逻辑拆分原始 query，识别其中独立的子问题；
- 对每个子问题进行标准化改写，将其转化为更贴近知识库表述风格、更适合检索的结构化查询。例如，将“能申请吗？”转化为“日本三年多次旅游签证的申请条件是什么？”，或将“需要什么材料？”明确为“申请日本三年多次旅游签证所需提交的财力证明材料有哪些？”；
- 每一路改写后的 query 独立进入召回流程，分别从签证QA库中并行检索相关片段。
3. 本节的多路召回和上节提到的query改写，其实是正向query**检索增强**的一种方式，即可以针对用户的query，生成一些假设的答案或者做query改写或扩写，然后通过这些中间内容去做检索召回。

#### 2.4.5. 重排序和后置过滤

![[IMG-20260405035402658.gif|800]]

1. 在当前的 签证QA 知识召回流程中，我们采用仅以用户问题（Question, Q）作为向量检索字段的方式，不依赖答案（Answer, A）内容。因为Q 更简洁、聚焦：相比可能较长、包含多信息点的答案，问题通常短小、意图明确，更适合做高精度语义匹配。
2. 为保证回答的全面性，系统通常会从向量索引中召回 Top 5 的候选文档。但随之而来的问题是：这 5 个结果究竟该保留哪些？由于初检仅基于 Q 与文档标题或摘要的语义相似度，召回结果可能包含：
- QA录入有误，导致问答不一致：部分知识条目在录入时 Q 与 A 不匹配，例如问题写的是“签证财力证明通用要求”，答案却只针对“日本单次旅游签证”，造成用户问通用问题时被错误导向特定国家；
- 问题表述过宽，答案却高度具体：如 Q 为“需要什么材料？”，有的问题描述的很宽泛，但是他的答案却很具体，是在描述日本单次旅游签证的材料清单。
3. 因此使用了重排序（Re-Rank）模块，在不改变召回集合的前提下，依据用户问题与文档内容的深层语义及业务匹配度，对候选结果重新打分排序。简要实现方式如下：
4. rerank 模块会逐一对用户 query 与每个召回文档的正文内容进行细粒度交互计算，输出一个 0~1 的相关性分数。相比初检阶段的粗略向量匹配，rerank 能更准确判断：
- 文档是否真正回答了用户关心的原始问题；
- 内容是否适用于当前上下文中的目标国家、签证类型、申请人身份等业务要素。
5. 最终，系统从 rerank 排序结果中选取过滤出 Top-3 最相关的问答对，作为大模型生成回答的上下文依据。

## 3. 办签助手响应太慢了怎么办？
1. 在项目上线前，业务逻辑都验证的差不多之后，就会自然而然收到内部体验同学的各类反馈
- 理解用户需求这一步，太慢了，这个得优化一下速度之后才能上
- 这个速度，很难给用户用啊
- 现在整体优化到几s了 可以同步一下
- 思考链的优化发了吗？为啥还是这么久
2. 我们通过一系列优化措施，我们将办签助手澄清场景的端到端平均响应时间从最初的8秒缩短至4–5秒，政策咨询场景正文回答链路的端到端平均响应时间也从15–17秒显著降低至8–9秒。
3. 在AI应用落地过程中，模型优化从来不是单纯追求“越快越好”，而是在响应速度、回答质量与用户体验之间寻找最佳平衡点。一方面，用户对等待时间极为敏感；另一方面，过于激进地压缩推理时间可能导致模型输出质量下降，比如逻辑断裂、信息遗漏或事实错误，反而损害信任。
4. 优化原则：在保障核心业务意图准确理解与高质量回复的前提下，尽可能压缩端到端延迟。

# 四、如何让AI“可靠地”完成多维度材料审核？
## 1. 整体架构

![[IMG-20260405035405214.gif|800]]

- 为应对复杂性和未来扩展需求，我们将系统划分为四层，并设计了两个横向支撑模块：
- **场景接入层（Scene Access Layer）**：作为系统统一入口，将各类签证预审需求（如护照审核、证件照检测、材料分类等）抽象为标准化业务场景，通过场景化处理器实现业务逻辑与底层AI能力解耦，支持快速扩展新场景。
- **任务处理层（Task Processing Layer）**：管理一次材料审核调用的完整生命周期，包括任务创建、状态流转、模型会话调用编排、异常处理与结果聚合解析。该层屏蔽了底层模型交互的复杂性，向上提供“一次触发即返回结构化结果”的抽象。
- **大模型会话层（LLM Session Layer）**：封装与大模型的交互细节，提供Prompt版本管理、多模态消息构造、流式/非流式调用、模型动态切换、Token 消耗统计等能力。确保模型调用的一致性与可追溯性。
- **基础设施层（Infrastructure Layer）**：提供任务/会话/数据存储、OSS/OTS资源访问、模型网关、配置中心及监控告警等基础服务，为上层应用构建稳定、可观测、高可用的技术底座。
- **横向支撑模块**：通过**数据标注模块**实现自动打标与评测数据闭环，结合**评测模块**进行准确率巡检、变更回放与人工复核，形成“发布—评估—优化”的持续进化机制。

### 2.1. 如何避免多审核点耦合，导致审核准确率不可控？
1. 问题描述：
2. 在我们的实际业务中，用户通常需要提交的材料有10余种，如身份证、护照、学历证明、银行流水等，而每份材料平均有6–8个独立维度的校验点，例如：护照有效期小于180天、护照图片存在反光、上传的护照图片存在手指等其他遮挡物、在职证明材料缺少公司抬头或公司印章、银行存款/理财姓名和申请人姓名不同等等。
3. 若将材料的全部校验规则塞入单次大模型调用，模型需同时处理数多个判断目标，注意力被严重稀释，关键校验点易被忽略。
4. 解决方案：子场景并行调度机制
5. 系统将整体审核任务解耦为多个语义独立的子场景（如“证件类型识别”、“签发机关合法性校验”、“有效期逻辑验证”等），每个子场景对应一个轻量、聚焦的LLM会话。调度引擎根据当前业务配置动态加载生效的子场景列表，并行发起多个独立审核推理请求。
6. 各子任务使用高度定制化的Prompt，仅处理其专属逻辑。最终，系统依据预设的业务规则（如“任一关键项失败即拒绝”或“多数通过即通过”）对各子结果进行融合，生成最终审核结论。
7. 带来的好处：
- 精准聚焦：每个模型推理只解决部分审核点判断问题，显著提升单点判断准确率；
- 弹性扩展：新增校验点只需注册新子场景，无需改动现有逻辑；
- 性能优化：并行执行使总响应时间趋近于最慢子场景任务，而非串行叠加。

### 2.2. 材料输入不可控，如何避免模型输入超限？
1. 问题描述：
2. 用户可能仅提交少量关键材料，也可能一次性上传数十张图片或包含20+页的长PDF（如整本护照、多月银行流水等）。大模型对输入Token有严格限制，若将全部内容直接送入，极易因超出上下文长度而失败，或因信息过载导致推理质量下降。
3. 更复杂的是，单个PDF文件内部常混杂多种材料类型（如第1页是封面、第2–5页是银行流水、第6页是在职证明），若将整个PDF作为整体进行分类，模型易受无关页面干扰，造成类别误判。
4. 解决方案：
5. 场景自适应的材料预处理与分片策略：系统在调用大模型前，先对原始材料进行清洗与路由，自动剔除分辨率过低、空白或损坏的内容；按业务场景动态分片：结构化证件审核场景（如护照、身份证），采用“截断策略”，仅保留前N页（通常关键信息集中在前1–3页），舍弃冗余页面，兼顾性能与准确性。材料完整性或全量分析场景（如材料分类）：采用“分片并行策略”，将全部有效材料按Token容量切分为多个批次，每个批次独立发起大模型调用，并行处理，确保全覆盖。
6. PDF 页面级分类投票机制：针对多页PDF内含异构内容的问题，将PDF逐页渲染为图像，并保留页序与源文件映射关系；将所有图片统一送入模型分类，每页独立输出类别。对同一PDF的所有页面分类结果进行多数投票聚合，得票最高的类别作为该PDF的最终判定结果。

### 2.3. 如何实现审核规则（Prompt）随政策快速迭代、验证与回滚？
1. 问题描述：审核点常因政策调整而变更，而 Prompt 是影响模型输出的核心。传统硬编码方式导致每次修改需重新发版，迭代慢、验证难、回滚风险高。
2. 核心方案：所有Prompt模板通过配置中心管理，支持按子场景独立发布、灰度、回滚。系统在运行时动态加载对应版本，并将使用的Prompt元信息（代码+版本）记录到会话日志中。
3. 结构化组织：Prompt 按 业务场景 → 子场景 → 版本 三级管理（如：护照审核 → 有效期检查 → 20250407），支持细粒度控制。
4. 动态热加载
- 所有模板托管于配置中心（如 Diamond），运行时按子场景实时拉取对应版本；
- 支持文字、多模态 Few-shot 示例等灵活配置；
- 灵活开关，可通过配置中心灵活动态启用/禁用某个校验点

### 2.4. 如何审核任务的状态一致性与系统可靠性？
1. 问题描述：多个会话可能同时更新同一任务的数据状态，易引发数据冲突；同时，AI调用本身存在超时、输出格式错误等不确定性，若缺乏容错机制，可能导致任务失败或状态错乱。
2. 核心方案：
3. 基于分布式锁的会话状态机
- 一次 LLM 调用不是“fire-and-forget”，而是一个需要严格管控的状态机。会话状态严格遵循在 INIT 、PROCESSING、SUSPEND、FINISHED、FAILED 之间按规则流转，确保每一步变更可控、可追溯。
- 关键状态变更加分布式锁（如 Tair Lock），确保原子性与一致性。
2. 柔性容错机制，保证系统可用性
- AI模型并非100%可靠。网络抖动、模型超时、输出格式错误都可能发生。若任一子场景失败就导致整个任务失败，用户体验将极差。
- 单个子场景失败时，返回安全默认结果（如“暂视为合规”），不阻断主流程；
- 同步告警通知，保障可运维性。
3. 全链路记录，签证材料审核属于强监管场景，所有AI决策必须完整留痕，支持精准回溯与人工复核。
- 完整留存 Prompt 版本快照、模型原始输出、结构化结果、耗时与 Token 消耗。

# 五、签证Agent评测的 What-Why-How
## 1. 什么AI业务应用评测？
1. AI 应用评测是一种系统化的方法，用于衡量和持续改进 AI 应用的质量和性能。
2. 这里的评测不是模型评测，我们经常可以看到各种模型打榜，不在本文的讨论范围内。
3. 模型评测聚焦于基础模型的通用能力（如语言理解、推理等），通常依赖 MMLU、C-Eval 等标准化基准测试集。
4. 而 AI 应用评测关注的是包含 Prompt 工程、工具调用、记忆机制、私有数据（如 RAG）等多组件协同下的整体业务表现。
5. 因此，构建贴合实际业务场景的专用评测数据集，是开展有效 AI 应用评测的前提和关键。
6. 相关概念：
7. **数据集（Dataset）**：本质上是由多个“样本”（example）组成的集合，这一命名源于机器学习传统——每条数据都是供模型学习的范例。一个标准的 AI 应用评测数据集中，每个样本通常包含三个关键属性：
8. input：用户输入或模块待处理的信息。
9. referenceOutput：理想输出，用于有标准答案场景下的评估。
10. metadata：元数据，用于标注来源、难度、业务类别等辅助信息，便于分析与筛选。
11. **评估（evaluation）**：指对 AI 应用（或其特定模块）输出结果在质量、效率或合规性等方面的系统性判断，通常围绕多个维度展开，例如是否存在幻觉、是否用户友好、格式是否正确等。
12. **实验（Experiment）**：是连接数据集与评估的核心环节：它将数据集视为“考题册”，评估标准视为“阅卷规则”，通过一个可控的验证流程，依次将数据集中的每个样本输入待测 AI 应用或其子模块，记录输出结果，并利用预设的评估器进行打分和分析，从而将主观感受转化为客观量化数据。

## 2. 为什么要做AI业务应用评测？
1. 什么是 AI 应用（Lm Based Application），顾名思义，区别于传统应用，就是加了 AI。在传统应用的基础上加了AI，会导致一个致命结果：**应用从一个有线故障面的应用转变为一个几乎拥有无限故障面的应用。**
2. 对于传统应用来说，它的逻辑是透明的，就是代码是我们写的，那它的逻辑是可控的。那么如果它出了问题，我们很容易去复现，去搞清楚问题出在哪里。
3. 所以在传统开发中，bug 实际上是我们写出来的，但是在AI应用中，bug可以说是聊出来。
4. 首先对于AI应用来说，LLM内部它本质上是在做概率统计，【transfermer】，模型的数据输出逻辑来源于统计学概率，这个对于它的调用方来说是个黑盒。另外调用方没办法很好复现，因为他的行为是概率性的。对于固定的输出，他都不一定会给出固定的输出。【测试提过一个bug来，你想复现，用相同的query输入，得到的确实符合预期的回复。😅】
5. 正是应为模型的泛化性，等应用真正上线之后，我们没有办法穷举用户的具体问题，进而我们没有办法知道我们应用中的核心LLM会给出什么样的输出。【测试同学提过来一个bug，认为模型不能输出某个回复，但开发却认为这不是一个bug，是模型自带的feature。😤】
6. 我们使用LLM的初衷就是希望模型可以代替人类、代替程序去解决一些泛化性的东西，我们又想利用模型的泛化性，又想保证模型的数据具有一定的稳定性，这本身就是一个寻找平衡的过程。
7. 模型具有以下特性，所以我们需要AI业务应用评测来保障效果：

|   |   |   |
|---|---|---|
|**LLM 特性**|**对应的研发挑战**|**典型表现 / 风险**|
|**1. 极度敏感**  <br>（对 Prompt 极其挑剔，差之毫厘，谬以千里）|**落地难**  <br>（如何稳定解决具体业务问题？）|模型能力强大，但微小输入变化可能导致输出严重偏离业务目标，难以构建可靠、一致的业务逻辑。|
|**2. 天生幻觉**  <br>（关注“通顺”而非“真实”）|**泛化难**  <br>（Demo 表现好，上线即崩）|在有限测试用例中表现良好，但面对真实用户多样、复杂甚至对抗性提问时，生成看似合理实则错误的内容，导致系统失效。|
|**3. 概率选择**  <br>（基于概率采样，非确定性推理）|**迭代难**  <br>（升级或微调易引发蝴蝶效应）|修改 Prompt、更新模型版本或调整参数后，局部优化可能意外破坏其他功能，整体表现不升反降，缺乏可预测性和可维护性。|

5. 目标：构建**“线上运行 → 自动标注 → 评测发现缺陷 → 模型/规则更新 → 效果验证”**的正向数据飞轮，另外，模型评测可以让我们对AI应用的迭代方向有一个较为清晰的方向。

## 3. 签证Agent评测怎么做？
### 3.1. 评测指标定义
1. 首先我们要面临的第一个问题就是，我们要评估什么？我们把办签助手的评估维度分为了两部分：**端到端的评测**和**路径评测**
2. **端到端的评测：**办签助手面临的是各种签证问题，包含指令办签、签证政策咨询、办签进度查询等，我们把两端的输入和最终的输出拿出来，是容易直接评估，办签助手回答的是好还是坏。

**A. 业务指标**

|   |   |   |   |

|**评估维度**|**具体要求说明**|**正向示例（符合要求）**|**反向示例（违反要求）**|

### 3.3. 自动化评测**工程链路**

关于评估方法，目前业界主要采用三种方式：

1. 代码评估：基于预定义规则的程序化判断，适用于有明确客观标准的场景。
2. LLM-as-Judge：利用大语言模型的语义理解能力，通过设计的 Prompt 使其充当“AI 裁判”，特别适合处理无标准答案的主观任务。
3. 人工评估：由人类专家直接打分或评判，但成本高、效率低、难以规模化。

![[IMG-20260405035411753.gif|800]]

#### 场景一：线上监控
- **实时流量异步诊断打分**：对线上用户的每一次QA，在异步处理链路中自动触发审核模型进行打分与判定；同时记录AI应用输出、流程中间态数据，及上下文信息，形成实时诊断日志，用于后续分析与告警。降低业务人员review线上回答质量的成本，简化每日评测指标的统计工作。
- **定时回放固定评测集，估算线上能力水位**：维护一套高质量、覆盖典型场景的评测集（包含正例、负例、边界 case）；定期用当前线上流程对该评测集进行全量回放，客观估算当前线上系统的整体能力水位，避免“黑盒运行”。
- **周期性评测报告**：定期从诊断日志与回放结果中提取打分数据、准确率等关键指标；自动汇总生成日报/周报，支持按时间、场景分类、领区、用户画像等维度下钻分析。

#### 场景二：变更的回放保障
- **上线前回归验证**：针对需求迭代、模型替换、流程变更（如新增知识库、办签场景铺盖扩充等）上线前，必须在当前线上环境对全量评测集进行回放，验证核心评测指标未发生显著下降，确保变更不破坏存量能力。

#### 场景三：多模态材料审核评测
1. 在签证售前 AI 咨询产品中，端到端问答场景的输入以文本为主，但多模态材料审核（如护照、证件照、银行流水等图像）引入了新的评测复杂度：

当系统对一张护照图片判定“不符合审核点 A（护照有效期需要大于180天，请及时更换护照。）和 B（护照镭射反光严重，使馆系统无法通过，请重新上传。）”时，如何客观评估该判断是否准确？

2. 传统人工逐条复核效率低、成本高，难以支撑大规模线上数据迭代。为此，我们构建了一套 “多模型投票 + 置信度过滤 + 人工兜底 + 评测集回流” 的闭环评测与标注机制：

3. 多模型协同判定与置信度控制

- 采用多个先进多模态大模型（如 QVQ-Max、Qwen-VL-Max 等）并行分析同一张材料图像；
- 设定投票规则：若 M 个模型中有 N 个及以上输出一致的审核结论（如“不通过，原因：A, C”），则视为高置信度样本，自动采纳该结果作为标注；
- 若未达阈值（如 M=5, N=3 未满足），则转交人工审核，确保底线质量。
2. 自动化数据标注链路建设

![[IMG-20260405035417034.gif|800]]

- 构建异步自动化标注流水线，对线上真实用户上传的材料图像进行抽样标注；触发多模型投票流程，高置信样本自动入库，低置信样本进入人工审核队列；
- 目标：将人工标注依赖度降低 50% 以上，同时保障评测集质量；
- 所有经确认的样本（含人工兜底结果）自动补充进多模态评测集，用于持续评估审核链路的准确率。

# 六、对签证Agent的未来展望
1. 当前，我们的签证AI应用已初步形成售前咨询Agent与售后材料审核Agent两大核心能力模块。尽管二者在功能上相对独立，但它们共同指向一个更宏大的愿景：**打造一站式、全自动、端到端的智能签证托管服务**。
2. 签证Agent的使命，是依托AI自动化技术，重构签证办理全链路（从用户咨询、材料审核、送签提交到结果回执），在显著提升用户体验的同时，大幅降低商家运营成本，并强化平台在签证服务领域的专业心智与商业价值（包括用户增长与交易转化）。
3. 要实现这一终态，全面自动化是必然方向：商家将聚焦于兜底审核与送签资质提供，而其余环节均由AI Agent高效、可靠地完成。
4. 这一演进路径，本质上要求我们从“单点智能”走向“系统智能”，构建多Agent协同作业体系。
5. 正如互联网架构从单体服务演进至微服务与Serverless，强调高内聚、低耦合，Agent系统的发展也将遵循相似规律。当单一Agent承载的领域知识日益厚重、业务复杂度持续攀升，拆分出专业化、角色化的子Agent将成为提升系统韧性与扩展性的关键。

![[IMG-20260405035441254.png|800]]

3. 未来已来。我们将持续沉淀单领域能力，推进从“单Agent提效”到“多Agent自治协同”的跃迁，最终实现智能签证新范式。这不仅是技术演进的方向，更是行业效率革命的大势所趋。
