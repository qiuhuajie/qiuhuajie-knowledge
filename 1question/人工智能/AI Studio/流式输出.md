# 一、流式输出是什么
1. 解决的问题：
	1. **业务复杂数据大耗时长**：例如Feeds类大列表涉及个性化推荐，响应较慢，同时页面布局等静态类数据与商品信息等动态数据在服务端合并渲染，会受动态数据的RT影响而导致页面加载慢，业务方希望能够通过==单次请求分批响应，页面分级渲染，提升用户体验==。
	2. **AI Chat 聊天里：** 流式输出主要承载模型文本；在 [签证Agent](签证Agent)、ReAct Agent、Coding Agent 这类复杂系统里，流式输出还会承载==规划阶段、工具调用、执行日志、中间结果和最终回答==。
2. AI 应用里的流式输出，本质上是把一次完整回答**拆成连续的增量事件**，让用户在模型尚未生成完整答案前，就能看到首个 token、阶段进度、工具执行结果或动态 UI 片段。
3. 它优化的第一目标不是模型真实推理速度，而是==用户感知速度==。非流式模式要求用户等待完整结果，流式模式则让用户先看到系统已经开始工作，**把等待过程变成阅读过程**。

# 二、 SSE 协议
## 1. SSE 的本质
1. 严格地说，[[7.1 HTTP 介绍|HTTP协议]]无法做到服务器主动推送信息。但是，有一种变通方法，就是服务器向客户端声明，接下来要发送的是流信息（streaming）。
	1. 也就是说，发送的不是一次性的数据包，而是一个数据流，会连续不断地发送过来。
	2. 这时，客户端不会关闭连接，会一直等着服务器发过来的新的数据流，视频播放就是这样的例子。
2. SSE，即 **Server-Sent Events**，就是利用这种机制，==基于 HTTP 的服务器到客户端单向事件流协议。
	![[Attachment/d321cd5f55f9ebda17e989351e904d72_MD5.png|428]]
3. SSE 与 [WebSocket](WebSocket) 作用相似，都是建立浏览器与服务器之间的通信渠道，然后服务器向浏览器推送信息。
	1. WebSocket 更强大和灵活。因为它是全双工通道，可以双向通信；
	2. SSE 是单向通道，只能服务器向浏览器发送，因为流信息本质上就是下载。如果浏览器向服务器发送信息，就变成了另一次 HTTP 请求。==但是，SSE 也有自己的优点==。
		- SSE 使用 HTTP 协议，现有的服务器软件都支持。WebSocket 是一个独立协议。
		- SSE 属于轻量级，使用简单；WebSocket 协议相对复杂。
		- SSE 默认支持断线重连，WebSocket 需要自己实现。
		- SSE 一般只用来传送文本，二进制数据需要编码后传送，WebSocket 默认支持传送二进制数据。
		- SSE 支持自定义发送的消息类型。

## 2. SSE 协议格式说明
1. SSE 响应头的关键约定是 `Content-Type: text/event-stream`、`Cache-Control: no-cache` 和长连接保持。每条消息由若干字段组成，以空行结束，常用字段包括 `data`、`event`、`id` 和 `retry`。

| 字段      | 含义      | AI 应用中的用法                                 |
| ------- | ------- | ----------------------------------------- |
| `data`  | 事件内容    | token、阶段状态、工具结果、错误对象                      |
| `event` | 自定义事件类型 | `message`、`progress`、`tool_result`、`done` |
| `id`    | 事件编号    | 支持 `Last-Event-ID`、去重和断点恢复                |
| `retry` | 重连间隔    | 控制浏览器自动重连策略                               |
![[IMG-20260625162232966.png|683]]

## 3. SSE 通信过程
- 客户端用 `EventSource` 发起一个不会立刻结束的 HTTP 请求，服务端把它变成一个持续输出的文本流，浏览器一边接收一边触发事件。
    1. 客户端先创建 **`EventSource`** 对象，向服务端发起一个普通 HTTP 请求。
    2. 服务端收到请求后，不立即返回完整响应，而是先返回 SSE 专用响应头：
        `Content-Type: text/event-stream`
        `Cache-Control: no-cache`
        `Connection: keep-alive`
    3. 连接建立后，客户端进入持续监听状态，等待服务端不断往这个 HTTP 连接里写入数据流。
    4. 服务端按 SSE 规定的文本格式持续推送消息。
        每条消息由若干行字段组成，比如 `data`、`id`、`event`、`retry`，消息之间用空行分隔。
    5. 客户端收到后按字段解析：默认触发 `message` 事件；如果服务端写了 `event` 字段，就触发对应的自定义事件。
    6. 如果连接断开，浏览器会按 **`retry`** 指定的时间自动重连；如果之前服务端发过 `id`，浏览器重连时会带上 `Last-Event-ID`，方便服务端从断点继续推送。
    7. 如果不再需要，客户端调用 **`source.close()`** 主动关闭连接。
        ![[IMG-20260625165958833.png|532]]
## 4. Spring Boot 中 SSE 案例
1. 假设我们正在开发一个实时股票价格监控应用，需要将股票价格实时推送给客户端。下面是在 Spring Boot 中使用 SSE 实现该场景的示例代码。
2. 首先，我们定义一个控制器来处理 SSE 请求并发送实时股票价格。
```java
import org.springframework.http.MediaType;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;
import org.springframework.web.servlet.mvc.method.annotation.SseEmitter;
import java.util.Random;

@RestController
public class StockController {
  
    @GetMapping(value = "/stock-price", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public SseEmitter streamStockPrice() {
        SseEmitter emitter = new SseEmitter();
        // 模拟生成实时股票价格并推送给客户端
        Random random = new Random();
        new Thread(() -> {
            try {
                while (true) {
                    // 生成随机的股票价格
                    double price = 100 + random.nextDouble() * 10;
                    // 构造股票价格的消息
                    String message = String.format("%.2f", price);
                    // 发送消息给客户端
                    emitter.send(SseEmitter.event().data(message));
                    // 休眠1秒钟
                    Thread.sleep(1000);
                }
            } catch (Exception e) {
                emitter.completeWithError(e);
            }
        }).start();
        return emitter;
    }
}
```

3. 上述代码中，我们定义了 `streamStockPrice()` 方法。该方法使用 `@GetMapping` 注解将 `/stock-price` 路径映射到该方法上，并指定 **`produces = MediaType.TEXT_EVENT_STREAM_VALUE`**，表明该方法会产生 SSE 事件流。
4. 在方法内部，创建了一个 **`SseEmitter`** ==对象作为事件发射器==，并在单独线程中不断生成随机股票价格，再将价格转换为字符串发送给客户端。通过 **`emitter.send()`** 方法发送的数据会被封装成 SSE 事件流，客户端可以通过监听该事件流实时接收数据。

# 三、AI 应用流式输出协议选择

1. 大模型对话的主链路通常是“一次用户请求，服务端持续返回”，它不是客户端和服务端高频互相发消息，所以 SSE 比 更贴合默认场景。SSE 可以复用 HTTP 的认证、网关、限流、日志和监控体系，实现成本也更低。
2. WebSocket 更适合双向实时协作、游戏、强交互控制台这类场景；长轮询可以做兼容兜底，但本质还是客户端主动查询；WebTransport 技术上更先进，但浏览器、网关和服务端生态还不如 SSE 成熟。

|协议|通信方向|优势|主要限制|适合场景|
|---|---|---|---|---|
|SSE|服务端到客户端单向|简单、HTTP 友好、自动重连|仅文本、主要单向|AI 对话、进度流、通知流|
|WebSocket|双向全双工|双向实时、支持二进制|连接管理和重连复杂|协作、游戏、强交互 Agent|
|长轮询|客户端主动请求|兼容性好|延迟和请求开销较高|简单状态查询兜底|
|WebTransport|双向、多路复用|低延迟、无 TCP 队头阻塞|生态和部署未完全成熟|未来低延迟多媒体或复杂流|## 4. 在 Spring Boot 中使用 SSE 的场景案例

# 四、MTOP 流式输出
## 1. 链路定位
1. MTOP 流式响应把传统的==“单请求—单响应”改为“单请求—多次响应”==。业务方法收到一次请求后，可以多次调用 **`write()`** 输出数据，最后通过 **`end()`** 或 **`endExceptionally()`** 结束流。
2. 流式请求使用 MTOP 新链路，由 MTOP HttpServer 承接。虽然它与原 HSF HttpServer 底层都基于 Netty，但长连接和多次写出会改变连接占用时间、单机容量及接口限流模型。
3. 在 AI 场景中，完整链路可以抽象为：
    ```mermaid
    flowchart LR
        A["客户端"] -->|"一次 MTOP 请求"| B["MTOP 流式 Provider"]
        B -->|"startStream()"| C["业务编排层"]
        C -->|"流式调用"| D["FAI工作流 / 大模型SDK"]
        D -->|"增量事件"| C
        C -->|"MtopStream.write()"| B
        B -->|"多帧响应"| A
    ```
## 2. 接入前提
1. 仅去中心化 MTOP API 支持该流式能力，API 需要先完成去中心化接入。
2. 应用管理需要迁移到 UltraMax，新控制台才支持流式 API 和流式流量分组配置。
3. MTOP 去中心化插件需升级到 `1.3.2` 及以上，对应 Pandora Sar 至少为 `2024-04-release-fix-fastjson`，并同步升级服务端 SDK。
4. 流式链路通过新端口 `14999` 进入：
    1. 原 VIPServer Key 使用 `12220` 时，可创建端口为 `14999` 的流式流量分组并将 API 路由过去。
    2. 前面存在 Nginx 等代理时，需要将流式 API 转发到 `14999`，同时关闭响应缓冲，否则服务端虽然持续写出，客户端仍可能等缓冲区攒满后一次性收到。
5. 流式与 NextRPC 属于同质能力，不建议叠加使用；如果必须同时使用，应保证第一次流式响应写出前，NextRPC 逻辑已经执行完毕。
## 3. 服务端 API 与生命周期
1. MTOP 服务端提供两个==入口方法==：
    1. **`MtopContext.acceptStream()`**：判断当前客户端和请求链路是否支持流式。
    2. **`MtopContext.startStream()`**：开启流式响应并返回 **`MtopStream`**。
2. **`MtopStream`** 的==核心方法==如下：

| 方法 | 语义 |
| --- | --- |
| `write(Object content)` | 写出一帧数据，连接继续保持 |
| `writeAndEnd(Object content)` | 写出最后一帧并正常结束 |
| `end()` | 不再写数据，正常结束 |
| `endExceptionally(Throwable throwable)` | 以异常状态结束 |

3. 典型服务端逻辑如下：

    ```java

    Mtop3Context context = Mtop3Contexts.getContext();

    if (context == null || !context.acceptStream()) {

        return Result.fail(...);

    }

    MtopStream stream = context.startStream();
    try {
        service.execute(chunk -> stream.write(Result.of(chunk)));
        stream.end();
    } catch (IllegalArgumentException e) {
        stream.writeAndEnd(Result.fail(...));
    } catch (Exception e) {
        stream.endExceptionally(e);
    }
    return null;
    ```
4. 流式响应需要遵循以下生命周期约束：
    1. 必须先判断 **`acceptStream()`**，不支持流式时强行调用 **`startStream()`** 会抛出异常。
    2. **`write()`** 应发生在 **`end()`** 之前；流结束后的 **`write()`** 和 **`writeAndEnd()`** 会被忽略。
    3. **`end()`** 与 **`endExceptionally()`** 只有第一次调用生效，后续重复结束会被忽略。
    4. 在 **`startStream()`** 之前抛出的异常仍按普通单响应处理；开启流之后，业务应显式调用 **`endExceptionally()`**，不能只把异常继续向外抛。
    5. 网络断开等非预期中断不等于业务正常结束，端侧需要结合超时、最后一帧标志或业务状态判断数据是否接收完整。
# 五、AI 预测出签率流式输出改造

## 1. 改造背景与目标
1. 原链路是典型的异步轮询模式：前端提交预测任务 → 后端异步调用 FAI 生成完整结果 → 前端轮询任务状态 → 完成后一次性展示。这种模式在传统业务里很稳，但在大模型生成场景下有一个明显的体验缺口：**模型已经开始输出，用户仍然只能看到”处理中”**。
2. 流式改造的核心目标是把”用户可见完成”和”后端全量完成”解耦：

| 输出段 | 策略 |
| --- | --- |
| 建议文本 `<diag>` | 逐 token 实时推送，打字机效果 |
| 预测分数 `<score>` | 标签闭合后一次性推送，不等 report |
| 完整报告 `<report>` | 流结束后后台落库，不阻塞前端 |

3. 前端在 `</score>` 到来时即可结束等待，后端仍继续把完整报告写库。

## 2. 整体链路
1. 新接口 `submitAndStreamPredict()` 贯穿五层：
    ```java
    MTOP 层
      MtopVisaApprovalPredictionServiceImpl
      └─ acceptStream() → startStream() → 传入 streamWriter (Consumer<PredictStreamChunkVO>)
            ↓
    应用编排层
      VisaApprovalPredictionAppService.submitAndStreamPredict()
      └─ 防抖检查 → initPredictTaskWithExecutionVersion() → 构建会话上下文
            ↓
    流式会话框架
      DefaultStreamMaterialLlmSessionManagerImpl<StreamTagParser>
      └─ initStreamSession → doStreamModelCall (阻塞直到流结束) → parseStreamResult → 写 session 记录
            ↓
    FAI 回调层
      WorkflowServiceFacadeImpl.streamStartWorkflow()
      └─ faiWorkflowClientService.startWithStream() + CompletableFuture 阻塞 600s
         WT_MESSAGE → onMessage(token) → parser.onToken(token)
         WF_FINISHED → completionFuture.complete()
         WF_FAULT    → completionFuture.completeExceptionally()
            ↓
    解析层
      StreamTagParser（五阶段状态机）
      └─ token 逐字符驱动，分段推送或缓冲业务事件
    ```
2. 应用层以 `Consumer<PredictStreamChunkVO>` 为边界抽象 `streamWriter`，MTOP 传输细节与业务事件生产完全解耦。

## 3. ⭐️StreamTagParser：五阶段状态机
* 这是本次改造的核心，负责==把”不可靠的 token 边界”转换成”可靠的业务边界”==。
    ```java
    INIT ──遇到 <diag>──▶ STREAMING ──遇到 </diag>──▶ BUFFERING_SCORE
                                                              │
                                                       遇到 </score>
                                                              │
                                                   BUFFERING_REPORT ──遇到 </report>──▶ DONE
    ```

### 3.1 各阶段行为

| 阶段 | 触发条件 | 核心行为 |
| --- | --- | --- |
| `INIT` | 初始 | `pendingTag` 逐字符匹配 `<diag>` |
| `STREAMING` | `<diag>` 识别 | 每次 `flushSuggestionBuffer()` 推送 `SUGGESTION_CHUNK`；识别 `<adv>` / `<risk>` 子标签给 chunk 打 `type` 字段；遇 `</diag>` 推 `SUGGESTION_COMPLETE` |
| `BUFFERING_SCORE` | `</diag>` | 内容入 `scoringBuffer`；遇 `</score>` 解析 XML → `pushScoreFromScoring()` → 触发 `onScoreComplete` 回调 |
| `BUFFERING_REPORT` | `</score>` | 内容入 `reportBuffer`；遇 `</report>` 置 `DONE` |
| `DONE` | `</report>` | 流结束，`assembleFullResult()` 可供落库 |

### 3.2 跨 Chunk 标签问题
* FAI 单次回调可能只返回 `<d`，下次才返回 `iag>`，不能用 `contains()` 做简单匹配。`StreamTagParser` 用 `pendingTag` 暂存所有以 `<` 开头但尚未完整匹配的字符序列，只有完整匹配到业务标签才切换阶段或刷出内容；未能匹配的字符会被追加回 `suggestionBuffer` 继续参与正文积累。

## 4. 前端事件协议

| 事件 | 推送时机 | 关键字段 |
| --- | --- | --- |
| `SUGGESTION_CHUNK` | `<diag>` 阶段每次 flush | `text`（增量文本）、`type`（`adv`/`risk`，可选） |
| `SUGGESTION_COMPLETE` | `</diag>` 闭合 | — |
| `SCORE` | `</score>` 闭合 | `predictScore`、`predictLevel`、`predictLevelText`、`dimensionDetails`（含 `dimensionName`/`levelText`） |
| `COMPLETED` | `</score>` 后 `persistScoreAndComplete()` 中推送 | `taskId`、`finished=true` |

* `SCORE` 数据由 `pushScoreFromScoring()` 在解析 XML 后通过 `enrichScoreData()` 补全派生字段（`predictLevel` / `predictLevelText` 由分数阈值计算，维度 `levelText` 由维度分映射），前端无需自行换算。

## 5. 两阶段落库

![[Attachment/be359df770250ead18eebb2e34713302_MD5.png|600]]

```java
</score> 触发
    onScoreComplete 回调
        → persistScoreAndComplete()
            ① updateTaskStatus(FINISHED)
            ② 推送 COMPLETED chunk（前端结束渲染）

流结束（doStreamModelCall 返回）
    assembleFullResult()（scoringBuffer + reportBuffer 合并）
        → parseStreamResult() → VisaApprovalPredictionResult
    → persistReportResult()
        ③ updateBizResultByExecutionVersion(taskId, result, executionVersion)
           条件：task_id = ? AND execution_version = ?
           匹配才写入；不匹配静默跳过（用户已重新提交）
```

* 阶段一和阶段二在同一个调用线程内串行执行（`doStreamModelCall` 阻塞等 FAI 流完成），不存在并发竞争；`executionVersion` 仅用于防用户重复提交时跨任务的数据覆盖。

### 5.1 两个回调各自的落库职责

1. 两阶段落库由两个回调驱动，它们在构建 `StreamTagParser` 时由应用层注入：
    * **`Consumer<PredictStreamChunkVO> streamWriter`**：每产生一个业务事件（`SUGGESTION_CHUNK`、`SCORE`、`COMPLETED` 等），parser 通过 `streamWriter.accept(chunk)` 推给 MTOP 层写出。它本身不直接落库，但在阶段一中，`onScoreCompleteCallback` 内部会借助 `streamWriter` 推送 `COMPLETED` chunk 通知前端。
    * **`Runnable onScoreCompleteCallback`**：`</score>` 闭合时由 parser 调用 `onScoreCompleteCallback.run()`，触发阶段一落库——`persistScoreAndComplete()`，内部做两件事：1️⃣ `updateTaskStatus(FINISHED)`，2️⃣通过 `streamWriter` 推送 `COMPLETED` chunk。
2. 阶段二的落库不经过这两个回调，而是在 `doStreamModelCall` 阻塞等流结束后，由请求线程串行执行 `persistReportResult()`，用 `task_id + execution_version` 条件更新写入完整 report。
3. 整体时序：
    * 流进行中 → parser 逐 token 解析 → `streamWriter.accept()` 不断推送 chunk 给前端
    * `</score>` 闭合 → `onScoreCompleteCallback.run()` → 阶段一落库（FINISHED + COMPLETED）
    * `</report>` 闭合 → 流结束 → 请求线程继续 → 阶段二落库（report 写入）
4. 两个回调都在**同一个调用线程**上同步执行（parser 的 `onToken` 由 `doStreamModelCall` 的阻塞循环驱动），不存在并发竞争，这是不需要加锁的前提。详见 [[CompletableFuture（CF） 介绍]]。### 5.1 两个回调各自的落库职责

### 5.3 异常情况处理

* 幂等：`WF_FINISHED` 可能重复回调，断线重连也可能重发，落库必须幂等；本方案用 `task_id + execution_version` 条件更新天然幂等。
* 异常与状态区分：`WF_FAULT` → `completeExceptionally`，请求线程 `get()` 抛出后应把任务落为失败状态，而不是把半截结果当成功落库。
* 超时兜底：`completionFuture.get(600s)` 超时同样抛异常，需走 `endExceptionally` 并落超时状态，避免任务永远停在“处理中”。
* 事务边界：两阶段是两次独立 update，不在同一事务，不追求强一致，最终一致由 `execution_version` 条件更新兜底。

## 6. executionVersion：防覆盖机制
1. 每次调用 `initPredictTaskWithExecutionVersion()` 生成一个 UUID 作为 `executionVersion`，写入 `visa_llm_task.execution_version`。
2. 同一个用户同一个出行人（`bizId = userId_visaPassengerId`）共用同一条 task 记录（`task_id` 不变），但每次重新提交会刷新 `executionVersion`。report 落库时用 `task_id + execution_version` 做条件更新：若用户在旧流执行期间重新提交，旧流的 report 写入会因 `execution_version` 不匹配而静默跳过，避免旧数据污染新结果。

### 6.1 本质：乐观锁的版本号

1. `execution_version` 的更新条件 `WHERE task_id = ? AND execution_version = ?`，本质就是[[超卖问题（乐观锁）|乐观锁]]的版本号比对：读时记下版本，写时校验版本一致才更新，不一致说明记录已被“另一条流”改写，本次写入作废。
2. 它和经典乐观锁的差别在语义而非机制：
    * 经典乐观锁（如库存扣减）用 version 防并发覆盖，失败后通常要重试。
    * `executionVersion` 用 version 防“过期的旧流”覆盖“新提交的结果”，失败应静默跳过而不是重试——旧流的结果本就该被丢弃。
3. 所以它是“版本号 + 不重试”的乐观锁变体：校验不通过即代表用户已重新提交，旧流落库直接放弃。

### 6.2 版本刷新与覆盖时序

1. 每次 `initPredictTaskWithExecutionVersion()` 生成新 UUID 作 `executionVersion`，UPSERT 进 `visa_llm_task`：同一 `bizId` 复用 `task_id`，只把 `execution_version` 刷成新值。
2. 旧流仍持有“提交那一刻”的旧版本；新提交把库里的版本改成新值。
3. 旧流落 report 时带的是旧版本，`AND execution_version = 旧值` 匹配不到当前记录，`update` 影响 0 行，静默跳过。

## 7. ⭐️⭐️断点与恢复：序号化续接机制

### 7.1 问题场景
1. 流式输出有一个持续的时间窗口，在这个窗口内用户可能遇到两类中断：
    * **弱网闪断**：电梯、地铁、Wi-Fi 切换导致网络短暂中断，用户还在当前页面等着。
    * **用户离开页面**：手动关闭页面、退出 App、切后台被系统杀掉 WebView，稍后再回来。
2. 在 MTOP 长连接方案下，这两类中断的结果是一样的——**连接断了，流就没了**。MTOP 底层是 HTTP 长连接，没有 SSE `Last-Event-ID` 自动重连语义（见[[#3. SSE 通信过程|二、3]]），每次 `startStream` 都是独立的 HTTP 请求，无法在原来的流上恢复。
3. 所以无论哪种中断，都必须靠**业务层的续接机制**来解决。

### 7.2 为什么不能直接重连原来的流
1. MTOP 流绑定在特定的 HTTP 连接上，连接断开后：
    * 服务端的 `streamWriter`（`Consumer<PredictStreamChunkVO>`）绑定的是原来那条连接的写入器，新连接拿不到这个引用。
    * 即使发起新的 `startStream`，服务端也不知道这是"同一个任务的续传请求"——它会当作一个新请求处理。
2. 所以如果没有额外的缓存机制，当用户返回页面查到 `task_status = PROCESSING` 时，只能展示"生成中"轮询等待，**打字机效果不可能续上**。

### 7.3 解决方案：序号化续接
1. 同项目的行程助手（f-ai-ask）已经实现了一套完整的**序号化续接机制**，可以直接复用。核心思路：弱网闪断和用户退出页面两类问题，用同一套 Tair 缓存 + 续接接口统一解决。

#### 7.3.1 写入侧：chunk 带序号写入 Tair
1. 每个 chunk 在推送给前端的同时，带上递增序号（`seq`），以 `{messageId}{seq}` 为 key 写入 Tair（TTL 2 小时）。
2. 关键逻辑（`JourneyPlanAiAssistantContentHelper.recordStreamMessageResult()`）：
    ```java
    synchronized (systemMessageId.intern()) {
        Integer seq = CONTENT_SEQ_MAP.getOrDefault(systemMessageId, 0);
        result.setSeq(seq);
        // chunk 写入 Tair，key = "ai:journey:content:map:{msgId}{seq}"
        mdbTairHelper.putValue(genContentKey(systemMessageId, seq), JSON.toJSONString(result), TTL);
        // 序号递增
        CONTENT_SEQ_MAP.put(systemMessageId, seq + 1);
        // 最终序号也写入 Tair，续接时用来判断是否已全部生成
        mdbTairHelper.putValue(genMessageSeqKey(systemMessageId), String.valueOf(seq), TTL);
    }
    ```

#### 7.3.2 续接侧：continueChat 从断点恢复
1. 前端断线重连时，调用独立的 `continueChat` 接口，携带当前已收到的最大 `seq`。服务端开启一条**全新的 MTOP 流**，从 Tair 中取出 `seq` 之后的 chunk 逐条写出，同时轮询等待尚未生成的 chunk。
2. 关键逻辑（`outputCacheContent()`）：
    ```java
    // 从前端传入的 seq 开始，逐条从 Tair 取 chunk
    while (true) {
        String key = genContentKey(systemMessageId, curSeq);
        Object value = mdbTairHelper.getValue(key);
        if (value == null) {
            if (已超时) break;
            Thread.sleep(20ms);  // 等待生成线程写入新 chunk
            // 每 50 次发一个心跳包保活
            continue;
        }
        mtopStream.write(String.valueOf(value));  // 通过新 MTOP 流推送
        curSeq++;
    }
    ```

![[Attachment/23776ad6df380d5f2e8abf03497ab8f1_MD5.png|723]]

#### 7.3.3 Tair 存储设计
1. 底层通过 `MdbTairHelper` 封装 Tair（阿里分布式缓存）的读写，namespace = `407`，核心 API：
    * `putValue(key, value, expireTime)` → `tairManager.put(407, key, value, 0, expireTime)`
    * `getValue(key)` → `tairManager.get(407, key)`
    * `mgetValue(keys)` → `tairManager.mget(407, keys)`（批量查询，用于 seq=0 全量回放）
2. 缓存 key 设计和 TTL：

    | 数据 | key 格式 | TTL | 用途 |

    | --- | --- | --- | --- |

    | 单条 chunk | `ai:journey:content:map:{msgId}{seq}` | 2 小时 | 按序号存储每一帧内容 |

    | 最终序号 | `ai:journey:content:seq:{msgId}` | 2 小时 | 续接时判断是否已全部生成 |

    | 消息状态 | `ai:journey:messageId:status:{msgId}` | 1 小时 | 标记流是否结束（END / STOP / BLOCK） |

    ![[Attachment/fbd5bdd7f891cf7ed77e5b1ed8b22fca_MD5.png|733]]

3. Tair 故障时的行为：`putValue` 返回 false 并记日志，不影响主流程推送；但续接接口会因读不到缓存而退化为等待超时。没有备用存储兜底。

#### 7.3.4 序号管理：内存 + Tair 双层
1. 写入侧维护一个 `ConcurrentMap<String, Integer> CONTENT_SEQ_MAP`，跟踪每个 `messageId` 当前的序号，写入 chunk 时 `synchronized(messageId.intern())` 保证同一消息的序号严格递增。
2. 每次写入 chunk 后，序号同时更新到内存 map 和 Tair（`ai:journey:content:seq:{msgId}`）。内存 map 保证活跃流的快速访问，Tair 保证服务重启后续接线程仍能从持久化序号恢复。
3. **seq=0 全量回放**：如果前端传入 `seq=0`（首次进入或完全丢失上下文），续接侧调用 `getAlreadyOutputStreamMessageResult()` 按 100 条一批从 Tair 批量查出所有 chunk（`mget`），合并为一条完整的 `StreamMessageResult` 一次性写出，然后再从 `maxSeq` 开始续接增量。

#### 7.3.5 心跳保活机制
1. 续接线程在轮询等待新 chunk 时，每等待 50 次（`50 × 20ms = 1 秒`）发送一个心跳包：
    ```java
    if (waitTime % 50 == 0) {
        mtopStream.write(JSON.toJSONString(StreamMessageResult.heartBeat()));
    }
    ```
2. `StreamMessageResult.heartBeat()` 只设 `heartBeat = true`，其余字段为空，前端收到后忽略内容、仅重置超时计时器。
3. 心跳的目的：
    * 防止 MTOP 长连接因空闲被客户端 SDK 或中间代理（Nginx / SLB）超时断开。
    * 告诉前端"服务端还在等数据生成，不是挂了"。

#### 7.3.6 超时与退出控制
1. 超时参数（均可通过 `@AppSwitch` 动态调整）：

    | 参数 | 默认值 | 含义 |

    | --- | --- | --- |

    | `WAIT_TIME_MS` | 20ms | 每次轮询间隔 |

    | `CONTINUE_CHAT_TIMEOUT_COUNT` | 1500 | 最大等待次数 |

    总超时 = `1500 × 20ms = 30 秒`。

2. 续接线程的退出条件有三个，任一满足即结束循环：
    * **正常完成**：消息状态变为 `END`，且 `curSeq >= maxSeq`（所有 chunk 已读完）。
    * **超时**：连续等待次数超过 `CONTINUE_CHAT_TIMEOUT_COUNT`，放弃等待。
    * **流被中止**：`messageStatusHelper.isStop()` 返回 true（用户主动停止或风控拦截 BLOCK）。
3. 超时计数器在每次成功读到 chunk 后**归零**——只要生成线程还在持续写入，续接线程就不会超时。只有当生成线程停止写入且消息状态未更新为 END 时，才会触发 30 秒超时退出。
4. 续接结束后，`continueChat` 发一条最终心跳（`writeAndEnd`）关闭 MTOP 流，前端据此判断续接完成。

#### 7.3.7 整体流程
1. 正常流程和续接流程通过 Tair 解耦，生成线程只管写，续接线程只管读：
    ```java
    【正常流程】
    生成线程 → streamWriter.accept(chunk)
                ├→ MTOP 流写出给前端（实时推送）
                └→ recordStreamMessageResult() → Tair 缓存（seq 递增）

    【弱网闪断 / 用户退出再返回】
    前端携带 seq → continueChat
                    → startStream() 开新 MTOP 流
                    → outputCacheContent()
                        → 从 Tair 读 seq 之后的 chunk → 新流写出（回放已有）
                        → 轮询等待新 chunk → 新流写出（续接增量）
                        → 每秒发心跳保活
                        → 正常结束 / 超时 30s / 流被中止 → 流结束
    ```
2. 对前端来说，续接接口返回的数据格式和正常流完全一致，效果等同于"续上了打字机"。

### 7.4 适配签证预测场景的改造点
1. 核心改造：在 `streamWriter.accept()` 的同时调用 `recordStreamMessageResult()` 把 chunk 写入 Tair，并提供 `continueChat` 续接入口。
2. 需要注意的差异：
    * 行程助手用 `systemMessageId` 作缓存维度，签证预测可以用 `taskId`。
    * 行程助手的 chunk 是纯文本增量，签证预测的 chunk 是 `PredictStreamChunkVO`（包含 `SUGGESTION_CHUNK`、`SCORE`、`COMPLETED` 等类型），续接时前端需要按类型分别处理。
