# 一、流式输出是什么
1. 解决的问题：
	1. **业务复杂数据大耗时长**：例如Feeds类大列表涉及个性化推荐，响应较慢，同时页面布局等静态类数据与商品信息等动态数据在服务端合并渲染，会受动态数据的RT影响而导致页面加载慢，业务方希望能够通过==单次请求分批响应，页面分级渲染，提升用户体验==。
	2. **AI Chat 聊天里：** 流式输出主要承载模型文本；在 [签证Agent](签证Agent)、ReAct Agent、Coding Agent 这类复杂系统里，流式输出还会承载==规划阶段、工具调用、执行日志、中间结果和最终回答==。
2. AI 应用里的流式输出，本质上是把一次完整回答**拆成连续的增量事件**，让用户在模型尚未生成完整答案前，就能看到首个 token、阶段进度、工具执行结果或动态 UI 片段。
3. 它优化的第一目标不是模型真实推理速度，而是==用户感知速度==。非流式模式要求用户等待完整结果，流式模式则让用户先看到系统已经开始工作，**把等待过程变成阅读过程**。

# 二、 SSE 协议
## 1. SSE 的本质
1. 严格地说，[[7.1 HTTP 介绍|HTTP协议]]无法做到服务器主动推送信息。但是，有一种变通方法，就是服务器向客户端声明，接下来要发送的是流信息（streaming）。
	1. 也就是说，发送的不是一次性的数据包，而是一个数据流，会连续不断地发送过来。
	2. 这时，客户端不会关闭连接，会一直等着服务器发过来的新的数据流，视频播放就是这样的例子。
2. SSE，即 **Server-Sent Events**，就是利用这种机制，==基于 HTTP 的服务器到客户端单向事件流协议。
	![[Attachment/d321cd5f55f9ebda17e989351e904d72_MD5.png|428]]
3. SSE 与 [WebSocket](WebSocket) 作用相似，都是建立浏览器与服务器之间的通信渠道，然后服务器向浏览器推送信息。
	1. WebSocket 更强大和灵活。因为它是全双工通道，可以双向通信；
	2. SSE 是单向通道，只能服务器向浏览器发送，因为流信息本质上就是下载。如果浏览器向服务器发送信息，就变成了另一次 HTTP 请求。==但是，SSE 也有自己的优点==。
		- SSE 使用 HTTP 协议，现有的服务器软件都支持。WebSocket 是一个独立协议。
		- SSE 属于轻量级，使用简单；WebSocket 协议相对复杂。
		- SSE 默认支持断线重连，WebSocket 需要自己实现。
		- SSE 一般只用来传送文本，二进制数据需要编码后传送，WebSocket 默认支持传送二进制数据。
		- SSE 支持自定义发送的消息类型。

## 2. SSE 协议格式说明
1. SSE 响应头的关键约定是 `Content-Type: text/event-stream`、`Cache-Control: no-cache` 和长连接保持。每条消息由若干字段组成，以空行结束，常用字段包括 `data`、`event`、`id` 和 `retry`。

| 字段      | 含义      | AI 应用中的用法                                 |
| ------- | ------- | ----------------------------------------- |
| `data`  | 事件内容    | token、阶段状态、工具结果、错误对象                      |
| `event` | 自定义事件类型 | `message`、`progress`、`tool_result`、`done` |
| `id`    | 事件编号    | 支持 `Last-Event-ID`、去重和断点恢复                |
| `retry` | 重连间隔    | 控制浏览器自动重连策略                               |
![[IMG-20260625162232966.png|683]]

## 3. SSE 通信过程
- 客户端用 `EventSource` 发起一个不会立刻结束的 HTTP 请求，服务端把它变成一个持续输出的文本流，浏览器一边接收一边触发事件。
    1. 客户端先创建 **`EventSource`** 对象，向服务端发起一个普通 HTTP 请求。
    2. 服务端收到请求后，不立即返回完整响应，而是先返回 SSE 专用响应头：
        `Content-Type: text/event-stream`
        `Cache-Control: no-cache`
        `Connection: keep-alive`
    3. 连接建立后，客户端进入持续监听状态，等待服务端不断往这个 HTTP 连接里写入数据流。
    4. 服务端按 SSE 规定的文本格式持续推送消息。
        每条消息由若干行字段组成，比如 `data`、`id`、`event`、`retry`，消息之间用空行分隔。
    5. 客户端收到后按字段解析：默认触发 `message` 事件；如果服务端写了 `event` 字段，就触发对应的自定义事件。
    6. 如果连接断开，浏览器会按 **`retry`** 指定的时间自动重连；如果之前服务端发过 `id`，浏览器重连时会带上 `Last-Event-ID`，方便服务端从断点继续推送。
    7. 如果不再需要，客户端调用 **`source.close()`** 主动关闭连接。
        ![[IMG-20260625165958833.png|532]]
## 4. Spring Boot 中 SSE 案例
1. 假设我们正在开发一个实时股票价格监控应用，需要将股票价格实时推送给客户端。下面是在 Spring Boot 中使用 SSE 实现该场景的示例代码。
2. 首先，我们定义一个控制器来处理 SSE 请求并发送实时股票价格。
```java
import org.springframework.http.MediaType;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;
import org.springframework.web.servlet.mvc.method.annotation.SseEmitter;
import java.util.Random;

@RestController
public class StockController {
  
    @GetMapping(value = "/stock-price", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public SseEmitter streamStockPrice() {
        SseEmitter emitter = new SseEmitter();
        // 模拟生成实时股票价格并推送给客户端
        Random random = new Random();
        new Thread(() -> {
            try {
                while (true) {
                    // 生成随机的股票价格
                    double price = 100 + random.nextDouble() * 10;
                    // 构造股票价格的消息
                    String message = String.format("%.2f", price);
                    // 发送消息给客户端
                    emitter.send(SseEmitter.event().data(message));
                    // 休眠1秒钟
                    Thread.sleep(1000);
                }
            } catch (Exception e) {
                emitter.completeWithError(e);
            }
        }).start();
        return emitter;
    }
}
```

3. 上述代码中，我们定义了 `streamStockPrice()` 方法。该方法使用 `@GetMapping` 注解将 `/stock-price` 路径映射到该方法上，并指定 **`produces = MediaType.TEXT_EVENT_STREAM_VALUE`**，表明该方法会产生 SSE 事件流。
4. 在方法内部，创建了一个 **`SseEmitter`** ==对象作为事件发射器==，并在单独线程中不断生成随机股票价格，再将价格转换为字符串发送给客户端。通过 **`emitter.send()`** 方法发送的数据会被封装成 SSE 事件流，客户端可以通过监听该事件流实时接收数据。

# 三、AI 应用流式输出协议选择

1. 大模型对话的主链路通常是“一次用户请求，服务端持续返回”，它不是客户端和服务端高频互相发消息，所以 SSE 比 更贴合默认场景。SSE 可以复用 HTTP 的认证、网关、限流、日志和监控体系，实现成本也更低。
2. WebSocket 更适合双向实时协作、游戏、强交互控制台这类场景；长轮询可以做兼容兜底，但本质还是客户端主动查询；WebTransport 技术上更先进，但浏览器、网关和服务端生态还不如 SSE 成熟。

|协议|通信方向|优势|主要限制|适合场景|
|---|---|---|---|---|
|SSE|服务端到客户端单向|简单、HTTP 友好、自动重连|仅文本、主要单向|AI 对话、进度流、通知流|
|WebSocket|双向全双工|双向实时、支持二进制|连接管理和重连复杂|协作、游戏、强交互 Agent|
|长轮询|客户端主动请求|兼容性好|延迟和请求开销较高|简单状态查询兜底|
|WebTransport|双向、多路复用|低延迟、无 TCP 队头阻塞|生态和部署未完全成熟|未来低延迟多媒体或复杂流|## 4. 在 Spring Boot 中使用 SSE 的场景案例

# 四、MTOP 流式输出
## 1. 链路定位
1. MTOP 流式响应把传统的“单请求—单响应”改为“单请求—多次响应”。业务方法收到一次请求后，可以多次调用 **`write()`** 输出数据，最后通过 **`end()`** 或 **`endExceptionally()`** 结束流。
2. 流式请求使用 MTOP 新链路，由 MTOP HttpServer 承接。虽然它与原 HSF HttpServer 底层都基于 Netty，但长连接和多次写出会改变连接占用时间、单机容量及接口限流模型。
3. 在 AI 场景中，完整链路可以抽象为：
    ```mermaid
    flowchart LR
        A["客户端"] -->|"一次 MTOP 请求"| B["MTOP 流式 Provider"]
        B -->|"startStream()"| C["业务编排层"]
        C -->|"流式调用"| D["FAI / 大模型工作流"]
        D -->|"增量事件"| C
        C -->|"MtopStream.write()"| B
        B -->|"多帧响应"| A
    ```
## 2. 接入前提
1. 仅去中心化 MTOP API 支持该流式能力，API 需要先完成去中心化接入。
2. 应用管理需要迁移到 UltraMax，新控制台才支持流式 API 和流式流量分组配置。
3. MTOP 去中心化插件需升级到 `1.3.2` 及以上，对应 Pandora Sar 至少为 `2024-04-release-fix-fastjson`，并同步升级服务端 SDK。
4. 流式链路通过新端口 `14999` 进入：
    1. 原 VIPServer Key 使用 `12220` 时，可创建端口为 `14999` 的流式流量分组并将 API 路由过去。
    2. 前面存在 Nginx 等代理时，需要将流式 API 转发到 `14999`，同时关闭响应缓冲，否则服务端虽然持续写出，客户端仍可能等缓冲区攒满后一次性收到。
5. 流式与 NextRPC 属于同质能力，不建议叠加使用；如果必须同时使用，应保证第一次流式响应写出前，NextRPC 逻辑已经执行完毕。
## 3. 服务端 API 与生命周期
1. MTOP 服务端提供两个入口方法：
    1. **`MtopContext.acceptStream()`**：判断当前客户端和请求链路是否支持流式。
    2. **`MtopContext.startStream()`**：开启流式响应并返回 **`MtopStream`**。
2. **`MtopStream`** 的核心方法如下：

| 方法 | 语义 |
| --- | --- |
| `write(Object content)` | 写出一帧数据，连接继续保持 |
| `writeAndEnd(Object content)` | 写出最后一帧并正常结束 |
| `end()` | 不再写数据，正常结束 |
| `endExceptionally(Throwable throwable)` | 以异常状态结束 |

3. 典型服务端逻辑如下：

    ```java

    Mtop3Context context = Mtop3Contexts.getContext();

    if (context == null || !context.acceptStream()) {

        return Result.fail(...);

    }

    MtopStream stream = context.startStream();
    try {
        service.execute(chunk -> stream.write(Result.of(chunk)));
        stream.end();
    } catch (IllegalArgumentException e) {
        stream.writeAndEnd(Result.fail(...));
    } catch (Exception e) {
        stream.endExceptionally(e);
    }
    return null;
    ```
4. 流式响应需要遵循以下生命周期约束：
    1. 必须先判断 **`acceptStream()`**，不支持流式时强行调用 **`startStream()`** 会抛出异常。
    2. **`write()`** 应发生在 **`end()`** 之前；流结束后的 **`write()`** 和 **`writeAndEnd()`** 会被忽略。
    3. **`end()`** 与 **`endExceptionally()`** 只有第一次调用生效，后续重复结束会被忽略。
    4. 在 **`startStream()`** 之前抛出的异常仍按普通单响应处理；开启流之后，业务应显式调用 **`endExceptionally()`**，不能只把异常继续向外抛。
    5. 网络断开等非预期中断不等于业务正常结束，端侧需要结合超时、最后一帧标志或业务状态判断数据是否接收完整。
# 五、AI 预测出签率流式输出改造
## 1. 为什么要改
1. 这次改造表面上是把 AI 预测出签率从“一次性返回”改成“边生成边展示”，但真正要解决的问题不是接口形式，而是用户等待模型生成时的空白体验。
2. 原链路是典型异步任务模式：前端提交预测任务，后端异步调用 FAI，前端轮询任务状态，直到完整结果生成后一次性展示。这种模式在传统业务里很稳，但放到大模型生成场景里会有一个明显缺口：模型已经开始输出了，用户却仍然只能看到“处理中”。
3. 流式改造的目标是重新定义“完成”的含义：建议内容可以先逐段展示，分数要尽快返回，完整报告可以继续在后端补齐并落库。换句话说，前端不必等到后端所有事情都结束，才开始消费模型已经产生的价值。

## 2. 改造后的链路
1. 新链路以 **`submitAndStreamPredict()`** 作为入口，MTOP 层先通过 **`acceptStream()`** 判断当前请求是否支持流式能力，再通过 **`startStream()`** 获取 **`MtopStream`**，后续用 **`write()`** 多次向前端写出事件。
	![[Attachment/2b4db560513ee65671a54509c59de4e4_MD5.png|577]]
2. 应用层没有直接绑定 MTOP，而是抽象出 **`Consumer<PredictStreamChunkVO>`** 作为 **`streamWriter`**。这个设计的核心价值是隔离职责：传输层只关心怎么写出，业务层只关心产出什么事件。
3. FAI 层通过 **`startWithStream()`** 接收真实增量输出，再把 **`WT_MESSAGE`** 交给 **`StreamTagParser`** 做结构化解析。最终前端看到的不是原始 token，而是被整理后的业务事件，例如 **`SUGGESTION_CHUNK`**、**`SUGGESTION_COMPLETE`**、**`SCORE`**、**`COMPLETED`**。
4. 这条链路可以拆成四层理解。
    1. MTOP 流式层负责长连接、多次写出和异常结束。
    2. 应用编排层负责预测任务、降级开关、事件封装和任务状态推进。
    3. FAI 回调层负责接收模型增量输出。
    4. 解析落库层负责把非结构化 token 流变成可展示、可持久化、可恢复的业务结果。

## 3. 关键难点
### 3.1 难点一：模型输出是 Token 流，不是稳定 JSON
1. 最容易低估的一点是，FAI 流式回调返回的不是一个天然完整的业务对象，而是一段段 token。模型可能在一次回调里返回 `<diag>`，也可能把 `<diag>` 拆成 `<d`、`ia`、`g>` 多次返回。
2. 如果解析逻辑只按字符串片段做简单匹配，就会在标签跨 chunk 时失效。轻一点的结果是少展示几个字，重一点的结果是把标签内容误当成正文，或者提前把半截分数、半截报告交给前端。
3. 所以这里不能把流式解析写成普通的 **`contains()`** 判断，而是要把它看成一个小型协议解析器。**`StreamTagParser`** 通过逐字符状态机维护当前解析位置，并用 **`pendingTag`** 暂存尚未确认的标签片段，直到确认进入 **`<diag>`**、**`<score>`**、**`<report>`** 这样的业务区间。
4. 这一步的本质是把“不可靠的 token 边界”转换成“可靠的业务边界”。只有业务边界稳定，后面才能谈前端展示、任务状态和数据库一致性。

### 3.2 难点二：不同标签的消费节奏完全不同
1. 同样是模型输出，**`<diag>`**、**`<score>`**、**`<report>`** 的消费方式不一样。**`<diag>`** 是用户可直接阅读的建议文本，适合一边生成一边 flush 给前端，形成打字机效果。
2. **`<score>`** 不能按 token 直接消费，因为分数是结构化结果，只有闭合标签出现以后才具备完整语义。如果提前推送，前端可能拿到不完整分数，也可能在模型继续修正时出现展示抖动。
3. **`<report>`** 更不能简单流给前端后立刻落库，因为它通常是完整报告数据，承担后续详情页或历史记录读取职责。它需要等内容完整后统一解析、校验和持久化。
4. 因此这条流里同时存在三种节奏。
    1. 即时展示型：**`<diag>`** 解析到正文后立刻输出 **`SUGGESTION_CHUNK`**。
    2. 闭合确认型：**`<score>`** 等到完整闭合后输出 **`SCORE`**。
    3. 后台沉淀型：**`<report>`** 等完整生成后通过 **`persistReportResult()`** 落库。
5. 这个设计让“用户可见完成”和“后端全量完成”分开了。前端可以更早结束等待，后端仍然可以继续把完整报告补齐。

### 3.3 难点三：流式接口不能只考虑成功路径
1. 普通接口失败时返回一个错误响应就结束了，但流式接口一旦调用 **`startStream()`**，响应就已经开始。后续再抛异常，不能再指望传统单响应异常处理器帮你兜底。
2. 因此流式链路需要显式管理生命周期：正常路径要 **`write()`** 后 **`end()`**，业务失败可以 **`writeAndEnd()`** 返回最后一帧错误，非预期异常要 **`endExceptionally()`** 结束连接。
3. 这里还要区分“业务完成”和“连接结束”。连接断开不一定代表模型工作流完成，模型完成也不一定代表 report 已经落库。端侧和后端都需要依赖业务事件判断状态，而不是只看 HTTP 连接是否关闭。

### 3.4 难点四：重复提交会带来结果覆盖问题
1. AI 预测出签率不是纯展示接口，它会写任务状态和结果数据。用户可能重复点击、刷新重试，或者前一次工作流慢返回，后一轮工作流先完成。
2. 如果只按 **`task_id`** 更新结果，就会出现旧执行覆盖新执行的问题。这个问题在线性接口里不明显，在流式和异步工作流里会被放大，因为多个执行版本可能同时处于运行中。
3. 当前方案给每次流式执行生成新的 **`executionVersion`**。完整 report 落库时按 **`task_id + execution_version`** 做条件更新，保证只有当前执行版本能写入自己的结果。
4. 这一步不是为了多一个字段，而是为了给异步模型链路补上“因果顺序”。谁发起、谁输出、谁落库，都必须能被同一个执行版本串起来。

## 4. 完成点与边界设计
1. 当前实现把前端完成点放在 **`</score>`**：当解析到完整分数后，服务端推送 **`SCORE`**，更新任务状态为 **`FINISHED`**，再推送 **`COMPLETED(finished=true)`**，前端即可结束展示。
2. 完整 **`<report>`** 仍会继续在后端生成，并通过 **`persistReportResult()`** 写入 **`biz_result_data`**。因此 **`COMPLETED`** 到 report 落库之间会存在一个短暂窗口。
3. 这个窗口需要被产品和技术同时承认：如果页面只展示本次流里的建议和分数，当前设计是成立的；如果页面在 **`COMPLETED`** 后立刻读数据库展示完整报告，就需要补阶段一 CAS 落库，或者增加“报告是否就绪”的状态。

## 5. 上线前检查点
1. MTOP 流式 API 与流量分组必须指向支持流式的新链路，代理层需要关闭响应缓冲，否则服务端持续 **`write()`**，客户端仍可能一次性收到。
2. **`execution_version`** 字段 DDL 必须先发布，否则流式链路无法做版本隔离，重复提交下的一致性保护会失效。
3. **`FAI_VISA_PREDICTION_STREAM_ENABLED`** 需要真正接入入口降级逻辑。流式能力应该可以灰度打开，也应该能在 FAI、MTOP 或前端兼容问题出现时快速退回原轮询链路。
4. 前端需要明确消费事件顺序：**`SUGGESTION_CHUNK`** 用于增量展示，**`SCORE`** 用于分数展示，**`COMPLETED`** 用于结束本次流式渲染，而不是代表所有后端持久化工作都已经完成。
