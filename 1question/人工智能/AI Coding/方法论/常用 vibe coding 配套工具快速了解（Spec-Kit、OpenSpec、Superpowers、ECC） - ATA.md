> [!abstract] 核心摘要
> 本文把 vibe coding 工具拆成“决策层”和“执行层”两层：前者负责在写代码前形成规格、变更与任务锚点，后者负责把工作流、约束与工具链真正落实到编码执行过程。作者通过 Spec-Kit、OpenSpec、Superpowers、ECC 四个代表性工具，说明结构化文档、轻量变更管理、自动触发技能和工具大全之间的差异，并用一个提醒应用示例串起从需求到实现的典型流程。对正在搭建 AI Coding 方法栈的人来说，这篇文章更像是一张选型地图，而不是单一工具教程。

# 一、我的整理
## 1. 一句话结论
* 这四个工具并不是互斥关系，而是分别覆盖“需求结构化”“变更管理”“执行工作流标准化”“工具与配置资产库”四种不同层次，真正有效的落地方式通常是组合使用，而不是只押注某一个框架。

## 2. 全文主线
1. 作者先提出一个很实用的划分方法：把 AI Coding 工具分成“决策层”和“执行层”。
2. 然后围绕四个代表工具，分别解释它们的定位、核心能力、优缺点与适用边界。
3. 最后用一个简单应用开发示例，演示从规格制定、变更提案到执行编排的协作路径，强调工具栈应服务于“少猜、少跑偏、可持续演进”。

## 3. 最值得记住的判断
* Spec-Kit 适合新项目从一开始就建立严格规格驱动流程，但对小需求偏重。
* OpenSpec 在“变更提案 → 实施 → 归档”这一闭环上更轻量，更适合增量演进。
* Superpowers 的价值不在功能多，而在用 skills 把优秀流程变成强制性执行规范。
* ECC 更像一个经过生产验证的 agents、skills、hooks、rules 工具箱，强在资产丰富，弱在需要自行筛选。
* 一个成熟团队往往会把“决策层工具”与“执行层工具”叠加，而不是只依赖 prompt 本身来约束 AI。

# 三、正文整理
## 1. 为什么要区分“决策层”和“执行层”
1. 作者认为，各种 vibe coding 工具的共同目标都是让 AI Coding 从“凭感觉写代码”走向“有结构、有锚点、有约束的规范化执行”。
2. 其中“决策层”负责写代码之前的动作，例如需求澄清、规格生成、方案设计与任务拆解；“执行层”则负责具体编码、测试、工作流触发与过程约束。
3. 这种拆法很重要，因为它说明很多工具并不是替代关系，而是前后衔接关系。

## 2. 四类工具的角色分工
### 2.1 Spec-Kit
1. Spec-Kit 的核心是 SDD，也就是规格驱动开发。它强调在实现前，先通过一整套命令生成结构化文档，明确“做什么”“为什么做”“如何拆解”。
2. 它的典型工作流包括：初始化项目、写项目宪法、产出需求 spec、澄清歧义、制定技术方案、拆解任务，再进入实现阶段。
3. 这种模式的优点是流程清晰、可控性强，特别适合从零启动的新项目；缺点是成本较高，对小型需求和明确 bugfix 来说过于笨重，而且文档质量一旦偏差，后续会被持续放大。

### 2.2 OpenSpec
1. OpenSpec 与 Spec-Kit 同属决策层工具，但它更强调轻量级变更管理，而不是完整的大流程治理。
2. 它用“propose / apply / archive”三件套管理功能迭代，让需求规范、技术方案和任务清单围绕一次变更增量演进。
3. 这种方式保留了规格生命周期，但减少了过重的流程负担，更适合已有项目的持续迭代；代价是当需求变化频繁、方案还没有稳定下来时，反复跑规范流程会比较浪费 token。

### 2.3 Superpowers
1. Superpowers 属于执行层工具，它的关键价值是通过 skills 自动触发，把优秀工作流固化成编码过程中的强约束。
2. 这意味着团队可以把“什么时候必须先计划”“什么时候必须写测试”“什么时候要检查风险”这些动作写成可执行规范，而不是只靠工程师记忆或口头约定。
3. 它的问题在于流程强制性较高，治理收益明显，但 token 成本和执行摩擦也会同步上升。

### 2.4 Everything Claude Code
1. ECC 更像一套围绕 Claude Code 沉淀下来的 agents、skills、hooks、rules 大全，强调拿来即用和按需组装。
2. 它的优势是经过真实生产环境验证，覆盖面广，方便团队快速搭建一套可工作的执行资产库。
3. 它的挑战在于内容很多，如果不做筛选就全量安装，很容易把上下文、token 和工作流复杂度一起拉高。

## 3. 四类工具的适用场景
1. 如果团队正在启动一个新项目，而且希望一开始就建立强规格、强验收、强拆解链路，Spec-Kit 会更合适。
2. 如果团队已有系统，希望在现有代码库上对功能变更做可追溯、可归档的轻量治理，OpenSpec 更贴近现实。
3. 如果团队已经知道“正确流程是什么”，只是难以让 AI 稳定遵守，那么 Superpowers 这类自动触发 skills 的方案会更有效。
4. 如果团队当前最缺的是一整套经验证的 agents、skills、hooks 与规则资产，而不是流程理论，那么 ECC 这类“工具大全”更有现实价值。

## 4. 组合使用的实践启发
1. 文章隐含的关键启发是：决策层与执行层应该解耦设计、组合落地。
2. 一个常见搭配是先用 Spec-Kit 或 OpenSpec 产出清晰的需求锚点和任务清单，再把实现阶段交给 Claude Code、Cursor、CodeX 这类执行环境，并通过 Superpowers 或 ECC 补足技能、规则与 hooks。
3. 这种组合比单纯把一大段 prompt 丢给模型更稳，因为它把“规格”“变更”“执行约束”“工具资产”拆到了不同层面管理。

## 5. 对提醒应用案例的理解
1. 作者用 macOS 提醒应用开发作为示例，本质上是在展示一个最小可验证闭环：先定需求，再定变更，再走实施，而不是一上来就让 AI 直接生成整套代码。
2. 这个案例说明，哪怕是一个功能不算复杂的小应用，也可以通过规范驱动的方式减少返工，提高过程透明度。
3. 真正值得借鉴的不是示例本身，而是“先让 AI 明白边界和约束，再让它进入执行”的顺序。

## 6. 对实际落地的启示
1. 如果团队当前 AI Coding 最大的问题是“写出来的东西很快，但不稳”，优先补的通常不是更长的 prompt，而是规格与流程层面的锚点。
2. 如果团队已经有不少规范，却总是散落在 review 评论、群聊和资深工程师经验中，那么执行层工具的真正价值是把这些隐性规则转成 skills、hooks 和 rule files。
3. 工具选型时应该优先明确团队当前的瓶颈属于“决策混乱”“变更失控”“执行不稳定”还是“资产缺失”，然后再决定先引入哪一类工具。
