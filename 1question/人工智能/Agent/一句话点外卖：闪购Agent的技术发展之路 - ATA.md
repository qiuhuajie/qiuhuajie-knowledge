# 一句话点外卖：闪购Agent的技术发展之路
> 作者：悟石、萦之等闪购Agent研发同学

## 一、背景

近年来，大模型技术正加速从底层基础能力向规模化应用纵深演进，其价值重心已由"提升内部效率"转向"重塑面向消费者的交互与服务范式"。AI不再仅是辅助工具，而是作为新一代人机接口，深度融入用户决策与交易闭环。

在此背景下，闪购技术中心于2025年3月启动对"AI应用"命题的体系化探索，聚焦五大核心方向：

- 消费者Agent
- 商家Agent
- 物流智能化
- AI服务助手
- 产品研发AI化

旨在构建以智能体（Agent）为核心的下一代本地生活服务基础设施。其中，在消费者侧，提出双轨并行的演进路径：

1. **现有交互形态升级**：通过生成式AI重构端内搜索、推荐与导购链路，提升服务精准度
2. **未来交互变革**：探索"一句话点单"新范式，将"商品理解、意图识别、交易下单与履约追踪"集成于单一智能体中

后续通过淘宝闪购APP（可视化AI搜）、天猫精灵×闪购（纯语音ASR）、OPPO小布助手（三方生态合作）等多场景验证，逐步沉淀出C端Agent架构。

至2025年11月，"千问C × 闪购Agent"项目中，双方达成共识：提供AI Native的全链路闭环体验，支持用户通过自然语言完成从导购、下单、支付到履约状态查询的完整流程，目标将传统3-5分钟的点单耗时压缩至30秒以内。

千问C春节红包活动中，整套Agent系统成功应对了超10倍于预估峰值的并发压力，充分验证了架构的高可用性与稳定性。

## 二、探索历程
### 1. 闪购端内AI智能体项目（2023年）

早在2023年，团队就开始了LLM Native产品的探索，推出"问小饿"产品：

- 识别用户意图
- 进行店品推荐
- 跳转到相应导购页面

定位：搜推之间的新型导购模式创新

![问小饿产品|371](https://oss-ata.alibaba.com/article/2026/03/80f408b9-f73a-4d12-8c72-ef0edf2a4cf3)

### 2. 天猫精灵×闪购智能体项目（2025年4月）

从"搜推导购"扩展为"导购+交易闭环"的全链路智能体服务，品类从"奶茶果汁"、"汉堡薯条"扩展到整个餐饮行业。

**下单流程：**

1. 用户唤醒天猫精灵，说出外卖需求
2. 账号绑定状态验证（首次）
3. 提出外卖意图（品类/店铺/商品）
4. 逐步引导用户选择菜品
5. 确认订单（菜品、数量、总价、优惠、地址）
6. 支付校验
7. 订单跟踪

2025年11月18日，千问产品团队发起合作对齐会。11月23日，闪购团队首批20余人入驻千问闭关区，全面投入项目攻坚。

## 三、闪购Agent技术架构
### 1. 整体架构

采用**意图分发 + LLM规划**双轮驱动模式：

- **简单意图** → 快思考的AI搜索模式（毫秒级响应）
- **复杂意图** → 慢思考的Planner进行规划和拆解
- **Memory Service** → Session级别的短期记忆

**技术栈：**

- Spring Boot 3.2.5 + Spring AI
- 支持多渠道：千问App（LUI+GUI）、天猫精灵（ASR）、淘宝闪购
- Fast/Auto双模式 + Memory模块三层架构
- MCP Server提供7+原子工具（主搜/地址/优惠/规格/加减购/提单/AI付）
- 模型：/8b/4b + 外卖领域微调SFT模型
- 流式响应：Reactor Flux逐字符推送

![Agent技术架构|869](https://oss-ata.alibaba.com/article/2026/03/60e4ab85-7768-462d-be24-a13bd7a744b5)

### 2. 意图识别设计

构建全链路LUI能力，覆盖29个关键用户意图：

- 初始搜索意图（"帮我点霸王茶姬"）
- 深度交互（修改规格、加入购物车、提交订单、物流售后）

**核心挑战：**

- 语义开放、表达多样、意图交织
- 高度依赖对话历史与上下文状态

![意图识别|800](https://oss-ata.alibaba.com/article/2026/03/8a7ae412-8a3b-4f29-aac2-d2c05fb41894.png)

**数据合成流程：**

```java
合成用户Query → 合成Agent Response → 规则及LLM质量校验 → 意图推理及投票
```
### 3. 意图驱动设计

采用**责任链模式 + 优先级排序**机制：

| 梯队 | 优先级 | 覆盖场景 |
|:---:|:---:|:---|
| 高 | 1-10 | 搜索、退出、取消等紧急意图 |
| 中 | 15-35 | 订单、购物车、地址等常规业务 |
| 低 | 40-999 | 确保所有请求都有响应 |

**核心优势：**

- 动态扩展：新增处理器只需 @Component 注解
- 优先级可控：灵活控制处理顺序
- 责任链协作：支持多处理器流转
- 循环防护：MAX_CHAIN_COUNT=5 防止死循环

![意图驱动设计|800](https://oss-ata.alibaba.com/article/2026/03/1fb87f05-b55b-45a6-b420-22ed820a5b59)

### 4. 交互形态

提供**LUI + GUI**结合的交互模式：

- LUI和卡片GUI：通过A2A方式调用闪购Agent服务
- 闪购外卖二级页GUI：直接通过mtop接口调用

![Agent新交互形态|800](https://oss-ata.alibaba.com/article/2026/03/1a2c8c17-d30c-47e6-aaeb-8ba4513b7180)

## 四、Memory Service

提升多轮交互体验，使Agent更智能、更懂用户。

### 核心能力
1. **上下文感知**：精准识别用户当前意图与历史行为关联（如："换一杯"）
2. **意图延续**：支持跨轮次业务逻辑串联（如：地址修改→搜索商品→订单提交）
3. **领域知识沉淀**：动态维护商品/活动/用户偏好等实时状态数据

### 实现方案

基于Redis的短期记忆：

- 命名空间隔离：sessionId维度
- 数据结构：Hash + List + String
- 压缩策略：模型Memory用对话摘要，业务上下文Memory用LZ4压缩

### 模型Memory历史压缩

将对话数据（文本+卡片）转为模型更容易理解的自然语言，减少token消耗。

![Memory历史压缩|800](https://oss-ata.alibaba.com/article/2026/03/e139af99-c7b7-47b6-97f7-d0ada52b37a7)

## 五、快思考-AI搜索

对传统搜索进行重构，适配千问场景的复杂Query。

### 架构图

用户一次自然语言搜索 → 与LLM对话上百次 → 返回店品卡片

**核心流程：**

1. 生成式改写：自然语言 → 机器可理解的搜索语言
2. 闪购主搜召回候选集
3. 生成式相关性二次筛选与语义对齐
4. SKU选择与组货
5. 结果组装：店品卡片

![AI搜索架构|800](https://oss-ata.alibaba.com/article/2026/03/c4062571-4f7c-4e08-a902-c0d00f75c022.png)

### 1. Query理解与改写

**挑战：**

- 多轮意图理解
- 从用户意图到商品供给的匹配

**技术方案：**

- 数据：零售、餐饮数据覆盖
- 改写策略：单品召回、店品组合召回、多品召回、店铺召回
- 模型：微调

![Query理解与改写|800](https://oss-ata.alibaba.com/article/2026/03/30550585-e86a-4404-8761-54faa7781abf.png)

### 2. 相关性

商品与用户需求匹配的"最后一道关卡"。

**模型方案：**

```java
Qwen3-235B数据标注 + 专业标注 → → 单token判别
```

**"全、精、准"三角度判别体系：**

| 维度 | 策略 |
|:---:|:---|
| 全 | 横向：餐饮/零售/医药全场景；纵向：商品全类型 |
| 精 | 分布感知重采样、歧义样本深挖、典型bad case定向补充 |
| 准 | 专业标注+大模型校验双决策、prompt迭代、AB实验验证 |
![相关性体系|800](https://oss-ata.alibaba.com/article/2026/03/46005339-31d4-4838-a6f2-aaf9008e2f6c.png)

### 3. SKU检索和识别

用户通过语音/自然语言一次性表达完整诉求，系统直接返回满足规格属性的店品供给。

**技术方案：**

- 检索字段增加规格属性信息
- 新增sku规格属性相关性分层
- 规格属性补全：召回品→品-属性映射表→检索所有规格属性
- 效率排序：去重，保留品下最大相关性sku

![SKU检索和识别|800](https://oss-ata.alibaba.com/article/2026/03/6faa044f-fa94-41fe-b860-0fe9553f5969.png)

**SKU选择策略：**

- **餐饮类**：提取温度、甜度、杯型、配料等意图，与SKU可选项匹配
- **零售类**：解析包装规格（如"6罐装"），结合量词类型判定购买份数

![SKU选择|800](https://oss-ata.alibaba.com/article/2026/03/6c9a91a8-6728-446f-bfe3-7421f59c4d27.png)

### 4. 多商品组合

用户在同一家店同时购买咖啡+面包等多品需求。

**技术方案：**

- 大模型改写词list（长度5）
- 多路召回：改写召回 + 店品全文检索
- 全文检索店命中置顶

![多商品组合架构|800](https://oss-ata.alibaba.com/article/2026/03/af907fad-1311-4a37-bb08-f5a53230b7a.png)

**组品策略：**

- 已有套餐优先
- 自定义组货备选
- 未指定分量默认单人餐

## 六、慢思考-Planner模式

解决复杂Query（如"点杯奶茶送到家里"需先解析地址再搜索）。

![Planner架构|800](https://oss-ata.alibaba.com/article/2026/03/e9b6eeb3-3347-4798-981b-7e8dd0df27bb.png)

### 1. 逻辑层划分
```java
┌─────────────────────────────────────┐
│   Orchestrator 层（编排层）          │  协调ReAct循环，流程编排和状态流转
├─────────────────────────────────────┤
│   Engine 层（引擎层）                │  LLM规划/调用/结果处理/工具执行/异常处理
├─────────────────────────────────────┤
│   Context 层（上下文层）             │  状态管理，数据转换，连接LLM与业务世界
└─────────────────────────────────────┘
```
### 2. 类ReAct执行框架

| 阶段 | 组件 | 核心动作 |
|:---:|:---:|:---|
| Reason | PlanningEngine.plan() | LLM分析状态，决定调用什么工具 |
| Act | PlanningEngine.executeTools() | 执行MCP工具，获取外部信息 |
| Observe | ContextManager.updateState() | 将结果加入对话历史 |
![ReAct执行框架|800](https://oss-ata.alibaba.com/article/2026/03/ca88045b-098a-413c-a447-e3efd7d8d0f0.png)

### 3. Planner模型设计

**三种特性：**

1. **灵活性**：面对不同情况完成任务的能力
2. **决策者**：Planner是决策者，Tool是执行者，决策者只需感知重点信息
3. **训推一体**：框架可用于线上使用，也可参与训练，保证离在线一致

![Planner模型|800](https://oss-ata.alibaba.com/article/2026/03/bec7f1c3-c227-4357-9a98-cbe53f0f6409.png)

**简化设计：**

- 业务逻辑放MCP Tool侧，Planner仅保留主要模型规划
- 重点信息传递：formatContent
- 随机性：固定话术 + 随机化部分

![简化Planner设计|800](https://oss-ata.alibaba.com/article/2026/03/d4a58658-9dfc-42f8-83ab-8658581a9f1d.png)

## 七、大规模万卡推理集群

Whale平台提供算力保障：

- 峰值：**1W张显卡**（等效H20）
- 相关性模型：**25W QPS**
- 成果：零宕机、低抖动

### 三大核心技术
1. **PD分离**：预填充和解码阶段解耦，降低首字延迟（TTFT）
2. **MTP投机采样**：小模型预测+大模型并行校验，提升Token生成速度
3. **FlexLB负载均衡**：状态感知与精确分发，Cache-Aware路由，高KV Cache命中率

## 八、总结与展望
### 总结

技术层面：

- "快思考+慢思考"双模架构
- Query理解、相关性判断、SKU识别、多品组合端到端优化
- Planner框架解决复杂意图动态规划
- Memory Service解决多轮对话上下文延续
- 春节红包活动：超10倍并发压力验证

业务层面：

- 多场景落地：端内AI搜索、千问C点单、天猫精灵语音
- 覆盖行业：餐饮、零售、医药

### 展望
1. **技术架构升级**：多Agent中枢协作、插件化意图处理器
2. **体验优化**：短期记忆→长期记忆、流式预加载降低延迟
3. **场景拓展**：从"点单"→"决策辅助"、跨平台Agent协作
