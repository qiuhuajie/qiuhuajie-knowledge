## 2. 核心摘要
* 核心摘要如下。
    > [!abstract] 核心摘要
    > 这篇文章从工程视角系统梳理了 AI Agent 的设计方法：先回答什么是 Agent，再拆出 Prompt、推理、工具、记忆、执行框架和协作进化等关键能力，最后落到研发范式与一个资损分析业务案例。
    > 文章的核心判断是：Agent 不是“会聊天的模型”，而是“以 LLM 为认知核心、以工具、记忆、工作流和评估体系为执行外壳”的完整系统。
    > 真正可落地的 Agent，通常不是纯粹的自主智能体，而是“确定性外壳 + 智能内核 + 知识管理 + 评估闭环”的混合架构。

## 3. 一句话结论
* 设计 AI Agent 系统，本质上是在构建一个“让 LLM 在可控边界内自主完成目标”的软件系统。

## 4. 全文主线
1. 先区分 LLM 和 Agent 的差异。
2. 再拆解 Agent 的核心能力模块。
3. 然后比较几种常见的执行范式与研发方式。
4. 最后通过资损分析案例说明如何在业务里真正落地。

## 5. 文章最值得记住的判断
* Agent 研发的重心，不是“把 Prompt 写得更花”，而是把目标、工具、记忆、流程和评估一起设计出来。
* 真正稳定的生产系统，往往不是纯自主 Agent，而是混合架构。
* 智能是加分项，稳定、可控、可追踪、可验证才是底盘。
* AI 消除了很多次要困难，但并没有替代领域建模和业务判断，反而对人的要求更高。

# 二、正文整理
## 1. 什么是 AI Agent
1. Agent 可以理解为：给 LLM 这个“大脑”装上感官、手脚、记忆和行动能力，让它从单次问答升级为可执行的软件系统。
    ![[IMG-20260407202543934.png|500]]
### 1.1 软件范式的演进
1. 作者借用了 Andrej Karpathy 的 Software 1.0 / 2.0 / 3.0 框架来解释软件范式变化。
    ![[IMG-20260407202544058.png|500]]
2. 这里的核心变化不是“代码会不会消失”，而是开发重点从“如何实现”转向“如何设定目标、上下文和约束”。
3. 下表概括了三种软件范式的差异。

| 维度 | Software 1.0 | Software 2.0 | Software 3.0 |
| --- | --- | --- | --- |
| 核心范式 | 编写逻辑 | 定义目标并训练模型 | 描述意图并管理上下文 |
| 编程内容 | 写代码 | 准备数据与训练模型 | 写 Prompt 与设计系统行为 |
| 执行核心 | CPU | GPU | LLM |
| 典型问题 | 我该如何实现功能 | 我需要什么数据训练模型 | 我如何设定目标让 Agent 自主完成 |

### 1.2 从 LLM 到 Agent
1. 文中的定义很直接：`AI Agent = LLM（Reasoning）+ Planning + Memory + Tool Use`。
    ![[IMG-20260407202544191.png|500]]
2. 这一定义背后的关键变化是，LLM 只是认知内核，而 Agent 已经是能够持续感知、规划、调用工具并执行任务的系统。
3. 下表可以快速区分 LLM 和 AI Agent。

| 维度 | LLM | AI Agent |
| --- | --- | --- |
| 本质 | 推理内核 | 可执行的软件系统 |
| 角色 | 被动回答问题 | 主动规划并执行任务 |
| 工作方式 | 单次静态交互 | 多步动态工作流 |
| 与环境交互 | 基本只限文本 | 可调用工具、改变环境 |
| 状态 | 近似无状态 | 有短期和长期记忆 |

### 1.3 Agent 研发与传统研发的区别
1. 作者进一步对比了传统软件研发与 Agent 研发的差异。
    ![[IMG-20260407202544314.png|500]]
2. 这里最重要的变化，不是技术栈换了什么，而是研发目标从“实现逻辑”转向“在约束下达成目标”。
3. 下表总结了两种研发方式的差异。

| 维度   | 传统软件研发               | Agent 研发                    |
| ---- | -------------------- | --------------------------- |
| 核心目标 | 实现功能与逻辑              | 让系统在目标约束下自主完成任务             |
| 思维模型 | 命令式                  | 目标导向式                       |
| 主要产物 | 代码、接口、程序             | Agent 架构、Prompt、工具、评估集      |
| 开发流程 | 需求 -> 设计 -> 编码 -> 测试 | 目标 -> 原型 -> 评估 -> 调优 -> 再评估 |
| 调试方式 | 看堆栈和变量               | 看推理链、工具调用和状态流               |
| 结果评判 | 确定性正确                | 概率性可靠                       |

4. 作者强调的重点是：Agent 研发的团队协作基础，不再只是接口文档，而是共同维护的评估集和对“什么是好结果”的共识。

## 2. Agent 的基础能力模型
1. 一套普通 Agent 至少要有四类扩展能力：
    * `Planning`：把宏观目标拆成可执行子任务。
    * `Memory`：突破上下文限制，管理知识和历史状态。
    * `Tools`：让 Agent 接触外部世界。
    * `Action`：把决策真正执行出去。

### 2.1 Prompt Engineering：智能体基石
1. Prompt 不是聊天措辞，而是 Agent 的“底层协议”。
2. 关键技术点包括：
    * `角色设定`：为模型建立稳定的专业身份和行为边界。
    * `Zero-shot / Few-shot`：通过任务说明或示例让模型快速进入目标模式。
    * `输出格式化`：强约束 JSON / XML / Markdown 等结构化输出。
    * `工具使用提示模板`：明确告诉模型有哪些工具、什么时候用、参数怎么传。
    * `自我一致性 / 自我优化`：多次生成、投票，或基于初稿继续反思优化。
3. 典型例子包括：
    * 内容审核 Agent 通过角色 Prompt 进入合规审计视角。
    * 商品标题生成 Agent 通过 few-shot 学会固定风格。
    * 属性抽取 Agent 通过 JSON Schema 输出可解析结果。
4. 工程关注点主要有两类：
    * 输出结构不稳定：要加鲁棒解析、Schema 校验与重试。
    * 指令依从性衰退：要做上下文管理、关键指令重申。

### 2.2 Reasoning：如何思考
1. 这一层要解决的是：别让模型直接“拍脑袋”，而是让它进行慢思考和可解释推理。
2. 关键技术点包括：
    * `CoT`：链式思考，适合多步计算和规则判断。
    * `ToT`：树式思考，适合多分支方案探索和规划问题。
    * `GoT`：图式思考，适合多视角融合和迭代优化。
    * `Self-Reflection`：先产出初稿，再让模型扮演批评者修正结果。
3. 典型例子包括：
    * 复杂订单价格计算适合 CoT。
    * 履约路径规划适合 ToT。
    * 冷启动营销策略适合 GoT。
    * 智能客服回答优化适合 Self-Reflection。
4. 工程关注点主要是：
    * 推理错误会级联放大。
    * 复杂推理会显著增加延迟和成本。
5. 更稳妥的工程手段包括：
    * 用确定性工具监督关键中间步骤。
    * 只对高难任务激活高成本推理。
    * 必要时用蒸馏或分层路由平衡性能和成本。

### 2.3 Tool Use：如何行动
1. 这一层让 LLM 从“文本生成器”变成“能操作世界的执行者”。
2. 关键技术点包括：
    * `Function Calling`：按 Schema 调用外部函数或 API。
    * `MCP`：统一 Agent 与外部系统、数据源、工具的标准协议。
    * `Claude Skills`：把领域知识、工具和行为规范打包成模块化能力包。
    * `Code Interpreter / Sandbox`：把模型写出的代码送进沙箱执行，用于分析、计算和可视化。
3. 典型例子包括：
    * 智能客服退换货通过函数调用接入售后系统。
    * 商品信息管理 Agent 通过 MCP 聚合 MySQL、OSS、Redis、ES 等异构数据。
    * 电商资损防控 Skill 把规则库、Prompt 模板和检查脚本打包。
    * 商家经营分析 Agent 通过 Python 沙箱读取 CSV 并出图。
4. 工程关注点主要是：
    * 工具调用幻觉与参数错误：要做前置参数校验和错误反馈闭环。
    * 安全与权限越界：要遵守最小权限原则，高危动作必须加人工审批。

### 2.4 Memory：长期记忆与上下文
1. 这一层负责让 Agent 不只“知道”，还“记得”。
2. 关键技术点包括：
    * `RAG`：把外部知识库变成可检索的长期记忆。
    * `对话上下文管理`：在多轮会话中维护关键状态，而不是无脑堆历史。
    * `反思与经验记忆`：把任务中的失败教训和成功经验沉淀下来，用于未来相似任务。
3. 典型例子包括：
    * 商家规则咨询适合 RAG。
    * 多轮导购适合实体记忆。
    * 活动配置经验积累适合反思记忆。
4. 工程关注点主要是：
    * 检索内容质量与相关性问题。
    * 上下文窗口溢出和噪音干扰。
5. 更关键的治理策略包括：
    * 知识要有版本与生命周期管理。
    * 检索结果要重排序和过滤。
    * 长文本和长历史要做摘要与压缩。

### 2.5 Execution Frameworks：工作流编排
1. 执行框架决定 Agent 是“敏捷响应”还是“深思熟虑”。
2. 关键执行范式包括：
    * `ReAct`：`Thought -> Action -> Observation` 的密集循环，适合动态环境。
    * `Plan-and-Execute`：先出完整计划，再逐步执行，适合结构化长流程。
    * `Reflexion`：在执行中显式加入反思与重试，提高迭代质量。
3. 典型例子包括：
    * 全网比价更适合 ReAct。
    * 新商家入驻更适合 Plan-and-Execute。
    * 营销人群圈选优化适合 Reflexion。
4. 工程关注点主要是：
    * 任务执行可能死循环或发散。
    * 多次模型调用后，问题定位和全链路追踪会很困难。
5. 工程上通常需要：
    * 最大步数和超时限制。
    * 环路检测。
    * 模型 I/O、工具调用和状态变更的全链路追踪。

### 2.6 Collaboration & Learning：协同与进化
1. 这一层讨论的是如何突破单 Agent 能力边界。
2. 关键方向包括：
    * `Multi-Agent`：多个独立 Agent 按角色协作。
    * `Agent RL`：Agent 在环境里通过试错不断学习最优策略。
3. 典型例子包括：
    * 全链路故障定位适合多 Agent 协作，每个 Agent 专注不同层面。
    * 个性化推荐策略优化适合用强化学习不断迭代。
4. 工程关注点主要是：
    * 多 Agent 沟通低效、冲突、死锁。
    * 奖励信号设计不合理会导致 Reward Hacking。
5. 对应治理思路包括：
    * 标准化通信协议和 SOP。
    * 仲裁机制。
    * 多维综合奖励体系。
    * 必要时引入人类反馈校准奖励模型。

### 3.2 Agent 设计范式
1. 作者把工程里常见的设计范式总结成三类。

#### 范式一：最小可用范式
1. 这种范式的本质是：`自然语言入口 + RAG + 单次工具调用`。
2. 它适合“问问 / 查查 / 解释一下”这类轻交互、弱流程场景。
3. 下图对应最小可用范式。
    ![[IMG-20260407202545021.png|500]]
4. 核心特征包括：
    * 单轮触发。
    * 线性流程。
    * 基本无长期状态。
5. 优点包括：
    * 响应快。
    * 成本低。
    * 实现与调试简单。
6. 缺点包括：
    * 缺乏多步规划能力。
    * 工具使用受限。
    * 上下文容易遗忘。

#### 范式二：工作流式
1. 工作流式的核心是：由开发者预先定义任务执行路径，而不是让模型在运行时自由发挥。
2. 常见的三种形态如下。
3. 链式工作流如下。

4. 路由工作流如下。
    ![[IMG-20260407202545348.png|500]]
5. 并行工作流如下。
    ![[IMG-20260407202545475.png|500]]
6. 核心特征包括：
    * 执行路径预定义。
    * 任务拆解结构化。
    * 控制流优先于自由探索。
    * 节点之间强调确定性。
7. 优点包括：
    * 可预测、可审计、可靠性高。
    * 好监控、好调试、好回放。
    * 并行能力强，适合 I/O 密集型任务。
8. 缺点包括：
    * 灵活性受限。
    * 设计和维护成本高。
    * 路由分类、上下文隔离和结果聚合都可能成为瓶颈。

#### 范式三：动态规划类
1. 动态规划式的核心是：让 Agent 在运行时根据环境反馈动态决定执行路径。
2. 下图对应动态规划类范式。
    ![[IMG-20260407202545611.png|500]]
3. 核心特征包括：
    * 运行时动态决策。
    * 推理驱动行动。
    * 高自治。
    * 工具调用顺序灵活。
4. 优点包括：
    * 适应未知环境的能力强。
    * 支持探索性任务。
    * 过程更透明，适合人机协同。
5. 缺点包括：
    * 稳定性和可靠性更难保证。
    * 极度依赖模型能力。
    * 延迟高、成本不可预测、安全风险更大。

### 3.3 Agent 开发方式
1. 从工程实现看，作者把常见开发方式分成三类。
2. 下表概括了三种开发方式的差异。

| 类型 | 特点 | 示例 |
| --- | --- | --- |
| 纯 Prompt | 只依赖模型能力，不接工具、不做状态和流程编排 | 直接调用 LLM，自定义 Prompt |
| 拖拽 / 低代码 | 可视化编排节点，具备基础流程控制和集成能力 | Dify、Coze、idealab |
| 纯代码型 | Prompt、工具、记忆、工作流、评估都用代码管理 | LangChain、Spring AI、AgentScope 等 |

## 4. 案例：AI 需求资损分析
* 这一部分是全文最有业务落地感的章节，也是作者真正把 Agent 方法论放到具体业务中的地方。

### 4.1 背景
1. 文章把传统资损防控的问题总结成三类：
    1. 资源投入与实际风险分布不匹配。
    2. 专家经验难以标准化、复制和传承。
    3. 人工评估流程是线性串行的，吞吐跟不上需求增长。

### 4.2 整体方案
1. 这个方案的定位不是一个“离线问答机器人”，而是嵌入研发 pipeline 的 AI 资损分析能力。
2. 它覆盖的环节包括：
    * 需求评审。
    * 技术评审。
    * 开发阶段。
    * 测试验证。
    * 发布阶段。
3. 下图概括了 AI 需求资损分析整体方案。
    ![[IMG-20260407202545728.png|500]]
4. 作者把整体方案拆成三个运行面：
    * `运行态`：面向当前需求，统一调度领域 Agent，为业务线提供接入和灰度能力。
    * `归档与保鲜`：面向长期演进，把结论、脚本、知识和效果指标沉淀下来并持续更新。
    * `研发态`：通过数据集、评估 Agent 和知识回流，实现分析能力持续迭代。

### 4.3 流程设计
1. 文章先从“人工如何分析需求资损”出发，拆出五个关键步骤：
    1. 判断需求是否涉及资损。
    2. 提炼需求改动点和技术实现点。
    3. 丰富业务语义和术语背景。
    4. 召回相似资损场景。
    5. 生成最终的资损布控场景分析。

4. 作者总结了这种设计的四个关键价值：
    * `多源输入标准化`：同时看 PRD、技术方案、代码变更，避免只看单一材料造成误判。
    * `需求点 / 实现点抽象`：先把长文档抽成结构化要素，再推理，降低漂移。
    * `变更影响面分析`：从局部代码变化扩展到完整资金链路。
    * `资金风险描述语言`：统一“业务预期 -> 链路特征 -> 校验策略 -> 校验点 -> 脚本”这条链路。

### 4.4 知识体系
1. 这一节的核心观点非常重要：资金安全不是“把文档喂给模型”就能解决的问题，而是强依赖上下文工程。
2. 作者先引用了 MSRA 对查询任务难度的分层。

    * `Level 3`：可解释推理查询，需要领域规则。

    * `链路落点能力`：让风险从抽象描述落到系统、接口、表字段和口径。

## 5. 个人实践体会
1. 这部分非常像作者真正踩过坑之后的总结，可以压成 8 条最值得记住的经验。

### 5.1 从最小可用范式开始
* 不要一上来就做全功能复杂系统。先用最简单的 `Prompt + LLM` 验证核心业务价值，再逐步引入 ReAct、Planning、多 Agent 等复杂机制。

### 5.2 混合架构是生产常态
1. 真正落地时，最常见的不是纯自主 Agent，而是混合架构。
2. 最常见的组合是：`工作流外壳 + 智能内核 + 知识管理`。
3. 这种组合能同时保留灵活性和可控性。

### 5.3 智能是奢侈品，稳定是必需品
1. 比起“偶尔惊艳”，业务更在意三件事：
    * 能否稳定完成任务。
    * 错误是否可预测。
    * 出问题后有没有兜底。

### 5.4 AI 解决的是次要困难，不是根本困难
1. AI 可以加速文本处理、内容生成和工具操作，但解决不了这些根本问题：
    * 到底要解决什么业务问题。
    * 什么才是好结果。
    * 领域模型该怎么建。

### 5.5 Agent 参与者都要深度理解业务
* 无论是架构设计还是 Prompt 工程，真正决定上限的都是业务理解深度。

### 5.6 领域知识壁垒比想象中高
1. 真正有价值的知识不仅包括文档，还包括：
    * 专家经验。
    * 历史案例。
    * 边界情况。
    * 平台长期沉淀。
2. 知识质量、持续更新和版本管理，比“有没有知识库”更重要。

### 5.7 没有评估，就没有迭代
1. “感觉效果还不错”不等于可用。必须建立：
    * 离线评估。
    * 在线评估。
    * 用户反馈。
    * 标注回流。
2. 只有这样，才能形成“发现问题 -> 产生新样本 -> 优化 -> 再验证”的飞轮。

### 5.8 Agent 会逐渐成为团队的一部分
1. 在规则清晰、边界明确的领域内，Agent 的稳定性可能会逐渐超过人。
2. 作者给出的视角很有意思：未来可以把 Agent 当成“团队成员”来培养与管理，包括明确职责、持续培训、知识更新和协作机制设计。

# 三、适合回看的结论
* Agent 不是一个 Prompt，而是一整套围绕目标、自主性和约束设计的系统。
* 设计 Agent 时，Prompt、推理、工具、记忆、执行框架和评估体系同等重要。
* 生产环境里，混合架构通常比纯自主 Agent 更现实。
* 领域知识和评估体系决定了 Agent 的能力上限。
* 真正的落地难点，从来不只是模型能力，而是业务建模、知识管理、流程编排和产品化。
