# 一、Pod 在网络上的问题
1. 在kubernetes中，pod是应用程序的载体，我们可以通过pod的ip来访问应用程序
2. 虽然每个Pod都会分配一个单独的 Pod IP，然而却存在如下两问题：
    - ==问题1：====**Pod IP 会随着Pod的重建产生变化**==

        ```Plain
         [root@master ~]# kubectl get deploy -n dev
         NAME        READY   UP-TO-DATE   AVAILABLE   AGE
         nginx-app   3/3     3            3           6m43s
         # 当前有3个 pod
         [root@master ~]# kubectl get pods -n dev -o wide
         NAME                        READY   STATUS    RESTARTS   AGE   IP               NODE     NOMINATED NODE
         nginx-app-d6ff45774-76kpj   1/1     Running   0          8m    10.244.196.144   node01   <none>
         nginx-app-d6ff45774-dnz8q   1/1     Running   0          7s    10.244.196.145   node01   <none>
         nginx-app-d6ff45774-w4zdt   1/1     Running   0          8m    10.244.196.143   node01   <none>
         # 删除其中一个
         [root@master ~]# kubectl delete pods nginx-app-d6ff45774-czr9f -n dev
         pod "nginx-app-d6ff45774-czr9f" deleted
         # 由于 pod 控制器的存在，会重新拉起一个新的pod来顶替被删除的pod
         # ⭐可以看到新建的pod 的 IP 与之前删除的 IP 发生了变化
         [root@master ~]# kubectl get pods -n dev -o wide
         NAME                        READY   STATUS    RESTARTS   AGE     IP               NODE     NOMINATED NODE
         nginx-app-d6ff45774-76kpj   1/1     Running   0          7m28s   10.244.196.144   node01   <none>
         nginx-app-d6ff45774-czr9f   1/1     Running   0          22s     10.244.140.78    node02   <none>
         nginx-app-d6ff45774-w4zdt   1/1     Running   0          7m28s   10.244.196.143   node01   <none>
        ```
    - ==问题2：====**Pod IP 仅仅是集群内可见的虚拟IP，外部无法访问**==
3. 这样对于访问这个服务带来了难度，因此，kubernetes 设计了Service来解决这个问题
# 二、Service 介绍
1. Service可以看作是==**一组同类 Pod**== ==**对外的访问接口**== ==，借助Service，应用可以方便地====**实现**====**服务发现**====**和**====**负载均衡**==
    1. Service会对提供同一个服务的多个pod进行聚合，并且提供一个统一的入口地址
    2. 通过访问Service的入口地址就能访问到后面的pod服务
    3. service 与 pod 的绑定是通过标签选择器来实现的
2. Service 的本质
    1. **Service在很多情况下只是一个概念，真正起作用的其实是** ==**kube-proxy 服务进程**==
    2. 每个Node节点上都运行着一个kube-proxy服务进程
    3. 当创建Service的时候会通过api-server向etcd写入创建的service的信息
    4. 而kube-proxy会基于监听的机制发现这种Service的变动，然后它会将最新的Service信息转换成对应的访问规则

    ![[IMG-20260404031956056.png|800]]

3. 查看访问规则：

    ```Plain
     [root@node1 ~]# ipvsadm -Ln
     IP Virtual Server version 1.2.1 (size=4096)
     Prot LocalAddress:Port Scheduler Flags
       -> RemoteAddress:Port           Forward Weight ActiveConn InActConn
     TCP  10.97.97.97:80 rr
       -> 10.244.1.39:80               Masq    1      0          0
       -> 10.244.1.40:80               Masq    1      0          0
       -> 10.244.2.33:80               Masq    1      0          0
    ```
    - 说明：
        - 10.97.97.97:80 是service提供的访问入口
        - **当访问这个入口的时候，可以发现后面有三个pod的服务在等待调用，kube-proxy会**==**基于rr（轮询）的策略**==**，**==**将请求分发**==**到其中一个pod上去**
        - 这个规则会同时在集群内的所有节点上都生成，所以在任何一个节点上访问都可以

# 三、kube-proxy 目前支持三种工作模式
1. userspace 模式
    1. **userspace模式下，kube-proxy会为每一个Service创建一个监听端口**，发向Cluster IP的请求被Iptables规则重定向到kube-proxy监听的端口上，kube-proxy根据LB算法选择一个提供服务的Pod并和其建立链接，以将请求转发到Pod上
    2. 该模式下，**kube-proxy充当了一个四层负责均衡器的角色**
    3. 由于kube-proxy运行在userspace中，在进行转发处理时会增加内核和用户空间之间的数据拷贝，虽然比较稳定，但是效率比较低

    ![[IMG-20260404031956084.png|800]]

2. iptables 模式
    1. iptables模式下，kube-proxy为service后端的每个Pod创建对应的iptables规则，直接将发向Cluster IP的请求重定向到一个Pod IP
    2. 该模式下kube-proxy不承担四层负责均衡器的角色，只负责创建iptables规则
    3. 该模式的优点是较userspace模式效率更高，但不能提供灵活的LB策略，当后端Pod不可用时也无法进行重试

    ![[IMG-20260404031956198.png|800]]

3. ipvs 模式
    1. ipvs模式和iptables类似，kube-proxy监控Pod的变化并创建相应的ipvs规则。ipvs相对iptables转发效率更高
    2. 除此以外，ipvs支持更多的LB算法

        ![[IMG-20260404031956229.png|800]]

    3. 此模式必须安装ipvs内核模块，否则会降级为iptables

        ```Plain
         # 开启ipvs
         [root@k8s-master01 ~]# kubectl edit cm kube-proxy -n kube-system
         # 修改mode: "ipvs"
         [root@k8s-master01 ~]# kubectl delete pod -l k8s-app=kube-proxy -n kube-system
         [root@node1 ~]# ipvsadm -Ln
         IP Virtual Server version 1.2.1 (size=4096)
         Prot LocalAddress:Port Scheduler Flags
           -> RemoteAddress:Port           Forward Weight ActiveConn InActConn
         TCP  10.97.97.97:80 rr
           -> 10.244.1.39:80               Masq    1      0          0
           -> 10.244.1.40:80               Masq    1      0          0
           -> 10.244.2.33:80               Masq    1      0          0
        ```
