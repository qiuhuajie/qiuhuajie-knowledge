# 1. 介绍
1. EventLoopGroup 是**一组 EventLoop** 的抽象，**Netty 为了更好的利用多核 CPU 资源般会有多个 EventLoop 同时工作，每个 EventLoop 维护着一个 Selector 实例**
1. EventLoopGroup 提供 `next` 接口，**可以从组里面按照一定规则获取其中一个 EventLoop 来处理任务**
# 2. EventLoopGroup 类及其实现类
![[IMG-20260405035413814.png|Untitled 371.png]]
- NioEventLoopGroup：异步非阻塞（常用）下面的 BossEventLoopGroup 和 WorkerEventLoopGroup 就是该实现类型
- OioEventLoopGroup：同步阻塞
# 3. **在 Netty 服务器端编程中的组件构成**
1. 一般都需要提供两个 EventLoopGroup，例如：BossEventLoopGroup 和 WorkerEventLoopGroup
1. 通常一个服务端口，即一个 ServerSocketChannel 对应一个 Selector 和一个 EventLoop 线程
1. BossEventLoop 负责接收客户端的连接并将 SocketChannel 交给 WorkerEventLoopGroup 来进行 IO 处理
    
    ![[IMG-20260404031815426.png]]
    
# 4. **流程分析 ⭐**
1. BossEventLoopGroup 通常是一个单线程的 EventLoop
    
    1. 这个 EventLoop 维护着一个注册了 ServerSocketChannel 的 Selector 实例的 BossEventLoop
    
    1. 并不断轮询 Selector 将连接事件分离出来（BossEventLoopGroup 的 EventLoop 只处理连接事件）
    
    1. 通常是 `OP ACCEPT` 事件，然后将接收到的 SocketChannel 交给 WorkerEventLoopGroup
    
    ![[IMG-20260405035413810.png|Untitled 368.png]]
    
1. WorkerEventLoopGroup 会由 `next` 选择其中一个 EventLoop
    
    1. 将这个 BossEventLoopGroup 发来的 SocketChannel 注册到其维护的 Selector 上
    
    1. 并对其后续的 IO 事件进行处理
    
![[IMG-20260405035413803.png]]

# 6. 常用方法
- `public NioEventLoopGroup()`：构造方法
- `public Future<?> shutdownGracefully()`：断开连接，关闭线程。会关闭所有的child Channel。关闭之后，释放掉底层的资源
# 7. 轮询机制的实现原理浅析
1. 不妨看一段 DefaultEventExecutorChooserFactory 的源码：
    
    ![[IMG-20260405035443371.png|Untitled 4 142.png]]
    
    ```Java
    private final AtomicInteger idx = new AtomicInteger();
    private final EventExecutor[] executors;
    
    @Override
    public EventExecutor next() {
        //idx.getAndIncrement()相当于idx++，然后对任务长度取模
        return executors[idx.getAndIncrement() & executors.length - 1];
    }
    ```
    
1. 它这里还有一个判断，如果线程数不是2的N次方，则采用取模算法实现
    
    ```Java
    @Override
    public EventExecutor next() {
        return executors[Math.abs(idx.getAndIncrement() % executors.length)];
    }
    ```
