# 一、概念区分
1. I/O 模型就是用什么样的通道进行数据的发送和接收， 很大程度上决定了程序通信的性能
2. 针对以上举例，和我们实际工作中的概念进行对比
    - 顾客：通道
    - 食堂：操作系统
    - 服务员：线程
    - 饭菜：数据
    - 饭菜好了：数据就绪
    - 端菜/送菜：数据读取
3. ==**阻塞与非阻塞**==
    - 菜没好，是否原地等待 -> 数据就绪前是否等待
    - 阻塞：**数据未就绪时，读操作阻塞直到有数据；缓冲区满时，写操作也会阻塞等待**。本质上是线程挂起，不能做其他事
    - 非阻塞：**请求直接返回**，本质上线程活跃，可以处理其他事情
4. ==**同步与异步**==
    - 菜好了，谁端？->数据就绪，是操作系统送过去，还是应用程序自己读取
    - 同步：**数据就绪后，应用程序自己读取**
    - 异步：**数据就绪后，操作系统直接回调应用程序**

# 二、分类
## 1. BIO（Blocking I/O）
1. ==**同步阻塞**== I/O，Java BIO 就是传统的 java io 编程，其相关的类和接口在 `java.io`
2. 在进行I/O操作时，必须等待数据读取或写入完成后才能进行下一步操作；但数据好后，仍需要程序自己去读
3. 适应场景：
    1. 适用于==连接数目比较小且固定==的架构
    2. 这种方式对服务器资源要求比较高，并发局限于应用中
    3. JDK1.4以前的唯一选择， 但程序简单易理解
4. **自选餐线**，一个服务员同时一刻只能服务等待队列的第一个顾客，其他人都得在队伍里排队等待，必须等待前面的同学打好菜才到我们，这就是同步阻塞模型BIO

    ![[IMG-20260405035413680.png|440]]

5. BIO 缺点：
    1. 每个请求都需要创建独立的线程，因此**当并发数较大时，需要创建大量线程来处理连接**，占用大量的虚拟机栈空间
        1. 即便内存空间足够，但 cpu 核数有限的
        2. 比如 4核 处理 10000个线程，**cpu 大量时间是耗费在线程调度而不是业务逻辑处理上**，会产生极大浪费
    2. 连接建立后，**如果当前线程暂时没有数据可读，则线程就阻塞在 Read 操作上**，造成线程资源浪费
6. [[BIO 详解]] 更有助于理解 BIO 工作机制
## 2. NIO（Non-blocking I/O）
1. ==**同步非阻塞**== I/O，是一种事件驱动的I/O模型
2. NIO 相关类都被放在 `java.nio` 包及子包下， 并且对原 `java.io`包中的很多类进行改写
3. 在进行I/O操作时，不需要等待操作完成，可以进行其他操作；但数据好后，仍需要程序自己去读
4. 适应场景：
    1. 适用于==连接数目多且连接比较短==（轻操作） 的架构
    2. 比如聊天服务器， 弹幕系统， 服务器间通讯等
    3. 编程比较复杂， JDK1.4开始支持
    4. BIO 以**流**的方式处理数据，而 NIO 以**块**的方式处理数据，块 I/O 的效率比流 I/O 高很多
5. **麻辣烫餐线**，会给我们发个叫号器，我们拿到叫号器（Selector 多路复用器）后不需排队原地等待，我们可以找个地方去做其他事情，等麻辣烫准备好，我们收到呼叫之后，自行取餐，这就是同步非阻塞

    ![[IMG-20260620092307869.png|769]]

## 3. AIO（Asynchronous I/O）
1. ==**异步非阻塞**== I/O，是一种更高级别的I/O模型
2. 在进行I/O操作时，不需要等待操作完成，就可继续进行其他操作，当操作完成后会自动回调通知
3. 适应场景：
    1. 使用于连接数目多且连接比较长（重操作） 的架构
    2. 比如相册服务器， 充分调用OS参与并发操作， 编程比较复杂， JDK7开始支持
4. **包厢模式**，我们只要点好菜，坐在包厢可以自己玩，等到饭做好，服务员亲自送，无需自己取，这就是异步非阻塞I/O模型AIO
5. AIO 的缺陷：
    1. 对AIO支持最好的是Windows系统，但是很少用Windows做服务器；linux常用来做服务器，但AIO的实现不够成熟
    2. linux下AIO相比NIO性能提升并不明显；维护成本过高
