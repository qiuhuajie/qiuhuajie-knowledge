# 一、先建立指标坐标系
## 1. 应用机器指标到底在看什么
* 应用机器指标不是一堆孤立的数字，而是一套判断系统处在什么状态的坐标系。排障时真正重要的，不是会不会背 `top`、`free -h`、`vmstat`、`pidstat`，而是能不能先判断出机器现在属于算力耗尽、资源排队、内存吃紧、IO 等待、网络异常，还是线程调度失控。

## 2. 核心指标地图
* 如果只看单个数，很容易误判；把常用指标放在一起看，系统状态会清楚很多。

| 指标               | 本质上在看什么                 | 常见查看方式                      | 最常见误判             |
| ---------------- | ----------------------- | --------------------------- | ----------------- |
| CPU 利用率          | 处理器有多忙                  | `top`、`mpstat`、`pidstat -u` | 把 CPU 低误当成系统没问题   |
| Load Average     | 系统里有多少任务在跑、在等，或卡在不可中断等待 | `uptime`、`top`              | 把 Load 当成 CPU 百分比 |
| `available` 内存   | 系统还剩多少可安全继续使用的内存        | `free -h`                   | 只看 `free`         |
| Swap             | 是否已经开始拿磁盘当内存            | `free -h`、`vmstat`          | 以为少量 Swap 永远无害    |
| `wa`             | CPU 有多少时间在等 IO          | `top`、`vmstat`              | 把 `wa` 当磁盘利用率     |
| `await` / `util` | IO 时延和设备忙碌度             | `iostat`                    | 只看吞吐不看时延          |
| 重传率 / 丢包         | 网络质量是否变差                | 监控平台、`sar -n TCP,ETCP`      | 只看带宽不看质量          |
| `r` / `b` / `cs` | 调度排队、阻塞和上下文切换           | `vmstat`                    | 不看调度层，只看 CPU      |
| 磁盘空间 / inode     | 容量是否耗尽                  | `df -h`、`df -i`             | 只看 IO 性能，不看容量     |

## 3. 最容易犯的几个误判
* `Load Average` 不是 CPU 百分比，前者更像任务队列长度，后者更像处理器忙碌程度。
* `free` 不是“还能继续放心分配的内存”，判断机器内存余量更应该看 `available`。
* `wa` 高说明 CPU 在等 IO，不等于某块磁盘利用率一定已经 100%。
* CPU 低不等于机器健康，因为应用完全可能卡在锁、连接池、磁盘、网络或外部依赖等待上。

# 二、Load Average
## 1. Load 是什么
* Load 表示系统在一段时间内处于可运行状态和不可中断等待状态的任务平均数。更准确地说，Linux 主要统计 `R` 状态和 `D` 状态的任务，所以 Load 表达的不是“CPU 用了多少百分比”，而是“系统里有多少任务正在跑、等着跑，或者卡在不可中断等待上”。

## 2. 为什么很多人把 Load 理解成排队长度
* 把 CPU 核数理解成超市收银台，把任务理解成来结账的人，Load 就很像“正在结账的人 + 排队等结账的人”。这个类比很好用，因为它能帮助直观理解为什么 Load 要和 CPU 核数一起看，但也要知道它只是近似类比，因为 `D` 状态的任务不一定真是在排 CPU。

## 3. Load 1 / 5 / 15
* `Load Average` 不是瞬时值，而是不同时间窗口下做过平滑处理的平均值。`load1` 最敏感，适合看当前压力；`load15` 最平滑，更适合看系统基线和持续高压是否已经存在一段时间。

| 指标 | 含义 | 用途 |
| --- | --- | --- |
| `load1` | 过去 1 分钟平均负载 | 看当前瞬时压力 |
| `load5` | 过去 5 分钟平均负载 | 看短期趋势 |
| `load15` | 过去 15 分钟平均负载 | 看整体基线 |

* 三个值放在一起，重点是看趋势。

| 组合 | 含义 |
| --- | --- |
| `load1=5, load5=2, load15=1` | 压力正在明显上升 |
| `load1=1, load5=2, load15=5` | 压力正在回落 |
| `load1=3, load5=3, load15=3` | 系统压力稳定 |

## 4. Load 和 CPU 的关系
* Load 和 CPU 相关，但它们不是一个维度。CPU 利用率回答的是“处理器忙不忙”；Load 回答的是“系统里平均有多少任务在跑、排队，或者卡在不可中断等待上”。前者更像处理能力使用率，后者更像处理队列长度。

| 场景 | Load | CPU | 原因 |
| --- | --- | --- | --- |
| 大量计算任务 | 高 | 高 | 线程都在真实使用 CPU |
| 大量 IO 不可中断等待 | 高 | 低或不高 | 任务卡在 `D` 状态，算进 Load，但不怎么吃 CPU |
| 少量热点线程 | 低到中等 | 某几个核高，总 CPU 可能不高 | 线程少，排队不多，但局部核心很忙 |
| 系统几乎空闲 | 低 | 低 | 没什么任务在跑，也没什么在等 |

## 5. 什么等待会算进 Load
* 不是所有等待都会算进 Load。真正容易把 Load 拉高的，是可运行但还没抢到 CPU 的任务，以及不可中断等待的任务；像 `sleep()`、`Object.wait()`、`park()`、线程池空闲等待、很多用户态锁等待和连接池队列等待，通常都不属于 `D` 状态，因此未必会反映到 Load 上。

| 等待类型 | 是否通常计入 Load | 例子 |
| --- | --- | --- |
| 可运行但没拿到 CPU | 是 | Runnable 线程排队等调度 |
| 不可中断等待 `D` | 是 | 磁盘 IO、NFS、块设备等待 |
| 用户态休眠 / `park` / `wait` | 否 | `sleep()`、线程池空闲等待 |
| 用户态连接池等待 | 通常否 | Java 线程等数据库连接池名额 |

## 6. 怎么判断 Load 是否正常
1. 判断 Load 是否正常，最常用的方法就是拿它和逻辑 CPU 核数比较。
2. 逻辑核数可以先这样看：

    ```bash
    nproc
    ```

3. 常见经验判断如下：

| 比较 | 状态 | 类比 |
| --- | --- | --- |
| `Load < 核数` | 正常 | 收银台没排长队 |
| `Load ≈ 核数` | 接近满载 | 每个收银台都有人，但没明显堆积 |
| `Load > 核数` | 过载 | 已经开始排队 |
| `Load > 核数 * 2` | 严重过载 | 队伍很长，系统明显吃紧 |

4. 例如 4 核机器上，`load 0.49` 说明整体很空闲，`load 4.0` 说明已经接近满载，`load 8.0` 说明平均有一倍以上的任务在排队，系统已经明显过载。

## 7. 排障时怎么用 Load
* `Load` 高且 CPU 高，更像是计算压力、GC 压力或线程真的跑满。
* `Load` 高但 CPU 低且 `wa` 高，更像是磁盘、块设备或远程存储等待。
* `Load` 不高但 RT 很高，不能掉以轻心，因为应用可能卡在用户态锁、线程池、连接池或外部 RPC 等待上，这类场景要继续结合 [[Java 应用运行状况诊断手册]]、[[ThreadPoolExecutor|线程池]]、[[5.2 数据库连接池|数据库连接池]] 和 [[线程池打满全局 504]] 去看。

# 三、CPU 核心指标
## 1. CPU 利用率在表达什么
* CPU 利用率表达的是“处理器时间是怎么花掉的”。它回答的不是“系统里有多少任务”，而是“CPU 自己到底在忙什么”，所以它适合判断算力是否被吃满，却不能单独解释系统为什么慢。

## 2. 最常看的 CPU 字段
* `top`、`vmstat`、`mpstat` 里最核心的，就是把 CPU 时间拆到用户态、内核态、等待 IO 和空闲这几类。

| 字段 | 含义 | 常见判断 |
| --- | --- | --- |
| `us` | 用户态 CPU | 业务代码、JVM 计算、普通用户态逻辑在吃 CPU |
| `sy` | 内核态 CPU | 系统调用、内核处理、上下文切换、协议栈等开销 |
| `id` | 空闲 CPU | 越高说明 CPU 越闲 |
| `wa` | IO wait | CPU 有时间，但在等 IO 结果 |
| `st` | steal | 虚拟化场景下，被宿主机“偷走”的 CPU 时间 |

## 3. 这些字段分别说明什么
* `us` 高通常说明业务线程真的在跑计算，或者 JVM 层有比较重的逻辑在消耗 CPU。
* `sy` 高通常说明系统调用开销大、网络栈或内核处理偏重、锁竞争明显，或者上下文切换已经很多；如果它同时伴随 `cs` 很高，往往要怀疑线程过多和调度成本过大，这和 [[ThreadPoolExecutor|线程池]]、[[线程#5.2 上下文切换]] 很有关系。
* `wa` 高说明 CPU 有时间但在等 IO，所以它更像 IO 问题的信号，而不是 CPU 本身忙。
* `st` 高主要出现在虚拟化环境，表示你的虚拟机本来想用 CPU，但时间片被宿主机拿走了。

## 4. 为什么 CPU 低不一定是好事
* 总 CPU 低只说明处理器没有忙起来，不说明请求一定顺畅。应用完全可能卡在锁、连接池、数据库、磁盘、网络或外部服务等待上，所以“CPU 很低但系统很卡”反而是排障里最要警惕的一类现象。

## 5. 为什么 CPU 高也不一定等于整机过载
* 如果机器有很多核，少数热点线程完全可以把某几个核打满，但整机平均 CPU 依然不高。这时要看单核和单线程视角，而不是只看整机平均值，常用命令是 `mpstat -P ALL`、`top -Hp` 和 `pidstat -u`。

# 四、内存核心指标
## 1. 为什么不能只看 Free
* Linux 会主动把空闲内存拿去做页缓存和缓冲区，因此 `free` 很小本身并不一定危险。真正更有判断价值的是 `available`，因为它更接近“系统还剩多少可以比较安全继续拿出来给程序用的内存”。

## 2. `free -h` 里最重要的字段
* 看 `free -h` 时，重点不是背字段名，而是分清哪些内存是真正还能继续使用的，哪些只是暂时被缓存占住但依然可回收。

| 字段 | 含义 |
| --- | --- |
| `total` | 总物理内存 |
| `used` | 已使用内存 |
| `free` | 完全空闲内存 |
| `buff/cache` | 缓存和缓冲区，可回收 |
| `available` | 实际可用内存 |
| `Swap` | 磁盘交换空间 |

## 3. 怎么判断机器是不是内存紧张
* 如果 `available` 持续很低，并且 Swap 开始增长，就说明机器已经进入内存压力区间。这时影响往往不只体现在内存本身，还可能带来业务 RT 变差、GC 抖动、系统调度变慢和 IO 放大。

## 4. Swap 为什么危险
* Swap 的本质是把一部分内存页换到磁盘上，一旦频繁换入换出，原本应该是内存速度的访问就会退化成磁盘速度，所以线上机器只要看到 Swap 持续增长，通常都值得重点关注。

## 5. 机器内存问题和 JVM 内存问题不是一回事
* 系统层看到的内存，包含 JVM 堆、Metaspace、直接内存、线程栈、本地库、mmap 和页缓存等多种来源，所以“机器内存紧张”不等于“Java 堆一定满了”，反过来“堆很满”也不一定会立刻把系统内存打爆。排障时要把系统层和 JVM 层拆开看，必要时继续回到 [[Java 应用运行状况诊断手册]] 做堆、GC 和进程内存的交叉验证。

# 五、IO 核心指标
## 1. IO 指标到底在说明什么
* 很多“CPU 不高但系统很慢”的故障，本质上不是算力不够，而是线程卡在磁盘、块设备、NFS、远程存储或日志刷盘这类 IO 路径上，所以 IO 指标经常是解释“系统为什么卡住”的关键线索。

## 2. 最常看的 IO 指标
* 判断 IO 健康度时，通常要一起看等待、频率、吞吐、时延和设备忙碌度，而不是只看某一个数。

| 指标 | 含义 | 说明 |
| --- | --- | --- |
| `wa` | IO 等待占比 | 先看有没有明显等待 |
| `r/s` `w/s` | 每秒读写次数 | IO 频率高不高 |
| `rkB/s` `wkB/s` | 吞吐量 | IO 数据量大不大 |
| `await` | 平均等待时延 | 排队时间 + 设备服务时间 |
| `util` | 设备忙碌度 | 越接近 100%，设备越忙 |

## 3. `wa`、`await`、`util` 不是一回事
* `wa` 是 CPU 视角，表示 CPU 花了多少时间在等 IO。
* `await` 是请求视角，表示一次 IO 从发出去到完成平均等了多久。
* `util` 是设备视角，表示设备有多忙。
* 这三个指标经常一起出现，但不能互相替代。比如 `wa` 高说明系统在等 IO，`await` 高说明单次 IO 已经变慢，`util` 高说明设备忙碌度很高。

## 4. 除了性能，还要看容量
* 应用机器的 IO 问题不只有“盘慢”，还有“盘满”。磁盘空间满、inode 耗尽、日志目录打爆、临时文件目录没有空间，都可能让应用出现写日志失败、落盘失败、临时文件创建失败甚至请求异常，所以排障时除了 `iostat`，也要记得看 `df -h` 和 `df -i`。

## 5. IO 问题通常长什么样
* IO 问题最常见的表现是 Load 偏高、CPU 不一定高、`wa` 变高、线程可能进入 `D` 状态，同时 RT 抖动明显，尤其是在日志刷盘、本地文件、数据库文件和网络文件系统参与请求链路时更明显。

# 六、网络核心指标
## 1. 网络指标在看什么
* 网络层问题很容易伪装成应用问题。对应用来说，它看到的是超时、重试、RT 升高和吞吐下降；对机器来说，底层可能是网络抖动、丢包、重传、连接堆积或者反向代理层出现阻塞。

## 2. 最值得看的网络指标
* 网络层最有价值的指标，不只是带宽，还有包速率、重传率、丢包和连接积压，因为这些指标更能解释“为什么延迟升高但 CPU 并不高”。

| 指标 | 含义 | 典型风险 |
| --- | --- | --- |
| 带宽使用率 | 网卡流量大小 | 流量打满会造成拥塞 |
| PPS | 每秒包数 | 小包场景下容易先打满 PPS |
| 重传率 | TCP 重新发送的比例 | 网络抖动、丢包、下游不稳定 |
| 丢包数 | 包被丢弃 | 网卡、交换机、队列拥塞 |
| 连接数 / 队列积压 | 连接状态异常 | 短连接风暴、反向代理拥堵 |

## 3. 为什么重传率特别值得看
* 重传率高不一定意味着带宽已经满了，但通常意味着网络质量不好，或者下游响应非常不稳定。这类问题特别容易造成“CPU 不高但 RT 突然升高”的现象，在 [[4.1 反向代理|反向代理]] 和 [[线程池打满全局 504]] 这类链路型故障里尤其常见。

## 4. 网络问题通常会怎样传导到应用层
* 网络问题最常见的传导路径，是请求变慢导致线程占用时间变长，线程池和连接池周转变慢，再进一步放大成排队、超时和全局雪崩。所以遇到 RT 抬高、超时增加但机器 CPU 并不高时，网络层一定要纳入判断链路。

# 七、调度、线程与进程指标
## 1. `r`、`b`、`cs` 分别在说明什么
* `vmstat` 里的 `r`、`b`、`cs` 很适合用来判断系统到底是在 CPU 排队、IO 阻塞，还是线程调度开销太大，因此它们是机器层排障里很容易被低估的一组指标。

| 字段 | 含义 | 典型判断 |
| --- | --- | --- |
| `r` | 等待运行的线程数 | 持续大于核数，说明 CPU 排队严重 |
| `b` | 不可中断阻塞线程数 | 持续大于 0，说明 IO 或内核资源等待多 |
| `cs` | 每秒上下文切换数 | 高说明线程切换频繁、调度成本高 |

## 2. 线程数应该怎么看
* 线程数多本身不一定有问题，关键要看线程数是否在异常增长，以及它们大部分到底是在运行、空闲等待，还是阻塞等待。如果线程数持续上涨，同时伴随 `cs` 高、`sy` 高、RT 变差，就很值得怀疑线程池配置、锁竞争或依赖等待问题，这时要继续结合 [[ThreadPoolExecutor|线程池]]、[[5.2 数据库连接池|数据库连接池]] 和 `jstack` 一起看。

## 3. 什么时候要从机器层切到 Java 层
* 如果机器层指标并不算夸张，但吞吐已经掉得很厉害，或者 RT 很高却看不出明显的 CPU、内存、IO 异常，就要尽快转到 Java 进程内部去看线程状态、线程池、连接池和阻塞点。这时要结合 [[Java 应用运行状况诊断手册]]、[[线程池打满全局 504]] 和 `jstack` 一起判断。

# 八、把指标组合起来看
## 1. 常见组合现象
* 单个指标更像局部现象，多个指标组合起来，才更接近真正的故障类型。

| 组合现象 | 更像什么问题 |
| --- | --- |
| `Load` 高 + CPU 高 | 计算压力、GC 压力、业务并发高 |
| `Load` 高 + CPU 低 + `wa` 高 | 磁盘或远程存储等待 |
| `Load` 低 + CPU 低 + RT 高 | 用户态等待、线程池、连接池、锁、外部 RPC |
| `available` 低 + Swap 上升 | 内存压力、可能发生换页 |
| `cs` 高 + `sy` 高 | 线程太多、上下文切换重、内核开销大 |
| 重传率高 + CPU 不高 | 网络抖动或下游网络问题 |
| 磁盘空间满 / inode 满 | 日志写入失败、临时文件失败、应用行为异常 |

## 2. “忙死了”和“卡死了”怎么区分
* 很多排障效率差，不是因为不会命令，而是没有先判断系统到底是忙死了还是卡死了。这个区分一旦做对，后面的排查方向会直接收敛。

| 维度 | 忙死了 | 卡死了 |
| --- | --- | --- |
| CPU | 往往高 | 往往不高 |
| Load | 可能高 | 可能高，也可能不高 |
| 吞吐 | 下降，但通常还有处理能力 | 可能接近停滞 |
| 线程状态 | 更多在跑代码 | 更多在等锁、等连接、等 IO、等下游 |
| 常见原因 | 计算重、GC 重、线程打满 | 锁竞争、连接池耗尽、磁盘 / 网络等待、外部依赖阻塞 |
| 下一步 | `top -Hp`、热点线程、GC | `jstack`、线程池、连接池、外部依赖链路 |

## 3. 为什么“CPU 很低但系统很卡”最要警惕
* 这类场景最容易误导人，因为看起来机器“很闲”，但实际上系统可能已经被等待链路拖死了。只要请求被慢 IO、锁竞争、连接池等待、网络抖动或慢下游占住，线程就会堆积，最终表现成 RT 飙升、超时增加甚至全局 504，而 CPU 依然不高。

# 九、常用命令
## 1. 机器层常用命令
* 下面这组命令适合先做一次机器层体检，再决定是否需要进入 JVM 内部继续深挖。

    ```bash
    top -bn1 | head -5
    uptime
    nproc
    free -h
    vmstat 2 5
    mpstat -P ALL 2
    pidstat -u 2
    pidstat -r 2
    pidstat -d 2
    iostat -x 2
    df -h
    df -i
    ss -s
    ```

## 2. 从机器层到 Java 层的衔接
1. 如果系统指标已经指向 CPU 热点，先用 `top -Hp` 找热点线程，再回到 [[Java 应用运行状况诊断手册]] 里用 `jstack` 对线程栈。
2. 如果系统指标更像资源等待，优先结合 [[ThreadPoolExecutor|线程池]]、[[5.2 数据库连接池|数据库连接池]]、[[4.1 反向代理|反向代理]] 和 [[线程池打满全局 504]] 去看线程池、连接池、网关和下游链路。
3. 如果机器内存吃紧但 CPU 不高，再继续核对 JVM 堆、GC 和堆外内存，避免只盯系统层而漏掉进程内部问题。
