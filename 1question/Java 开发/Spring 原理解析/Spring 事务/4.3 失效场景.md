# 4.3 失效场景

> 💡 
> 在 Spring Boot 中，造成事务不自动回滚的场景有很多，比如以下这些：
> 1. 非 public 修饰的方法中的事务不自动回滚
> 2. 当 @Transactional 遇上 try/catch 事务不自动回滚
> 3. 调用类内部的 @Transactional 方法事务不自动回滚
> 4. 抛出检查异常时事务不自动回滚
> 5. 数据库不支持事务，事务也不会自动回滚

# 1. 非 Public 修饰的方法

1. 非 public 方法中事务不回滚的直接原因是
    1. **在非 public 方法上添加的 `@Transactional` 关键字是无效的**
    2. 也就是此方法本身是以非事务的方式运行的，所以它当然不会自动回滚事务了
2. **因为 `@Transactional` 使用的是 Spring AOP 实现的**
    1. 而 Spring AOP 是通过动态代理实现的
    2. **而 `@Transactional` 在生成代理时会判断，如果方法为非 public 修饰的方法，则不生成代理对象**，这样也就没办法自动回滚事务了
    3. 它的部分实现源码如下：

        ```java
        protected TransactionAttribute computeTransactionAttribute(Method method, Class<?> targetClass) {
           // Don't allow no-public methods as required.
           // 非 public 方法，设置为 null
           if (allowPublicMethodsOnly() && !Modifier.isPublic(method.getModifiers())) {
              return null;
           }
           // 后面代码省略....
         }
        ```


# 2. 遇上 try/catch

1. 当程序中出现了 `try/catch` 代码时，事务不会自动回滚
2. 这是因为 `@Transactional` 注解在其实现时，**需要感知到异常才会自动回滚**，而用户自行在**代码中加入了 `try/catch` 之后，`@Transactional` 就无法感知到异常了**，那么也就不能自动回滚事务了
3. 此问题的解决方案有两种：

# 3. 调用类内部的 @Transactional 方法

1. 调用类内部 `@Transactional` 的方法不自动回滚事务的原因是
    1. `@Transactional` 是基于 Spring AOP 实现的，而 Spring AOP 又是基于动态代理实现的
    2. 而**当调用类内部的方法时，不是通过代理对象完成的，而是通过 `this` 对象实现的，这样就绕过了代理对象**，从而事务就失效了

# 5. 数据库不支持事务

1. 当**在程序中添加了 `@Transactional`，相当于给调用的数据库发送了：开始事务、提交事务、回滚事务的指令**
2. 但是如果数据库本身不支持事务
    1. 比如 **MySQL 中设置了使用 `MyISAM` 引擎，因为它本身是不支持事务的**
    2. 这种情况下，即使在程序中添加了 `@Transactional` 注解，那么依然不会有事务的行为，也就不会执行事务的自动回滚了
3. 在这种情况下，**只需要设置 MySQL 的引擎为 `InnoDB` 就可以解决问题了**
    1. 因为 InnoDB 是支持事务的，当然 MySQL 5.1 之后的默认引擎就是 InnoDB

# 6. 异步线程中有事务处理

1. 如果在异步线程种有事务处理，这种是没办法用事务去保证的
2. 因为异步线程拿到的连接和主线程拿到的连接肯定不是同一个，所以只有当一个数据库同一个连接，才能做事务控制
3. 这种异步线程的情况在 Spring 事务控制里是不支持的
