# 1. Spring AOP 与 AspectJ 框架

1. Spring AOP 和 AspectJ AOP 有什么区别？
    1. Spring AOP 属于**运行时增强**，而 AspectJ 是**编译时增强**。Spring AOP 基于代理（Proxying），而 AspectJ 基于字节码操作
    2. Spring AOP 已经集成了 AspectJ ，**AspectJ 应该算的上是 Java 生态系统中最完整的 AOP 框架了**
2. 如何选用：
    1. AspectJ 相比于 Spring AOP 功能更加强大，但是 Spring AOP 相对来说更简单
    2. 如果我们的**切面比较少**，那么两者性能差异不大。但是，当**切面太多**的话，最好选择 AspectJ 它比Spring AOP 快很多

# 2. AOP 如何实现

1. 使用**代理模式**
2. 举例：

    ![[IMG-20260620231629118.png|661]]

    1. 理发店会给顾客安排理发师，但实际接待顾客的助理
    2. 理发店通过助理来为顾客提供服务
    3. 首先助理会带顾客排队、洗头，然后才由理发师来剪发，之后又由助理带顾客洗头、吹头、收费
    4. 理发师只提供了理发服务，而其他的**增强服务**都是由助理来提供
    5. 这里的助理就是**代理，代理能够伪装成主体，从而骗过调用方，然后在主体执行的前后，夹带私货**
    6. 概念对应：
        1. 顾客：调用方
        2. 理发师：主对象
        3. 助理：代理
        4. 洗头等额外任务：增强 Advice
        5. 在剪发之前做还是在剪发之后做的时机：连接点 JointPoint
        6. 哪些行为需要被增强的规则：切点 PointCut

# 3. Spring AOP 代理如何实现

1. 实现代理通常分为两类：静态代理和动态代理[（详见 设计模式 代理模式）](../../%E8%AE%BE%E8%AE%A1%E6%A8%A1%E5%BC%8F/%E7%BB%93%E6%9E%84%E5%9E%8B%E6%A8%A1%E5%BC%8F/%E4%BB%A3%E7%90%86%E6%A8%A1%E5%BC%8F%20%E2%AD%90%20842740694a744703bae301ee954a103e.md)
    - AspectJ 框架使用的是静态代理
    - Spring AOP 使用的是动态代理
2. 在原始 Spring 中，如果目标类实现了任意接口，则会使用 JDK 动态代理，否则使用 CgLib 代理
3. 而在 SpringBoot 里，无论什么情况都会默认使用 CgLib 代理，弱化了有使用限制的 JDK 动态代理

# 4. 源码流程

1. 在启动时，会创建 IOC 容器，同时将容器内的 Bean 进行三个连续的动作：构造、填充属性、初始化[（详见 IOC Bean 的生命周期）](2%204%20Bean%20的生命周期%20649a42ae86884a96a518b52cb5cb0569.md)
2. AOP 功能就是通过容器内一个专门处理 AOP 的 **`DefaultAdvisorAutoProxvCreator`** （AOP Bean 后置处理器）进行方法增强的，可以算作 IOC 容器的附加功能
3. 所有的后置处理器都是在 Bean 构造完并且填充了属性之后执行的，当然也包括上面说的这个 AOP Bean 后置处理器
4. 在每一个 Bean 初始化后，都会调用这个后置处理器的 `postProcessAfterlnitialization()` 方法，在这个方法里会为需要使用 AOP 的 Bean 创建代理对象

    ![**这个方法是 `DefaultAdvisorAutoProxvCreator` 从其父类 `AbstractAutoProxyCreator` 中继承来的**](IMG-20260620231629192.png)

    **这个方法是 `DefaultAdvisorAutoProxvCreator` 从其父类 `AbstractAutoProxyCreator` 中继承来的**

    1. 先通过 `getAdvicesAndAdvisorsForBean()` 方法获取所有的增强 Advice ，同时判断当前bean是否满足我们配置的切面条件，如果满足条件，就会为这个 Bean 构造代理对象，来实现 AOP

        ![Untitled](IMG-20260620231629219.png)

    2. 为了更统一、更方便的构造代理对象，会先搭建一个专门用来构造生成代理对象的工厂 `procyFactory`，会告诉这个工厂具体选择哪个方式进行代理，分别是 CgLib 代理和 JDK 动态代理

        ![Untitled](IMG-20260620231629246.png)

        ![Untitled](IMG-20260620231629275.png)

        1. 开发人员可以通过添加 `@EnableAspect]AutoProxy` 注解，并且将其中 `proxyTargetClass` 配置改为 `true`，来强制使用 CgLib 代理

            ![Untitled](IMG-20260620231629296.png)

        2. 当这个配置为 false，同时该类实现了任意接口，才会使用 JDK 动态代理
    3. procyFactory 知道了使用哪种代理方式之后，就会构造 JdkDynamicAopProxy 或者 CglibAopProxy，然后就可以通过 **`getProxy()`** 方法获得真正的代理对象了
        1. **JDK 动态代理方式中**
            1. 在 `getProxy()` 中会构造一个实现同样 bean 接口的代理对象，将真实 Bean 作为代理对象中的一个成员变量

                ![Untitled](../../%E8%AE%BE%E8%AE%A1%E6%A8%A1%E5%BC%8F/%E7%BB%93%E6%9E%84%E5%9E%8B%E6%A8%A1%E5%BC%8F/%E4%BB%A3%E7%90%86%E6%A8%A1%E5%BC%8F%20%E2%AD%90/Untitled%203.png)

            2. 在调用 Bean 方法的时候，就会执行代理对象中的 `invoke()` 方法，这个 `invoke()` 方法只有两步：
                1. 第一步，我会通过之前提到的 `execution` 表达式，获取所有与该方法匹配的所有增强方法，并将它们组成**调用链**，同时进行排序
                2. 第二步：按顺序执行这些调用链，这里的调用方法就是经典的责任链模式[（详见 设计模式 责任链模式）](../../%E8%AE%BE%E8%AE%A1%E6%A8%A1%E5%BC%8F/%E8%A1%8C%E4%B8%BA%E5%9E%8B%E6%A8%A1%E5%BC%8F/%E8%B4%A3%E4%BB%BB%E9%93%BE%E6%A8%A1%E5%BC%8F%20%E2%AD%90%203512b3bd8cfe4a49a9935baee9352faf.md)，在调用中间会插入并执行 Bean 的真实方法

                ![Untitled](IMG-20260620231629339.png)

        2. **CgLib 代理方式中**
            1. 同样会在 getProxy() 中构造代理对象，用增强器 `Enhancer` 来设置代理基本信息以及增强方法的调用链，接着执行 `enhancer.create()` 方法来生产代理对象
            2. 和 JdkDynamicAopProxy 不同的是：CgLib 代理方式是基于 JDK rt jar 包中的 **`ASM`** 来生产**一组新的 `.class` 文件**，然后实例化它的对象，所以对于没有实现接口的 Bean 也能生成代理对象

                ![[IMG-20260620231629370.png|458]]

            3. 在调用 Bean 方法的时候，会先执行代理对象的 `intercept()` 方法，与 JDK 动态代理一样，也会通过责任链来执行所有的方法增强

                ![Untitled](IMG-20260620231629433.png)
