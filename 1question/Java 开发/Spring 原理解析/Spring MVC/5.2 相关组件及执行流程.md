# 前言 **Servlet**

## Servlet 的使用

1. 首先我们通常要编写一个自己的Servlet然后继承自HttpServlet
2. 然后重写其doGet()与doPost()方法
    1. 这两个方法都会将HttpServletRequest与HttpServletResponse作为参数传递进去，然后我们从Request中提取前端传来的参数，在相应的doXXX方法内调用事先编写好的Service接口
    2. Dao接口即可将数据准备好放置到Response中并跳转到指定的页面即可，跳转的方式可以选择转发或者重定向
3. Servlet使用的是模板方法的设计模式，在Servlet顶层将会调用service方法，该方法会构造HttpServletRequest与HttpServletResponse对象作为参数调用子类重写的doXXX()方法。然后返回请求
4. 最后我们需要将我们编写的自定义Servlet注册到web.xml中，在web.xml中配置servlet-mapping来为该servlet指定处理哪些请求

## Servlet 存在的问题

- 每个Servlet只能处理一个请求，这样当系统比较大，业务比较复杂的时候可能会存在成百上千的Servlet，找起来都眼花
- 每次我们都需要手动的从Request中获取请求参数，然后封装成我们想要的对象，这其中可能还要对参数进行校验，在调用业务逻辑层获取到数据之后，我们还要手动的设置到响应中，同时手动的选择转发或者重定向进行跳转
- 我们的请求的url是硬配置到web.xml中的，缺乏灵活性，如果可以动态的配置这种请求url与处理的对应关系就好了
- 我们的Servlet与前端的渲染框架紧耦合在一块，这样当前端换一种显示技术的时候就需要改动较大的代码，如果能把数据的处理与数据的显示分离，让其松散耦合就更好了

## 如何优化的思考

1. 能不能进一步的来抽离业务逻辑的开发呢？
    1. 大概思路就是编写一个BaseServlet
    2. 然后我们自己定义的Servlet继承自BaseServlet，前端的请求需要指定Servlet的哪个方法进行处理，这样请求的时候将需要带上一个method参数
    3. 例如这样：http://localhost:8080/myProject/MyServlet?method=getInfo
    4. 在BaseServlet中将提取该参数信息，并使用反射的方法调用子类的该方法
        1. 子类方法统一返回String类型的结果，代表要返回的逻辑视图名，也就是要跳转的路径
        2. 然后父类拿到结果，使用重定向或者转发进行跳转
2. 总结：
    1. 如果我们想要在Servlet上更进一步，想要进一步的将业务逻辑与其他工作相分离，那么就需要在Servlet之上，构建一个事无巨细，任劳任怨的**超级Servlet**，来为我们做这些工作
    2. Spring 人员基于这个思路开发了这个超级Servlet，并正式命名为 **`DispatcherServlet`**

## **Spring MVC** 两级控制器方式

1. Spring MVC 把超级 Servlet 叫做 **`DispatcherServlet`**，称之为**前端控制器**
2. Spring MVC管定义我们的业务逻辑处理的类叫做 **`Handler`**
    1. 可以理解成业务逻辑的自定义 Servlet
    2. 只不过他不再是一个 Servlet 了，而是一个普普通通的类，因为很多的 Servlet 处理都由 DispatcherServlet 做了
    3. 而这个Handler就叫做**次级控制器**
    4. Hander是一个抽象的，Spring MVC 选择使用 **`Controller`** 来实现 Handler

# 1. **Spring MVC** 相关**组件**

## 1.1 DispatcherServlet：**前端控制器**

1. DispatcherServlet 是整个 Spring MVC 的核心
2. DispatcherServlet 和其他组件一起完成了很多的工作，包括：
    - 请求参数的自动绑定
    - 参数的自动校验
    - 请求url的自动匹配
    - 逻辑视图名到真实页面的跳转
    - 数据获取与数据渲染显示的分离
    - 等等
3. 在此过程中他更像是一个指挥者，整个流程控制的中心，由它调用其它组件处理用户的请求，并最终完成服务端的响应数据

## 1.2 HandlerMapping：**处理器映射器**

1. Servlet 编写代码的时，请求的映射工作是交给了 web.xml
    1. 首先 DispatcherServlet 也是一个Servlet，DispatcherServlet 它能处理所有的请求，因此配给 DispatcherServlet 的是所有请求
    2. 现在所有的请求都被映射到了 DispatcherServlet，那么 **DispatcherServlet 现在就有责任将请求分发至具体的次级控制器**
2. **如何找到**或者说**如何保存 请求到具体的次级控制器的这种映射关系**
    1. 就需要 HandlerMapping 配合 DispatcherServlet
    2. 在 HandlerMapping 中，保存了特定的请求 url 应该被哪一个 Handler（也就是通常的Controller）所处理

    [（详见 SpringMVC **DispatcherServlet 源码** **`mappedHandler = getHandler(processedRequest)` 的分析**）](处理请求(调用)的过程%2030b970903f244fbf8d9757ca1a4df4d7.md)

3. HandlerMapping 根据映射策略的不同，大概有下面几种**映射查找方式：**
    1. org.springframework.web.servlet.handler.**SimpleUrlHandlerMapping**：通过配置请求路径和Controller映射建立关系，找到相应的Controller
    2. org.springframework.web.servlet.mvc.support.**ControllerClassNameHandlerMapping**：通过 Controller 的类名找到请求的Controller
    3. org.springframework.web.servlet.handler.**BeanNameUrlHandlerMapping**：通过定义的 beanName 进行查找要请求的Controller
    4. org.springframework.web.servlet.mvc.annotation.**DefaultAnnotationHandlerMapping**：通过注解 `@RequestMapping(“/userlist”)` 来查找对应的Controller

## 1.3 **HandlerInterceptor：Handler 拦截器**

1. DispatcherServlet 根据 url 从 HandlerMapping 查出了 Handler。此时，DispatcherServlet 并不能直接执行这些 Handler
2. 因为不能把所有请求都全部交给 Handler 执行，最起码要过滤一下不合理的请求，并将这些请求拦截，比如
    - 跳转页面的时候检查Session，如果用户没登录跳转到登录界面
    - 以及一些程序的异常以统一的方式跳转等等
3. **HandlerInterceptor 与 Filter的区别**：[（详见 SpringBoot 拦截器）](../../SpringBoot/4%20Web%20%E5%BC%80%E5%8F%91%E6%89%8B%E5%86%8C/4%206%20%E6%8B%A6%E6%88%AA%E5%99%A8%20906de244e54049788c3c13972a0e5ff8.md)

## 1.4 Handler：次级**控制器（处理器）**

1. Handler 就是次级控制器，是**真正执行业务逻辑的类**
2. 通常在 Spring MVC 中，Handler 就是很熟悉的 Controller，调用封装好的业务逻辑接口（Service）就是在这里进行处理的
3. 可以说 Spring MVC 已经将业务逻辑与其他不相关的繁杂工作分离的较为彻底了。这样可以在 Handler（Controller）中对业务逻辑更专注

## 1.5 **HandlerExecutionChain：执行链**

1. DispatherServlet 在将 url 交给特定的 HandlerMapping 之后，HandlerMapping 返回给 DispaterServlet 的却不是一个可执行的 Handler（Controller）
2. 而是一个 HandlerExecutionChain 对象，为什么？
    1. HandlerInterceptor 与 Handler是怎样联系在一起的方式就是 HandlerExecutionChain
    2. 它就是若干的 HandlerInterceptor 与 Handler 的组合
3. 那么是怎么组合的呢？
    1. 使用了设计模式中的责任链设计模式
    2. 其维护着一个 `List<HandlerInterceptor> interceptorList` ，保存着所有注册的拦截器`HandlerInterceptor` [（详见 设计模式 责任链模式）](../../%E8%AE%BE%E8%AE%A1%E6%A8%A1%E5%BC%8F/%E8%A1%8C%E4%B8%BA%E5%9E%8B%E6%A8%A1%E5%BC%8F/%E8%B4%A3%E4%BB%BB%E9%93%BE%E6%A8%A1%E5%BC%8F%20%E2%AD%90%203512b3bd8cfe4a49a9935baee9352faf.md)

        ![Untitled](../../%E8%AE%BE%E8%AE%A1%E6%A8%A1%E5%BC%8F/%E8%A1%8C%E4%B8%BA%E5%9E%8B%E6%A8%A1%E5%BC%8F/%E8%B4%A3%E4%BB%BB%E9%93%BE%E6%A8%A1%E5%BC%8F%20%E2%AD%90/Untitled%202.png)

    3. HandlerExecutionChain 将 HandlerInterceptor 与 Handler 串成一个执行链的形式
        1. 首先请求会被第一个 HandlerInterceptor 拦截
        2. 如果返回false，那么直接短路请求
        3. 如果返回true,那么再交给第二个HandlerInterceptor处理
        4. 直到所有的HandlerInterceptor都检查通过，请求才到达Handler(Controller)，交由Handler正式的处理请求
        5. 执行完成之后再逐层的返回
4. 因此，DispatcherServlet 拿到的就是这样一个串联好的 HandlerExecutionChain，然后顺序的执行请求

## 1.6 HandlerAdapter：**处理器适配器**

1. 拿到了 HandlerExecutionChain 执行链后，可以从中取出 Handler
2. 但是会发现 Spring MVC 并没有一个名为 Handler 的类，**在所有源码用到 handler 的地方，它都是一个 `Object` 对象**
3. 因此要执行它，**首先得判断它是一个什么类型的处理器**，判断完成后在调用对应类型的处理方法
4. 而 HandlerAdapter 就是用来做判断的[（详见 设计模式 适配器模式）](../../%E8%AE%BE%E8%AE%A1%E6%A8%A1%E5%BC%8F/%E7%BB%93%E6%9E%84%E5%9E%8B%E6%A8%A1%E5%BC%8F/%E9%80%82%E9%85%8D%E5%99%A8%E6%A8%A1%E5%BC%8F%20%E2%AD%90%202b037a9aa2cd47c59d02cdc4054ad619.md)

## 1.7 **ModelAndView：解耦的关键**

1. DispatcherServlet 使用了 ModelAndView 保存数据和想要跳转的路径
2. 调用业务逻辑层获取数据后，将数据封装到 ModelAndView 中，同时设置 ModelAndView 的 view 逻辑视图名称
    1. 从ModelAndView 的名称可以看出，它保存了 **Handler 执行完成之后所需要发送到前端的数据**，以及**需要跳转的路径**
    2. 这些是DispatcherServlet需要用到的

## 1.8 **ViewResolver：视图渲染查找**

1. 这一步是 Spring MVC 将数据的获取与数据的显示渲染相分离的关键
2. DispatcherServlet 拿到 ModelAndView 后，这里面有执行完成请求后返回的响应结果数据，还有逻辑视图的路径
    1. 但由于前端可能采用各种各样的方式显示数据，可能是Jsp，可能是Html，也可能是其他的方式
    2. 这个时候 DispatcherServlet 就需要使用 ViewResolver，根据拿到的逻辑视图的路径去**查找谁能把数据进行解析与渲染[（详见 SpringBoot 视图解析器与视图原理）](../../SpringBoot/4%20Web%20%E5%BC%80%E5%8F%91%E6%89%8B%E5%86%8C/4%205%20%E8%A7%86%E5%9B%BE%E8%A7%A3%E6%9E%90%20%E4%B8%8E%20%E6%A8%A1%E6%9D%BF%E5%BC%95%E6%93%8E/%E8%A7%86%E5%9B%BE%E8%A7%A3%E6%9E%90%E5%99%A8%E4%B8%8E%E8%A7%86%E5%9B%BE%20%E5%8E%9F%E7%90%86%2019a66d91644248748452ddf0a9a09532.md)**
3. 通常的寻找策略有以下几种：
    - BeanNameViewResolver：将逻辑视图名解析为一个Bean,Bean的id等于逻辑视图名
    - XmlViewResolver：和BeanNameViewResolver类似，只不过目标视图Bean对象定义在一个独立的XML文件中，而非定义在DispatcherServlet上下文的主配置文件中
    - InternalResourceViewResovlver：将视图名解析为一个URL文件，一般使用该解析器将视图名映射为保存在WEB-INF目录中的程序文件（如JSP）
    - XsltViewResolver：将视图名解析为一个指定XSLT样式表的URL文件
    - JasperReportsViewResolver：JasperReports是一个基于java的开源报表工具，该解析器将视图名解析为报表文件对应的URL
    - FreeMarkerViewResolver：解析为基于FreeMarker模板技术的模板文件
    - VelocityViewResolver和VelocityLayoutViewResolver：解析为基于Velocity模板技术的模板文件

## 1.9 **View：数据渲染**

1. 在根据逻辑视图名借助 ViewResolver 查找到对应的 View 实现类之后
2. DispatcherServlet 就会**将 ModelAndView 中的数据交给 View 实现类来进行渲染**
3. 待该 View 渲染完成之后，会将渲染完成的数据交给 DispatcherServlet，这时候 DispatcherServlet 将其封装到 Response 返回给前端显示

# 2. **Spring MVC 执行流程**

![Untitled](IMG-20260620224904944.png)

1. 用通过客户端向服务器发送请求后，请求会被 SpringMVC 的前端控制器 **`DispatcherServlet`** 所 **①** 拦截
2. `DispatcherServlet` 拦截到请求后，会 **②** 调用 **`HandlerMapping`** 处理器映射器
3. `HandlerMapping` 根据请求 URL 找到具体的 **`Handler`** （处理器）
    1. 这里的处理器指的就是程序中编写的 Controller 类，也称后端控制器
    2. 但其实返回的是一个执行链（**`HandlerExecutionChain`**），链中包含处理器对象 以及 处理器拦截器（如果有就生成），返回给 `DispatcherServlet`
4. `DispatcherServlet` 会通过返回信息 **③** 选择合适的 **`HandlerAdapter`**
5. `HandlerAdapter` 会调用并 **④** 执行 `Handler`
6. Controller 执行完成后，会 **⑤** 返回一个 **`ModelAndView**` 对象，该对象中会包含视图名和模型（或只含视图名），模型 `**Model`** 中存着响应回前端的数据
7. `HandlerAdapter` 将 `ModelAndView` 对象返回给 `DispatcherServlet`
8. `DispatcherServlet` 会根据 `ModelAndView` 对象 **⑥** 选择一个合适的 **`ViewResolver`** （视图解析器）
9. `ViewResolver` 解析后，会向 `DispatcherServlet` 中返回具体的 **`View`**（视图）
10. `DispatcherServlet` 对 `View` 进行 **⑦** 渲染（即将模型中的数据填充至视图中）
11. 视图渲染结果会 **⑧** 返回给客户端浏览器显示
