# 1. 什么是循环依赖
1. 从字面上来理解就是A依赖B的同时B也依赖了A
1. 可能有下面这三种情况
![[IMG-20260405035413841.webp]]
![[IMG-20260404031821564.webp]]
![[IMG-20260405035420224.webp]]
1. 代码示例：
    
    ```Java
    @Component
    public class A {
        // A中注入了B
    		@Autowired
    		private B b;
    }
    
    @Component
    public class B {
        // B中也注入了A
    		@Autowired
    		private A a;
    }
    ```
    
1. 产生的问题：
    
    如果两个bean出现了相互依赖的情况，如果Spring没有处理循环依赖，那么出现的结果就是**在bean的创建过程中出现相互依赖，导致这个bean永远无法创建出来，则就导致一直在相互创建**
    
# 2. **什么情况下的循环依赖可以被 Spring 解决**
Spring 解决循环依赖是有前置条件的
1. 条件一：⚠️**出现循环依赖的 Bean 必须要是单例**
1. 条件二：⚠️**依赖注入的方式不能全是构造器注入的方式**[[2.1 IOC 介绍]]
    
    1. 当全使用构造器方式注入时：
        
        ```Java
        @Component
        public class A {
        		public A(B b) {
        		
        		}
        }
        
        @Component
        public class B {
        	 public B(A a){
        	
        	 }
        }
        ```
        
        **这个时候循环依赖是无法被 Spring 解决的**，因此如果的项目中有两个这样相互依赖的Bean，在启动时就会报出以下错误
        
        ```Java
        Caused by: org.springframework.beans.factory.BeanCurrentlyInCreationException: Error creating bean with name 'a': Requested bean is currently in creation: Is there an unresolvable circular reference?
        ```
        
    
    1. 很多博客上说，只能解决 setter 方法的循环依赖，这是错误的，为此做了测试，测试结果如下

# 3. **Spring 是如何解决的循环依赖**🙋‍♂️
## 3.1 前提知识
1. **Bean创建的几个关键点**[[2.4 Bean 的生命周期]]
1. **AOP的原理**[[3. AOP]]
1. **Java 调用方法时参数的传递方式**[[调用方法时参数的传递方式]]
1. `**getBean()**` **返回的是什么**
    
    1. 当按 beanname 从 `BeanFactory.getBean(beanname)` 一个Bean时，返回的不一定是A类对应的实例
    
    1. 因为，当A需要创建代理对象时，在这种情况下 `getBean()` 得到是代理对象的引用，而不是A类对应的原始实例
    
## 3.2 **三级缓存**
1. 缓存：即把创建好的Bean缓存起来
1. 源码
    
    ![[IMG-20260405035431489.png|Untitled 1 236.png]]
    
    - `**singletonObjects**`
        
        - 第一级缓存
        
        - 里面存放的都是创建好的==**成品Bean**==
        
    
    - **`earlySingletonObjects`**
        
        - 第二级缓存
        
        - 里面存放的都是==**半成品的Bean**==**（只有一个引用地址，里面的属性都是空的，存在的目的是为了在解决循环依赖时，用作检查缓存）**
        
    
    - **`singletonFactories`**
        
        - 第三级缓存
        
        - 不同于前两个存的是 Bean对象引用，此缓存**存的是** ==**bean 工厂对象**==，也就存的是专门创建Bean的一个工厂对象
        
        - 此缓存用于解决循环依赖
        
    
## 3.3 举例说明 Spring 是如何使用三级缓存解决循环依赖的
1. 循环依赖：
    
    ```Java
    public class A {
    		@Autowired
        B b;
        public A() {
        }
    }
     
    class B{
        @Autowired
        A a;
        public B() {
        }
    }
    ```
    
1. **一级缓存能不能解决循环依赖**❓
    
    - 展开
        
        ![[IMG-20260405035443815.png]]
        
        - 从A获取开始，从缓存里查看，没有开始创建A实例，执行构造方法，填充属性时发现需要依赖B
        
        - 尝试从缓存中获取B
        
        - 开始创建B实例，执行构造方法，填充属性时，发现需要依赖A，取缓存找A
        
        - A正在创建没有完成
        
        - 死结
        
    
1. **二级缓存能不能解决循环依赖**❓
    
    - 展开
        
        ![[IMG-20260405035443840.png]]
        
        - A引用创建后，提前暴露到半成品缓存中
        
        - 依赖B，创建B ，B填充属性时发现依赖A， **先从成品缓存查找，没有再从半成品缓存查找，发现可以取到A的早期引用**
        
        - **B顺利走完创建过程，**将B的早期引用从半成品缓存移动到成品缓存
        
        - B创建完成，A获取到B的引用，继续创建
        
        - A创建完成，将A的早期引用从半成品缓存移动到成品缓存
        
        - 解决循环依赖
        
    
1. **两个缓存就能解决**🙀❓**，为啥需要三个缓存**❓
    
    1. 上面两个缓存的地方，**==没有考虑⭐代理的情况==**
    
    1. Bean 的代理
        
        1. ==Bean在创建的最后阶段，会检查是否需要创建代理==，如果创建了代理，那么通过 bean name 获取到的，应该是代理实例的引用
        
        1. ==即当 Bean 被 AOP 代理扩展后，那些依赖该 Bean 的 Bean 也应该依赖的时扩展后的代理引用，而不是为代理扩展前的原始引用==
        
    
    1. **也就是说当在考虑代理的情况下，只用二级缓存是存在问题的：**
        
        1. 假设A最终会创建代理，提前暴露A的引用， B填充属性时填充的是A的原始对象引用
        
        1. A最终放入成品库里是代理的引用。那么B中依然是A的早期引用。这样 B 中岂不是依赖错了，并没有依赖到 A 执行扩展逻辑（代理）后的对象
        
    
1. ⭐**三级缓存完美解决循环依赖**
    
    ![[IMG-20260405035504459.png]]
    
    - A绑定到 ObjectFactory 注册到工厂缓存 singletonFactory 中
    
    - **B在填充A时，先查成品缓存有没有，再查半成品缓存有没有，最后看工厂缓存有没有单例工厂类**
        
        - 发现有A的 ObjectFactory
        
        - 调用 getObject() ，执行扩展逻辑，可能返回的代理引用，也可能返回原始引用
        
        - ⭐==**这样不管 A 有没有被代理扩展，B 中都能依赖到正确的 A 对象**==，因为在第三级缓存中取到的是 A 的 ObjectFactory
            
            - 如果 A 需要代理扩展，则这个工厂中是可以调用到 A 的扩展代理逻辑的，因此根据这个工厂得到的 A 对象就是代理扩展后的对象引用（解决了只用二级缓存时出现的问题）
            
            - 如果 A 不需要代理扩展，拿就按一般逻辑走，得到的也是对应正确的原始引用
            
        
    
    - 使用A的 ObjectFactory成功创建得到A的早期引用后，会将A放入到半成品缓存中
    
    - B会从二级缓存中获取到 A 的对象引用（**由上得，不管是原始引用还是代理引用都能正确地获得**），然后会删除 A 的三级缓存
    
    - 接着 B填充A引用完毕
    
    - 代理问题， 循环依赖问题都解决了
