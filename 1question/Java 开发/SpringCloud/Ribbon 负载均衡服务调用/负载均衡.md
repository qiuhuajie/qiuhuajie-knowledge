# 一、介绍
1. 负载均衡的**两个基本点**：
    - 选择哪个服务器来处理客户端请求
    - 将客户端请求转发出去
2. **一个核心原理**：
    1. 通过硬件或软件的方式**维护一个**==**服务列表清单**==
    2. 当用户发送请求时，会将请求发送给负载均衡器，然后**根据**==**负载均衡算法**==**从可用的服务列表中选出一台服务器的地址，将**==**请求进行转发**==**，完成负载功能**
3. **负载均衡的特性**
    - **高性能**：可根据不同的分配规则自动将流量进行分摊
    - **可扩展性**：可以很方便增加集群中设备或链路的数量
    - **高可靠性**：系统中某个设备或链路发生故障，不会导致服务中断
    - **易配置性**：配置和维护方便
    - **透明性**：用户感知不到如何进行负载均衡的，也不用关心负载均衡

# 二、分类
## 1. 按软件或硬件分类
### 1.1 硬件负载均衡
- **==F5==** 就是常见的硬件负载均衡产品
    1. 优点：性能稳定，具备很多软件负载均衡不具备的功能，如应用交换，会话交换、状态监控等
    2. 缺点：设备价格昂贵、配置冗余，没有软件负载均衡灵活，不能满足定制化需求

### 1.2 软件负载均衡
- ==**Nginx：**==
    1. 性能好，可以负载超过 1W
    2. 工作在网络的**7层**之上，可以针对http应用做一些分流的策略
    3. Nginx也可作为静态网页和图片服务器
    4. Nginx仅能支持http、https和Email协议
- ==**LVS（Linux Virtual Server）：**==
    1. 是一个虚拟服务器集群系统，采用 IP 地址均衡技术和内容请求分发技术实现负载均衡
    2. 接近硬件设备的网络吞吐和连接负载能力
    3. 抗负载能力强、是工作在网络**4层**之上仅作分发之用
    4. 相比 Nginx 软件本身不支持正则表达式处理，不能做动静分离

## 2. 按服务器列表存放的位置分类
### 2.1 服务端负载均衡
- **==Nginx==** 和 ==**F5**== 属于服务端的负载均衡
    1. 后端的服务器地址列表是**存储在后端服务器**中或者存在专门的 **Nginx 服务器**或 **F5 上**
    2. 服务器的地址列表的来源是通过注册中心或者手动配置的方式来的
- **==请求方只管发请求，我发给服务端就不管了，由服务端为请求做负载均衡==**
### 2.2 请求端负载均衡
- **==Ribbon==** 属于请求端负载均衡器
    1. **Ribbon 会从 Eureka Server 读取服务信息列表，存储在 Ribbon 中**
    2. 请求端**自己维护**一份服务器的地址列表
        1. 这个维护的工作就是由 Ribbon 来干的
        2. 如果服务器宕机了，Ribbon 会从列表剔除宕机的服务器信息
    3. Ribbon 有多种负载均衡算法，我们可以自行设定规则从而请求到指定的服务器
- **==消费方从服务注册中心获知服务信息列表，然后自己再从这些地址中选择出一个合适的服务器，再发请求==**
- 个人理解：这里的请求端
    - 可以是用户端
    - 也可以是服务端中某两个和微服务之间调用的服务调用方

## 3. 按工作在网络的哪一层
### 3.1 四层负载均衡
1. 四层负载均衡，指的是**工作在 OSI 的第四层，即传输层**，其**==通过 IP + 端口的方式进行路由转发==**
2. 转发过程：
    1. 客户端向负载均衡发送 SYN 请求建立第一次连接
    2. 四层负载均衡器通过配置的负载均衡算法选择一台后端服务器，并且将客户端报文中的 IP 地址信息修改为后台服务器的 IP 地址信息
    3. 因此 **TCP 三次握手连接是**==**客户端与后端服务器直接建立起来的**==
3. 常见的四层负载均衡设备有：F5、LVS 等
### 3.2 七层负载均衡
1. 七层负载均衡，指的是**工作在 OSI 的第七层，即应用层**，其一般是==**基于请求 URL 地址的方式进行代理转发**==
2. 转发过程：
    1. 客户端向负载均衡发送连接请求，==**客户端先与负载均衡设备进行 TCP 连接**==
    2. 四层负载均衡器通过配置的负载均衡算法选择一台后端服务器
    3. **然后，**==**负载均衡设备再与后端服务器建立另外一条 TCP 连接通道**==
    4. 因此，七层设备在网络性能损耗会更多一些
3. 常见的七层负载均衡设备有：**Nginx** 等

    ```JSON
    # Nginx 配置文件
    worker_processes  1;
    events {
         worker_connections  1024;
    }
    # ⭐ 7 层 http 负载
    http {
    			 ...
    }
    ```
### 3.3 四七异同
1. ==**工作层级不同**==
    1. **四层负载均衡工作在传输层**
    2. **七层负载均衡工作在应用层**
2. ==**建立连接不同**==
    1. **四层负载均衡，客户端直接与后端服务器建立连接**
    2. **七层负载均衡，则作为中间用户与上下游建立连接**
3. ==**效率不同**==
    1. **四层负载均衡相对效率较高**
    2. **七层负载均衡效率较低**
4. ==**灵活性方面**==
    1. **七层负载均衡可以做更多的改造，例如可以对流量进行清洗，避免 DDOS 攻击**
    2. **而四层负载均衡，由于是客户端与后端服务器直连，负载均衡设备无法介入**

# 三、均衡策略
## 1. 轮循均衡（Round Robin）
1. 原理：如果给服务器从 0 到 N 编号，轮询均衡策略会==从 0 开始依次选择==一个服务器作为处理本次请求的服务器
2. 场景：适合所有父亲都有相同的软硬件配置，且请求频率相对平衡
## 2. 权重轮询均衡（Weighted Round Robin）
1. 原理：==按照服务器的不同处理能力，给服务器分配不同的权重==，然后请求会按照权重分配给不同的服务器
2. 场景：服务器的性能不同，充分利用高性能的服务器，同时也能照顾到低性能的服务器
## 3. 随机均衡（Random）
1. 原理：将请求==随机分配==给不同的服务器
2. 场景：适合客户端请求的频率比较随机的场景
## 4. 响应速度均衡（Response Time)
1. 原理：负载均衡设备对每个服务器==发送一个探测请求，看看哪台服务器的响应速度更快==
2. 场景：适合服务器的响应性能不断变化的场景
3. 注意：响应速度是针对负载均衡设备和服务器之间的
