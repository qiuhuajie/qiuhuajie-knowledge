# 一、Nacos 注册中心的实现原理
1. Nacos 作为注册中心主要提供以下三个功能
    - **服务注册**
    - **服务发现**
    - **健康监测**
2. 原理图示

    ![[IMG-20260405035413895.png|800]]

    1. **服务注册：**
        1. 服务提供方使用 OpenAPI 发起 HTTP 服务注册请求，==提供注册服务的信息==，包括自身 IP 和端口，元数据等
        2. Nacos 接受请求后，将==信息储存在内存 Map== 中
    2. **服务发现：**
        1. 服务消费者查询服务的==服务实例列表==，==缓存在本地==（参考 Ribbon）
        2. 同时开启==定时任务==，每 10s 拉取一次 Nacos 上的服务列表，保证及时获取新上线的服务
        3. 当检测到服务提供者==异常==或有服务==新增==时，使用 UDP 将服务的变更推送给服务消费者
    3. **健康监测：**
        1. 建立==心跳==， Nacos 每 5s 监测一次服务提供者上服务的状态
        2. 当前时间 - 上一次心跳的时间
            1. 如果超过 15s，则将此服务设置为非健康服务，并进行广播
            2. 如果超过 30s，则将此服务直接移除

# 二、Nacos 配置中心的实现原理
1. Nacos 客户端使用**长轮询**[[Export-e8249ac1-4ae1-42cd-83d5-041a266841fe/1question/基础知识/计算机网络/应用层协议/WebSocket|WebSocket]]的方式向 Nacos 服务端去发起配置更新的请求
2. 流程：首先==客户端拿着本地缓存的配置信息，去服务端上，进行配置的比较，一旦发现有差异，则表示远程的配置有更新，则将远程配置缓存到本地==
3. 在这个过程中，如果配置比较多，可能导致对比的时间比较长，使得配置同步的效率非常低
4. Nacos 针对上面的问题做了==两个优化==：
    1. 减少网络通信的数量，客户端==将需要比对的配置进行分片==，每个分片的大小是 3000 项
    2. ==分段比较和增量更新==
        1. 客户端将 3000 个配置项（`key-value`）对应的 MD5 值拼成一个字符串，发送到服务端上进行比较，服务端会把有更新的 `key` 返回给客户端
        2. 客户端拿到有更新的 `key` 后，逐个去服务端获取最新值，并缓存到本地
    3. 这样做的优点是减少了每次网络通信的数据量，提高效率
