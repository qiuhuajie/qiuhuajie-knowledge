# 一、Seta 分布式事务的角色
- `Transaction Coordinator(TC)`
    - 全局事务协调者
    - 用来==协调全局事务和各个分支事务（不同服务）的状态==， ==驱动全局事务和各个分支事务的回滚或提交==
- `Transaction Manager(TM)`
    - 事务管理者
    - 用来==在业务层中开启/提交/回滚一个整体事务==（在调用服务的方法中用注解 **`@GlobalTransactional`** 开启事务）
- `Resource Manager(RM)`
    - 资源管理者
    - 一般情况下，==一个业务数据库代表了一个分支事务==（`Branch Transaction`）
    - `RM` ==管理分支事务==与 `TC` 进行==协调注册分支事务==并且==汇报分支事务的状态==，==驱动分支事务的提交或回滚==
- `UNDO_LOG`
    - Seata 实现分布式事务，设计了一个关键角色 `UNDO_LOG` （回滚日志记录表）
    - **在每个应用分布式事务的业务库中都需要创建这张表**
    - 这个表的核心作用就是：**将业务数据在更新前后的数据镜像组织成回滚日志，备份在 `UNDO_LOG` 表中，以便业务异常能随时回滚**

# 三、Seta 的 AT 事务模式
1. Seata 也是从两段提交演变而来的一种分布式事务解决方案，**Seata AT 模式是增强型的 2PC 模式**
2. 提供了 `AT`、`TCC`、`SAGA` 和 `XA` 等事务模式，这里重点介绍 `AT` 模式
## 1. 情况 1：一阶段 ✅ 提交，二阶段 ✅ 提交
![[IMG-20260405035422024.png|800]]
### 1.1 一阶段
- 比如：更新 `user` 表的 `name` 字段。

    ```SQL
    update user set name = '小富最帅' where name = '程序员内点事'
    ```
1. 首先 Seata 的 `JDBC` 数据源代理通过==**对业务 SQL 解析，提取 SQL 的元数据**==
    - 也就是得到 SQL 的类型（`UPDATE`），表（`user`），条件（`where name = '程序员内点事'`）等相关的信息
2. 查询**==数据前镜像==**：根据解析得到的条件信息，生成查询语句，定位一条数据

    ```SQL
    select name from user where name = '程序员内点事'
    ```
3. 紧接着执行业务 SQL，根据前镜像数据主键查询出**==后镜像数据==**，==**把业务数据在更新前后的数据镜像组织成回滚日志**==

    ```SQL
    select name from user where id = 1
    ```
    1. 回滚记录数据格式如下：包括 `afterImage` 后镜像、`beforeImage` 前镜像、 `branchId` 分支事务ID、`xid` 全局事务ID
        - 回滚记录示例：

    ```JSON
    {
    	"branchId": 641789253,
    	"undoItems": [{
    		"afterImage": {					//记录着业务SQL执行前的数据状态，⭐用于和其他SQL元数据生成回滚SQL语句
    			"rows": [{
    				"fields": [{
    					"name": "id",
    					"type": 4,
    					"value": 1
    				}, {
    					"name": "name",
    					"type": 12,
    					"value": "GTS"
    				}, {
    					"name": "since",
    					"type": 12,
    					"value": "2014"
    				}]
    			}],
    			"tableName": "product"
    		},
    		"beforeImage": {			//记录着业务SQL执行后的数据状态，⭐用于数据校验是否有脏写
    			"rows": [{
    				"fields": [{
    					"name": "id",
    					"type": 4,
    					"value": 1
    				}, {
    					"name": "name",
    					"type": 12,
    					"value": "TXC"
    				}, {
    					"name": "since",
    					"type": 12,
    					"value": "2014"
    				}]
    			}],
    			"tableName": "product"
    		},
    		"sqlType": "UPDATE"
    	}],
    	"xid": "xid:xxx"
    }
    ```
    2. 这样就可以保证，任何提交的业务数据的更新一定有相应的回滚日志
4. 最后将 **业务数据的更新** 和 **回滚日志** 在同一个本地事务中提交，分别插入到业务表和 `UNDO_LOG` 表中
5. 在本地事务提交前
    1. 管理各分支事务的 `RM` 需向全局事务协调者 `TC` 注册分支 ( `Branch Id`) ，为要修改的记录申请**全局锁** （利用 `SELECT FOR UPDATE` 语句，为这条数据加锁）
    2. 而如果一直拿不到锁那就需要回滚本地事务
    3. `TM` 开启事务后会生成全局唯一的 `XID`，会在各个调用的服务间进行传递
    > 💡 `UNDO_LOG` ➕ 全局锁 机制带来的好处
    > 1. 本地事务分支在全局事务的第一阶段就可以提交，意味着可以马上释放本地事务锁定的资源。相比于传统的 `XA` 事务在第二阶段释放资源，`Seata` 降低了锁范围提高效率
    > 2. 同时，即使第二阶段发生异常需要回滚，也可以快速 从 `UNDO_LOG` 表中找到对应回滚数据并反解析成 SQL 来达到回滚补偿
6. 最后本地事务提交，业务数据的更新和前面生成的 `UNDO LOG` 数据一并提交，并将本地事务提交的结果上报给全局事务协调者 TC
### 1.2 二阶段
- 第二阶段是根据各分支的决议做提交或回滚：
- 如果决议是全局提交
    - 此时各分支事务已提交并成功，这时 全局事务协调者（`TC`）会向分支发送第二阶段的请求
    - 收到 TC 的分支提交请求，该请求会被放入一个==**异步任务队列**==中，并马上返回提交成功结果给 TC
    - 异步队列中会**异步和批量地根据 `Branch ID` 查找并==删除相应== `UNDO LOG` ==回滚记录==**

### 2.1 一阶段
同上
### 2.2 二阶段
- 第二阶段是根据各分支的决议做提交或回滚：
- 如果决议是全局回滚
    - 过程比全局提交麻烦一点
    - `RM` 服务方收到 `TC` 全局协调者发来的回滚请求，通过 `XID` 和 `Branch ID` 找到相应的回滚日志记录
    - 接着通过回滚记录==**生成反向的更新 SQL 并执行**==，以完成分支的回滚

# 四、是如何保证写隔离的❓（避免脏写）
1. 两个全局事务 tx1 和 tx2，分别对 a 表的 m 字段进行更新操作，m 的初始值 1000
2. tx1 先开始，开启本地事务，拿到本地锁，更新操作 m = 1000 - 100 = 900。本地事务提交前，先拿到该记录的**全局锁** ，本地提交释放本地锁
3. tx2 后开始，开启本地事务，拿到本地锁，更新操作 m = 900 - 100 = 800。本地事务提交前，也尝试拿该记录的**全局锁** ，但是，==tx1 全局提交前，该记录的==**==全局锁==**==被 tx1 持有，tx2 需要重试等待====**全局锁**==

    ![[IMG-20260405035432689.png|800]]
4. **如果 txq 的二阶段全局提交：**那么直到 tx1 二阶段全局提交，释放**全局锁** 。tx2 才能拿到 **全局锁** 提交本地事务
5. **如果 tx1 的二阶段全局回滚：**
    1. 则 ==tx1 需要重新获取该数据的本地锁==，进行反向补偿的更新操作，实现分支的回滚
    2. 此时，如果 tx2 仍在等待该数据的 **全局锁**，同时持有本地锁，则 tx1 的分支回滚会失败
    3. ==tx1 分支的回滚会一直重试，直到 tx2 的====**全局锁**== ==等锁超时==，放弃尝试获取**全局锁**并回滚本地事务，并释放**本地锁**，tx1 的分支回滚最终成功

    ![[IMG-20260405035444748.png|800]]
6. ==因为整个过程====**全局锁**== ==在 tx1 结束前一直是被 tx1 持有的，tx2 一直拿不到全局锁，也就无法提交本地事务==
7. 故不会发生 **脏写** 的问题
# 五、是如何保证读隔离的❓（避免脏读）
1. 在数据库本地事务隔离级别 **读已提交** 或以上的基础上，==Seata（AT 模式）的默认==**==全局隔离级别==**==是== ==**读未提交**==
    1. ==**读未提交**== 就会出现问题：tx_2 会读到 tx_1 全局提交之前的数据
    2. 而 tx_1 在之后可能执行了回滚，那么 tx_2 读到的数据就是错的（脏读）
2. 解决办法：
    1. **如果在特定场景下，必需要求全局的** ==**读已提交**== **，目前 Seata 的方式是通过对 `SELECT FOR UPDATE` 语句进行代理**

    ![[IMG-20260405035504935.png|800]]
    2. SELECT FOR UPDATE 语句的执行
    3. 会申请 **全局锁** ，如果 **全局锁** 被其他事务持有
        1. 则释放本地锁（回滚 `SELECT FOR UPDATE` 语句的本地执行）
        2. 并重试获取全局锁
    4. 这个过程中，`FOR UPDATE` 查询是被阻塞住的，直到 **全局锁** 拿到，即读取的相关数据是 **已提交** 的，才返回
    5. 出于总体性能上的考虑，Seata 目前的方案并没有对所有 SELECT 语句都进行代理，仅针对 FOR UPDATE 的 SELECT 语句
