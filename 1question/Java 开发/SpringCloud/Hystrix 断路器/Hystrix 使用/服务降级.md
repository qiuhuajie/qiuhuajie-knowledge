# 一、在服务提供端 8001 添加服务降级
1. **主启动类：****`@EnableCircuitBreaker`**

    ```Java
    @SpringBootApplication
    @EnableEurekaClient
    @EnableCircuitBreaker //开启服务降级
    public class PaymentHystrixMain8001 {
        public static void main(String[] args) {
            SpringApplication.run(PaymentHystrixMain8001.class,args);
        }
    }
    ```
2. **在需要做降级的方法上添加注解：`@HystrixCommand`**
    - 一旦调用服务方法失败并抛出了错误信息后，会自动调用 `@HystrixCommand` 标注好的 `fallbackMethod` 调用类中的指定方法
    - 即只要是当前服务不可用了，就做服务降级，兜底的方案都是 `paymentInfo_TimeOutHandler`
    > 注意：配置了热部署方式对java代码的改动明显，但对@HystrixCommand内属性的修改建议重启微服务

    ```Java
    @Service
    public class PaymentService {
        ...
        //⭐声明当下面的方法超时或异常时，应该怎么办
        @HystrixCommand(fallbackMethod = "paymentInfo_TimeOutHandler",commandProperties = {
                //三秒以内就是正常逻辑，如果超了3秒就会去执行兜底方法
                @HystrixProperty(name="execution.isolation.thread.timeoutInMilliseconds",value="3000")
        })
        //原方法：
        public String paymentInfo_TimeOut(Integer id) {
            //模拟当前这个业务逻辑需要执行 5 秒
            int outTime = 5;
            try { TimeUnit.SECONDS.sleep(outTime); } catch (InterruptedException e) { e.printStackTrace(); }
            return "线程池:"+Thread.currentThread().getName()+"，调用的方法：paymentInfo_TimeOut，输入的参数："+id+"\t"+"调用状态：O(∩_∩)O，耗时："+outTime+"秒";
        }
        //⭐当上面的方法出现超时或异常时，就执行这个兜底方法：
        public String paymentInfo_TimeOutHandler(Integer id){
           return "当前执行了服务提供端的超时兜底：支付系统繁忙，响应状态：o(╥﹏╥)o";
        }
    }
    ```
3. **测试：**

    [http://localhost:8001/payment/hystrix/timeout/31](http://localhost:8001/payment/hystrix/timeout/31)

# 二、在服务消费端 80 添加服务降级
> 💡 **虽然服务端（8001）和客户端（80）都能使用，但 hystrix 服务降级一般放在 客户端**
1. **yaml 配置文件**

    ```YAML
    server:
      port: 80
    eureka:
      client:
        register-with-eureka: false
        service-url:
          defaultZone: http://eureka7001.com:7001/eureka/
    # 在Feign中开启Hystrix
    feign:
      hystrix:
        enabled: true
    ```
2. **主启动类：****`@EnableHystrix`****，与 8001 启动类的注解不用**

    ```Java
    @SpringBootApplication
    @EnableFeignClients
    @EnableHystrix //开启 Hystrix
    public class OrderHystrixMain80 {
        public static void main(String[] args) {
            SpringApplication.run(OrderHystrixMain80.class,args);
        }
    }
    ```
3. **Controller：****`@HystrixCommand`**
    > 注意：
    >
    > 1. 当80 调用 8001 时：服务要跑3秒，即使8001容忍5秒，但调用服务的消费者80 只能容忍2秒，故80调用8001依旧会报错
    >
    > 2. 服务提供端 8001 和 服务消费端 80 都配了服务降级，且业务执行时间都超过了，8001 和 80 设置的容忍时间，那么当访问超时时，会优先执行80端的服务降级兜底方法

    ```Java
    @RestController
    @Slf4j
    public class OrderHystirxController {
        @Resource
        private PaymentHystrixService paymentHystrixService;
        //⭐声明当下面的方法超时或异常时，应该怎么办
        @GetMapping("/consumer/payment/hystrix/timeout/{id}")
        @HystrixCommand(fallbackMethod = "paymentTimeOutFallbackMethod",commandProperties = {
                //1.5秒以内就是正常逻辑，如果超了1.5秒就会去执行兜底方法
                @HystrixProperty(name="execution.isolation.thread.timeoutInMilliseconds",value="1500")
        })
        //原方法：
        public String paymentInfo_TimeOut(@PathVariable("id") Integer id) {
            String result = paymentHystrixService.paymentInfo_TimeOut(id);
            return result;
        }
        //⭐当上面的方法出现超时或异常时，就执行这个兜底方法：
        public String paymentTimeOutFallbackMethod(@PathVariable("id") Integer id){
            return "当前执行了服务消费端的超时兜底：对方支付系统繁忙，响应状态：o(╥﹏╥)o";
        }
    }
    ```
4. **测试：**
    1. [http://localhost/consumer/payment/hystrix/timeout/31](http://localhost/consumer/payment/hystrix/timeout/31)

    ![[IMG-20260404031839344.png|800]]

    2. **上面的是超时故障**，⭐如果此时直接将 8001 停掉，**模拟服务宕机**

# 三、服务降级的代码优化
## 1. 优化 1：分离原方法与降级方法的代码
1. 解决思路：
    1. 之前 80 controller 中所有的方法，其实都是需要去调用 Feign 接口里的方法，才能实现调用 8001 服务；==**那么还不如将兜底处理的逻辑，直接写在 Feign 接口中**==
2. 解决方法：
    1. **根重新新建一个类（PaymentFallbackService）实现 PaymentHystrixService 接口（Feign 接口），统一为接口里面的方法进行异常处理**

    ```Java
    @Component //必须加
    public class PaymentFallbackService implements PaymentHystrixService {
        @Override
        public String paymentInfo_OK(Integer id) {
            return "当前执行了服务消费端解耦合之后的超时兜底：对方支付系统繁忙，响应状态：o(╥﹏╥)o";
        }
        @Override
        public String paymentInfo_TimeOut(Integer id) {
            return "当前执行了服务消费端解耦合之后的超时兜底：对方支付系统繁忙，响应状态：o(╥﹏╥)o";
        }
    }
    ```
    2. **在原 Feign 接口 `@FeignClient` 注解中设置 `fallback` 属性：**
        - fallback：
            - 定义容错的处理类，当调用远程接口失败或超时时，会调用对应接口的容错逻辑
            - fallback 指定的类必须实现 @FeignClient 标记的接口

    ```Java
    @Component
    //标明当Feign 接口中的方法，执行出错去哪里找兜底的方法
    @FeignClient(value = "CLOUD-PROVIDER-HYSTRIX-PAYMENT",fallback = PaymentFallbackService.class)
    public interface PaymentHystrixService {
        @GetMapping(&quot;/payment/hystrix/ok/{id}&quot;)
        String paymentInfo_OK(@PathVariable(&quot;id&quot;) Integer id);
        @GetMapping(&quot;/payment/hystrix/timeout/{id}&quot;)
        String paymentInfo_TimeOut(@PathVariable(&quot;id&quot;) Integer id);
    }
    ```
3. 测试：
    1. 将OrderHystirxController中的 @HystrixCommand 注解和指定的兜底方法（解耦之前）注释掉，以测试解耦之后的都的方法
    2. 启动 8001、80
    3. [http://localhost/consumer/payment/hystrix/timeout/31](http://localhost/consumer/payment/hystrix/timeout/31)

    ![[IMG-20260405035422111.png|800]]

## 2. 优化 2：添加全局默认的降级方法
1. 解决思路：==**指定一个默认的兜底方法，如果没明确注明具体的兜底方法，那就执行这个默认的**==
2. 解决方法：
    1. 在 OrderHystirxController 上标注 `@DefaultProperties`

    ```Java
    @DefaultProperties(defaultFallback = "payment_Global_FallbackMethod")
    ```
    2. 同时在 OrderHystirxController 里的想要添加服务降级的方法上都添加 注解 `@HystrixCommand`

    ```Java
    @HystrixCommand //加了@HystrixCommand属性注解，但并且没有写具体自定义降级方法名字的，就用统一全局的
    public String paymentInfo_TimeOut(@PathVariable("id") Integer id) {
        String result = paymentHystrixService.paymentInfo_TimeOut(id);
        return result;
    }
    //添加全局默认兜底方法
    public String payment_Global_FallbackMethod() {
        return "当前执行了服务消费端的 默认全局超时兜底：对方支付系统繁忙，响应状态：o(╥﹏╥)o\"";
    }
    ```
