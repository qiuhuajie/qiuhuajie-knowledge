# 一、介绍
1. 限流，也称流量控制
    1. 是指系统在面临**高并发**，或者**大流量请求**的情况下，**限制新的请求对系统的访问**，来避免瞬时的大量请求对系统造成负荷，从而**保证系统的稳定性**
    2. 限流会导致部分用户请求处理不及时或者被拒，这就影响了用户体验。所以一般需要在系统稳定和用户体验之间**平衡**一下
    3. 它可防止 DoS 攻击和限制 Web 爬虫
2. ==限流只是保证系统稳定性中的一个环节，通常会将它和熔断、降级搭配在一起使用==
# 二、常见的限流算法
> 💡 算法选型
> - 漏桶和令牌桶其实比较适合**阻塞式限流**场景，即没令牌我就等着，这样就不会误杀了，而漏桶本就是等着，比较适合后台任务类的限流
> - 而基于时间窗口的限流比较适合**对时间敏感**的场景，请求过不了就快点儿告诉我
## 1. 固定窗口算法
1. 首先**维护一个计数器，将单位时间段当做一个窗口**，计数器记录这个窗口接收请求的次数
    - 当次数少于限流阀值，就允许访问，并且计数器+1
    - 当次数大于限流阀值，就拒绝访问
    - 当前的时间窗口过去之后，计数器清零
2. 假设单位时间是1秒，限流阀值为3
    1. 在单位时间1秒内，每来一个请求，计数器就加1，如果计数器累加的次数超过限流阀值3，后续的请求全部拒绝
    2. 等到1s结束后，计数器清0，重新开始计数
3. 如下图

    ![[IMG-20260405035413903.png|800]]
- 算法实现

    ```Java
    @Slf4j
    public class FixedWindowRateLimiter {
        // 时间窗口大小，单位毫秒
        private long windowSize;
        // 允许通过请求数
        private int maxRequestCount;
        // 当前窗口通过的请求计数
        private AtomicInteger count = new AtomicInteger(0);
        // 窗口右边界
        private long windowBorder;
        public FixedWindowRateLimiter(long windowSize,int maxRequestCount){
            this.windowSize = windowSize;
            this.maxRequestCount = maxRequestCount;
            windowBorder = System.currentTimeMillis()+windowSize;
        }
        public synchronized boolean tryAcquire(){
            long currentTime = System.currentTimeMillis();
            if (windowBorder < currentTime){
                log.info("window  reset");
                do {
                    windowBorder += windowSize;
                }while(windowBorder < currentTime);
    						// 计数器置零
                count = new AtomicInteger(0);
            }
            if (count.intValue() < maxRequestCount){
                count.incrementAndGet();
                log.info("tryAcquire success");
                return true;
            }else {
                log.info("tryAcquire fail");
                return false;
            }
        }
    }
    ```
1. 存在的问题：
    1. 这种算法有一个很明显的==**临界问题**==
    2. 假设限流阀值为5个请求，单位时间窗口是1s，如果我们在单位时间内的前 `0.8-1s` 和 `1-1.2s`，分别并发5个请求
    3. 虽然都没有超过阀值，但是如果算 `0.8-1.2s`，则并发数高达10，已经**超过单位时间 1s 不超过 5 阀值**的定义了

## 2. 滑动窗口限流算法
1. 滑动窗口限流解决固定窗口临界值的问题
2. 它将单位时间周期分为 n 个小周期，分别记录每个小周期内接口的访问次数，并且根据时间滑动删除过期的小周期
3. 滑动窗口算法规则：

    ![[IMG-20260405035420311.png|800]]
    1. 假设单位时间还是1s，滑动窗口算法把它划分为 5 个小周期，也就是滑动窗口（单位时间）被划分为 5 个小格子。每格表示 0.2s
    2. 每过 0.2 s，时间窗口就会往右滑动一格
    3. 然后呢，每个小周期，都有自己独立的计数器，如果请求是 0.83s 到达的，0.8~1.0s 对应的计数器就会加 1
4. 下滑动窗口是如何解决临界问题的？
    1. 假设我们 1s 内的限流阀值还是5 个请求，0.8~1.0s 内（比如0.9s的时候）来了5个请求，落在黄色格子里。时间过了1.0s这个点之后，又来5个请求，落在紫色格子里
    2. 如果**是固定窗口算法，是不会被限流的**
    3. 但是**滑动窗口的话，每过一个小周期，它会右移一个小格**。也就是过了 1.0s 这个点后，会右移一小格，当前的单位时间段是0.2~1.2s，这个区域的请求已经超过限定的5了，已触发限流啦，实际上，紫色格子的请求都被拒绝了
    > 💡 当滑动窗口的格子周期划分的越多，那么滑动窗口的滚动就越平滑，限流的统计就会越。
5. 存在的问题
    1. 在应对==**突发流量**==的情况时，一旦到达限流后，请求都会直接暴力被拒绝
    2. 但==**直接暴力拒绝不太友好，**==可能会损失一大部分请求
- 算法代码实现

    ```Java
    @Slf4j
    public class SlidingWindowRateLimiter {
        // 时间窗口大小，单位毫秒
        private long windowSize;
        // 分片窗口数
        private int shardNum;
        // 允许通过请求数
        private int maxRequestCount;
        // 各个窗口内请求计数
        private int[] shardRequestCount;
        // 请求总数
        private int totalCount;
        // 当前窗口下标
        private int shardId;
        // 每个小窗口大小，毫秒
        private long tinyWindowSize;
        // 窗口右边界
        private long windowBorder;
        public SlidingWindowRateLimiter(long windowSize, int shardNum, int maxRequestCount) {
            this.windowSize = windowSize;
            this.shardNum = shardNum;
            this.maxRequestCount = maxRequestCount;
            shardRequestCount = new int[shardNum];
            tinyWindowSize = windowSize/ shardNum;
            windowBorder=System.currentTimeMillis();
        }
        public synchronized boolean tryAcquire() {
            long currentTime = System.currentTimeMillis();
            if (currentTime > windowBorder){
                do {
                    shardId = (++shardId) % shardNum;
                    totalCount -= shardRequestCount[shardId];
                    shardRequestCount[shardId]=0;
                    windowBorder += tinyWindowSize;
                }while (windowBorder < currentTime);
            }
            if (totalCount < maxRequestCount){
                log.info("tryAcquire success,{}",shardId);
                shardRequestCount[shardId]++;
                totalCount++;
                return true;
            }else{
                log.info("tryAcquire fail,{}",shardId);
                return false;
            }
        }
    }
    ```
## 3. 漏桶算法
1. 漏桶算法面对限流，就更加的柔性，不存在直接的粗暴拒绝
    1. 流量整形：无论请求多少，请求的速率有多大，都按照固定的速率流出，对应的就是服务按照固定的速率处理请求
    2. 它的特点就是**宽进严出**
2. 它的原理很简单，可以认为就是**注水漏水**的过程

    ![[IMG-20260405035422022.png|800]]
    1. 往漏桶中以任意速率流入水
    2. 以固定的速率流出水
    3. 当水超过桶的容量时，会被溢出，也就是被丢弃。因为桶容量是不变的，保证了整体的速率
3. 这种思想的其他应用：
    1. [[ThreadPoolExecutor]]
    2. [[1.1 同步调用与异步调用]]

    ![[IMG-20260405035426859.png|800]]
4. 存在的问题
    1. ==**不管当前系统的负载压力如何，所有请求都得进行排队**==
    2. 即使此时服务器的负载处于相对空闲的状态，这样会造成系统资源的浪费
5. 应用：
    1. 阿里开源的限流框架 `Sentinel` 中的匀速排队限流策略，就采用了漏桶算法
    2. Nginx 中的限流模块 `limit_req_zone`，采用了漏桶算法
    3. 还有 OpenResty 中的 `resty.limit.req` 库

## 4. 令牌桶算法
1. 面对**突发流量**的时候，可以使用令牌桶算法限流
2. **令牌桶算法原理**：

    ![[IMG-20260405035426884.png|800]]
    - 有一个令牌管理员，根据限流大小，定速往令牌桶里放令牌
    - 如果令牌数量满了，超过令牌桶容量的限制，那就丢弃
    - 系统在接受到一个用户请求时，都会先去令牌桶要一个令牌
        - 如果拿到令牌，那么就处理这个请求的业务逻辑
        - 如果拿不到令牌，就直接拒绝这个请求
    - 算法代码实现

    ```Java
    public class TokenBucket {
      public long timeStamp = System.currentTimeMillis();  // 当前时间
      public long capacity; // 桶的容量
      public long rate; // 令牌放入速度
      public long tokens; // 当前令牌数量
      public boolean grant() {
        long now = System.currentTimeMillis();
        // 先添加令牌
        tokens = Math.min(capacity, tokens + (now - timeStamp) * rate);
        timeStamp = now;
        if (tokens < 1) {
          // 若不到1个令牌,则拒绝
          return false;
        } else {
          // 还有令牌，领取令牌
          tokens -= 1;
          return true;
        }
      }
    }
    ```
3. 这样即使瞬间大量请求进入，只有令牌桶里有足够的令牌，或者有一部分令牌，那么这些请求就无需再排队等待，可以在短时间内响应
    > 💡 举例
    > - 可以看出令牌桶在应对突发流量的时候
    > - 桶内假如有**预存的 100 个令牌**，那么这 100 个令牌可以马上被取走，而不像漏桶那样匀速的消费
    > - 所以在==**应对**====**突发流量**====**的时候，令牌桶表现的更佳**==
4. 类比：[[Semaphore 工具锁]]
5. 缺点：
    1. ==存在一定的误杀：==请求的访问其实是随机到达的，假设令牌桶每20ms放入一个令牌，桶内初始没令牌，这请求就刚好在第一个20ms内有两个请求，再过20ms里面没请求。但其实从40ms来看，只有2个请求，应该都放行的，但现在有一个请求就直接被拒了
    2. ==需要预热==：假设没预热，上线时候桶里没令牌，请求过来就直接拒了
6. 应用
    1. Guava 中的 `RateLimiter` 限流
    2. Spring Cloud的 `gateway` 限流[[Gateway 项目实战]]
    3. Sentinel 在 `WarmUpController` 中运用到了令牌桶算法，在这里可以实现对系统的预热，设定预热时间和水位线，对于预热期间多余的请求直接拒绝掉

    ```Java
    public boolean canPass(Node node, int acquireCount, boolean prioritized) {
      long passQps = (long) node.passQps();
      long previousQps = (long) node.previousPassQps();
      syncToken(previousQps);
      // 开始计算它的斜率
      // 如果进入了警戒线，开始调整他的qps
      long restToken = storedTokens.get();
      if (restToken >= warningToken) {
        long aboveToken = restToken - warningToken;
        // 消耗的速度要比warning快，但是要比慢
        // current interval = restToken*slope+1/count
        double warningQps = Math.nextUp(1.0 / (aboveToken * slope + 1.0 / count));
        if (passQps + acquireCount <= warningQps) {
          return true;
        }
      } else {
        if (passQps + acquireCount <= count) {
          return true;
        }
      }
      return false;
    }
    ```
# 三、服务调用方如何对接口进行限流保护
当然，在客户端对调用接口进行限流是保护下游服务、避免自身资源被上游打垮（例如收到 `429 Too Many Requests` 错误）的一种非常重要的“防御性”编程实践。这也被称为“出口限流”或“调用方限流”。
在Java中实现客户端限流，主要有以下几种主流且高效的方式。
##\#核心思想
客户端限流的核心思想是在发起实际的HTTP请求之前，先通过一个“关卡”。这个关卡会根据预设的规则（如每秒允许通过N个请求）来决定是立即放行、等待一段时间后放行，还是直接拒绝。
最常用的算法是 **令牌桶（Token Bucket）** 算法：
1. 系统以一个恒定的速率往桶里放入令牌。
2. 如果桶满了，多余的令牌会被丢弃。
3. 每次客户端要发起请求时，必须先从桶里获取一个令牌。
4. 如果桶里有令牌，则取走一个，请求被放行。
5. 如果桶里没有令牌，请求必须等待，直到有新的令牌被放入，或者直接被拒绝。
这种算法既能限制平均速率，也允许一定程度的突发流量（桶里积攒的令牌数）。

## 1. 实现方案
下面介绍三种在Java中非常流行的实现方案，从简单到功能强大。
## 2. 方案一：Google Guava `RateLimiter` (最经典、简单)
Guava库中的`RateLimiter`是实现单机限流最简单、最经典的选择。它基于令牌桶算法。
**优点：**
- API极其简单，易于上手。
- 稳定、高效，经过Google大规模应用验证。
**缺点：**
- 只适用于单个JVM实例（单机限流），不适用于分布式环境。
- 功能相对基础。
**1. 添加依赖 (Maven):**
```XML
<dependency>
    <groupId>com.google.guava</groupId>
    <artifactId>guava</artifactId>
    <version>32.1.2-jre</version>
</dependency>
```
**2. 代码示例:**
假设我们希望限制调用某个API的速率为**每秒2次 (2 QPS)**。
```Java
import com.google.common.util.concurrent.RateLimiter;
import java.time.LocalDateTime;
import java.time.format.DateTimeFormatter;
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;
public class GuavaRateLimiterDemo {
    // 创建一个RateLimiter，设置为每秒生成2个令牌
    private static final RateLimiter rateLimiter = RateLimiter.create(2.0);
    public static void callApi(int taskId) {
        // acquire()会阻塞，直到获取到令牌
        double waitTime = rateLimiter.acquire();
        System.out.printf("[%s] Task-%d 准备调用API, 等待了 %.2f 秒%n",
                getCurrentTime(), taskId, waitTime);
        // 模拟调用API
        // System.out.println("... Calling downstream service ...");
    }
    public static void main(String[] args) {
        ExecutorService executor = Executors.newFixedThreadPool(10);
        for (int i = 0; i < 10; i++) {
            final int taskId = i;
            executor.submit(() -> callApi(taskId));
        }
        executor.shutdown();
    }
    private static String getCurrentTime() {
        return LocalDateTime.now().format(DateTimeFormatter.ofPattern("HH:mm:ss.SSS"));
    }
}
```
**输出分析:**
你会看到，尽管10个任务是并发提交的，但它们的执行时间被均匀地分开了，大约每隔0.5秒执行一次，严格遵守了2 QPS的限制。`acquire()`方法的返回值是线程为了获取令牌而等待的时间（秒）。
## 3. 方案二：Resilience4j `RateLimiter` (功能强大，推荐在Spring生态中使用)
Resilience4j是一个轻量级的容错库，除了限流，还提供了断路器、重试、舱壁隔离等功能。它的`RateLimiter`功能比Guava更强大。
**优点：**
- 配置灵活，可以动态修改限流配置。
- 完美融入函数式编程风格。
- 提供事件监听，可以监控到限流事件（成功、失败）。
- 与Spring Boot有非常好的集成。
**缺点：**
- 比Guava稍复杂。
- 原生同样是单机限流。
**1. 添加依赖 (Maven):**
```XML
<dependency>
    <groupId>io.github.resilience4j</groupId>
    <artifactId>resilience4j-ratelimiter</artifactId>
    <version>2.2.0</version>
</dependency>
```
**2. 代码示例:**
```Java
import io.github.resilience4j.ratelimiter.RateLimiter;
import io.github.resilience4j.ratelimiter.RateLimiterConfig;
import io.github.resilience4j.ratelimiter.RateLimiterRegistry;
import java.time.Duration;
import java.time.LocalDateTime;
import java.time.format.DateTimeFormatter;
import java.util.function.Supplier;
public class Resilience4jRateLimiterDemo {
    public static void main(String[] args) {
        // 1. 创建限流器配置
        RateLimiterConfig config = RateLimiterConfig.custom()
                // 刷新周期：1秒
                .limitRefreshPeriod(Duration.ofSeconds(1))
                // 周期内允许的请求数：2
                .limitForPeriod(2)
                // 获取令牌的超时等待时间：0秒（获取不到立即失败）
                .timeoutDuration(Duration.ofSeconds(0))
                .build();
        // 2. 创建限流器注册中心
        RateLimiterRegistry registry = RateLimiterRegistry.of(config);
        // 3. 从注册中心获取一个限流器实例
        RateLimiter rateLimiter = registry.rateLimiter("apiCaller");
        // 模拟要被限流的业务逻辑
        Supplier<String> restrictedSupplier = RateLimiter.decorateSupplier(rateLimiter, () -> {
            // 模拟调用API
            return String.format("[%s] API Call Successful!", getCurrentTime());
        });
        // 模拟并发调用
        for (int i = 0; i < 5; i++) {
            new Thread(() -> {
                try {
                    String result = restrictedSupplier.get();
                    System.out.println(result);
                } catch (io.github.resilience4j.ratelimiter.RequestNotPermitted e) {
                    System.err.printf("[%s] API Call Blocked by Rate Limiter.%n", getCurrentTime());
                }
            }).start();
        }
    }
    private static String getCurrentTime() {
        return LocalDateTime.now().format(DateTimeFormatter.ofPattern("HH:mm:ss.SSS"));
    }
}
```
**输出分析:**
你会看到前两次调用成功，后续的调用因为在1秒周期内超出了2次的限制而被立即拒绝，并抛出`RequestNotPermitted`异常。这就是`timeoutDuration`设为0的效果。如果把它设为正数，线程则会等待。
## 4. 方案三：Bucket4j (高性能，支持分布式)
Bucket4j是一个专门为限流设计的库，性能非常高，并且其最大亮点是原生支持分布式限流（使用Redis, Hazelcast, Ignite等作为后端存储）。
**优点：**
- 性能极高。
- API灵活，支持非常复杂的限流策略（如多带宽）。
- **核心优势：** 原生支持分布式限流，是多客户端实例场景下的首选。
**1. 添加依赖 (Maven for local):**
```XML
<dependency>
    <groupId>com.bucket4j</groupId>
    <artifactId>bucket4j-core</artifactId>
    <version>8.6.0</version>
</dependency>
```
_(如果用Redis，则需要添加 `bucket4j-redis` 依赖)_
**2. 代码示例 (单机版):**
```Java
import io.github.bucket4j.Bandwidth;
import io.github.bucket4j.Bucket;
import io.github.bucket4j.Refill;
import java.time.Duration;
public class Bucket4jDemo {
    public static void main(String[] args) throws InterruptedException {
        // 定义带宽：10秒内允许5个请求
        Refill refill = Refill.intervally(5, Duration.ofSeconds(10));
        Bandwidth limit = Bandwidth.classic(5, refill);
        // 创建一个本地（内存中）的桶
        final Bucket bucket = Bucket.builder()
                .addLimit(limit)
                .build();
        // 模拟连续请求
        for (int i = 1; i <= 10; i++) {
            if (bucket.tryConsume(1)) {
                System.out.println("第 " + i + " 次请求: 成功!");
            } else {
                System.out.println("第 " + i + " 次请求: 被限流!");
            }
            Thread.sleep(500); // 模拟请求间隔
        }
    }
}
```
**输出分析:**
前5次请求会成功，因为桶的初始容量是5。随后的请求会被限流，直到时间流逝，桶内有新的令牌被补充进来。

## 5. 如何与常用HTTP客户端结合 (以`RestTemplate`为例)
在实际项目中，我们不会在每个调用点都写限流代码，而是通过**拦截器（Interceptor）** 来统一处理。
这是一个使用`RestTemplate`和Guava `RateLimiter`的例子：
**1. 创建一个限流拦截器:**
```Java
import com.google.common.util.concurrent.RateLimiter;
import org.springframework.http.HttpRequest;
import org.springframework.http.client.ClientHttpRequestExecution;
import org.springframework.http.client.ClientHttpRequestInterceptor;
import org.springframework.http.client.ClientHttpResponse;
import java.io.IOException;
public class RateLimitingInterceptor implements ClientHttpRequestInterceptor {
    private final RateLimiter rateLimiter;
    public RateLimitingInterceptor(double permitsPerSecond) {
        this.rateLimiter = RateLimiter.create(permitsPerSecond);
    }
    @Override
    public ClientHttpResponse intercept(HttpRequest request, byte[] body, ClientHttpRequestExecution execution) throws IOException {
        // 在执行请求前，先获取令牌
        rateLimiter.acquire();
        // 获取到令牌后，执行原始请求
        return execution.execute(request, body);
    }
}
```
**2. 将拦截器配置到****`RestTemplate`****:**
```Java
import org.springframework.boot.web.client.RestTemplateBuilder;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.web.client.RestTemplate;
@Configuration
public class RestTemplateConfig {
    @Bean("downstreamServiceTemplate")
    public RestTemplate downstreamServiceTemplate(RestTemplateBuilder builder) {
        return builder
                // 添加我们的限流拦截器，限制为5 QPS
                .additionalInterceptors(new RateLimitingInterceptor(5.0))
                .build();
    }
}
```
**3. 使用:**
现在，只要是通过注入的`downstreamServiceTemplate`发出的任何请求，都会自动被5 QPS的速率限制。
```Java
@Service
public class MyApiClient {
    @Autowired
    @Qualifier("downstreamServiceTemplate")
    private RestTemplate restTemplate;
    public String fetchData() {
        // 这个调用会自动被限流
        return restTemplate.getForObject("<http://downstream-service/api/data>", String.class);
    }
}
```
对于`WebClient`（响应式编程），可以使用`ExchangeFilterFunction`实现类似的功能。

## 6. 总结与最佳实践
1. **选择合适的工具**：
    - **单机简单场景**：`Guava RateLimiter` 足矣。
    - **Spring生态或需要更复杂配置**：`Resilience4j` 是更好的选择，还能整合其他容错模式。
    - **分布式/集群环境**：**必须**使用`Bucket4j` + Redis等外部存储，或者其他分布式限流方案，否则每个客户端实例都会有自己的一套限流器，总速率会是 `实例数 * 单机速率`，起不到保护下游的作用。
2. **限流速率可配置**：不要硬编码速率值（如`2.0`）。应将其放在配置文件中（如`application.properties`），方便在不同环境下调整。
3. **结合其他容错模式**：客户端限流通常与**断路器（Circuit Breaker）** 和**重试（Retry）** 结合使用。
    - **限流**：防止正常情况下流量过大。
    - **断路器**：当下游服务故障（大量超时或错误）时，快速失败，停止发送请求，保护客户端自身资源，也给下游恢复时间。
    - **重试**：对于偶然的网络抖动或服务临时不可用，可以进行有限次数的重试。
4. **与下游服务团队沟通**：与你要调用的服务的提供方沟通，了解他们能承受的QPS，并以此为依据设置你的限流值。这是一种负责任的合作方式。
