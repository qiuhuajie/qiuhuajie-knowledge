# 一、🙋‍♂️ 为什么选择 SpringCloud Gateway❓
1. **Zuul**
    1. 实现方式比较老式，基于 Servlet IO 实现
        1. ==阻塞式编程==：当请求进入时，每次 I/ O 操作都是从工作线程中选择一个执行，请求线程被阻塞，一直到工作线程完成
        2. 问题：==在高并发下性能性能不佳==
    2. 已经进入了维护阶段
    > ℹ️ Home
    > Zuul is a gateway service that provides dynamic routing, monitoring, resiliency, security, and more.
    > [https://github.com/Netflix/zuul/wiki](https://github.com/Netflix/zuul/wiki)
2. **Spring Cloud Gateway 的优点**
    1. 基于==响应式编程 WebFlux 框架==实现的，框架底层则使用了高性能的 Reactor 异步非阻塞模型通信框架 Netty，能够充分利用 CPU 资源，提高系统的吞吐量和响应性能
    2. 提供了==灵活的路由规则==，可以根据请求的 URI、HTTP 方法、请求头、请求参数等来路由请求
    3. ==支持集成各种服务发现和配置中心==，例如 Consul、Zookeeper、Eureka 等
    4. 还提供了==负载均衡==的能力，可以根据服务实例的健康状态、权重等因素来动态地分配请求到不同的服务实例上
    5. 在请求压力激增时，支持集成==请求限流、断路器==，保护集群

# 二、Gateway 可以做什么

![[IMG-20260405035413888.png|800]]

1. Gateway 是在 Spring 生态系统之上构建的 API 网关服务
2. **网关是介于客户端和服务器端之间的中间层**
    1. 微服务只接收来自网关的请求，而其它直接访问微服务本身的请求拒绝，这样可以极大保护微服务免受不法侵害
    2. ==因此，API 的实现方面更多的考虑业务逻辑，====而安全、性能、监控可以交由 网关来做，这样既提高业务灵活性又不缺安全性==
3. 可以实现
    - 请求路由和负载均衡（前端请求路由到不同的后端服务）
    - 鉴权（URL 过滤，Token 校验）
    - 熔断、流量控制（限流）、重试
    - 日志监控
