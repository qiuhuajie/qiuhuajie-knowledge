# 一、Map 接口概述
1. 双列数据，存储 key-value对 的数据
2. Map 接口有三个实现类：**HashMap、TreeMap、Hashtable**
3. **Map 结构的理解**：
    1. Map中的 **key**：
        1. 无序的、**不可重复的**，使用Set存储所有的key
        2. ==key 所在的类要重写 equals( ) 和 hashCode( )==
    2. Map中的 **value**：
        1. 无序的、**可重复的**，使用Collection存储所有的value
        2. ==value 所在的类要重写 equals( )==
    3. 一个键值对：
        1. ==key-value 构成了一个== ==**`Entry`**== ==**对象**==
        2. Map中的 **Entry**：无序的、**不可重复的**，也使用Set存储所有的entry

# 二、HashTable
- **底层数据结构：**
    - `Hashtable` 底层就是普通数组和链表 结合在一起使用，也就是 链表散列
- **线程是否安全：**
    - `Hashtable` 是**线程安全**的，**因为 `Hashtable` 内部的方法基本都经过 `synchronized` 修饰⭐**
- **效率：**
    - 效率低，因为是线程安全的，`Hashtable` 基本被淘汰，不要在代码中使用它
- **对 Null key 和 Null value 的支持：**
    - `Hashtable` 不允许有 null 键和 null 值，否则会抛出 `NullPointerException`
- **初始容量大小和每次扩充容量大小的不同 ：**
    - 创建时如果不指定容量初始值
        - `Hashtable` 默认的初始大小为 11，之后每次扩充，容量变为原来的 2n+1
    - 创建时如果给定了容量初始值
        - `Hashtable` 会直接使用你给定的大小

# 三、TreeMap ⭐
- **底层数据结构：**
    - `TreeMap` 的底层是红黑树
- **线程是否安全：**
    - `TreeMap` 和 `HashMap` 都是非线程安全的，如果你要保证线程安全的话就使用 `ConcurrentHashMap`
- **效率：**
    - `TreeMap` 比 `HashMap` 的性能低
    - 因为 `TreeMap` 主要是保证数据平衡
- **对 Null key 和 Null value 的支持：**
    - `TreeMap` 不允许有 null 键，但可以有多个 null 值
- **使用场景：**
    - 在需要按自然顺序或自定义顺序遍历键的情况下
- **相比于** `HashMap` **来说** `TreeMap` **还多了对集合中的元素根据键排序的能力以及对集合内元素的搜索的能力**
    - `TreeMap`它还实现了`NavigableMap`接口，而`NavigableMap`接口继承自`SortedMap` 接口

        - 实现 **`NavigableMap` 接口**让 `TreeMap` 有了对集合内元素的==**搜索的能力**==（Navigable：可驾驶的，可航行的）
        - 实现 **`SortedMap` 接口**让 `TreeMap` 有了对集合中的元素==**根据键排序的能力**==

# 四、HashMap ⭐
## 1. 介绍
- **底层数据结构：**
    - JDK1.8 以后的 `HashMap` 在解决哈希冲突时有了较大的变化，当链表长度大于阈值（默认为 **`8`**）时，将链表转化为红黑树），以减少搜索时间
- **线程是否安全：**
    - `HashMap` 是==非线程安全==的，如果你要保证线程安全的话就使用 `ConcurrentHashMap`
    - 如果既要保证线程安全又要保证顺序，可以使用 `Collections.synchronizedMap()` 方法转化为线程安全的集合
- **效率：**
    - 因为线程安全的问题，`HashMap` 要比 `Hashtable` 效率高一点
- **对 Null key 和 Null value 的支持：** （[[ConcurrentHashMap]]）
    - `HashMap` 可以存储 null 的 key 和 value
    - 但 null 作为键只能有一个，null 作为值可以有多个
- **初始容量大小和每次扩充容量大小 ：**
    - 创建时如果不指定容量初始值
        - `HashMap` 默认的初始化大小为 16，之后每次扩充，容量变为原来的 2 倍
    - 创建时如果给定了容量初始值
        - `HashMap` 会将其扩充为 2 的幂次方大小，也就是说 `HashMap` 总是使用 2 的幂作为哈希表的大小
- **使用场景：**
    - 在 Map 中插入、删除和定位元素时

## 2. HashMap 的底层实现 ⭐
### 2.1 JDK1.8 之前
1. JDK1.8 之前 `HashMap` 底层是 **数组和链表** 结合在一起使用，也就是 **链表散列**

    ![[IMG-20260620221528878.png|562]]

2. 这种结构设计的原因：**使用链地址法来解决哈希冲突**
3. ==**put() 插入过程：**==**⭐**
    1. 先将 key 经过 `hash()` 得到哈希值 `hash`

    ![[IMG-20260620221528971.png|505]]

    2. 然后通过 `hash & (n - 1)` 得到当前元素存放的位置
        1. 如果当前位置不存在元素，则生成一个 Entry 和链表，挂在当前位置（多线程并发环境下，会发数据覆盖，[[ConcurrentHashMap]] ）
        2. 如果当前位置存在元素的话，就判断以该元素为头结点的链表上，是否存在与要存入的元素的 `hash` 值以及 `key` 相同的 Entry
            1. 如果能找到满足条件的 Entry 的话，直接覆盖更新 value
            2. 找不到，则将新的 Entry 对插入到链表头部（多线程并发环境下，会发生死循环，[[ConcurrentHashMap]] ）

    ![[IMG-20260620221528971.png|555]]

### 2.2 JDK1.8 之后
1. 首先，新的 key-value 对会**插入到链表尾部**
2. 其次，相比于之前的版本， JDK1.8 之后的变化
    1. **当 数组的长度大于** **`64`** **&& 链表长度大于等于阈值（默认为 `8`）时，将链表转化为红黑树，以减少搜索时间，可以将时间复杂度从 O(n) 降低到 O(log n)**
    2. **一旦链表的长度小于** **`6`** **，红黑树会退化成链表**

    ![[IMG-20260620221529021.png|625]]

    > 💡 **红黑树**
    > 1. TreeMap、TreeSet 以及 JDK 1.8 下的 HashMap 都使用了红黑树
    > 2. ==红黑树就是====**为了解决二叉查找树的缺陷，因为二叉查找树在某些情况下会退化成一个线性结构**==
    > 💡 **为什么是 8 和 6 ❓**
    > - 首先元素在桶中的分布遵循泊松分布，链表长度达到 8 个元素的概率为 0.00000006，几乎是不可能事件
    > - 再从平均查找长度来看
    >     - **红黑树的平均查找长度是 logn**，如果长度为 8，则 **log8=3**，而**链表的平均查找长度为 n/2**，长度为 8 时，**8/2=4**，所以阈值 8 能大大提高搜索速度
    >     - 当长度为 6 时红黑树退化为链表是因为 **log6 约等于 2.6**，而 **6/2 = 3**，虽然为 6 时红黑树也比链表快，但红黑树节点占用更多的内存空间，所以此时退化回去更好
    > - 至于为什么转化为红黑树的阈值 8 和退化为链表的阈值 6 不一样，是为了避免频繁来回转化
    > 💡 **解决哈希冲突的方法**
    > 1. **开放地址法：**
    >     2. 冲突时，直接在哈希表中寻找下一个空位置并插入该键值对
    >     3. 不会生成链表，可以提高哈希表的性能；缺点：当数组大小不够时，无法扩容
    >     4. 常见的开放地址法有线性探测和二次探测等方法
    > 5. **链地址法：**
    >     6. 冲突时，将其插入到该位置的链表中，形成一个键值对列表
    >     7. 这种方法能够有效地解决哈希冲突问题，并且允许扩容；缺点：额外的内存开销
    > 8. **再哈希法：**
    >     9. 冲突时，使用第二个、第三个、哈希函数计算地址，直到无冲突时
    >     10. 缺点：计算时间增加
## 3. HashMap 的长度为什么是 2 的幂次方🙋‍♂️
1. Hash 值的范围值是一个 `40` 亿的映射空间
2. 所以 Hash 值是不能直接拿来用的，**用之前还要先做对数组的长度取模运算，得到的余数才能得到在数组中要存放的位置**
    1. 我们首先可能会想到采用 `%` 取余的操作来实现
    2. 由于，**采用二进制位操作 `&`，相对于 `%` 能够提高运算效率**
    3. 如何转化：取余（`%`）操作中，如果除数是 2 的幂次，则等价于：与其除数减一的与（`&`）操作
    4. **也就是说 `hash % n == hash & (n - 1)` 的前提是： `n` 是 2 的幂次方**

## 4. HashMap 的扩容机制🙋‍♂️⭐
1. 首先需要明确几个概念：
    - capacity **容量**：默认16
    - loadFactor **负载因子**：默认是 0.75
    - threshold **阈值**：阈值=容量*负载因子，默认 12
2. ==**当元素数量超过阈值时便会触发扩容，每次扩容的容量都是之前容量的 2 倍**==
    1. 也即，第 1 次扩容的动作会在元素个数达到 12 的时候触发，扩容的大小是原来的 2 倍
        1. 新容量 = 旧容量 x 2，再计算新的阈值
        2. HashMap的最大容量是 `Integer.MAX_VALUE` 也就是 2的 31 次方减1
    2. 基本的扩容逻辑就是==新建一个更长的数据，然后把原来数组里面的数据重新== **`rehash`** ==到新的数组里面就可以了==
3. 🙋‍♂️**HashMap 扩容时，为什么要扩容至原来的 2 倍**❓
    1. 一方面：容量扩大一倍可以降低负载因子，==使得桶中的元素更加均匀，减少哈希碰撞的概率==，从而提高数据的查找效率
    2. 另一方面：由于 HashMap 的==长度是 2 的幂次方时==，计算索引位置时，可以使用 `&` 来==提高运算性能==（见上）
4. 🙋‍♂️**为什么负载因子要设置为** **`0.75`** **❓**
    1. 负载因子的值越大，如 1.0 ，会导致触发扩容时的元素个数就越多，Hash 冲突的概率也会增加
    2. 负载因子的值越小，如 0.5 ，又会导致内存空间利用率降低，而且还会增加扩容的频率
    3. 因此，扩容因子的值的设置为 `0.75`，空间利用率较高，同时避免了很多哈希冲突，底层的链表或红黑树高度都比较低
    4. ==本质上就是一个冲突的概率以及空间利用率之间的一个平衡。和统计学里面的泊松分布有关系==
5. 🙋‍♂️**HashMap 的扩容机制的扩容机制是线程不安全的（**[[ConcurrentHashMap]] **）**
