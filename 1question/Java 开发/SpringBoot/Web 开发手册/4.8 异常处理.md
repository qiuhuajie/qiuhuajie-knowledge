# 二、异常处理自动配置原理
1. **ErrorMvcAutoConfiguration** 中自动配置异常处理规则
2. 配置类中注册的组件:
    1. **DefaultErrorAttributes**：定义错误页面中可以包含哪些数据，也是系统默认的一个 异常处理解析器

    ```Java
     public DefaultErrorAttributes errorAttributes() {
         return new DefaultErrorAttributes();
     }
    ```
    ```Java
     //DefaultErrorAttributes.class
     if (!options.isIncluded(Include.EXCEPTION)) {
         errorAttributes.remove("exception");
     }
     if (!options.isIncluded(Include.STACK_TRACE)) {
         errorAttributes.remove("trace");
     }
     if (!options.isIncluded(Include.MESSAGE) && errorAttributes.get("message") != null) {
         errorAttributes.remove("message");
     }
     if (!options.isIncluded(Include.BINDING_ERRORS)) {
         errorAttributes.remove("errors");
     }
    ```
    2. **BasicErrorController**：

    ```Java
     public BasicErrorController basicErrorController(ErrorAttributes errorAttributes, ObjectProvider<ErrorViewResolver> errorViewResolvers) {
         return new BasicErrorController(errorAttributes, this.serverProperties.getError(), (List)errorViewResolvers.orderedStream().collect(Collectors.toList()));
     }
    ```
        1. **这个组件可以处理：默认 /error 路径的请求**，用于 异常处理解析器 都无法解析后，发送 /error 来返回一个错误页 或 JSON 对象

    ```Java
     @RequestMapping({"${server.error.path:${error.path:/error}}"})
    ```
        2. BasicErrorController 中的响应：
            1. 浏览器访问错误时：返回一个页面

    ```Java
     public ModelAndView errorHtml(...) {
         ...
         //将状态码，数据等封装成一个 ModelAndView
         ModelAndView modelAndView = this.resolveErrorView(request, response,status, model);
         //这里传入 参数 error ，去找到 id=error的 View 组件，并返回
         return modelAndView != null ? modelAndView : new ModelAndView("error", model);
     }
    ```
            2. 机器客户端访问错误时：返回一个JSON对象

    ```Java
     public ResponseEntity<Map<String, Object>> error(...) {
             ...
             return new ResponseEntity(body, status);
     }
    ```
    3. **View**：用于返回 页面，响应默认错误页，id 是 error

    ```Java
     @Bean(name = {"error"})
     @ConditionalOnMissingBean(name = {"error"})
     public View defaultErrorView() {
         return this.defaultErrorView;
     }
    ```
    4. **BeanNameViewResolver**：用于解析得到视图
        1. 按照返回的视图名作为组件的id去容器中找View对象
        2. 如果想要返回页面；就会找 error 视图（**StaticView**）(默认是一个白页)

    ```Java
     public BeanNameViewResolver beanNameViewResolver() {
         BeanNameViewResolver resolver = new BeanNameViewResolver();
         resolver.setOrder(2147483637);
         return resolver;
     }
    ```
    5. **DefaultErrorViewResolver**：
        1. **⭐如果发生错误，并且遍历的所有 默认的异常处理解析器，都没法处理该异常**
        2. **⭐就会使用DefaultErrorViewResolver，返回一个页面**
        3. 这个页面的地址有 由 error/ 和 HTTP的状态码 拼接而来（viewName）

    ```Java
     //HTTP的状态码
     String errorViewName = "error/" + viewName;
    ```
        4. 这也是：**`templates/error/`** 目录下的4xx，5xx页面会被自动解析的原因

    ```Java
     @Bean
     @ConditionalOnBean({DispatcherServlet.class})
     @ConditionalOnMissingBean({ErrorViewResolver.class})
     DefaultErrorViewResolver conventionErrorViewResolver() {
         return new DefaultErrorViewResolver(this.applicationContext, this.resources);
     }
    ```
# 三、异常处理流程
1. 请求执行到 执行目标方法这一步时，目标方法运行期间有任何异常都会被catch，而且将 当前请求 标志为结束
2. 比如为了模拟5xx在handler里写的异常

    ```Java
     @GetMapping("/basic_table")
     public String basic_table(){
      int a = 10 / 0;
      return "table/basic_table";
     }
    ```
3. **==并且异常会被用==** ==**`dispatchException`**== **==来封装==**
4. **==异常被捕获后，进入视图解析流程（页面渲染）==**

    ```Java
     // DispatcherServlet.class
     //processDispatchResult()的参数：
         //request
         //response
         //mappedHandler(哪个handler有异常)
         //mv
         //exception（Handler产生的异常）
     private void processDispatchResult(HttpServletRequest request, HttpServletResponse response, @Nullable HandlerExecutionChain mappedHandler, @Nullable ModelAndView mv, @Nullable Exception exception) throws Exception {
         //对产生的异常判断
         if (exception != null) {
             if (exception instanceof ModelAndViewDefiningException) {
                ...
             } else {
                 Object handler = mappedHandler != null ? mappedHandler.getHandler() : null;
                 //如果 产生的异常 不是 ModelAndViewDefiningException
                 //就进行异常处理 processHandlerException()，处理完成后返回一个 mv
                 mv = this.processHandlerException(request, response, handler, exception);
                 errorView = mv != null;
             }
         }
    ```
5. 其中 **processHandlerException** 是如何处理异常的

    ```Java
     protected ModelAndView processHandlerException(HttpServletRequest request, HttpServletResponse response, @Nullable Object handler, Exception ex)
         ModelAndView exMv = null;
         if (this.handlerExceptionResolvers != null) {
             //遍历 handlerExceptionResolvers 中java所有的异常处理解析器，看哪个能处理当前异常
             Iterator var6 = this.handlerExceptionResolvers.iterator();
             while(var6.hasNext()) {
                 HandlerExceptionResolver resolver = (HandlerExceptionResolver)var6.next();
                 //如果当前 异常处理器 不能处理该异常，则会返回 null
                 exMv = resolver.resolveException(request, response, handler, ex);
                 //DefaultErrorAttributes 首次处理后，默认写死的会返回null，不会 break
                 //继续遍历下面的三个异常处理解析器
                 if (exMv != null) {
                     break;
                 }
             }
         }
         if (exMv != null) {
             ....
         } else {
             //如果所有异常处理解析器都不能处理该异常，则会将该异常抛出
             throw ex;
         }
     }
    ```
    1. 系统提供的异常处理器：

    ![[IMG-20260405035444370.png|561]]
    2. 首先所有异常都会先用 **DefaultErrorAttributes** 的 resolveException() 来处理异常

    ```Java
     public ModelAndView resolveException(HttpServletRequest request, HttpServletResponse response, Object handler, Exception ex) {
         //把异常信息保存到request域，并且返回null
         this.storeErrorAttributes(request, ex);
         return null;
     }
    ```
    3. 遍历剩下的三个解析器，如果遍历到的当前异常处理器 不能处理该异常，则会返回 null
        1. **ExceptionHandlerExceptionResolve** 这个解析器，会找标了 @ExceptionHandler（详见 下面第四节） 这个注解的 handler 是否能处理当前异常
        2. **ResponseStatusExceptionResolver** 这个解析器，会使用注解工具 找标了 @ResponseStatus（详见 下面第四节）这个注解的异常，并判断当先异常是否属于这些标了 @ResponseStatus的异常，如果是，则会使用

    ```Java
     //将 错误信息 statusCode, resolvedReason封装成一个mav 返回
     response.sendError(statusCode, resolvedReason)
    ```
            虽然这个解析器也会处理异常，但它最终_也是返回一个modelAndView错误页_
        3. **DefaultHandlerExceptionResolver** 处理框架底层的异常
    4. **如果 所有的默认异常处理解析器都 不能处理该异常**，所以异常会被抛出
    5. **最终底层就会发送 /error 请求，会被底层的BasicErrorController处理**
    6. 解析错误视图；遍历所有的 **ErrorViewResolver** （最底层的异常处理解析器，前面的解析器不能处理的，都会由他捕获，返回错误页）看谁能解析

    ![[IMG-20260405035504751.png|508]]
    7. 默认的 DefaultErrorViewResolver （详见 上面第三节），作用是把响应状态码作为错误页的地址：error/500.html

    ```Java
     String errorViewName = "error/" + viewName
    ```
    8. 模板引擎最终响应这个页面，返回一个页面：error/500.html
    9. **==总结：==**
        - 如果发生错误，并且遍历的所有**默认的异常处理解析器**和**自定义的异常处理解析器**，都没法处理该异常就会使用**`DefaultErrorViewResolver`**，返回一个错误页面

# 四、自定义异常处理
> 自定义异常处理 有以下几种方式：
## 1. 自定义错误页
1. 在**`templates/error/`** 目录下放 4xx，5xx页面
2. 有精确的错误状态码页面就匹配精确，没有就找 4xx.html；如果都没有就触发白页
3. 例如：只有 404.html，当遇到 400 错误后，仍然会使用默认的 白页（将改成 **4xx.html** 即可）
4. 并且 将 请求状态码 输出到页面，以区分 4xx 的各种错误

    ```HTML
     <section class="error-wrapper text-center">
         <h2 th:text="${status}">OOOPS!!!</h2>
         <h3 th:text="${message}">Something went wrong.</h3>
         <p class="nrml-txt" th:text="${trace}"></p>
         <a class="back-btn" th:href="@{/main.html}"> Back To Home</a>
     </section>
    ```
    ![[IMG-20260405035426813.png|800]]
## 2. @ControllerAdvice + @ExceptionHandler 处理全局异常⭐
1. **==在 handler 中写异常处理逻辑，而不是返回一个错误页==**
2. 编写自定义的 异常handler

    ```Java
     @Slf4j
     @ControllerAdvice
     public class GlobalExceptionController {
         // @ExceptionHandler 的参数value：该handler能处理哪些异常，放在一个数组中
         @ExceptionHandler(value = {ArithmeticException.class,NullPointerException.class})
         public String handlerGlobalException(Exception e){
             //被这个handler捕获到的异常，会被以参数方式传入
             log.info("当前捕获的异常是：{}",e);
             //当前处理异常的方法：简单地将页面转到 main.html
             return "main";
         }
     }
    ```
3. **这样当遇到** **`ArithmeticException`** **和** **`NullPointerException`** **这两种异常后，不会再跳转到 4xx.html 了，而是直接 转到 main.html**
4. 原理
    1. 由于 我们自定义的 handler 标注了 `@ExceptionHandler` 注解
    2. 底层在遍历所有异常处理解析器时，会由**ExceptionHandlerExceptionResolver** 找到自定义的 handler（标了注解，且能处理当前异常）

## 3. @ResponseStatus + 自定义异常
1. 自定义一个异常

    ```Java
     // @ResponseStatus 的 value 表示返回什么错误码， reason 描述异常信息
     //这里 HttpStatus.FORBIDDEN = 403
     @ResponseStatus(value = HttpStatus.FORBIDDEN,reason = "用户数量太多")
     public class userTooMany extends Exception{
         //有参无参构造器
         public userTooMany() {
         }
         public userTooMany(String message){
             super(message);
         }
     }
    ```
2. 在controller 中模拟异常

    ```Java
     @GetMapping("/dynamic_table")
     public String dynamic_table(Model model) throws userTooMany {
         List<User> users = Arrays.asList(new User("zhangsan", "123456"),
                 new User("lisi", "111111"),
                 new User("wangwu", "222222"),
                 new User("qhj", "666666"));
         model.addAttribute("users",users);
         //当用户数大于3时，产生自定义的异常，并抛出
         if (users.size() > 3){
             throw new userTooMany();
         }
         return "table/dynamic_table";
     }
    ```

4. 原理：
    1. 由于 我们自定义异常 标注了 `@ResponseStatus` 注解，底层在遍历所有异常处理解析器时，会由**ResponseStatusExceptionResolver** 处理异常
    2. **ResponseStatusExceptionResolver** 这个解析器，会使用注解工具 找标了 `@ResponseStatus`（见2.8.4） 这个注解的异常，并判断当先异常是否属于这些标了 @ResponseStatus的异常，如果是，则会使用


# 五、自定义实现 HandlerExceptionResolver 异常处理解析器
1. 自定义的异常处理解析器

    ```Java
     //需要设置优先级，否则系统遍历解析器时，会优先遍历默认的解析器，如果这些解析器能处理，我们自定义的解析器就不会生效，所以，需要将优先级设置的高一些
     @Order(value = Ordered.HIGHEST_PRECEDENCE) //优先级：数字越小优先级越高
     @Component
     public class CustomerHandlerExceptionResolver implements HandlerExceptionResolver {
         @Override
         public ModelAndView resolveException(HttpServletRequest request,
                                              HttpServletResponse response,
                                              Object handler,
                                              Exception ex) {
             try {
                 //使用sendError发送一个错误，参数携带的 status 和 message 会被封装到一个ModelAndView中，显示在错误页中
                 response.sendError(555,"当前异常是由自定义的异常处理解析器处理");
             } catch (IOException e) {
                 e.printStackTrace();
             }
             return new ModelAndView();
         }
     }
    ```
