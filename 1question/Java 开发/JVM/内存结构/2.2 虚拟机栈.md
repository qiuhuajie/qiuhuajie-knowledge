# 1. **简介**
1. Java Virtual Machine Stacks （Java 虚拟机栈），==每个线程运行时所需要的内存==，称为虚拟机栈
1. 虚拟机栈与线程同时创建，是==线程私有==的
1. 每个栈由==多个栈帧==（Frame）组成，一个栈帧对应着一次方法调用时所占用的内存
    
    ![[IMG-20260404031750928.png|Untitled 284.png]]
    
1. ==栈内存不涉及垃圾回收==，在每次方法调用结束后，栈帧就被弹出栈帧了，不需要垃圾回收来管理栈内存
# 2. **栈帧⭐**
1. 栈帧
    
    1. 一个栈帧对应着==一次方法调用时所占用的内存==
    
    1. 用来==存储数据和部分过程结果==，同时也被用来处理动态链接（Dynamic Linking）、方法返回值和异常分派（Dispatch Exception）
    
1. 生命周期
    
    1. 每次调用一个方法就 push 一个栈帧，调用完毕则 poll 一个栈帧
    
    1. 栈帧==随着方法调用而创建，随着方法结束而销毁==——无论方法是正常完成还是异常完成（抛出了在方法内未被捕获的异常）都算作方法结束
    
1. 当前栈帧（活动栈帧）：每个线程只能有一个活动栈帧，对应着当前正在执行的那个方法，这个**当前正在执行的方法的栈帧就是活动栈帧**
1. **每一个栈帧存储的数据包括**
    
    - ==局部变量表==
        
        - 如果是基本类型的变量则直接存的是这个变量的值
        
        - 如果是引用类型的变量则存的是值的地址，指向堆中具体的值
        
    
    - ==操作数栈==
    
    - ==指向当前方法所属类的运行时常量池引用==
    
    ![[IMG-20260404031751029.png|Untitled 1 213.png]]
    
1. 栈帧分配在Java虚拟机栈，而Java虚拟机栈是私有的，所以==**栈帧也是线程私有的**==

> [!important] ==**一个方法执行完毕后执行**==
> 
> `**return**`**，**`**return**` ==**的内容在哪里❓**==
> 
> 1. 本质上，**方法的退出就是当前栈帧出栈的过程**
> 
> 1. 此时，**==需要恢复上层方法的局部变量表、操作数栈，并将==** `**return**` **==返回值压入调用者栈帧的操作数栈、设置 PC 寄存器值等，让调用者方法继续执行下去==**
> 
> 1. 正常完成出口和异常完成出口的区别在于：**通过异常完成出口退出的，不会给他的上层调用者产生任何的返回值**

> [!important] **方法内的局部变量是否线程安全？**
> 
> - 阅读全文
>     
>     1. ==**总的来说就是要看，那些引用了对象的局部变量，是不是能被多个线程操作到**==
>     
>     1. 如果方法内局部变量没有逃离方法的作用范围，它是线程安全的
>         
>         - 普通方法内局部变量**在每个线程的虚拟栈内都有一份**，每个线程都操作自己的变量，互不影响
>         
>         ```Java
>          public static void m1() {
>              StringBuilder sb = new StringBuilder();
>              sb.append(1);
>              sb.append(2);
>              sb.append(3);
>              System.out.println(sb.toString());
>          }
>         ```
>         
>     
>     1. 如果是局部变量引用了对象，并逃离方法的作用范围，需要考虑线程安全
>         
>         - 例如：`static`变量由**每个线程共享同一份**，每个线程都会对其操作，会互相影响
>         
>         ```Java
>          public static void main(String[] args) {
>              StringBuilder sb = new StringBuilder();
>              sb.append(4);
>              sb.append(5);
>              sb.append(6);
>              new Thread(()->{
>                  m2(sb);                 //两个线程共同对sb对象操作，需要考虑线程安全
>              }).start();
>          }
>          
>          public static void m2(StringBuilder sb) {
>              sb.append(1);
>              sb.append(2);
>              sb.append(3);
>              System.out.println(sb.toString());
>          }
>         ```
>         
>         ```Java
>          public static StringBuilder m3() {
>              StringBuilder sb = new StringBuilder();
>              sb.append(1);
>              sb.append(2);
>              sb.append(3);
>              return sb;      //局部变量引用了对象，并逃离方法的作用范围（作为返回，就有可能被其他线程操作），需要考虑线程安全
>          }
>         ```
>         
>     
# 3. **栈内存诊断**
1. 默认的栈内存大小为：`1024k`
1. 可以在运行代码时，通过一个虚拟机指令来指定： `Xss256k`
    
    ![[IMG-20260404031751111.png|Untitled 2 178.png]]
    
    > 注意：栈内存分配并不是与大越好，因为物理内存是固定的，如果每个线程的栈内存变大，会导致线程的数量减少，并不会增强运行效率
    
1. 栈内存溢出的问题：
    
    1. 原因一：栈帧过大导致栈内存溢出（因为栈帧中存放的局部变量、参数等大多很小，而默认的栈内存大小为1M，故这种情况不多见）
    
    1. 原因二：栈帧过多导致栈内存溢出
        
        1. 示例代码：
            
            ```Java
             public class stackDemo2 {
                 private static int count;
             
                 public static void main(String[] args) {
                     try {
                         method1();
                     } catch (Throwable e) {
                         e.printStackTrace();
                         System.out.println(count);
                     }
                 }
             
                 private static void method1() {
                     count++;
                     method1();
                 }
             }
            ```
            
        
        1. 默认的栈内存大小 `1024k`，运行结果：栈溢出错误 `java.lang.StackOverflowError`，count 打印为 39019
            
            ![[IMG-20260405035426087.png|Untitled 3 135.png]]
            
        
        1. 此时修改栈内存为 `Xss256k`，count 打印为 24031

# 5. **线程运行诊断**
1. **案例1： cpu 占用过多**
    
    1. 先用top定位哪个进程对cpu的占用过高
    
    1. 再用ps命令进一步定位是哪个线程引起的cpu占用过高
        
        ```Plain
         ps H -eo pid,tid,%cpu | grep 进程id
        ```
        
    
    1. 进一步查看问题线程，进而定位到问题代码的源码行号
        
        ```Plain
         jstack 进程id
        ```
        
    
1. **案例2：程序运行很长时间没有结果**
    
    1. 有可能是线程死锁造成
    
    1. 同样使用 `jstack` 查看问题线程
