# 一、引用计数法 ⛔

> 💡 要进行垃圾回收，最为重要的一个问题是：
> **判断谁是垃圾？**
1. 思想：
    1. 日常生活中，如果一个东西经常没被使用，那么这个对象可以说就是垃圾
    2. 在 Java 中也是如此，如果一个对象不可能再被引用，那么这个对象就是垃圾，应该被回收
2. **引用计数法**
    - 在一个对象被引用时加一，被去除引用时减一
    - 这样就可以==通过判断==**==引用计数是否为零==**==来判断一个对象是否为垃圾==
3. 存在的问题
    1. 引用计数法虽然简单，但是其存在一个致命的问题，那就是**循环引用**
        ![[IMG-20260620005521781.png|375]]
    2. A 引用了 B，B 引用了 A，它们各自的引用计数都为 1
    3. 但是**它们两个对象却从未被其他对象引用，只有它们自身互相引用**
    4. 从垃圾的判断思想来看，它们两个确实是不被其他对象引用的，但是此时它们的引用计数却不为零

# 二、可达性分析法 GC Root Tracing
1. GC Root Tracing 算法：现今的 Java 虚拟机判断垃圾对象使用的算法
2. **算法思想**
    - 从 GC Root 出发，==扫描堆中所有对象==
        - 所有==可达==（有引用直接或简介指向）的对象都是存活的对象
        - 而所有==不可达==的对象都是垃圾
    ![[IMG-20260620005521857.png|487]]
3. 🙋‍♂️**哪些对象可以作为 GC Root ？**
    1. **GC Root 就是经过精心挑选的**==**一组活跃引用**==**，**==**这些引用肯定是存活的**==**。那么**==**通过这些引用延伸到的对象，自然也一定是存活的**==
    2. 通常包括：
        1. `System Class`：**Java 的系统类对象**，在运行期间肯定会用到的，如 Object，String，HashMap...，以及**异常对象**和**系统类加载器**
        2. `Busy Monitor`：**被加了锁的对象**
        3. `Thread`：**运行中的线程使用的对象（**Java 类中定义的**引用类型的静态变量、在虚拟机栈中引用的对象）**
        4. `Native Stack`：Java 虚拟机调用的操作系统的方法所引用的对象
4. 当对象被标记为可回收后，可以人为地控制让其不会被回收
    1. 当对象不可达（可回收）并发生 GC 时，会先判断对象是否执行了 `finalize` 方法，如果未执行，则会先执行 `finalize` 方法！
    2. 所以在 `finalize` 方法中，可以将当前对象与 GC Root 相关联
