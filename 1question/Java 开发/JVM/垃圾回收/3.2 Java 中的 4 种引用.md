> [!important] Java 中提供这 四种引用类型主要有
> 
> **两个目的**
> 
> 1. **有利于 JVM 进行**==**垃圾回收**==
> 
> 1. **可以让程序员**==**通过代码的方式决定某些**====**对象的生命周期**==
>     
>     - 如果希望某些对象具备一定的生命周期的话，可以使用软引用和弱引用，当如内存不足时，JVM 就会自动根据不同的引用来回收对象
>     
>     - 所有的引用类型，都是抽象类 `java.lang.ref.Reference` 的子类：
>         
>         ![[IMG-20260404031755676.png|Untitled 1 217.png]]
>         
>     
>     - 这个抽象类提供了 `get` 方法用来**获取引用指向的对象**（referent）
>         
>         ![[IMG-20260405035420126.png|Untitled 2 182.png]]
>         
>     
>     - 因此，可以将访问到的**软/弱引用对象**，重新指向**强引用**，也就人为的改变了对象的生命周期
>

# 1. **强引用**
1. 强引用在程序代码之中是**普遍存在**的
    
    ```Java
    // 比如下面这段代码中的object和str都是强引用
    Object object = new Object();
    String str = "hello";
    ```
    
1. **强引用关联对象的回收：**
    
    1. ==只要某个对象有强引用与之关联，JVM== **==必定不会回收==**==这个对象==，即使在内存不足的情况下，JVM宁愿抛出 `OutOfMemory` 错误也不会回收这种对象
    
    1. 只有所有 GC Roots 对象都不通过强引用引用该对象，该对象才能被垃圾回收
        
        - 如上图中，当C 对象和 B对象两个 GC Roots 对象 指向 A1对象的强引用都取消时，A1对象才能被垃圾回收
        
    
    1. 如果想中断强引用和某个对象之间的关联，可以显示地将引用赋值为 `**= null**`，这样一来的话，JVM 在合适的时间就会回收该对象
        
        - `object= null; System.gc();` 让栈帧中的 object 引用指向空，再执行垃圾回收，那么堆中原来由 object 引用指向的对象就可以被回收了
        
    
# 2. **软引用**
1. 软引用是用来描述一些**有用但并不是必需**的对象，在Java中用`java.lang.ref.SoftReference`类来表示
1. **软引用关联对象的回收：**
    
    1. ==仅有软引用引用该对象时，在==**==发生垃圾回收后， 且内存仍然不足时，会再次垃圾回收==**==，此时就会回收软引用对象==
        
        - 如上图中，当B对象指向 A2对象的强引用取消时，此时只有一个软引用指向 A2对象，那么垃圾回收后，仍内存不足时，A2 对象会被回收
        
    
    1. 由于软引用自身也是一个对象，自身也占用了一定的内存，可以配合引用队列来释放软引用自身
        
        - 如上图中，当 A2对象被回收后，软引用对象没有指向的对象了，那么他会被放在引用队列中， 便于释放自身内存
        
    
1. Java 声明
    
    ```Java
    SoftReference<MyObject> softReference = new SoftReference<>(new MyObject());
    ```
    
1. ==应用：==
    
    1. [[Threadlocal]]
    
    1. 根据软引用对象回收的时机，可以很好地用来解决 OOM (OutOfMemory) 的问题
        
        - 可以用来实现缓存，比如网页缓存、图片缓存等
            
            - **示例：**
                
                - 假如有一个应用需要读取大量的本地图片，但这些图片又不是必须的
                
                - 如果每次读取图片都从硬盘读取，则会严重影响性能
                
                - 但是如果全部加载到内存当中，又有可能造成内存溢出，此时使用软引用可以解决这个问题
                
            
            1. 代码：没有使用软引用的情况下
                
                ```Java
                 public class gcDemo2 {
                     /**
                      * -Xmx20m -XX:+PrintGCDetails -verbose:gc
                      */
                     private static final int _4MB = 4 * 1024 * 1024;    //模拟需要加载的一些图片
                 
                     public static void main(String[] args) throws IOException {
                         List<byte[]> list = new ArrayList<>();
                         for (int i = 0; i < 5; i++) {
                             list.add(new byte[_4MB]);       //将图片加入 list
                         }
                         System.in.read();
                     }
                 }
                ```
                
            
            1. 运行会造成堆空间的内存溢出（设置了堆内存上限）
                
                ![[IMG-20260405035439566.png|Untitled 4 114.png]]
                
            
            1. 但是如果这些图片是无关紧要的一些消息，那么就需要在他造成内存溢出时，对其进行回收！**不能让无关紧要的内容造成内存溢出错误**
            
            1. 那么就需要使用软引用
                
                1. 代码：
                    
                    ```Java
                     public class gcDemo2 {
                         /**
                          * 演示软引用
                          * -Xmx20m -XX:+PrintGCDetails -verbose:gc
                          */
                         private static final int _4MB = 4 * 1024 * 1024;
                     
                         public static void main(String[] args) throws IOException {
                             List<SoftReference<byte[]>> list = new ArrayList<>();
                             for (int i = 0; i < 5; i++) {
                                 SoftReference<byte[]> ref = new SoftReference<>(new byte[_4MB]);    //⭐
                     
                                 list.add(ref);
                                 //打印当前循环下，list中的内容
                                 for (SoftReference<byte[]> w : list) {
                                     System.out.print(w.get()+" ");
                                 }
                                 System.out.println();
                             }
                         }
                     }
                    ```
                    
                
                1. ⭐现在装入 `**list**` 的不再是一个 `**byte[]**` ，而是一个指向 `**byte**` 的 `**SoftReference**` 的对象
                    
                    > [!important] list ----强引用----> 软引用对象SoftReference -----软引用-----> byte[]
                    
                
                1. 运行后发现并不会产生堆内存溢出的错误，当循环结束后，将 `**list**` 中的内容遍历打印，发现有的 `**byte**` 对象被回收了
                    
                    ![[IMG-20260405035501815.png|Untitled 5 93.png]]
                    
                
            
            1. 这就是软引用的应用场景：
                
                - ⭐==避免一些无关紧要的对象，造成堆内存的内存溢出错误，从而影响整个应用程序的正常运行==
                
                - 让这些对象在发生内存不足时，被自动回收
                
            
            1. 还应该注意到，最后软引用对象被回收后，`**list**` 中存放着很多 `**null**` 对象，如何处理它们呢？
                
                1. 可以使用软引用**配合一个引用队列** `**new ReferenceQueue<>();**`，来回收软引用对象本身所占的空间
                
                1. 代码：
                    
                    ```Java
                     public class gcDemo3 {
                         /**
                          * 演示软引用, 配合引用队列
                          */
                         private static final int _4MB = 4 * 1024 * 1024;
                     
                         public static void main(String[] args) {
                             List<SoftReference<byte[]>> list = new ArrayList<>();
                     
                             // 引用队列
                             ReferenceQueue<byte[]> queue = new ReferenceQueue<>();
                     
                             for (int i = 0; i < 5; i++) {
                                 // ⭐关联了引用队列， 当软引用所关联的 byte[]被回收时，软引用自己会加入到 queue 中去
                                 SoftReference<byte[]> ref = new SoftReference<>(new byte[_4MB], queue);
                     
                                 list.add(ref);
                                 for (SoftReference<byte[]> w : list) {
                                     System.out.print(w.get()+" ");
                                 }
                                 System.out.println();
                             }
                     
                             System.out.println("开始从队列中获取无用的 软引用对象，并移除");
                             // 先取出第一个队列中的软引用对象
                             Reference<? extends byte[]> poll = queue.poll();
                             // 遍历list中所有为null的软引用对象
                             while( poll != null) {
                                 list.remove(poll);
                                 poll = queue.poll();
                             }
                     
                             for (SoftReference<byte[]> w : list) {
                                 System.out.print(w.get()+" ");
                             }
                             System.out.println();
                         }
                     }
                    ```
                    
                
                1. 运行输出：可以看到最后 `**list**` 中的 `**null**` 都被回收了
                    
                    ![[IMG-20260405035503692.png|Untitled 6 80.png]]
                    
                
            
        
    
# 3. **弱引用**
1. 弱引用也是用来描述**非必需对象**的，在java中用`java.lang.ref.WeakReference`类来表示
1. **弱引用关联对象的回收：**
    
    1. ==仅有弱引用引用该对象时，在==**==垃圾回收时，无论内存是否充足，都会回收==**==弱引用对象==
        
        - 如上图中，当B对象指向 A3对象的强引用取消时，此时只有一个弱引用指向 A3对象，那么垃圾回收后，无论内存是否充足，A2对象都会被回收
        
    
    1. 由于弱引用自身也是一个对象，自身也占用了一定的内存，可以配合引用队列来释放弱引用自身
        
        - 如上图中，当 A3对象被回收后，弱引用对象没有指向的对象了，那么他会被放在引用队列中， 便于释放自身内存
        
    
1. Java 声明
    
    ```Java
    WeakReference<MyObject> weakReference = new WeakReference<>(new MyObject());
    ```
    
- **应用场景演示**
    
    1. 代码：
        
        ```Java
         public class gcDemo4 {
             /**
              * 演示弱引用
              * -Xmx20m -XX:+PrintGCDetails -verbose:gc
              */
             private static final int _4MB = 4 * 1024 * 1024;
         
             public static void main(String[] args) {
                 //  list --> WeakReference --> byte[]
                 List<WeakReference<byte[]>> list = new ArrayList<>();
                 for (int i = 0; i < 5; i++) {
                     WeakReference<byte[]> ref = new WeakReference<>(new byte[_4MB]);
                     list.add(ref);
                     //打印当前循环下，list中的内容
                     for (WeakReference<byte[]> w : list) {
                         System.out.print(w.get()+" ");
                     }
                     System.out.println();
                 }
             }
         }
        ```
        
    
    1. 运行输出：弱引用也可以保证不发生堆内存溢出的问题，其与软引用的区别在于
        
        1. 垃圾回收的方法不同，这里并没有 `**full gc**`
            
            > JAVA内存模型中，弱引用对象在发生GC的时候就会被GC线程回收，但并不意味着在年轻代就会被回收，当JVM垃圾回收器的GC thread还没有执行到回收垃到时刻，如果此时被修饰为弱引用的对象已经通过某个途径copy到了老年代（如：该弱引用对象占独立内存块比较大，首次放入堆内存的时候，由于年轻代的Eden和From zone无足够连续内存空间存放，在老年代的担保策略下，直接存入old generation），此时此刻，这个弱引用对象就存在于老年代，当下一次full GC发生时，被GC线程回收掉，所以不能说弱引用是在young GC回收。
            > 
            > 软引用对象一般会经历多次GC，自然晋升到old generation，一般情况下，当内存不足时，才会在GC线程运行时，经由标记算法，一次回收此类对象，因此可以说，软引用对象一般都在full gc过程中被回收。
            
        
        ![[IMG-20260405035509921.png|Untitled 7 67.png]]
        
    
# 4. **虚引用**
1. **虚引用关联对象的回收：**
    
    1. 虚引用并==不会决定对象的生命周期==
    
    1. ==如果一个对象仅持有虚引用，那么它就和没有任何引用一样，在任何时候都可能被垃圾回收==
    
1. **设置虚引用的目的：**
    
    1. 虚引用不能单独使用也不能通过它访问对象，虚引用必须和**引用队列（ReferenceQueue）**联合使用
    
    1. 因为虚引用的构造函数必须指定引用队列
        
        ![[IMG-20260405035517933.png|Untitled 8 55.png]]
        
    
    1. ==引用队列：当某个被引用的对象（referent）被回收的时候，JVM 会将指向它的引用（reference）加入到引用队列的队列末尾==
    
    1. 因此可以 get 引用队列中的元素来知道哪些对象（被引用的对象）被回收掉了
    
    1. 通过这种方式，就==可以在对象被回收掉之后，做一些我们自己想做的事情，例如通知操作==
    
1. 应用
    
    - **数据库连接池**
        
        1. 数据库连接池 Connection Pool 应该具备的一个优点就是能够有效的避免连接资源泄露，同时能够对连接资源进行回收
            
            ![[IMG-20260405035519015.png|Untitled 9 48.png]]
            
        
        1. 如果引用队列中能够拿到引用，说明连接对象被 GC 掉了，此时我们就应该对连接池执行相应的清理逻辑
            
            ![[IMG-20260405035519074.png|Untitled 10 41.png]]
