# 一、介绍
1. CMS 回收器全称为 `Concurrent Mark Sweep`，意为==并发标记清除==
2. **CMS 收集器==只收集老年代==，可以配合串行回收器的新生代回收器或并行回收器中的 `ParNew` 收集器一起使用**
3. CMS 回收器主要关注==系统停顿时间==

# 二、CMS 的工作流程
1. **初始标记（CMS initial mark）**
    - 先==将与 GC Roots 直接关联的对象标记起来==，速度快，stw 的时间很短

        ![[IMG-20260620011547991.png|290]]

2. **并发标记（CMS concurrent mark）**
    - 正式从 GC Roots 的直接关系对象开始==遍历整个对象图，标记处所有垃圾对象==
    - 这个过程耗时较长，但是==用户线程可以与垃圾收集线程一起并发运行==，不需要阻塞
        ![[IMG-20260620011548070.png|301]]
3. **重新标记（CMS remark）**
    - 上面并发标记中，由于用户线程在并发执行，可能会产生新的对象变动
    - 这个阶段就是为了==修正并发标记期间的对象新变动==，这个阶段的停顿时间通常也很短
4. **并发清理（CMS concurrent sweep）**
    - ==清理所有之前被标记为垃圾的对象==
    - 由于不需要移动存活对象，所以这个阶段也是==可以与用户线程同时并发运行==的

# 三、CMS 的特点
1. **优点：**
    1. CMS 回收器主要关注==系统停顿时间==
    2. 在**==并发标记==**==和==**==并发清理==**==阶段可以和用户线程一起执行，因此大大降低了系统停顿时间 STW==
2. **缺点：**
    1. 如果在并发清理阶段，用户线程在并行执行，如果此时产生的新的垃圾对象，如果此时又导致内存不足，会造成 CMS 并发失败
    2. 那么 CMS 就会退化为老年代串行回收器，效率会大大降低
3. 主要参数
    1. `XX:+UseConcMarkSweepGC`：启动 CMS 回收器
    2. `XX:ConcGCThreads` 或 `XX:ParallelCMSThreads` ：设定线程并发数
    3. `XX:CMSInitiatingOccupancyFraction` ：
        - 指定老年代空间使用阈值，当**老年代空间使用率达到这个阈值时**，会执行一次 CMS 回收，而不像其他回收器一样等到内存不够用的时候才进行 GC
        - 如果等内存满了再回收（因为并发清理的同时其他线程也是在并发执行的，也会产生新的垃圾）这些新垃圾会没处放
        - 设置的越小，垃圾回收触发的越早
    4. `XX:+UseCMSCompactAtFullCollection` ：让 CMS 在完成垃圾回收后，进行一次内存碎片整理
