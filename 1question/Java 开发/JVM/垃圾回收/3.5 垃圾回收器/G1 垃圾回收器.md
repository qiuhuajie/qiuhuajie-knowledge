# 一、特点 1：G1 对堆划分的改进

> 💡 G1 回收器主要关注的也是系统停顿时间 STW，尽可能得降低 STW
> 可以使用参数：`-XX:MaxGCPauseMillis` 来设置目标最大停顿时间
1. 在 G1 回收器之前，所有的垃圾回收器其内存分配都是**连续的一块内存**，如下图所示
    ![[IMG-20260620011609406.png|630]]
2. **G1 对堆划分的改进**
    1. G1 中新生代、老年代**不再是之前的一块连续的内存地址**，而是由各个**离散的、大小相等的 `Region` 组成的**
        ![[IMG-20260620011609496.png|629]]
    2. `Region` 的==角色是动态变化==的，每个`Region` 都可以成为 `Eden`空间、`Survivor`空间、老年代`Old`空间
    3. Region 还有一类特殊的 `Humongous` 区域，专门用来==存储大对象==（G1 之前是直接进入老年代）
        - G1 认为只要大小超过了一个 Region 容量一半的对象即可判定为大对象
        - 如果是那些超过了整个 Region 容量的超大对象，将会放在==连续的 N 个== Humongous Region区域
    4. 注：这里 `Eden` 区与两个`Survivor` 区在新生代中的比例仍为 `8:1:1`
3. 对堆划分**改进的好处**
    1. ✅使得 G1 可以==面向堆内存任何部分==来组成回收集来进行回收
    2. 因此 G1 可以被称为==全年代的垃圾回收器==
        - 之前的垃圾回收，都是针对新生代和老年代配了不同的垃圾收集器
        - 但 G1 一个垃圾回收器就可以负责 回收新生代和老年代
4. 参数配置
    - 堆内存中一个 Region 的大小可以通过 `-XX:G1HeapRegionSize` 参数指定
5. **线上监控水位示例**
	![[Attachment/b77baa09157695ef3deddd6cdd47c2fa_MD5.png|654]]
	- `g1_eden_space` 红线
	    表示 **新生代 Eden 区**，新对象大多先分配在这里。它频繁上下波动是正常的，因为对象创建后很快会被 Young GC 清掉或转移。
	- `g1_survivor_space` 黄线
	    表示 **Survivor 区**，Young GC 后还活着、但还没进入老年代的对象会先放这里。一般不会特别大。
	- `g1_old_gen` 绿线
	    表示 **老年代**，存放存活时间更长的对象。这个值持续上涨又不回落，通常要关注是不是有对象堆积或内存泄漏。
	- `metaspace` 紫线
	    表示 **类元数据区**，放类定义、方法信息、反射相关元数据，不是业务对象本身。一般跟“加载了多少类”有关。
	- `compressed_class_space` 浅蓝线
	    表示 **压缩类指针相关的类空间**，本质上也是类元数据的一部分，通常和 `metaspace` 一起看。
	- `code_cache` 深蓝线
	    表示 **JIT 编译后的机器码缓存**。HotSpot 会把热点方法编译后放这里，太满会影响 JIT 性能，但平时一般比较稳定。

# 二、特点 2：G1 根据回收价值选择回收
1. G1 回收器：garbage one（first）**带有**==**优先级**==**的选择回收**
    1. 每个 `Region` 都有一个隐藏属性：==**回收价值**==，回收价值就是根据 `Region` 内部垃圾的比例、垃圾回收的预计耗时等等，评估出来的一个指数
        1. G1 通过跟踪每一个 `Region` 的回收价值，**在垃圾回收的时候，**==**组合一些有价值的**== **`Region` 进行回收**
        2. 而不是像之前的垃圾回收器：例如，在新生代回收时，扫描所有的新生代对象
    2. ✅==这样就可以降低垃圾回收带来的时间停顿时间，也可以做到在有限的时间内回收更多的垃圾==
2. 可以使用参数：`XX:MaxGCPauseMillis` 来设置目标最大停顿时间
# 三、G1 的工作机制
## 1. 新生代的垃圾回收
1. 当**新生代的 `Region` 占比达到 `60%` 时，就会触发一次新生代垃圾回收 `Minor GC`**
2. 采用的是==复制算法==：
    1. 首先需要 stop the world， 对 `Eden` `Region` 进行标记
    2. 把存活对象拷贝到 `Survivor` `Region` ，再把`Eden` `Region` 清空

## 2. 对象进入老年代的时机
1. 方式一：**年龄阈值**
    1. ==对象经过多次的复制后，达到年龄阈值，就会晋升到老年代==
    2. 可以通过参数 `XX：MaxTenuringThreshold` 进行年龄的设置
2. 方式二：**提前晋升**
    1. 如果一旦==发现某个新生代 GC 过后，同年龄的存活对象超过了====`Survior`== ==的== ==`50%`==
    2. 比如此时有1岁的，2岁的，3岁的，5岁的，发现 3 岁的对象大小总和已经超过了 `Survior` 的 `50%`
    3. 那么**3岁及以上**的对象都会全部进入老年代，不需要达到年龄阈值，而是进行提前晋升

## 3. 老年代的垃圾回收
1. **当老年代的 `Region` 占比达到 `45%` 时，就会触发一次老年代垃圾回收**，这个比例可以通过参数来调整：`-XX:InitiatingHeapOccupancyPercent`
2. 之前的垃圾回收器在老年代使用的是标记清除或标记整理，而 ==G1 在老年代使用的和在新生代一样的算法：复制算法==
3. 本次回收的触发条件虽然是老年代的 `Region` 占比达到阈值，但==**在垃圾回收的时候，是一个混合回收，也就是会对**== **`Young Region`**、**`Old Region`**、**`Humongous Region`** ==**都回收**==
4. 混合回收的详细过程：
    1. **1️⃣初始标记：**同 CMS 垃圾回收
    2. **2️⃣并发标记：**同 CMS 垃圾回收
    3. **3️⃣最终标记：**同 CMS 垃圾回收
    4. **4️⃣混合回收：**
        1. 首先会==计算一下老年代中各个== ==`Region`== ==的回收价值==，然后 STW
        2. 选中 `Young` `Region` 、 `Old` `Region` 、 `Humongous` `Region` 中的一些==回收价值高==的 `Region` 进行回收
        3. 这样是为了让回收消耗的停留时间 STW 控制在设置的预期时间内
        4. 如果回收不完，后续再进行回收，一部分一部分回收直到回收完毕。但是会保证一次回收停顿的时长在预期时间内
            - `XX：G1MixedGCCountTarget`，可以设置在一次混合回收的过程中，最后一个阶段执行几次混合回收，默认值是8次！这样设置的目的也是能让每次回收停顿的时长记得到保证同时又能间隙的让系统接着运行
            - `XX：G1HeapWastePercent`，默认值是5%，意思是当混合回收的时候，一旦空闲出来的Region数量达到了堆内存的5%，此时就会立即停止混合回收

## 4. 可能退化为串行回收器的 Full CG
1. 由于当在进行混合回收的过程中，由于无论是新生代还是老年代都是==基于复制算法==进行的，都需要将各个`Region`中的存活对象拷贝到别的`Region`中
2. 此时如果==一旦出现拷贝的过程中发现====**没有空闲的**====`Region`====**可以进行存储了**====，就会触发一次失败==
3. 那么这个时候系统会立马切换为 Seiral（串行） 收集器进行单线程的 `Full GC` ：标记、清理和压缩整理，该过程就变得非常的慢了！
