# 1. **相关 JVM 参数**
|   |   |
|---|---|
|**含义**|**参数**|
|**堆初始大小**|`-Xms`|
|**堆最大大小**|`-Xmx` 或 -XX:MaxHeapSize=size|
|新生代大小|`-Xmn` 或 (-XX:NewSize=size + -XX:MaxNewSize=size )|
|幸存区比例（动态）|-XX:InitialSurvivorRatio=ratio 和 -XX:+UseAdaptiveSizePolicy|
|**幸存区比例**|`-XX:SurvivorRatio`=ratio|
|晋升阈值|-XX:MaxTenuringThreshold=threshold|
|晋升详情|-XX:+PrintTenuringDistribution|
|**GC详情**|`-XX:+PrintGCDetails` -verbose:gc|
|FullGC 前 MinorGC|-XX:+ScavengeBeforeFullGC|
下面演示所用的参数设置：
```Plain
 -Xms20M -Xmx20M -Xmn10M -XX:+UseSerialGC -XX:+PrintGCDetails -verbose:gc -XX:-ScavengeBeforeFullGC
```
常量设置：
```Java
 private static final int _512KB = 512 * 1024;
 private static final int _1MB = 1024 * 1024;
 private static final int _6MB = 6 * 1024 * 1024;
 private static final int _7MB = 7 * 1024 * 1024;
 private static final int _8MB = 8 * 1024 * 1024;
```

# 3. **Minor GC**
1. 代码：
    
    ```Java
     public class gcDemo5 {
     
         public static void main(String[] args) throws InterruptedException {
             ArrayList<byte[]> list = new ArrayList<>();
             list.add(new byte[_7MB]);
             list.add(new byte[_512KB]);
         }
     }
    ```
    
1. 运行输出：
    
    ![[IMG-20260404031753358.png|Untitled 1 338.png]]
    
    - 可以看到进行了一次 `GC` ：之前 `Eden` 区占了 `26%` ，有分配了`7MB + 512KB` 的对象进去，显然已经放不下，故需要进行 `Minor GC`，将 `Eden` 和 `From`中存活的对象放入 `To`，清理要回收的对象，最后将 `From` 和 `To` 对调
    
    - 因此 最后 `From` 中存了一些保留下来的对象，占了 `60%`
    
1. 如果再分配
    
    1. 代码
        
        ```Java
         ArrayList<byte[]> list = new ArrayList<>();
         list.add(new byte[_7MB]);
         list.add(new byte[_512KB]);
         list.add(new byte[_512KB]);
        ```
        
    
    1. 运行输出
        
        ![[IMG-20260405035420335.png|Untitled 2 277.png]]
        
        - 可以看到有两次 `GC` ，第一次就是上面那个，第二次是第二个 `512KB` 分配导致
        
        - 看以看到 `7MB + 512KB + 512KB + 基础对象` ， `9M` 的新生代已经完全放不下了，因此，一些对象会被晋升到老年代
        
    
# 4. **大对象直接晋升**
1. 代码：
    
    ```Java
     public class gcDemo5 {
     
         public static void main(String[] args) throws InterruptedException {
             ArrayList<byte[]> list = new ArrayList<>();
             list.add(new byte[_8MB]);
         }
     }
    ```
    
1. 运行输出：
    
    ![[IMG-20260405035422042.png|Untitled 3 208.png]]
    
    - 可以看到 `8M` 的大内存是一下子分配的，jvm 发现新生代完全放不下，会直接将其晋升到老年代
    
    - 上面的例子中 `8M` 是 `7MB + 512KB + 512KB` 依次分配的，会触发 `GC` ，而本例子中是直接一次性分配，并不会触发 `GC`
    
# 5. **Full GC && 内存溢出**
1. 代码：
    
    ```Java
     public class gcDemo5 {
     
         public static void main(String[] args) throws InterruptedException {
             ArrayList<byte[]> list = new ArrayList<>();
             list.add(new byte[_8MB]);
             list.add(new byte[_8MB]);
         }
     }
    ```
    
1. 运行输出：
    
    ![[IMG-20260405035433475.png|Untitled 4 161.png]]
    
    - 可以看到第一个 `8M` 创建时，会直接放在老年代
        
        - 但第二个 `8M` 创建时，也会直接向老年代里放，但此时老年代一共才 `10M` 没有足够的空间了（这里可能是由 JVM 直接**预测**得到的，因为上次晋升了 `8M`，而此时老年代只剩余 `2M` 空间 ），那么他就会进行 `Full GC`
        
    
    - 可以看到先执行了一个 `Minor GC` ，这个 `Minor GC` 并不是由于新生代 `Eden` 不足触发的，而是**每次执行** `**Full GC**` **都会先执行一次** `**Minor GC**`
    
    - 经过一次 `Full GC` 后发现，老年代剩余的空间仍放不下一个 `8M` 的对象，那么就会导致堆内存溢出的问题 `OOM`
    
1. **复习回顾：**
    
    如果 `8M` 对象是非必要的内容，可以使用 软引用 来避免 `OOM` 的发生[[3.2 Java 中的 4 种引用]]
    
    ```Java
     public class gcDemo5 {
     
         public static void main(String[] args) throws InterruptedException {
             ArrayList<SoftReference<byte[]>> list = new ArrayList<>();
     
             SoftReference<byte[]> softReference1 = new SoftReference<byte[]>(new byte[_8MB]);
             SoftReference<byte[]> softReference2 = new SoftReference<byte[]>(new byte[_8MB]);
     
             list.add(softReference1);
             list.add(softReference2);
         }
     }
    ```

# 6. **异步线程导致的 OOM**
1. 代码：
    
    ```Java
     public class gcDemo5 {
     
         public static void main(String[] args) throws InterruptedException {
             new Thread(() -> {
                 ArrayList<byte[]> list = new ArrayList<>();
                 list.add(new byte[_8MB]);
                 list.add(new byte[_8MB]);
             }).start();
     
             System.out.println("sleep....");
             Thread.sleep(1000L);
         }
     }
    ```
    
1. 运行输出：
    
    ![[IMG-20260405035440225.png|Untitled 6 108.png]]
    
    - 可以看到 发生 `OOM` 的是异步线程，并不会影响到主线程的执行
