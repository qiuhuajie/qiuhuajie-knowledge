# 一、分代算法回收演示
1. 对象首先分配在伊甸区域`Eden`
    ![[IMG-20260620011336071.png|761]]
2. 当==新生代==`Eden`==空间不足==时，触发 `minor gc`，`Eden`和 `from` 存活的对象使用 copy 复制到 `to` 中，存活的==对象年龄==加 `+1`

    ![[IMG-20260620011336219.png|758]]

3. 将`Eden`空间和 `from` 中要删除的对象回收，并且交换 `from` 和 `to`

    ![[IMG-20260620011336381.png|750]]

4. 当新生代`Eden`空间再次不足时，触发 `minor gc`，`Eden`和 `from` 存活的对象使用 copy 复制到 `to` 中，存活的对象年龄再加 `+1`

    ![[IMG-20260620011336461.png|749]]

5. 将`Eden`空间中要删除的对象回收，并且交换 `from` 和 `to`

    ![[IMG-20260620011336529.png|748]]

6. 当对象寿命超过阈值时，会晋升至老年代，==最大寿命==是 **`15`**（4bit）

    ![[IMG-20260620011336653.png|743]]

7. 接着如果新生代`Eden`空间再次不足时，再次触发 `minor gc`，可能 `from` 区中会有新的对象晋升到老年代
    - 注意：如果 JVM 发现新创建的对象太大了，新生代完全放不下，会直接将其晋升到老年代
8. 那么如果晋升的对象越来愈多，导致==老年代的空间不足==，那么就会触发那么就会触发 `Full GC`

    ![[IMG-20260620011336698.png|740]]

    - 由于老年代中大部分对象更可能存活，且 Full GC 在清理老年代时，使用的垃圾回收算法是标记整理算法或标记清除算法，故它的 STW （世界暂停时间）会更长
    - 每次 Full GC 执行前，都会先执行一个 minor gc
9. 经过一次 `Full GC` 后，如果老年代剩余的空间仍不足，那么就会导致堆内存溢出的问题 `OOM`
![[IMG-20260620011336752.png|724]]
