# 一、先要优化代码

1. GC 只是应用程序调优的一个方向，但是是影响比较明显的一个调优方向，尽量减少 STW
2. 其他调优方向：内存、锁竞争、CPU 占用、IO，只有各个方面综合起来，才能对应用程序有一个大的提升
3. 垃圾回收的调优目标：
    1. 科学计算的应用程序：注重高吞吐量，适合使用 并行回收器 Parallel GC
    2. 网络应用程序：注重低延迟，适合使用 CMS、G1、ZGC

# 二、新生代 GC 调优
1. 新生代的特点
    1. 所有的 `new` 操作，这个对象首先会在 `Eden` 区分配，这个分配速度是非常快的
        > 每个线程都会在 Eden 区给它分配一个私有的区域：TLAB 全称 thread-local allocation buffer，以解决线程内存分配时的线程安全问题
    2. 死亡对象的回收代价是零
    3. 新生代中的对象 **98% 是朝生夕死的**，因此大新生代中大部分的对象，大概率不会存活很久
    4. `Minor GC` 的时间远远低于 `Full GC`
2. 如何给新生代进行 GC 调优
    1. 将**新生代大小**上调，但不是越大越好 `Xmn`
        - 新生代能容纳所有【并发量 * (请求-响应)】的数据
        - 幸存区大到能保留【当前活跃对象+需要晋升对象】
    2. **晋升年龄阈值**配置得当，让长时间存活对象尽快晋升
        - `XX:MaxTenuringThreshold=threshold`
        - `XX:+PrintTenuringDistribution`

# 三、老年代 GC 调优
> 以 CMS 为例
1. CMS 如果在并发清理阶段，用户线程会产生的新的垃圾对象，如果此时又导致内存（老年代）不足，造成 CMS 并发失败，那么 CMS 就会**退化为老年代串行回收器**，效率会大大降低，因此 CMS 的老年代内存越大越好
2. 遇到问题，查看日志
    1. 如果没有 Full GC，那么就先不要去调试老年代，要先尝试调优新生代
    2. 观察如果发生 Full GC 时，老年代内存占满，可以将老年代内存预设调大 `1/4 ~ 1/3`
        - `XX:CMSInitiatingOccupancyFraction=percent`

# 四、案例

> 案例1：
> **Full GC 和 Minor GC频繁**
> 1. 分析：可能是由于新生代区空间太小，`Eden` 区容易占满，导致 Minor GC 频繁，`Survivor` 区小，由于放不下那么多的幸存对象，导致很多生命周期短的对象，也会由于内存不足晋升到老年代，老年代存放的对象数量会大大增加，从而导致 Full GC 也很频繁
> 2. 解决方法：
>     1. 上调新生代的内存大小
>     2. 提高晋升年龄阈值，让短生命周期的对象，尽可能得留在新生代的`Survivor` 区
> 案例2：
> **请求高峰期发生 Full GC，单次暂停时间特别长 （CMS）**
> * 分析：可以在 GC 日志中查看是 CMS 哪个阶段时间较长
> 案例3：
> **老年代充裕情况下，发生 Full GC**
> 1. 如果 Old 区并未占满，但仍然频繁 Full GC，需要检查触发原因是否来自方法区 / 类元数据，而不是只盯着老年代水位。
> 2. JDK 7 及以前，HotSpot 使用永久代 PermGen 实现方法区；永久代空间不足时，可能触发 Full GC，JVM 会尝试卸载无用类并回收永久代空间。此时可以考虑调整 `-XX:PermSize` / `-XX:MaxPermSize`。
> 3. JDK 8 及以后，永久代被 Metaspace 取代；Metaspace 使用本地内存，但仍由 JVM 管理，达到阈值或上限时可能触发 `Metadata GC Threshold`，JVM 会通过 GC / Full GC 尝试卸载类。此时应检查 `-XX:MetaspaceSize` / `-XX:MaxMetaspaceSize`，并排查动态类生成、重复类加载或类加载器泄漏。
> 4. 因此，JDK 7 及以前是调高永久代参数；JDK 8 及以后是评估 Metaspace 参数和类卸载问题。线上不能只靠调大内存，尤其要确认类元数据为什么持续上涨。
