# 2. 详解
- 在Java虚拟机中类的全过程，包括==**加载、验证、准备、解析和初始化**==这5个阶段所执行的具体动作，这些==**都是由类加载器来实现的**==
## 2.1 加载 `.class` 文件
1. **🙋‍♂️类加载有三种方式：**
    
    1. 命令行启动应用时候由JVM初始化加载
    
    1. 通过 **`Class.forName()`** 方法动态加载
        
        - `Class.forName()` 将类的.class文件加载到jvm中之外，还会对类进行解释，执行类中的static块
        
    
    1. 通过 **`ClassLoader.loadClass()`** 方法动态加载
        
        - `ClassLoader.loadClass()` 只干一件事情，就是将.class文件加载到jvm中，不会执行static中的内容，只有在newInstance才会去执行static块
        
    
1. **🙋‍♂️类加载过程的第一步，主要完成下面 3 件事情：**
    
    1. 1️⃣ ==**类加载器**==**通过**==**类的全限定名**==**来获取此类的二进制字节流**
        
        - 这个二进制字节流来源有：class文件、zip、jar、war包中读取、网络中读取，比如Applet、运行时计算生成，比如动态代理技术、由其他文件生成，比如 JSP
        
    
    1. 2️⃣ **将字节流所代表的静态存储结构转换为**==**类的信息**==**（**==**`instanceKlass`**==**），并加载在**==**方法区**==**中**
        
        - 内部采用 C++ 的一个数据结构： `instanceKlass` 来描述一个 java 类的各种信息
        
        - 它的重要 field 有：
            
            - 展开
                
                - `_java_mirror` 即 java 的类镜像，例如对 String 来说，就是 String.class，作用是把 klass 暴露给 java 使用
                
                - `super` 即父类
                
                - `_fields` 即成员变量
                
                - `methods` 即方法
                
                - `_constants` 即常量池
                
                - `class_loader` 即类加载器
                
                - `_vtable` 虚方法表
                
                - `itable` 接口方法表
                
                - `Static Field Table` 静态变量表
                
            
        
    
    1. 3️⃣**同时在**==**堆**==**中会生成一个** ==**Class 对象**==**（** ==**`_java_mirror`**== **），作为方法区中 `instanceKlass` 的访问入口**
        
        1. Class 对象中会存储 `instanceKlass` 的地址，而 `instanceKlass` 中也会存有类对象的地址
        
        1. 也即， Class 对象包含了完整的类的结构信息，==可以通过这个 Class 对象看到类的结构==[[Export-e8249ac1-4ae1-42cd-83d5-041a266841fe/1question/Java 开发/Java/反射|反射]]
        
        1. 每当 new 一个实例对象出来时，会被放在堆中，==在其对象头中会存放这个 Class 对象的地址，但不论产生多少实例，Class 对象是唯一的==
            
            ![[IMG-20260405035420132.png]]

## 2.2 验证
1. **验证是为了**==**确保 Class 文件的字节流中包含的信息符合当前虚拟机的要求**==**，并且**==**不会危害虚拟机自身的安全**==
    
    1. 当加载的class文件不符合虚拟机的要求，java 虚拟机是无法执行这个字节码的
    
    1. 所以要先看看有没有符合，符合了才给虚拟机执行后续操作
    
1. 例如：用 UE 等支持二进制的编辑器修改 HelloWorld.class 的魔数，在控制台运行，会报下面的错误

## 2.3 准备
1. **准备阶段的主要目的是确保类的静态变量在使用前已经被分配内存，并设置了默认值。在此阶段，不会执行任何 Java 代码**
1. **在准备阶段会为 `static` 变量**==**分配空间**==**，并设置**==**默认值**==
    
    1. 为类的静态变量分配内存空间，静态变量包括类变量和静态成员变量
    
    1. 设置类的静态变量的默认值，对于基本数据类型，如 int、float 等，其默认值是 `0`；对于引用类型，其默认值是 `null`
    
    1. 如果类中存在常量（使用 `final` 关键字修饰的静态变量），则在准备阶段中已经将常量的值设置为声明时的值，而不是默认值
    
## 2.4 解析
1. 解析是虚拟机==将常量池内的==**==符号引用==**==替换为==**==直接引用==**的过程
1. 类加载器**编译**的时候，不知道某个类 `com.xxx.xxx` 的实际内存地址，此时用符号引用
1. 但是此时**经过加载阶段**，**`com.xxx.xxx` 类信息已经加载到内存了**，此时就==可以====**改为直接引用**====，指向实际的内存位置==
## 2.5 初始化
1. 🙋‍♂️**调用** `**==<clinit>()==**` **进行类的初始化，设置显示**==**初始值**==
    
    1. `<clinit>()`方法由 **静态变量显示赋值代码** 和 **静态代码块**[**（静态代码块）**](https://www.notion.so/8f9c2e8c195c45f1ab16febf29cb5226?pvs=21)组成，变量显示赋值代码和静态代码块代码从上到下顺序执行
    
    1. `<clinit>()`方法**只执行一次**
    
    1. 举例：
        
        ```Java
        public class Test {
            static int a; // 在准备阶段将 a 分配为 0
            static String s = "Hello"; // 在准备阶段将 s 分配为 null；在后面的初始化阶段会执行显示赋值语句，因此被覆盖为初始值 "Hello"
            static {
                a = 10; // 静态代码块中的初始化，将 a 赋为 10
            }
        }
        ```
        
    
1. `static` 变量在 JDK 7 之前存储于 `instanceKlass` 末尾，从 JDK 7 开始，存储于 `_java_mirror` 末尾
1. **类初始化是【懒惰的】**
    
    1. **会触发初始化的情况**
        
        1. **main 方法所在的类**，总会被**首先初始化**
        
        1. **首次访问**这个类的**静态变量**或**静态方法**时[[单例模式 ⭐]]
        
        1. 子类初始化，如果父类还没初始化，会引发父类的初始化，但子类访问父类的静态变量，只会触发父类的初始化
        
        1. **`new`** 关键字也会导致初始化
        
    
    1. **不会触发类初始化的情况**
        
        1. 访问**类的 static final 静态常量，且为基本类型或字符串**，不会触发初始化，因为在准备阶段就已经完成赋值了
        
        1. 类对象 .class 不会触发初始化
        
        1. 创建该类的数组不会触发初始化
        
        1. 类加载器的 loadClass 方法
        
        1. Class.forName 的参数 2 为 false 时
