# 一、任务顺序执行
## 1. thenRun / thenAccept / thenApply
1. 这三个 API 都表示任务 A 执行完成后继续执行任务 B，区别在于任务 B 是否需要任务 A 的结果，以及任务 B 是否产生新的返回值。

| API | 是否接收上一步结果 | 是否返回新结果 | 典型含义 |
| --- | --- | --- | --- |
| `thenRun(Runnable runnable)` | 否 | 否 | A 执行完后执行 B，B 只做动作 |
| `thenAccept(Consumer action)` | 是 | 否 | A 执行完后执行 B，B 消费 A 的结果 |
| `thenApply(Function fn)` | 是 | 是 | A 执行完后执行 B，B 基于 A 的结果计算新结果 |

## 2. 示例代码
* 下面的例子可以直观看到三个 API 的返回结果差异。

```Java
public class CompletableFutureAPIDemo2 {
    public static void main(String[] args) throws ExecutionException, InterruptedException, TimeoutException {

        System.out.println(CompletableFuture.supplyAsync(() -> "resultA").thenRun(() -> {}).join());

        System.out.println(CompletableFuture.supplyAsync(() -> "resultA").thenAccept(resultA -> {}).join());

        System.out.println(CompletableFuture.supplyAsync(() -> "resultA").thenApply(resultA -> resultA + " resultB").join());
    }
}

执行结果：
null
null
resultA resultB
```

# 二、普通方法和 Async 方法
## 1. 核心区别
* `xxx()` 与 `xxxAsync()` 的区别主要体现在后续任务由哪个线程执行。普通方法通常由完成上一个阶段的线程继续执行；`xxxAsync()` 会把后续阶段提交给线程池执行。

## 2. 未传入自定义线程池
* 如果没有传入自定义线程池，异步阶段默认使用 `ForkJoinPool.commonPool()`。

```Java
public class CompletableFutureAPIDemo2 {
    public static void main(String[] args) throws ExecutionException, InterruptedException, TimeoutException {
        ExecutorService threadPool = Executors.newFixedThreadPool(5);
        try {
            CompletableFuture<Void> completableFuture = CompletableFuture.supplyAsync(() -> {
                try { TimeUnit.MILLISECONDS.sleep(500); } catch (InterruptedException e) { e.printStackTrace(); }
                System.out.println("task1 use thread : " + Thread.currentThread().getName());
                return "task1 finish";
            }).thenRun(() -> {
                try { TimeUnit.MILLISECONDS.sleep(500); } catch (InterruptedException e) { e.printStackTrace(); }
                System.out.println("task2 use thread : " + Thread.currentThread().getName());
            }).thenRun(() -> {
                try { TimeUnit.MILLISECONDS.sleep(500); } catch (InterruptedException e) { e.printStackTrace(); }
                System.out.println("task3 use thread : " + Thread.currentThread().getName());
            }).thenRun(() -> {
                try { TimeUnit.MILLISECONDS.sleep(500); } catch (InterruptedException e) { e.printStackTrace(); }
                System.out.println("task4 use thread : " + Thread.currentThread().getName());
            });

            System.out.println(completableFuture.join());
        } catch (Exception e) {
            e.printStackTrace();
        } finally {
            threadPool.shutdown();
        }
    }
}
```

## 3. 第一个任务传入自定义线程池
### 3.1 thenRun
* 第一个任务传入自定义线程池后，后续调用 `thenRun()` 时，第二个任务通常会继续由完成上一个阶段的线程执行，因此实际效果上会复用前面传入的自定义线程池。

![[IMG-20260619225046733.png|677]]

### 3.2 thenRunAsync
* 第一个任务传入自定义线程池后，如果后续调用的是未指定线程池的 `thenRunAsync()`，第二个任务会提交到默认线程池 `ForkJoinPool.commonPool()` 执行。

# 三、异步回调要传线程池
## 1. 推荐做法
* 在 [[CompletableFuture（CF） 介绍]] 中，创建异步任务时建议==**强制传线程池，且根据实际情况做线程池隔离**==。

## 2. 不传线程池的问题
1. 当不传递线程池时，异步任务会使用 `ForkJoinPool` 中的公共线程池 `CommonPool`。
2. `CommonPool` 会被所有未指定线程池的异步调用共享，核心线程数通常为处理器数量减 1，单核环境下核心线程数为 1。
3. 所有异步回调共用同一个 `CommonPool` 时，核心业务和非核心业务会竞争同一批线程，容易成为系统瓶颈。

## 3. 手动传递线程池的好处
* 可以更方便地调节线程池参数。
* 可以给不同业务分配不同线程池，实现资源隔离，减少业务之间的相互干扰。

# 四、线程池隔离
## 1. 循环引用导致死锁
* 下面代码的问题是父任务和子任务使用同一个线程池，父任务内部又同步等待子任务完成。

```Java
public class CompletableFutureAPIDemo4 {
    public static void main(String[] args) throws ExecutionException, InterruptedException, TimeoutException {
        ExecutorService threadPool = Executors.newFixedThreadPool(10);
        CompletableFuture<String> parentTask = CompletableFuture.supplyAsync(() -> {
            try { TimeUnit.SECONDS.sleep(1); } catch (InterruptedException e) { e.printStackTrace(); }
            // 子任务
            return CompletableFuture.supplyAsync(() -> {
                System.out.println("childTask started");
                return "childTask finish";
            }, threadPool).join();
        }, threadPool);
        System.out.println(parentTask.join());
    }
}
```

## 2. 死锁链路
1. `main()` 方法中，父任务通过 `supplyAsync` 向 `threadPool` 请求线程，父任务内部的子任务又继续向同一个 `threadPool` 请求线程。
2. 如果同一时刻有 `10` 个请求到达，`threadPool` 被父任务打满，子任务请求线程时只能进入阻塞队列排队。
3. 父任务的完成依赖子任务执行结果，但子任务拿不到线程，所以父任务无法完成。
4. 主线程执行 `parentTask.join()` 后进入阻塞状态，并且永远无法恢复。

## 3. 修复方式
* 为了修复该问题，需要将父任务与子任务做线程池隔离，让两个任务请求不同线程池，避免循环依赖导致的阻塞。
