# 1. **程序**
1. 是为完成特定任务、用某种语言编写的一组指令的集合
1. 即指一段静态的代码，静态对象
# 2. **进程⭐**
1. 程序的一次执行过程，或是正在运行的一个程序
1. 是一个动态的过程：有它自身的产生、存在和消亡的过程
1. ==**进程间的通讯方式：**==
    
    1. **共享内存（Shared Memory）：**
        
        1. 共享内存是一种在不同进程之间共享内存区域的机制
        
        1. 多个进程可以==访问同一块物理内存区域==，这种方式==效率很高==
        
    
    1. **管道（Pipe）：**
        
        1. 管道是一种==半双工==的通信方式，通过操作系统内核中的一个==缓冲区==，实现两个进程之间的数据传输
        
    
    1. **消息队列（Message Queue）**：
        
        1. 消息队列是一种通过操作系统内核中的==缓冲区==实现的进程间通信机制
        
        1. 它可以实现不同进程之间的==异步通信==
        
    
    1. **信号量（Semaphore）：**
        
        1. 信号量是一种==计数器==，==用于多个进程之间的同步和互斥操作==
        
    
    1. **套接字（Socket）：**
        
        1. 套接字是一种用于进程间==网络通信==的通信机制
        
    
    1. **RPC（Remote Procedure Call）：**
        
        1. 远程过程调用是一种进程间通信方式，它可以使得在==不同主机上的进程之间==进行函数调用
        
    
# 3. **线程⭐**
1. 操作系统的调度的最小执行单元，也叫轻量级进程（Light Weight Process）
1. 一个进程中可以有多个线程
    
    1. **多个线程**==**共享进程的堆和方法区**==**（JDK1.8 之后的元空间）资源，但是每个**==**线程有自己私有的程序计数器、虚拟机栈 和 本地方法栈**==
    
    1. **这使得处理器可以在这些线程上高速切换，让使用者感觉到这些线程在同时执行**
    
    1. 例如：在 Java 中，当我们启动 main 函数时其实就是启动了一个 JVM 的进程，而 main 函数所在的线程就是这个进程中的一个线程，也称主线程
    
1. **线程与进程的区别**
    
    1. 线程是进程划分成的更小的运行单位
    
    1. 线程执行开销小，是运行调度的最小单位；而进程利于资源的管理和保护，是资源管理的最小单位
    
    1. 各进程是独立的，而同一进程中的线程极有可能会相互影响
    
1. **为什么进程切换比线程切换慢❓**
    
    1. 进程是操作系统中==资源分配的基本单位==
        
        1. 每个进程都有自己的==上下文、内存空间，以及系统资源==
        
        1. 所以切换进程时，除了需要保存当前进程的执行上下文信息外，还要保存内存页表等资源信息，并将系统资源分配给新的进程，所以比较耗时
        
    
    1. 而线程是==进程中的执行单元==，它与其他线程共享进程的地址空间和系统资源。线程切换只需要保存和恢复执行上下文即可
    
# 4. **使用多线程的优点**
- **切换成本：**线程可以比作是轻量级的进程，是程序执行的最小单位，线程间的切换和调度的成本远远小于进程
- **切换次数：**另外，多核 CPU 时代意味着多个线程可以同时运行，这减少了线程上下文切换的开销
- **适应高并发系统：**多线程并发编程是开发高并发系统的基础，线程机制可以大大提高系统整体的并发能力以及性能
- **单核情况下：**当使用多线程的时候，一个线程被 IO 阻塞，其他线程还可以继续使用 CPU。从而提高了效率
- **多核情况下：**创建多个线程，这些线程可以被映射到底层多个 CPU 上执行，任务执行的效率会有显著性的提高
# 5. **使用多线程的弊端**
## 5.1 死锁
1. 死锁
    
    - ==**在并发环境下，各进程因竞争资源而造成的一种互相等待对方手里的资源，导致各进程都阻塞，都无法向前推进的现象**==，就是“死锁”
    
    - 每个人都占有一个资源，同时又在等待对方手里的资源
    
    - 发生死锁后若无外力干涉，这些进程都将无法向前推进
    
1. **死锁产生的必要条件⭐**，必须同时满足四个条件，只要有一个条件不成立，死锁就不会发生
    
    1. ==**互斥条件：**==
        
        1. 只有对必须互斥使用的资源的争抢才会导致死锁（如哲学家的筷子、打印机设备）
        
        1. 像内存、扬声器这样的可以同时让多个进程同时使用的资源是不会导致死锁的（因为进程不用阻塞等待这种资源）
        
    
    1. ==**不剥夺条件：**==
        
        1. 进程所获得的资源在未使用完之前，不能由其他进程强行夺走
        
        1. 只能是主动释放资源
        
    
    1. ==**请求和保持条件：**==
        
        1. 进程已经至少保持了一个资源，但又提出了一个新的资源请求，而该资源又被其他进程占有
        
        1. 此时请求进程被阻塞，但又对自己持有的资源保持不放
        
    
    1. ==**循环等待条件：**==
        
        1. 存在一种进程资源的循环等待链
        
        1. 链中的每一个进程以获得的资源同时被下一个进程所请求等待
        
    
1. **死锁的处理策略⭐**[[2.7 死锁]]
    
    - **预防死锁：**破坏四个条件的某一个，不让死锁发生
        
        - **破坏互斥条件**：允许系统资源都能共享使用，则系统不会进行死锁状态。这种方案并不太可行，因为有些资源根本就不能同时访问，比如打印机
        
        - **破坏不剥夺条件**：当一个已经保持了某些不可剥夺资源的进程，请求新的资源时得不到满足，它必须释放已经保持的所有资源，待以后需要时再重新申请
        
        - **破坏请求和保持条件**：采用预先静态分配方法，即进程在运行前一次申请完他所需要的全部资源，在他的资源未满足前，不把它投入运行。一旦运行后，这些资源就一直归它所有，也不再提出其他资源请求
        
        - **破坏循环等待条件**：采用顺序资源分配法。首先给系统中的资源编号，规定每个进程，必须按编号递增的顺序请求资源，同类资源一次申请完。也就是说，只要进程提出申请分配资源，则该进程在以后的资源申请中，只能申请编号比之前大的资源
        
    
    - **避免死锁：**避免死锁就是在资源分配时，借助于算法（比如银行家算法）对资源分配进行计算评估，使系统总处于安全状态
    
    - **死锁的检测和解除：**允许死锁的发生，但操作系统会负责检测出死锁的发生，然后采取某种措施解除死锁
    
## 5.2 上下文切换
1. **上下文：**
    
    1. 线程在执行过程中属于自己的==运行条件和状态==（也称上下文）
    
    1. 比如：**==程序计数器，栈信息==**等
    
1. **上下文切换：**
    
    1. 需要==保存当前==线程的上下文，留待线程下次占用 CPU 的时候恢复现场
    
    1. 并==加载下一个==将要占用 CPU 的线程上下文
    
    1. 由于涉及到数据的保存和切换，也就意味着==会有一定损耗==，如果频繁切换上下文就会造成效率降低
    
1. **出现上下文切换的时机**
    
    1. 线程主动让出 CPU，比如调用了 `sleep()`，`wait()` 等
    
    1. 时间片用完
    
    1. 调用了阻塞类型的系统中断，比如请求IO，线程被阻塞
    
# 6. 线程的生命周期
## 6.1 操作系统中的五种状态
1. **初始状态：（新建状态）**
    
    - 仅是在语言层面创建了线程对象，还未与操作系统线程关联
    
1. **可运行状态：（就绪状态）**
    
    - 指该线程已经被创建（与操作系统线程关联），可以由 CPU 调度执行
    
1. **运行状态：**
    
    - 指获取了 CPU 时间片运行中的状态
    
    - 当 CPU 时间片用完，会从运行状态转换至可运行状态，会导致线程的上下文切换
    
1. **阻塞状态：**
    
    - 如果调用了阻塞 API，如 BIO 读写文件，这时该线程实际不会用到 CPU，会导致线程上下文切换，进入阻塞状态
    
    - 等 BIO 操作完毕，会由操作系统唤醒阻塞的线程，转换至可运行状态
    
    - 与可运行状态的区别是，对阻塞状态的线程来说只要它们一直不唤醒，调度器就一直不会考虑调度它们
    
1. **终止状态：**
    
    - 表示线程已经执行完毕，生命周期已经结束，不会再转换为其它状态

- **`NEW`：线程刚被创建，但是还没有调用** `**start()**` **方法**
- `**RUNNABLE**`：
    
    - 当调用了 `start()` 方法之后
    
    - **Java API 层面的 RUNNABLE 状态涵盖了操作系统层面的【就绪状态】、【运行状态】**
    
    - 可能正在运行，也可能正在等待 CPU 时间片
    
- **`BLOCKED` ， `WAITING` ， `TIMED_WAITING`：都是 Java API 层面对操作系统层面的【阻塞状态】的细分**
    
    - **`BLOCKED` ：**
        
        - **等待获取一个排它锁，如果其线程释放了锁就会结束此状态**
        
    
    - **`WAITING` ：**
        
        - **等待其它线程显式地唤醒，否则不会被分配 CPU 时间片**
        
    
    - `**TIMED_WAITING**`**：**
        
        - **无需等待其它线程显式地唤醒，在一定时间之后会被系统自动唤醒**
        
    
- **`TERMINATED`：当线程代码运行结束**
    
    > [!important] 当 Java 线程终止时，会进行什么操作
    > 
    > 1. 会自动释放线程所持有的资源，避免资源泄漏
    > 
    > 1. 同时，线程终止后，可以返回一个终止状态，通常是一个整数或枚举值，用于表示线程的执行结果或终止的原因
    > 
    > 1. 最后，线程的 Thread 对象如果没有再被引用的话，会被垃圾回收
    
# 7. 用户线程和守护线程
- Java 线程分为用户线程和守护线程
- **用户线程：**
    
    - 线程的 `daemon` 属性为 `false` 表示是用户线程
    
    - 是系统的工作线程，它会完成这个程序需要完成的业务操作
    
    - 注意：main() **主线程**也属于用户线程
    
- **守护线程：**
    
    - 线程的 `daemon` 属性为 `true` 表示是守护线程
    
    - Daemon 线程，是一种特殊的线程，在后台默默地完成一些系统性的服务，比如**垃圾回收线程**，主要被用作程序中后台调度以及支持性工作
    
    - 当所有非守护线程结束时，程序也就终止，同时会杀死所有守护线程
    
    - 当系统只剩下守护进程的时候，Java 虚 拟机会自动退出
    
    - 可以通过 `thread.setDaemon(``_true_``);` 将线程设置为 Daemon 线程
        
        > [!important] Daemon 属性需要在启动线程之前设置，不能在启动线程之后设置。否则会报错

# 8. **线程的优先级**
1. 线程的优先级等级，**让优先级更高的线程**==**更大概率（但不是一定优先获得）获得 CPU 时间片**==
    
    1. `MAX_PRIORITY`： 10
    
    1. `MIN _PRIORITY`： 1
    
    1. `NORM_PRIORITY`： 5（默认）
    
1. 如何获取和设置当前线程的优先级值
    
    1. `**getPriority()**`：返回线程优先值
    
    1. `**setPriority(int newPriority)**`：改变线程的优先级
    
    > [!important] 注意：低优先级只是获得调度的
    > 
    > **概率低**，并非一定是在高优先级线程之后才被调用
    
1. 示例：
    
    ```Java
     class MyThread extends Thread{
         @Override
         public void run() {
             for (int i = 0; i < 100; i++) {
                 if(i % 2 == 0){
                     System.out.println(i+Thread.currentThread().getName()+Thread.currentThread().getPriority());//获取优先级
                 }
             }
         }
     }
     public class ThreadTest {
         public static void main(String[] args) {
             MyThread myThread1 = new MyThread();
             myThread1.start();
             myThread1.setPriority(Thread.MAX_PRIORITY);//设置
             myThread1.setName("线程一");
     
             MyThread myThread2 = new MyThread();
             myThread2.start();
             myThread2.setPriority(Thread.MIN_PRIORITY);//设置
             myThread2.setName("线程二");
         }
     }
    ```
