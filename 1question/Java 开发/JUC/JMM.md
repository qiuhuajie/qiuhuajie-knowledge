# 1. 介绍
1. JMM（Java Memory Model）
    
    1. **JMM 是 Java 定义的并发编程相关的一组规范**
        
        1. 定义了==在 Java 中多线程程序中如何与内存交互==
        
        1. 定义了 ==Java 虚拟机如何在多线程环境下保证内存==**==可见性==**==、==**==原子性==**==和==**==有序性==**，指导 JVM 如何在多处理器环境下正确地执行Java程序（并发问题出现的三个根源：可见性、原子性、有序性）
        
    
    1. 在编写多线程程序时，开发人员需要遵守 JMM 的规则，以确保程序能够正确地执行，确保线程安全
    
    1. 具体来说，JMM 定义了：
        
        - **各种内存访问操作的顺序和同步语义**，包括读和写操作、锁的获取和释放、volatile 变量的读写操作等
        
        - **Happens-Before 规则**
        
    
1. **主要目的**
    
    1. 对 线程 和 主内存 之间的关系进行抽象，**简化多线程编程**，开发人员直接使用并发相关的一些关键字和类（比如 `volatile`、`synchronized`、各种 `Lock`）即可开发出并发安全的程序
    
    1. Java 语言是跨平台的，Java 自己提供一套内存模型以**屏蔽系统差异**，增强程序可移植性
    
1. **JMM 规范下，多线程对变量的读写过程⭐（图很重要）**
    
    1. 定义的所有共享变量都储存在物理主内存中
    
    1. 每个线程都有自己独立的工作内存，里面保存该线程使用到的变量的副本（主内存中该变量的一份拷贝）
        
        1. 写：线程对共享变量所有的操作都必须**先在线程自己的工作内存中进行后写回主内存**，不能直接从主内存中读写（不能越级）
        
        1. 读：不同线程之间也**无法直接访问其他线程的工作内存中的变量**，线程间变量值的传递需要通过主内存来进行（同级不能相互访问）

# 2. **可见性**
## 2.1 CPU 缓存**引起**
1. CPU 缓存则是**为了解决 CPU 处理速度和内存处理速度不对等的问题**
    

    ![[IMG-20260404031737124.png|Untitled 1 46.png]]

    
    1. CPU缓存模型是指CPU在执行程序时所使用的一种内存管理方式，**设计目的是为了**==**减少CPU访问主内存的次数**==**，从而提高计算机的性能和运行速度**
    
    1. **CPU缓存模型可以分为三级缓存，分别是L1、L2、L3缓存**
        
        1. L1缓存是CPU内部的缓存，速度最快，也是最小的缓存，它通常分为指令缓存和数据缓存两部分，用于存储最常用的指令和数据
        
        1. L2缓存位于CPU和主内存之间，它的容量比L1缓存大，速度比主内存快，但比L1缓存慢。
        
        1. L3缓存则是最大的缓存，通常位于CPU和主内存之间，它的容量比L2缓存大，但速度比L2缓存慢，同时也比主内存快
        
    
    1. 在CPU的缓存模型中
        
        1. 当CPU需要**读取数据**时，它首先会在L1缓存中查找，如果没有找到，则会继续在L2缓存和L3缓存中查找，最后才会到主内存中查找
        
        1. 同样地，当CPU需要**写入数据**时，它首先会在L1缓存中写入，然后再将数据写入L2缓存和L3缓存，最后才会写入主内存
        
    
    1. CPU缓存一致性
        
        1. 由于多个处理器或核心可以访问同一块内存区域，当其中一个处理器或核心更新了这块内存区域的数据时，其他处理器或核心可能会访问到过期的缓存数据，导致数据不一致的情况发生
        
        1. **通常采用**==**“缓存一致性协议”**==**来解决缓存一致性问题**
            
            1. 它通过在各级缓存之间传递**控制信息**来保证缓存数据的一致性
                
                1. ==当一个处理器或核心==**==修改了某个内存区域的数据==**==时，它会发送一个信号给其他处理器或核心，通知它们将该内存区域的==**==缓存数据==**==置为==**==无效状态==**
                
                1. 从而保证其他处理器或核心在后续的访问中可以获得最新的数据
                
            
            1. 需要注意的是，由于缓存一致性协议的实现需要进行一定的**额外开销**，因此它可能会影响系统的性能
            
        
    
1. 假若执行线程 1 的是 CPU1，执行线程 2 的是 CPU2
    
    ```Java
    //线程1执行的代码
    int i = 0;
    i = 10;
     
    //线程2执行的代码
    j = i;
    ```
    
    1. 由上面的分析可知，当线程 1 执行 `i = 10` 这句时，会先把 `i` 的初始值加载到 CPU 1 的高速缓存中，然后赋值为 10，那么在 CPU1 的高速缓存当中 `i` 的值变为10了，却没有立即写入到主存当中
    
    1. 此时线程 2 执行 `j = i`，它会先去主存读取 `i` 的值并加载到 CPU2 的缓存当中，注意此时内存当中 `i` 的值还是 `0`，那么就会使得 `j` 的值为 `0`，而不是`10`
    
    1. 这就是可见性问题，线程1 对变量 `i` 修改了之后，线程 2 没有立即看到线程 1 修改的值

# 3. **原子性**
## 3.1 **分时复用引起**
1. 问题示例：
    
    ```Java
    int i = 1;
    
    // 线程1执行
    i += 1;
    
    // 线程2执行
    i += 1;
    ```
    
1. 这里需要注意的是：`i += 1` 需要三条 CPU 指令
    
    1. 将变量 `i` 从内存读取到 CPU寄存器
    
    1. 在CPU寄存器中执行 `i + 1` 操作
    
    1. 将最后的结果 `i` 写入内存（缓存机制导致可能写入的是 CPU 缓存而不是内存）
    
1. 由于CPU分时复用（线程切换）的存在，线程 1 执行了第一条指令后，就切换到线程 2 执行，假如线程 2 执行了这三条指令后，再切换会线程 1 执行后续两条指令，将造成最后写到内存中的 i 值是 2 而不是 3
## 3.2 保证原子性的方法
1. **==一次操作或者多次操作，要么所有的操作全部都得到执行并且不会受到任何因素的干扰而中断，要么都不执行==**
1. 在 Java 中，可以借助 `synchronized`、CAS 以及各种原子类实现原子性[[CAS，Unsafe 和 原子类]]
# 4. **有序性**
## 4.1 指令重排引起
1. 存在指令重排，是 CPU 和 JIT 编译器在运行时的一些优化
1. 造成代码的执行顺序未必就是编写代码时候的顺序，但指令重排==只保证了==**==单线程==**==串行时的不对结果造成影响==
1. 所以在多线程下，指令重排序可能会导致一些问题（这个问题需要通过大量测试才能复现）
    
    ```Java
    int num = 0;
    boolean ready = false;
    
    // 线程1 执行此方法
    public void actor1(I_Result r) {
    		if(ready) {
    				r.r1 = num + num;
    		} else {
    				r.r1 = 1;
    		}
    }
    
    // 线程2 执行此方法
    public void actor2(I_Result r) {
    		// 下面两个指令可能会重排序，所以 r.r1 可能为 0
    		num = 2;
    		ready = true;
    }
    ```

# 5. Volatile 关键字
## 介绍
1. 在 Java 中，`volatile` 关键字可以禁止指令进行重排序优化，`volatile` 可以修饰成员变量和静态成员变量
1. **可以使用 `volatile` 实现了 JMM 中的==可见性==和==有序性==，但 `volatile` 无法保证原子性**[[CAS，Unsafe 和 原子类]][[CAS，Unsafe 和 原子类]][[CAS，Unsafe 和 原子类]]
1. `volatile` **的底层实现原理是**==**内存屏障**==，Memory Barrier（Memory Fence）
    
    - 对 `volatile` 变量的**写指令后会加入写屏障**
    
    - 对 `volatile` 变量的**读指令前会加入读屏障**
    
## 原理：**如何保证可见性❓**
1. 写屏障（sfence）保证在该屏障**之前**的，对共享变量的改动，都同步到主存当中
    
    ```Java
    public void actor2(I_Result r) {
    		num = 2;
    		ready = true; // ready 是 volatile 赋值带写屏障
    		// 写屏障：可以保证之前的所有代码都刷新到主存中
    }
    ```
    
1. 而读屏障（lfence）保证在该屏障**之后**，对共享变量的读取，加载的是主存中最新数据
    
    ```Java
    public void actor1(I_Result r) {
    		// 读屏障：保证之后的所有代码读的都是变量的在主存中的最新值
    		// ready 是 volatile 读取值带读屏障
    		if(ready) {
    				r.r1 = num + num;
    		} else {
    				r.r1 = 1;
    		}
    }
    ```
    
## 原理：**如何保证有序性❓**
1. 写屏障会确保指令重排序时，不会将写屏障**之前**的代码排在写屏障之后
    
    ```Java
    public void actor2(I_Result r) {
    		num = 2;
    		ready = true; // ready 是 volatile 赋值带写屏障
    		// 写屏障：保证之前的代码不会被重排到写屏障之后
    }
    ```
    
1. 读屏障会确保指令重排序时，不会将读屏障**之后**的代码排在读屏障之前
    
    ```Java
    public void actor1(I_Result r) {
    		// 读屏障：保证之后的代码不会被重排到读屏障之前
    		// ready 是 volatile 读取值带读屏障
    		if(ready) {
    				r.r1 = num + num;
    		} else {
    				r.r1 = 1;
    		}
    }
    ```

# 5. Happens-before
1. "JMM" 是指 Java Memory Model，它定义了在 Java 中多线程程序中内存访问的行为
1. 其中，"happens-before" 原则是 JMM 中的一个关键概念，用于**描述内存访问操作之间的顺序关系**
1. **通过遵循 "happens-before" 原则，Java 程序员可以确保多线程程序中的内存访问操作的顺序关系，从而避免一些常见的并发问题，如竞态条件和死锁等**
1. "happens-before" 原则指出
    
    - 如果一个操作 happens-before 另一个操作
        
        - 那么==第一个操作的==**==执行结果==**==将对第二个操作==**==可见==**
        
        - 而且==第一个操作的==**==执行顺序==**==排在第二个操作==**==之前==**
            
            > 但并不意味着一定要按照 happens-before 原则制定的顺序来执行，如果重排序之后的执行结果与按照 happens-before 关系来执行的结果一致，那么这种重排序并不非法
