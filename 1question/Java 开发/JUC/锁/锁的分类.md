# 一、乐观锁与悲观锁

![[IMG-20260619224349903.png|725]]

## 1. 悲观锁
1. 核心思想：
    1. 认为线程安全问题==**一定会发生**==
    2. **因此在操作数据之前先获取同步锁，确保线程串行执行，其他拿不到锁的线程都要阻塞**
2. 例如 **`Synchronized`**、**`Lock`** 都属于悲观锁
3. 优点：简单粗暴
4. 缺点：性能一般
5. 场景：**适合写操作多的场景，先加锁可以保证写操作时数据正确**
6. 悲观锁实现：[[一人限购一单（悲观锁）]]
## 2. 乐观锁
1. 核心思想：
    1. 认为线程安全问题==**不一定会发生**==
    2. **因此**==**不加锁**==**，只是在更新数据时，去**==**判断有没有其它线程对数据做了修改**==
        1. 如果没有修改则认为是安全的，自己才更新数据
        2. 如果已经被其它线程修改说明发生了安全问题，此时可以重试或异常
2. 优点：性能好
3. 缺点：成功率低
4. 场景：**适合读操作多的场景，不加锁的特点能够使其读操作的性能大幅提升**
5. 实现方式：
    - 1️⃣ **版本号法**
        1. 每次对共享资源修改之前，**根据**==**比较版本号**==**`version`** ==**字段**==**的结果**，判断在我拿到资源到即将修改这段时间内，有没有其他线程对共享资源进行了修改操作
            ![[IMG-20260619224350012.png|338]]
        2. MyBatis-Plus 乐观锁插件会自动维护、查询比较 `version` 字段，只需要对操作结果判断，再加错误重试的代码

            ![[IMG-20260619224350089.png|481]]

        3. 实现：[[MyBatis-Plus 乐观锁插件]]
    - 2️⃣ **`CAS`**（`compare and swap`）
        1. 版本号法需要多维护一个字段，但 **CAS 法**==**直接在判断修改时使用已有字段**==

            ![[IMG-20260619224350127.png|192]]

        2. CAS 的思想很简单，就是==用一个==**==预期值==**==和要==**==更新的变量值==**==进行比较，==**==两值相等才会进行更新==**
        3. 如，由于对库存的操作会使库存减少，因此可以比较当前库存（要更新的值）与之前查到的库存数（预期值）是否一致，进而可以判断共享资源是否有修改过

            ![[IMG-20260619224350158.png|525]]

# 二、公平锁与非公平锁
## 1. 公平锁
1. 公平锁：
    1. 多个线程按照申请锁的顺序来获取锁，线程直接进入队列中排队
    2. 队列中的第一个线程才能获得锁
2. 优点：等待锁的线程不会饿死
3. 缺点：是整体吞吐效率相对非公平锁要低，等待队列中除第一个线程以外的所有线程都会阻塞，CPU唤醒阻塞线程的开销比非公平锁大
## 2. 非公平锁
1. 非公平锁：
    1. 多个线程都可以直接尝试获取锁，获取不到才会到等待队列的队尾等待
    2. 如果一个后来的线程直接去获取，此时锁刚好可用，那么这个线程可以无需阻塞直接获取到锁
    3. 所以非公平锁有可能出现后申请锁的线程先获取锁的场景
2. 优点：
    1. 因为 CPU 恢复一个挂起状态的线程是有一定开销的
    2. 而非公平锁中，==**线程有几率不进入队列阻塞，就直接获得锁，所以 CPU 不必唤醒所有线程，避免了一定的开销，提高了吞吐量**==
3. 缺点：是处于等待队列中的线程可能会饿死，或者等很久才会获得锁
# 三、可重入锁不可重入锁
## 1. 可重入锁
1. 可重入锁，又名递归锁
    1. **是指在同一个线程**==**在外层方法获取锁的情况下**==**，**==**再进入该线程的内层方法会自动获取锁**==**（前提：⭐锁对象得是同一个锁，即同一个对象或者 class）**
    2. **不会因为之前已经获取过还没释放而阻塞**==**（不会自己锁自己）**==
2. **优点：**可一定程度==避免死锁==
3. 与可重入锁相反，不可重入锁不可递归调用，递归调用就发生死锁
## 2. 可重入锁的种类
### 2.1 A. Synchronized
1. synchronized 关键字使用的锁，默认是可重入锁
2. 代码示例：

    ```Java
    public class Widget {
        public synchronized void doSomething() {
            System.out.println("方法1执行...");
            doOthers();
        }
        public synchronized void doOthers() {
            System.out.println("方法2执行...");
        }
    }
    ```
    - 在上面的代码中，类中的两个方法都是被内置锁 `synchronized` 修饰的，`doSomething()` 方法中调用 `doOthers()` 方法
    - 因为**内置锁 `synchronized` 是可重入锁**
        - 所以同一个线程在调用 `doOthers()` 时==可以直接获得当前对象的锁==，进入 `doOthers()` 进行操作
    - **如果是一个不可重入锁**
        - 那么当前线程在调用 `doOthers()` 之前需要将执行 `doSomething()` 时获取当前对象的锁释放掉，实际上该对象锁已被当前线程所持有，且无法释放。所以此时==会出现死锁==
3. synchronized 可重入原理
    1. **每个锁对象拥有一个锁计数器 `recursions` 和一个指向持有该锁的线程的指针 `owner`**
    2. 当执行 `monitorenter` 获取锁时
        1. 如果目标锁对象的计数器 `recursions == 0`，那么说明它没有被其他线程所持有，Java 虚拟机会将该锁对象的持有线程设置为当前线程，并且将其计数器 `+ 1`
        2. 在目标锁对象的 `recursions != 0` 的情况下
            1. **如果**==**锁对象的持有线程是当前线程**==**，那么将其计数器 `+ 1`**
            2. 否则需要等待，直至持有线程释放该锁
    3. 当执行 `monitorexit` 时
        1. **则需将锁对象的计数器 `- 1`**
        2. 计数器为零代表锁已被释放

### 2.2 B. ReentrantLock
1. 显式锁 Lock 中，也有 ReentrantLock 这样的可重入锁
2. 代码示例：

    ```Java
    public class ReEntryLockDemo {
        static Lock lock = new ReentrantLock();
        public static void main(String[] args) {
            new Thread(() -> {
                lock.lock();
                try {
                    System.out.println("----外层调用lock");
                    lock.lock();
                    try {
                        // 第二块同步代码，可以自动获取锁 
                        System.out.println("----内层调用lock");
                    } finally {
                        lock.unlock();
                    }
                } finally {
                    lock.unlock();
                }
            }, "a").start();
        }
    }
    ```
3. 可重入锁 ReentrantLock 在使用时，**加锁次数和释放次数一定要匹配**，加锁几次就要解锁几次

# 四、独占锁和共享锁
1. 独占锁
    1. 也叫排他锁，是指==**该锁一次只能被一个线程所持有**==，如果线程 T 对数据 A 加上排它锁后，则其他线程不能再对 A 加任何类型的锁
    2. 获得排它锁的线程**既能读数据又能修改数据**
    3. JDK中的 synchronized 和 JUC 中 Lock 的实现类就是独占
2. 共享锁
    1. 是指==**该锁可被多个线程所持有**==，但如果线程 T 对数据 A 加上共享锁后，则其他线程**只能对 A 再加共享锁，不能加独占锁**
    2. 获得共享锁的线程**只能读数据，不能修改数据**
    3. 如 [[读写锁]]
3. 独占锁与共享锁也是通过 AQS 来实现的，通过实现不同的方法，来实现独享或者共享
# 五、自旋锁与非自旋锁
## 1. 自旋锁
1. 自旋锁解决的问题：
    1. CPU 需要切换状态来阻塞或唤醒一个 Java 线程，需要耗费处理器时间
    2. ==如果==**==同步代码块中的内容过于简单==**==，==**==状态转换消耗的时间==**==有可能比==**==用户代码执行的时间==**==还要==**==长==**
    3. 为了这一小段时间去切换线程，线程挂起和恢复现场的花费可能会让系统得不偿失
2. 自旋锁
    1. 如果物理机器有多个处理器，能够让两个或以上的线程同时并行执行
        1. 可以==让后面那个请求锁的线程不放弃 CPU 的执行时间，原地自旋等待持有锁的那个线程释放锁==
        2. 如果在自旋完成后前面锁定同步资源的线程已经释放了锁，那么当前线程就==可以不必阻塞而是直接获取同步资源==
        3. 从而避免 CPU 切换状态唤醒换线程的开销
    2. 自旋锁与非自旋锁获取锁的流程如下图
        ![[IMG-20260619224350197.png|536]]
3. 自旋锁的缺点：
    1. 首先，它不能代替阻塞
    2. 其次，自旋等待虽然避免了线程切换的开销，==但它要让后面那个线程==**==占用处理器时间来自旋重试==**
        1. 如果锁被占用的时间很短，自旋等待的效果就会非常好
        2. 反之，**如果锁被占用的时间很长，那么自旋的线程只会白浪费处理器资源**
    3. 所以，==自旋等待的时间必须要有一定的限度==，如果自旋超过了限定次数（**默认是10次**，可以使用`-XX:PreBlockSpin`来更改）没有成功获得锁，就应当挂起线程

## 2. 自适应自旋锁
1. JDK 6中默认开启自旋锁，并且引入了自适应的自旋锁
2. **自适应意味着**==**自旋的时间（次数）不再固定**==**，而是**==**由前一次在同一个锁上的自旋时间 及 锁的拥有者的状态 来决定**==
    1. 如果在同一个锁对象上，自旋等待刚刚成功获得过锁，并且持有锁的线程正在运行中，那么虚拟机就会认为这次自旋也是很有可能再次成功，进而它将允许自旋等待**持续相对更长的时间**
    2. 如果对于某个锁，自旋很少成功获得过，那在以后尝试获取这个锁时将可能**省略掉自旋过程，直接阻塞线程**，避免浪费处理器资源

## 3. 自旋锁的实现原理
1. 自旋锁的实现原理同样也是 ==**CAS + 循环条件**==
2. 例如：AtomicInteger 中调用 unsafe 进行自增操作的源码中，`do-while` 循环就是一个自旋操作，如果修改数值失败则通过循环来执行自旋，直至修改成功
