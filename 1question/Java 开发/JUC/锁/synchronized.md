# 一、介绍
1. `synchronized` 翻译过来是**同步**的意思，它是 Java 中一个关键字
    1. 是 JVM 层面提供的==同步锁机制==，用于==保证多线程访问同一资源的可见性、互斥性==
    2. 即当一个线程已经获取资源锁时，其他试图获取的线程只能等待或者阻塞在那里
2. 在 Java 5 以前，`synchronized` 是仅有的同步手段，Java 5 才有了 JUC 并发包实现锁的
3. 是一个 可重入、非公平的悲观锁
# 二、Synchronized 的使用
1. ==**普通同步方法**==
    - 对**当前实例加锁**，通常指 `this`，进入同步代码前要获得当前实例的锁
    - 加锁的对象在堆中

    ```Java
    synchronized void method() {
    }
    ```
2. ==**静态同步方法**==
    - 对**当前类 `.class` 加锁**，进入同步代码前要获得当前类对象的锁
    - 加锁的对象在方法区中
    - 由于加锁对象不同，普通 `synchronized` 方法和静态 `synchronized` 方法之间的调用是不互斥的

    ```Java
    synchronized static void method() {
    }
    ```
3. ==**同步类**==
    - 对**当前类 `.class` 加锁**，也就是说两个线程调用同一个类的不同对象上的这种同步语句，也会进行同步

    ```Java
    synchronized (SynchronizedExample.class) {
    }
    ```
4. ==**同步代码块**==
    - 对**括号里配置的对象加锁**

    ```Java
    Object o = new Object();
    synchronized(o) {
    }
    ```
> 阿里开发手册：
> **高并发时，同步调用应该去考量锁的性能损耗**
> - 能用无锁数据结构，就不要用锁；能锁区块就不要锁整个方法体；能用对象锁就不要用类锁
> - 尽可能使加锁的代码块工作量尽可能的小
> - 避免在锁代码块中调用 RPC 方法
# 三、synchronized 底层原理
## 1. Monitor 管程
1. Monitor 被翻译为**监视器**或**管程**
    1. 在底层 C++ 为：**`ObjectMonitor.java`** → **`ObjectMonitor.cpp`** → **`objectMonitor.hpp`**
    2. ObjectMonitor 中有几个关键属性

        ![[IMG-20260619221925884.png|572]]

    3. ==线程在获取锁的时候，相当于去争抢修改 Monitor 中的锁标识==
2. 多个线程对同一对象锁的竞争过程

    ![[IMG-20260619221925959.png|608]]

    - WaitSet 中的 Thread-0，Thread-1 是之前获得过锁，但条件不满足**进入 WAITING 状态的线程，会释放锁**
    - 此时 Monitor 中 Owner 为 null
    - 当 Thread-2 执行 synchronized(obj) 就会**将 Monitor 的所有者 Owner 置为 Thread-2，Monitor中只能有一个 Owner**
    - 在 Thread-2 上锁的过程中，如果 Thread-3，Thread-4，Thread-5 也来执行 synchronized(obj)，就会**进入 EntryList BLOCKED**
    - **Thread-2 执行完同步代码块的内容，然后唤醒 EntryList 中等待的线程来竞争锁，竞争的时是非公平的**
3. ⭐**如何给对象上锁**❓
    1. 每个 Java 对象，在加了 `synchronized` 关键字后，会**关联一个 Monitor 对象**
    2. 如果使用 `synchronized` 给对象上锁（重量级）之后，该**对象头的 `Mark Word` 中就被设置为指向 Monitor 对象的指针**
    3. 不上锁的对象不会关联监视器，不遵从以上规则

## 2. 同步代码块实现方式
1. 由一对 **`monitorenter`** / **`monitorexit`** 指令实现。

    ![[IMG-20260619221925995.png|445]]

2. ==**Monitor 的 monitorenter 和 monitorexit 两个指令是怎么保证同步的❓**==
    1. 虚拟机可以从方法常量池中的方法表结构中的 `ACC SYNCHRONIZED` 访问标志得知一个方法是否被声明为同步方法
    2. `monitorenter`：管程的获取
        1. `monitorenter` 指令指向同步代码块的开始位置
        2. 当调用同步方法时，执行线程就要求先成功持有管程，然后才能执行方法，持有管程即修改 monitor 的所有者 `_owner` 这字段
        3. 在方法执行期间，执行线程持有了管程，其他任何线程都无法再获取到同一个管程
    3. `monitorexit`：管程的释放
        1. `monitorexit` 指令则指明同步代码块的结束位置
        2. 当方法完成（无论是正常完成还是非正常完成）时释放管程
            1. 如果一个同步方法执行期间抛出了异常，并且在方法内部无法处理此异常，那这个同步方法所持有的管程，将在异常抛到同步方法边界之外时自动释放
            2. 上面指令代码中中发现有两个 `monitorexit`，其实后面这个是保证同步代码块抛出异常时，锁也能得到正确的释放

# 四、锁升级
1. 使用锁升级机制的原因：
    1. 在 Java6 之前，`synchronized` 属于==重量级锁，效率低下==，因为依靠操作系统内部的管程 **`Monitor`** ，所以 CPU 在阻塞或唤醒一个 Java 线程中要消耗很多的时间
    2. 从 Java 6 之后，==为了减少获得锁和释放锁所带来的性能消耗==，**引入偏斜锁（Biased Locking）、轻量级锁和重量级锁**，大大改进了其性能
2. ⭐所谓锁的升级，就是 JVM **优化 `synchronized` 运行的机制**
    1. ==当 JVM 检测到==**==不同的竞争状况==**==时，会==**==自动切换到适合的锁实现==**==，这种切换就是锁的升级==
    2. 切换过程：无锁 ➡️ 偏向锁 ➡️ 轻量级锁 ➡️ 重量级锁
    3. **主要出于对以下两种性能消耗的考虑：**
        1. CPU 频繁唤醒 / 挂起线程所带来的资源消耗
        2. 自旋等待锁释放时，占用 CPU 资源
3. 该优化主要是基于 `Mark Word` 来实现的
## 1. Mark Word
1. JVM 内存中，对象由三部分组成：对象头、实例数据、字节对齐
2. 其中对象头由 `Mark Word` 和 `Klass Point` 构成
    - ==**Mark Word**====（标记字段）：用于存储对象自身的运行时数据==

## 3. 偏向锁
1. 解决的问题：只有在一个线程执行同步时，提高性能
2. 偏向锁的思想：
    1. 如果当==一段同步代码一直被同一个线程多次访问==，这个线程会被认为是==锁的偏向线程==，偏向线程在访问时，会==自动获得锁==
    2. 类似于：老顾客来访，直接老规矩
3. ==**偏向锁的获取流程：**==**⭐**
    1. 在锁第一次被拥有的时候，使用 CAS 操作在 Mark Word 记录下偏向线程的 ID，这样偏向线程就一直持有着锁
    2. 当线程再次进入时，首先会比较 Mark Word 中存着的偏向线程的 ID
        1. 如果一致，则直接进入同步，无需再获取锁
        2. 如果不一致，表示有新的线程来竞争锁了，见下 轻量级锁

        ![[IMG-20260619221926262.png|472]]

## 4. 轻量级锁
1. 解决的问题：
    1. 如果当前只有**==少量的线程竞争==**，比如==两个线程交替进行竞争==
    2. 轻量级锁可以通过 CAS，减少重量级锁使用操作系统互斥量产生的性能消耗，说白了先自旋再阻塞
    3. **但只适用**==**多核 CPU**==**，如果只有一个 CPU，在上面自旋反而浪费资源**
2. 偏向锁到轻量级锁的升级：
    1. 当关闭偏向锁功能
    2. 或多线程竞争偏向锁
3. ==**偏向锁到轻量级锁的升级过程：**==**⭐**
    1. 假如开始只有线程 A ，已经拿到锁，这个锁是一个偏向锁
    2. 这时线程 B 又来抢该对象的锁，线程B在争抢时发现对象头 Mark Word 中的线程 ID 不是线程 B 自己的线程 ID，而是线程 A的 ID，那线程 B 就会进行 CAS 操作希望能获得锁
        1. 如果 A 此时不在执行，B 可以获取锁成功，直接替换 Mark Word 中的线程 ID 为 B 自己的 ID，此时 **B 成为了当前锁的偏向线程**，锁还是会保持偏向锁状态
        2. 如果 A 此时在执行，B 锁获取失败，则**偏向锁升级为轻量级锁**
            1. 此时轻量级锁由原持有偏向锁的线程 A 持有，继续执行其同步代码，此时 Mark Word 中存的指针指向的是线程 A 栈帧中的锁记录
            2. 而正在竞争的 **线程 B ，并不会阻塞，而是会进入自旋等待** 获得该轻量级锁

    ![[IMG-20260619221926354.png|642]]

    3. **Mark Word 中存的指针：指向的是线程**==**栈帧中的锁记录**==**，相对于重量级锁中指向 Monitor 更轻量**

    ![[IMG-20260619221926387.png|580]]

4. **轻量级锁本质就是一个自旋锁**[（详见 自旋锁）](https://www.notion.so/832965d633444d279d2cc30b03f53b99?pvs=21)

## 5. 重量级锁
1. 解决的问题：
    1. 适用于**==竞争激烈==**的情况
    2. 如果同步方法/代码块执行时间很长，那么使用轻量级锁自旋带来的性能消耗就比使用重量级锁更严重（线程一直自旋占用 CPU）
    3. 这时候就需要升级为重量级锁
2. **==偏向锁到轻量级锁的升级过程：==⭐（锁膨胀）**
    1. 轻量级锁中
        1. 如果 A 此时在执行，B 锁获取失败，则偏向锁升级为轻量级锁，此时轻量级锁由原持有偏向锁的线程 A 持有，继续执行其同步代码
        2. 而正在竞争的线程 B 会进入自旋等待获得该轻量级锁
    2. 如果
        1. **竞争的线程数太多**，比如上来就是10000个，那么这里 CAS 要转很久才可能交换值，同时 CPU 光在这 10000 个活着的线程中来回切换中就耗费了巨大的资源
        2. **CAS 如果自旋 10 次依然没有获取到锁**

            ![[IMG-20260619221926462.png|513]]

    3. 满足上面两种条件的话，那么轻量级锁会升级为重量级锁
    4. **此时，**==**对象头的 Mark Word 指向的就不是锁记录了，而是向操作系统申请的 Monitor 对象**==

        ![[IMG-20260619221926537.png|597]]

    5. **而获得锁失败失败的线程会进入 EntryList，被阻塞**

        ![[IMG-20260619221926626.png|551]]
