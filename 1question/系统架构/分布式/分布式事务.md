# 1. 介绍
1. 分布式事务是相对本地事务而言的，对于本地事务，利用数据库本身的事务机制，就可以保证事务的 ACID 特性
2. 而在分布式环境下，会涉及到多个数据库

    ![[IMG-20260404032014348.png|Untitled 83.png]]

3. 分布式事务其实就是将对同一库事务的概念扩大到了对多个库的事务
4. ⭐**分布式事务是指一个业务流程跨越多个分布式系统或服务的事物处理。它需要确保在多个参与者之间的数据一致性和原子性**
5. ==**分布式事务处理的关键是：**==
    1. **需要记录事务在任何节点所做的所有动作**
    2. **事务进行的所有操作要么全部提交，要么全部回滚**

# 2. **分布式事务场景**
1. **跨库事务：**跨库事务指的是，⼀个应⽤**某个功能需要操作多个库**，不同的库中存储不同的业务数据

    ![[IMG-20260405035513311.png|Untitled 1 51.png]]

2. **分库分表**
    1. 通常⼀个库数据量⽐较⼤或者预期未来的数据量⽐较⼤，都会进⾏⽔平拆分，也就是分库分表

        ![[IMG-20260405035504085.png|Untitled 2 40.png]]

    2. 对于分库分表的情况，要保证操作两个库要不都成功，要不都失败
    3. ⼀般开发⼈员都会使用一些**数据库中间**件来降低 sql 操作的复杂性
3. ==**微服务化（SOA 面向服务架构）**==
    1. 微服务架构将原先复杂的业务读物拆分成不同的独⽴服务，以简化业务逻辑
    2. 拆分后，独⽴服务之间通过 RPC 框架来进⾏远程调⽤，实现彼此的通信
    3. Service A 完成某个功能需要直接操作数据库，同时需要调⽤Service B和Service C，⽽Service B⼜同时操作了2个数据库， Service C也操作了⼀个库

        ![[IMG-20260405035511169.png|Untitled 3 33.png]]

    4. 需要保证这些跨服务的对多个数据库的操作要不都成功，要不都失败，实际上这可能是最典型的分布式事务场景

# 3. 实现方案
- 分布式常见的实现方案有 **2PC**、**3PC**、**TCC**、**本地消息表**、**MQ消息事务**、**最大努力通知**、**SAGA事务** 等等
- 分布式事务实现⽅案必须要**考虑性能的问题**，如果为了严格保证ACID特性，导致性能严重下降，那么对于⼀些要求快速响应的业务，是⽆法接受的
## 3.1 刚性事务
### 2PC 两阶段提交
> [!important] **XA 协议**
> 
> 1. 分布式事务中的 XA 协议，XA 接口提供资源管理器与事务管理器之间进行通信的标准接口
> 
> 2. 有三个角色：
> 
>     - **AP（Application）：应用系统（服务）**
> 
>     - **TM（Transaction Manager）：事务管理器（全局事务管理）**
> 
>     - **RM（Resource Manager）：资源管理器（数据库）**
> 
> ![[IMG-20260405035510119.png|Untitled 4 27.png]]
> 
>
1. **两阶段提交 2PC**
    1. 优点：尽量保证了数据的强一致，实现成本较低
    2. 两阶段协议可以用于单机集中式系统，由事务管理器协调多个资源管理器
        - Mysql 的事务就是通过**「日志系统」** 来完成两阶段提交的[[面试题汇总]][[面试题汇总]][[面试题汇总]]
    3. 也可以用于分布式系统，**「由一个全局的事务管理器协调各个子系统的局部事务管理器完成两阶段提交」**
    4. ⭐**两阶段提交的思路：==参与者将操作成败通知协调者，再由协调者根据所有参与者的反馈情况决定各参与者是否要提交操作还是回滚操作==**
        - **准备阶段：（投票阶段）**

            ![[IMG-20260405035519131.png|Untitled 5 25.png]]

            1. 协调者首先将命令写入日志，再发一个 `prepare` 命令给 B 和 C 节点这两个参与者
            2. **B 和 C 收到消息后，根据自己的实际情况，判断自己的实际情况是否可以提交**
            3. 将处理结果记录到日志系统，最后将结果返回给协调者
        - **提交阶段：（决定阶段）**

            ![[IMG-20260405035525572.png|Untitled 6 22.png]]

            1. **当 A 节点收到 B 和 C 参与者所有的确认消息后，判断所有协调者是否都可以提交**
                - 如果可以则 「写入日志」 并且发起 `commit` 命令
                - 有一个不可以则 「写入日志」 并且发起 `abort` 命令
            2. 参与者收到协调者发起的命令后执行命令，并将执行命令及结果写入日志，最后返回结果给协调者

    ![[IMG-20260405035530958.png|Untitled 7 20.png]]

2. ==**缺点:**==
    - **单点故障**：
        - 一旦事务管理器出现故障，整个系统不可用
    - **数据不一致**：
        - 在阶段二，如果事务管理器只发送了部分 commit 消息，此时网络发生异常，那么只有部分参与者接收到 commit 消息
        - 也就是说只有部分参与者提交了事务，使得系统数据不一致
    - **响应时间较长（同步阻塞）**：
        - 整个消息链路是串行的，要等待响应结果，不适合高并发的场景
    - **不确定性**：
        - 当事务管理器发送 commit 之后，并且此时只有一个参与者收到了 commit
        - 那么当该参与者与事务管理器同时宕机之后，重新选举的事务管理器无法确定该条消息是否提交成功

### 3PC 三阶段提交
1. 三阶段提交又称 3PC，是 2PC 的一种改进
    1. 上边提到两段提交，当协调者崩溃时，参与者不能做出最后的选择，就会一直保持阻塞锁定资源
    2. 2PC 中只有协调者有超时机制**，3PC 在协调者和参与者中都引入了超时机制，并增加了 CanCommit 阶段**
        1. 如果参与者一段时间内没有收到协调者的 commit 请求（协调者出现故障），那么就会自动进行 commit，参与者就不会一直阻塞
        2. ==解决了 2PC 单点故障和阻塞问题的问题==，==但是性能问题和不一致问题仍然没有根本解决==
2. 三阶段流程：

    ![[IMG-20260405035532434.png|Untitled 8 17.png]]

    - 流程详情
        - 第一阶段：**CanCommit阶段**
            1. 这个阶段所做的事很简单，就是协调者询问事务参与者，你是否有能力完成此次事务
                - 如果都返回 yes，则进入第二阶段
                - 有一个返回 no 或等待响应超时，则中断事务，并向所有参与者发送 `abort` 请求
        - 第二阶段：**PreCommit阶段**
            1. 此时协调者会向所有的参与者发送 `PreCommit` 请求
            2. 参与者收到后开始执行事务操作，并将 Undo 和 Redo 信息记录到事务日志中
            3. 参与者执行完事务操作后（此时属于未提交事务的状态），就会向协调者反馈 Ack 表示我已经准备好提交了，并等待协调者的下一步指令
        - 第三阶段：**DoCommit阶段**
            1. 在阶段二中如果所有的参与者节点都可以进行 PreCommit 提交，那么协调者就会从“预提交状态”转变为“提交状态”
                1. 然后向所有的参与者节点发送 `doCommit` 请求，参与者节点在收到提交请求后就会各自执行事务提交操作，并向协调者节点反馈“Ack”消息
                2. 协调者收到所有参与者的 Ack 消息后完成事务
            2. 相反，如果有一个参与者节点未完成 PreCommit 的反馈或者反馈超时，那么协调者都会向所有的参与者节点发送 abort 请求，从而中断事务

## 3.2 柔性事务
### TCC**（Try Confirm Cancel）**
1. TCC其实就是采用的补偿机制
    1. 核心思想：**针对每个操作，都要注册一个与其对应的确认和补偿（撤销）操作**
    2. **只不过是这里的补偿和撤销操作都是是针对业务上的**
2. 它分为三个阶段：
    1. `Try` 阶段主要是对**业务系统做检测及资源预留**，其主要分为两个阶段
    2. `Confirm` 阶段主要是对**业务系统做确认提交**，Try 阶段执行成功并开始执行 Confirm 阶段时，默认 Confirm 阶段是不会出错的。即：只要 Try 成功，Confirm 一定成功
    3. `Cancel` 阶段主要是在业务执行错误，需要回滚的状态下执行的业务取消，**预留资源释放**
3. 比如下一个订单减一个库存的执行流程：

    ![[IMG-20260405035535356.png|Untitled 9 14.png]]

    - Try 阶段：订单系统将当前订单状态设置为支付中，库存系统校验当前剩余库存数量是否大于 1，然后将可用库存数量设置为库存剩余数量 -1
        - 如果 Try 阶段 **「执行成功」** ，执行 Confirm 阶段，将订单状态修改为支付成功，库存剩余数量修改为可用库存数量
        - 如果 Try 阶段 **「执行失败」** ，执行 Cancel 阶段，将订单状态修改为支付失败，可用库存数量修改为库存剩余数量
4. **优点：**
    - **解决了协调者单点：**由主业务方发起并完成这个业务活动。业务活动管理器也变成多点，引入集群
    - **同步阻塞：**
        - 引入超时，超时后进行补偿，是保证最终一致性，不会一直持有资源的锁
        - 并且不会锁定整个资源，而是将资源转换为业务逻辑形式，资源粒度变小
    - **数据一致性：**有了补偿机制之后，由业务活动管理器控制一致性
5. **缺点：**
    1. TCC 的 Try、Confirm 和 Cancel 操作功能需业务提供，并且很大程度的**增加了业务代码复杂度**
    2. TCC **对业务的侵入较大，和业务紧耦合**，需要根据特定的场景和业务逻辑来设计相应的操作，因此，这种模式并不能很好地被复用

### 本地消息表（最终一致性）
1. 本地消息表的核心思想：==将分布式事务拆分成本地事务进行处理==
    1. 例如，可以在**订单库新增一个消息表**，将新增订单和新增消息放到一个事务里完成
    2. 然后通过轮询的方式去查询消息表，将消息推送到 MQ，库存服务去消费 MQ

    ![[IMG-20260405035535491.png|Untitled 10 14.png]]

2. **执行流程：**
    1. **订单服务：**
        1. **添加一条订单和一条消息，在一个事务里提交**
        2. **使用定时任务轮询查询状态为未同步的消息表，发送到 MQ，如果发送失败，就重试发送**
    2. **库存服务：**
        1. **接收 MQ 消息，修改库存表，需要保证幂等操作**
        2. **如果修改成功，调用 rpc 接口修改订单系统消息表的状态为已完成或者直接删除这条消息**
        3. **如果修改失败，可以不做处理，等待重试**
3. 额外的保证机制：
    1. 为了避免订单服务一直重复发送消息的情况，可以记录一下发送次数，当达到次数限制之后报警，人工接入处理
    2. 库存服务需要保证幂等，避免同一条消息被多次消费造成数据不一致
4. 缺点：
    1. 需要在业务系统里增加消息表，业务逻辑中多一次插入的 DB 操作，所以性能会有损耗
    2. 而且最终一致性的间隔主要由定时轮询查询任务的间隔时间决定

### **MQ 消息事务**（最终一致性）
1. 消息事务的原理：==**将两个事务通过消息中间件进行异步解耦**==
    1. 订单服务执行自己的本地事务，并发送 MQ 消息；库存服务接收消息，执行自己的本地事务
    2. 和上述的本地消息表有点类似，但是是通过消息中间件的机制去做的，其本质就是：“将本地消息表封装到了消息中间件中”
    3. 消息事务一定要保证业务操作与消息发送的一致性，即如果业务操作成功，这条消息也一定投递成功

    ![[IMG-20260405035540856.png|Untitled 11 13.png]]

2. **执行流程：**
    1. **发送 prepare 消息到消息中间件**
    2. **发送成功后，执行本地事务**
        1. **如果本地事务执行成功，则向中间件发送 commit 消息，消息中间件将消息下发至消费端**
        2. **如果本地事务执行失败，则回滚，消息中间件将以前的 prepare 消息删除**
    3. **消费端接收到消息进行消费，如果消费失败，则不断重试**
3. 对比本地消息表实现方案，不需要再建消息表，对性能的损耗和业务的入侵更小，所以这种方案更适用于高并发的场景
4. 目前市面上实现该方案的只有阿里的 RocketMQ
