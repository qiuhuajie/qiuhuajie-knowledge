# 1. 介绍
![[IMG-20260315213656234.png|Untitled 275.png]]
**推卸责任**
1. **介绍：**
    
    1. 顾名思义，责任链模式（Chain of Responsibility Pattern）为请求创建了一个接收者对象的链。这种模式给予请求的类型，对请求的发送者和接收者进行解耦。这种类型的设计模式属于行为型模式
    
    1. 在这种模式中，通常每个接收者都包含对另一个接收者的引用。如果一个对象不能处理该请求，那么它会把相同的请求传给下一个接收者，依此类推
    
1. **意图：**避免请求发送者与接收者耦合在一起，让多个对象都有可能接收请求，将这些对象连接成一条链，并且沿着这条链传递请求，直到有对象处理它为止
1. **主要解决：**职责链上的处理者负责处理请求，客户只需要将请求发送到职责链上即可，无须关心请求的处理细节和请求的传递，所以职责链将请求的发送者和请求的处理者解耦了
1. **何时使用：**在处理消息的时候以过滤很多道
1. **如何解决：**拦截的类都实现统一接口
1. **关键代码：**Handler 里面聚合它自己，在 HandlerRequest 里判断是否合适，如果没达到条件则向下传递，向谁传递之前 set 进去
1. **应用实例：** 
    
    1. JS 中的事件冒泡
    
    1. JAVA WEB 中 Apache Tomcat 对 Encoding 的处理，Struts2 的拦截器，jsp servlet 的 Filter
    
1. **优点：**
    
    1. 降低耦合度。它将请求的发送者和接收者解耦
    
    1. 简化了对象。使得对象不需要知道链的结构
    
    1. 增强给对象指派职责的灵活性。通过改变链内的成员或者调动它们的次序，允许动态地新增或者删除责任
    
    1. 增加新的请求处理类很方便
    
1. **缺点：**
    
    1. 不能保证请求一定被接收
    
    1. 系统性能将受到一定影响，而且在进行代码调试时不太方便，可能会造成循环调用
    
    1. 可能不容易观察运行时的特征，有碍于除错
    
1. **使用场景：**
    
    1. 有多个对象可以处理同一个请求，具体哪个对象处理该请求由运行时刻自动确定
    
    1. 在不明确指定接收者的情况下，向多个对象中的一个提交一个请求
    
    1. 可动态指定一组对象处理请求
    
1. **注意事项：**在 JAVA WEB 中遇到很多应用
# 2. 登场角色
## 2.1 Chain of Responsibility 模式中有以下登场角色
1. **Handler(处理者)**
    
    1. Handler 角色定义了处理请求的接口(API)
    
    1. **==Handler 角色知道“下一个处理者”是谁，如果自己无法处理请求，它会将请求转给“下一个处理者”，其实逻辑过程和==** ==`**if-else**`== **==很相似==**
    
    1. 当然，**“下一个处理者”也是 Handler 角色**
    
1. **ConcreteHandler(具体的处理者)**
    
    ConcreteVisitor角色是处理请求的具体角色
    
1. **Client(请求者)**
    
    Client角色是向第一个ConcreteHandler角色发送请求的角色

# 3. 代码示例 🙋‍♂️
1. `Handler` ➡️ **Handler(处理者)**
    
    ```Java
    public abstract class Handler {     // 抽象类
    
        String name;
    
        public Handler(String name) {
            this.name = name;
        }
    
        Handler next = null;             //下一个处理者
    
        void setNext(Handler next) {
            this.next = next;
        }
    
        public abstract void process(String param);   //处理审批请求的方法，得到一个请求, 处理是子类完成，因此该方法做成抽象
    
    }
    ```
    
1. `Handler1` ➡️ **ConcreteHandler(具体的处理者 1)**
    
    ```Java
    public class Handler1 extends Handler{
    
        public Handler1(String name) {
            super(name);
        }
    
        @Override
        public void process(String param) {
            if (param.equals("handler1")) {
                System.out.println("handler1..process");
            } else {
                System.out.println("handler1..can not process");
                this.next.process(param);
            }
        }
    }
    ```
    
1. `Handler2` ➡️ **ConcreteHandler(具体的处理者 2)**
    
    ```Java
    public class Handler2 extends Handler{
    
        public Handler2(String name) {
            super(name);
        }
    
        @Override
        public void process(String param) {
            if (param.equals("handler2")) {
                System.out.println("handler2..process");
            } else {
                System.out.println("handler2..can not process");
                this.next.process(param);
            }
        }
    }
    ```
    
1. `Demo` ➡️ **Client(请求者)**
    
    ```Java
    public class Main {
        public static void main(String[] args) {
            Handler1 handler1 = new Handler1("handler1");
            Handler2 handler2 = new Handler2("handler2");
    
            //需要将各个审批级别的下一个设置好 (处理人构成环形)
            handler1.setNext(handler2);
            handler2.setNext(handler1);
    
            String param = "handler2";
            handler1.process(param);
        }
    }
    ```
    
1. 运行结果
    
    ```Java
    handler1..can not process
    handler2..process
    ```
    
# 4. 源码应用分析
## 4.1 ⭐职责链模式在 SpringMVC 框架应用的源码分析
1. `DispatcherServlet` 在处理请求调用的过程中，调用到类中的 `doDispatch()` 方法[[处理请求(调用)的过程]][[处理请求(调用)的过程]][[处理请求(调用)的过程]]
1. **DispatcherServlet.class** ：⭐`**doDispatch()**`
    
    ```Java
     protected void doDispatch(HttpServletRequest request, HttpServletResponse response) throws Exception {
         HttpServletRequest processedRequest = request;
         HandlerExecutionChain mappedHandler = null;
         boolean multipartRequestParsed = false;
     
         WebAsyncManager asyncManager = WebAsyncUtils.getAsyncManager(request);
     
         try {
             ModelAndView mv = null;
             Exception dispatchException = null;
     
             try {
                 processedRequest = checkMultipart(request);
                 multipartRequestParsed = (processedRequest != request);
     
                 // 根据 request 找到对应处理请求的 mappedHandler
                 mappedHandler = getHandler(processedRequest);
                 if (mappedHandler == null) {
                     noHandlerFound(processedRequest, response);
                     return;
                 }
     
                 // 通过控制器方法创建相应的处理器适配器，调用所对应的控制器方法
                 HandlerAdapter ha = getHandlerAdapter(mappedHandler.getHandler());
     
                 ...
    
                 // ⭐调用拦截器的 preHandle()
                 if (!mappedHandler.applyPreHandle(processedRequest, response)) {
                     return;
                 }
     
                 // 由处理器适配器调用具体的handle()控制器方法，最终获得ModelAndView对象
                 mv = ha.handle(processedRequest, response, mappedHandler.getHandler());
     
                 if (asyncManager.isConcurrentHandlingStarted()) {
                     return;
                 }
     
                 applyDefaultViewName(processedRequest, mv);
    
                 // ⭐调用拦截器的postHandle()
                 mappedHandler.applyPostHandle(processedRequest, response, mv);
             }
             catch (Exception ex) {
                 dispatchException = ex;
             }
             catch (Throwable err) {
    
                 dispatchException = new NestedServletException("Handler dispatch failed", err);
             }
    
             // 后续处理：处理模型数据和渲染视图
             processDispatchResult(processedRequest, response, mappedHandler, mv, dispatchException);
         }
    
         ...
     }
    ```
    
1. 之前在适配器模式中介绍过，在根据 `request` 找到对应处理请求的 `mappedHandler` 后，会通过控制器方法创建相应的处理器适配器，调用所对应的控制器方法[[适配器模式 ⭐]]
1. 接下来就由处理器适配器调用具体的 `handle()` 控制器方法
1. 但是在执行调用具体的 `handle()` 控制器方法**前后**，需要先由找到的`mappedHandler`执行注册好的拦截器，用于对处理器进行**预处理**和**后处理**
    
    > [!important] **拦截器**
    > 
    > ：详见 [（SpringBoot 拦截器）](https://www.notion.so/2-6-5e730a05d8704033ab587ce99142113a?pvs=21)
    > 
    > 1. 拦截器类似于 Servlet 开发中的过滤器 Filter，用于对处理器进行**预处理**和**后处理**
    > 
    > 1. 所有拦截器 实现 `HandlerInterceptor` 接口，并重写其中的三个方法
    >     
    >     - `preHandle()`：处理之前操作
    >     
    >     - `postHandle()`：处理之后操作
    >     
    >     - `afterCompletion()`：页面渲染完后的一些操作（如：清理操作）
    >     
    
    1. 首先，找到的`mappedHandler` 是一个 `HandlerExecutionChain` 对象，其维护着一个 `List<HandlerInterceptor> interceptorList` ，保存着所有注册的拦截器`HandlerInterceptor`
        
        ![[IMG-20260315213657297.png|Untitled 2 173.png]]
        
        `interceptorList` 中的拦截器可以在`HandlerExecutionChain` 对象的构造方法中设置，也可以调用本类中的`addInterceptor()` 来向 `interceptorList` 中 `add` 拦截器
        
    
    1. **预处理：`mappedHandler.applyPreHandle(processedRequest, response)`**
        
        1. 在预处理中会**从前向后**遍历`interceptorList` 中所有的拦截器对象，并执行当前拦截器的`preHandle()`
            
            ![[IMG-20260315213657989.png|Untitled 3 130.png]]
            
            ![[IMG-20260315213658469.png|Untitled 4 107.png]]
            
        
        1. `preHandle()` 的返回值是一个 `boolean` ，如果返回`false` 表示当前请求被预处理拦截掉了，不会再继续执行，而是直接**倒序执行**所有已经执行了的拦截器的 `afterCompletion()`
        
        1. 分析源码看到，接下来就会执行⭐**`triggerAfterCompletion()`** ，该方法中会执行那些**已经执行过的拦截器的`afterCompletion()`** ，分析源码可以看到，通过使用 `interceptorIndex` 可以控制当前请求是被拦截器链上的第几个拦截所拦截，从而执行这个拦截器之前所有拦截器的`afterCompletion()`
            
            ![[IMG-20260315213658969.png|Untitled 5 87.png]]
            
            ![[IMG-20260315213659459.png|Untitled 6 75.png]]
            
        
    
    1. **后处理：`mappedHandler.applyPostHandle(processedRequest, response, mv);`**
        
        1. 在后处理中是**从后向前**遍历所有拦截器
        
        1. 执行遍历到的当前拦截器的 `postHandle()`
            
            ![[IMG-20260315224944293.png|Untitled 7 64.png]]
            
        
    
    1. **清理操作：**是在 `doDispatch()` 方法最后调用 `processDispatchResult()` 中执行的[[处理请求(调用)的过程]][[处理请求(调用)的过程]][[处理请求(调用)的过程]]
    
1. **总结：**
    
    1. **`HandlerExecutionChain`** 主要负责的是请求拦截器的执行和请求处理，但是他本身不处理请求，只是**将请求分配给链上注册处理器执行**
    
    1. 这是职责链实现方式，可以减少职责链本身与处理逻辑之间的耦合，规范了处理流程
