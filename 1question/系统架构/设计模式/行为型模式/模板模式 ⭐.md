# 1. 介绍
![[IMG-20260315213656199.png|Untitled 266.png]]
1. **介绍：**
    
    1. 在模板模式（Template Pattern）中，**一个抽象类公开定义了执行它的方法的方式/模板**
    
    1. 它的**子类可以按需要重写方法实现，但调用将以抽象类中定义的方式进行**。这种类型的设计模式属于行为型模式
    
    1. 模板方法法只存在于一个地方，也就是在父类中，容易修改。 需要修改算法时，只要修改父类的模板方法或者已经实现的某些步骤，子类就会继承这些修改，实现了最大化代码复用
    
    使用
    
1. ==**意图：**==
    
    1. **==定义一个操作中的算法的骨架，而将一些步骤延迟到子类中==**
    
    1. **==模板方法使得子类可以不改变一个算法的结构即可重定义该算法的某些特定步骤==**
    
1. **主要解决：**一些方法通用，却在每一个子类都重新写了这一方法
1. **何时使用：**
    
    - ⭐==**模板方法中可能会调用三种方法：**==
        
        1. ==**本类已经实现的方法：**==**这些方法是固定的也就是模板的东西**
        
        1. ==**本类中定义的一些抽象方法：**==**这些方法留给子类（或实现类）去实现，实现在模板基础上的定制化**
        
        1. ==**钩子方法：**==里面是一些空实现或者默认实现，如果某些子类需要某些特殊的操作，即可重写钩子方法，参照下面的`boolean customerWantCondiments()`
        
    
    - **模板方法模式使用场景：** 当要完成在某个过程， 该过程要执行一系列步骤 ， 这一系列的步骤基本相同，但其个别步骤在实现时 可能不同，通常考虑用模板方法模式来处理
    
1. **如何解决：**将这些通用算法抽象出来
1. **关键代码：**在抽象类实现，其他步骤在子类实现
1. **应用实例：**
    
    1. 在造房子的时候，地基、走线、水管都一样，只有在建筑的后期才有加壁橱加栅栏等差异
    
    1. 西游记里面菩萨定好的 81 难，这就是一个顶层的逻辑骨架
    
    1. spring 中对 Hibernate 的支持，将一些已经定好的方法封装起来，比如开启事务、获取 Session、关闭 Session 等，程序员不重复写那些已经规范好的代码，直接丢一个实体就可以保存
    
1. **优点：**
    
    1. 封装不变部分，扩展可变部分
    
    1. 提取公共代码，便于维护
    
    1. 行为由父类控制，子类实现
    
1. **缺点：**每一个不同的实现都需要一个子类来实现，导致类的个数增加，使得系统更加庞大
1. **使用场景：**
    
    1. 多个子类共有的方法，且逻辑相同
    
    1. 重要的、复杂的方法，可以考虑作为模板方法
    
1. **注意事项：==为防止恶意操作，一般模板方法都加上==** ==`**final**`== **==关键词==**
# 2. 登场角色
## 2.1 Template 模式中有以下登场角色
1. **AbstractClass(抽象类)**
    
    AbstractClass 角色不仅负责实现模板方法，还负责声明在模板方法中所使用到的抽象方法。这些抽象方法由子类ConcreteClass角色负责实现
    
1. **ConcreteClass(具体类)**
    
    该角色负责具体实现AbstractClass角色中定义的抽象方法。这里实现的方法将会在AbstractClass角色的模板方法中被调用

# 3. 代码示例
## 3.1 需求描述
1. 制作咖啡的流程 **选材--->添加配料--->冲泡**
1. 通过添加不同的配料，可以制作出不同口味的咖啡
1. **选材和冲泡几个步骤对于制作每种口味的咖啡都是一样的**
## 3.2 Template 模式解决
1. `Coffee` ➡️ **AbstractClass(抽象类)**
    
    - ⭐==**模板方法中可能会调用三种方法：**==
        
        1. ==**本类已经实现的方法：**==**这些方法是固定的也就是模板的东西**
        
        1. ==**本类中定义的一些抽象方法：**==**这些方法留给子类（或实现类）去实现，实现在模板基础上的定制化**
        
        1. ==**钩子方法：**==里面是一些空实现或者默认实现，如果某些子类需要某些特殊的操作，即可重写钩子方法，参照下面的`boolean customerWantCondiments()`
        
    
    ```Java
    public abstract class Coffee {
        //⭐模板方法 make , 模板方法可以做成final , 不让子类去覆盖
        final void make() {
            select();
            if(customerWantCondiments()) {
                addCondiments();    // 模板方法中调用本类中的抽象方法，而这些抽象方法的实现由子类根据自己的需求去实现
            }
            beat();
        }
    
        //选材料（1️⃣ 本类已经实现的方法）
        void select() {
            System.out.println("第一步：选择好的新鲜咖啡豆  ");
        }
    
        //添加不同的配料，抽象方法，子类具体实现（2️⃣ 本类中定义的一些抽象方法）
        abstract void addCondiments();
    
        void beat() {
            System.out.println("第三步：冲泡  ");
        }
    
        // 决定是否需要添加配料（3️⃣ 钩子方法）
        boolean customerWantCondiments() {
            return true;
        }
    }
    ```
    
1. `ACoffee` ➡️ **ConcreteClass(具体类)**
    
    ```Java
    public class ACoffee extends Coffee{
        @Override
        void addCondiments() {
            System.out.println(" 加入配料 A ");
        }
    }
    ```
    
1. `BCoffee` ➡️ **ConcreteClass(具体类)**
    
    ```Java
    public class BCoffee extends Coffee{
        @Override
        void addCondiments() {
            System.out.println(" 加入配料 B ");
        }
    }
    ```
    
1. `PureCoffee` ➡️ **ConcreteClass(具体类)**
    
    ```Java
    public class PureCoffee extends Coffee{
        @Override
        void addCondiments() {
    
        }
    
        // 重写钩子方法，返回 false 控制不加任何配料（其它风味的咖啡是不需要重写这个方法的，使用抽象类中已经实现的默认方法返回 true 就行）
        @Override
        boolean customerWantCondiments() {
            return false;
        }
    }
    ```
    
1. `TemplateDemo`
    
    ```Java
    public class TemplateDemo {
        public static void main(String[] args) {
    
            ACoffee aCoffee = new ACoffee();
            aCoffee.make();
            System.out.println("===================================");
            BCoffee bCoffee = new BCoffee();
            bCoffee.make();
            System.out.println("===================================");
            PureCoffee pureCoffee = new PureCoffee();
            pureCoffee.make();
        }
    }
    ```
    
1. 运行结果
    
    ```Java
    第一步：选择好的新鲜咖啡豆  
     加入配料 A 
    第三步：冲泡  
    ===================================
    第一步：选择好的新鲜咖啡豆  
     加入配料 B 
    第三步：冲泡  
    ===================================
    第一步：选择好的新鲜咖啡豆  
    第三步：冲泡
    ```

# 6. **AQS 底层使用了模板方法模式**
1. AQS使用了模板方法模式
1. 自定义同步器时，使用者继承 AbstractQueuedSynchronizer 并重写 AQS提供的 指定的模板方法：
    
    ```Java
    isHeldExclusively()  //该线程是否正在独占资源。只有用到condition才需要去实现它。
    tryAcquire(int)      //独占方式。尝试获取资源，成功则返回true，失败则返回false。
    tryRelease(int)      //独占方式。尝试释放资源，成功则返回true，失败则返回false。
    tryAcquireShared(int)//共享方式。尝试获取资源。负数表示失败；0表示成功，但没有剩余可用资源；正数表示成功，且有剩余资源。
    tryReleaseShared(int)//共享方式。尝试释放资源，成功则返回true，失败则返回false。
    ```
    
    - 这些重写方法很简单，无非是对于共享资源state的获取和释放
    
    - 将 AQS 组合在自定义同步组件的实现中，并调用其模板方法，而这些模板方法会调用使用者重写的方法
    
1. 默认情况下，每个方法都抛出 UnsupportedOperationException。 这些方法的实现必须是内部线程安全的，并且通常应该简短而不是阻塞
1. AQS 类中只有这几个方法可以被其他类使用，其他方法都是 `final` ，无法被其他类使用
1. 以 ReentrantLock 为例
    
    1. state 初始化为 0，表示未锁定状态
    
    1. A 线程 `lock()` 时，会调用 `tryAcquire()` 独占该锁并将 `state + 1`，此后，其他线程再 `tryAcquire()` 时就会失败
    
    1. 直到 A 线程 `unlock()` 到 `state = 0` （即释放锁）为止，其它线程才有机会获取该锁
        
        1. 当然，释放锁之前，A线程自己是可以重复获取此锁的（ state 会累加），这就是可重入的概念
        
        1. 但要注意，获取多少次就要释放多么次，这样才能保证 state 是能回到零态的
