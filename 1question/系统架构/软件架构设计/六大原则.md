# 一、开闭原则（Open Close Principle）
## 1. 介绍
1. 开闭原则的意思是：**对扩展开放（对提供方），对修改关闭（对使用方）**
2. 在程序需要进行拓展的时候，尽量**通过扩展软件的实体行为来实现变化**，**而不是通过修改已有的代码来实现**，即实现一个热插拔的效果
3. **用抽象构建框架，用实现扩展细节**
4. 简言之，是为了使程序的扩展性好，易于维护和升级
## 2. 代码示例
1. 一个画图形的功能

    ```Java
    /**
     * 图形类
     */
    class Shape {
        int m_type;
    }
    /**
     * 矩形类
     */
    class Rectangle extends Shape {
        Rectangle() {
            super.m_type = 1;
        }
    }
    /**
     * 圆形类
     */
    class Circle extends Shape {
        Circle() {
            super.m_type = 2;
        }
    }
    /**
     * 绘图工具类
     */
    class GraphicEditor {
        // 当需要扩展一个绘制三角形的功能时：
        public void drawShape(Shape s) {
            if (s.m_type == 1) {
                drawRectangle(s);
            }
            else if (s.m_type == 2) {
                drawCircle(s);
            }
            // 1. 这里需要新加一个判断条件
        }
        private void drawRectangle(Shape r) {
            System.out.println("矩形");
        }
        private void drawCircle(Shape r) {
            System.out.println("圆形");
        }
        // 2. 这里需要新加一个方法
    }
    public class Ocp {
        public static void main(String[] args) {
            GraphicEditor graphicEditor = new GraphicEditor();
            graphicEditor.drawShape(new Rectangle());
            graphicEditor.drawShape(new Circle());
        }
    }
    ```
2. 存在的问题：当需要扩展一个绘制三角形的功能时，需要修改 `GraphicEditor` 已有的代码，违反了开闭原则
3. **==优化后的代码：==**

    ```Java
    /**
     * 图形抽象类
     */
    abstract class Shape {
        int m_type;
        abstract void draw();
    }
    /**
     * 矩形类
     */
    class Rectangle extends Shape {
        Rectangle() {
            super.m_type = 1;
        }
        @Override
        public void draw() {
            System.out.println("矩形");
        }
    }
    /**
     * 圆形类
     */
    class Circle extends Shape {
        Circle() {
            super.m_type = 2;
        }
        @Override
        public void draw() {
            System.out.println("圆形");
        }
    }
    /**
     * 绘图工具类类
     */
    class GraphicEditor {
        public void drawShape(Shape s) {
            s.draw();
        }
        // ⭐当需要扩展一个绘制三角形的功能时，这里的代码无需修改。只需要增加一个三角形类继承 Shape 类，并重写抽象方法 draw() 即可
    }
    public class OcpImprove {
        public static void main(String[] args) {
            GraphicEditor graphicEditor = new GraphicEditor();
            graphicEditor.drawShape(new Rectangle());
            graphicEditor.drawShape(new Circle());
        }
    }
    ```
# 二、里氏代换原则（Liskov Substitution Principle）
## 1. 介绍
1. 面向对象中的继承包含这样一层含义：父类中凡是已经实现好的方法， **实际上是在设定规范和契约**，虽然它不强制要求所有的子类必须遵循这些契约，但是如果子类对这些已经实的方法任意修改，就会对整个继承体系造成破坏
2. ==因此里氏替换原则在对继承性的使用时，做出了要求：==
    1. **一个软件实体如果适用于一个父类，那么一定适用于其子类**
    2. **所有引用父类的地方必须能透明地使用其子类的对象，子类对象能够替换父类对象，而程序逻辑不变**
    3. **子类可以扩展父类的功能，但是不能改变父类的原有功能**
        1. 子类可以实现父类的抽象方法，但不能覆盖父类的非抽象方法
        2. 子类可以增加自己特有的方法
        3. 当子类的方法重载父类的方法时，方法的前置条件（即方法的输入\入参）要比父类方法的输入参数更宽松
        4. 当子类的方法实现父类的方法时（重写/重载或实现抽象方法），方法的后置条件（即方法的输出/返回值）要比父类更严格或者与父类一样
## 2. 代码示例
1. 原程序
    ```Java
    class A {
        public int func1(int num1, int num2) {
            return num1 - num2;
        }
    }
    /**
     * B 类继承了 A 类
     */
    class B extends A {
        /**
         * B 类将 A 的 func1() 方法重写了，这个可能是程序员无意的重写⚠️
         */
        @Override
        public int func1(int a, int b) {
            return a + b;
        }
        public int func2(int a, int b) {
            return func1(a, b) + 9;
        }
    }
    public class Lsp {
        public static void main(String[] args) {
            A a = new A();
            System.out.println("11-3=" + a.func1(11, 3));
            B b = new B();
            // ⭐这里想要得到的结果是 11-3，但是由于B 类无意将 A 的 func1() 方法重写了，这里导致结果为 11+3
    				// ⚠️ 显然这里违反了里氏代换原则：子类对象b替换了父类对象a后，原来的减的功能变成了加的功能
            System.out.println("11-3=" + b.func1(11, 3));
            System.out.println("11+3+9=" + b.func2(11, 3));
        }
    }
    ```
2. 存在的问题：
    由于子类 `B` 类无意将 父类`A` 的 `func1()` 方法重写了，这里导致结果偏离预期，造成原有功能出现错误
3. 解决方法：
    1. 在实际编程中，常常会通过重写父类的方法完成新的功能，这样写起来虽然简单，但整个继承体系的复用性会比较差。特别是运行多态比较频繁的时候
    2. 通用的做法是：**原来的父类和子类都继承一个更通俗的基类，**==**原有的继承关系去掉，采用依赖，聚合，组合等关系代替**==
4. ==**优化后的代码：**==
    ```Java
    class Base {
        /**
         * 将更基础的方法和成员写到 Base 类中
         */
    }
    /**
     * A 类继承了 Base 类
     */
    class A extends Base{
        public int func1(int num1, int num2) {
            return num1 - num2;
        }
    }
    /**
     * B 类继承了 Base 类
     */
    class B extends Base {
        /**
         * 现在 A B 之间不再具有继承关系了，这里的 func1() 方法成了 B 类自己的方法
         */
        public int func1(int num1, int num2) {
            return num1 + num2;
        }
        public int func2(int num1, int num2) {
            return func1(num1, num2) + 9;
        }
        // 如果 B 需要使用 A 的方法，可以使用组合的方式
        A a = new A();
        public int func3(int num1, int num2) {
            return a.func1(num1, num2);
        }
    }
    public class LspImprove {
        public static void main(String[] args) {
            A a = new A();
            System.out.println("11-3=" + a.func1(11, 3));
            B b = new B();
            // b 调用自己的 相加的方法逻辑
            System.out.println("11+3=" + b.func1(11, 3));
            System.out.println("11+3+9=" + b.func2(11, 3));
            // b 还可以通过组合的方式，使用 a 的 相减的方法逻辑
            System.out.println("11-3=" + b.func3(11, 3));
        }
    }
    ```
# 三、依赖倒转原则（Dependence Inversion Principle）
## 1. 介绍
1. 依赖倒转原则（Dependence Inversion Principle）是指：
    1. **高层模块不应该直接依赖底层模块，应该依赖其底层的抽象**
    2. **抽象不应该依赖细节，细节应该依赖抽象**
    3. 依赖倒转(倒置)的中心思想是**==面向接口编程==**
        1. 相对于细节的多变性，抽象的东西要稳定的多。以抽象为基础搭建的架构比以细节为基础的架构要稳定的多
        2. 在Java中，抽象指的是接口或抽象类，细节就是具体的实现类
        3. 使用接口或抽象类的目的：是制定好规范，而不涉及任何具体的操作，把展现细节的任务交给他们的实现类去完成
## 2. 代码示例
1. 源程序：人具有接受信息的功能
    ```Java
    class Email {
        public String getInfo() {
            return "电子邮件信息：xxx";
        }
    }
    class Person {
        // ⭐这里的接受方法依赖的是一个Email实体（具体）
        public void receive(Email email) {
            System.out.println(email.getInfo());
        }
    }
    public class Dip {
        public static void main(String[] args) {
            Person person = new Person();
            person.receive(new Email());
        }
    }
    ```
2. 存在的问题：
    1. `receive(Email email)` 方法现在依赖的是一个具体的 `Email` 实体
    2. 当需要新加一个接受 微信消息的功能时，需要在 `Person` 类中再写一个 `receive(Wechat wechat)` 方法才能实现
3. 解决方法：==**依赖接口，而不是依赖细节**==
4. ==**优化后的代码：**==
    ```Java
    // 新加一个IReceiver 接口
    interface IReceiver {
        String getInfo();
    }
    class Email implements IReceiver{
        @Override
        public String getInfo() {
            return "电子邮件信息：xxx";
        }
    }
    class Wechat implements IReceiver{
        @Override
        public String getInfo() {
            return "微信信息：xxx";
        }
    }
    class Person {
        // ⭐这里的 receive 方法依赖的是一个接口（抽象）
        public void receive(IReceiver iReceiver) {
            System.out.println(iReceiver.getInfo());
        }
    }
    public class DipImprove {
        public static void main(String[] args) {
            //客户端无需改变
            Person person = new Person();
            person.receive(new Email());
            person.receive(new Wechat());
        }
    }
    ```
## 3. 依赖关系传递的三种方式⭐
> 下面依赖关系传递的代码示例中，开关依赖电视
1. **方式 1：接口参数传递**
    ```Java
    /**
     * 电视接口
     */
    interface ITV {
    	void play();
    }
    /**
     * 开关接口
     */
    interface IOpenAndClose {
    	/**
    	 * 抽象方法，接收接口
    	 */
    	void open(ITV tv);
    }
    class ChangHong implements ITV {
    	@Override
    	public void play() {
    		System.out.println("长虹电视机，打开");
    	}
    }
    class OpenAndClose implements IOpenAndClose {
    	@Override
    	public void open(ITV tv) {
    		tv.play();
    	}
    }
    public class DependencyPass {
    	public static void main(String[] args) {
    		ChangHong changHong = new ChangHong();
    		OpenAndClose openAndClose = new OpenAndClose();
    		// ⭐调用方法时将依赖的对象传入
    		openAndClose.open(changHong);
    	}
    }
    ```
2. **方式 2：构造方法传递**
    ```Java
    /**
     * 电视接口
     */
    interface ITV {
    	void play();
    }
    /**
     * 开关接口
     */
    interface IOpenAndClose {
    	void open();
    }
    class ChangHong implements ITV {
    	@Override
    	public void play() {
    		System.out.println("长虹电视机，打开");
    	}
    }
    class OpenAndClose implements IOpenAndClose {
    	/**
    	 * 定义一个成员变量用于传递依赖
    	 */
    	public ITV tv;
    	/**
    	 * 通过构造器将依赖赋值
    	 */
    	public OpenAndClose(ITV tv) {
    		this.tv = tv;
    	}
    	public void open() {
    		this.tv.play();
    	}
    }
    public class DependencyPass {
    	public static void main(String[] args) {
    		ChangHong changHong = new ChangHong();
    		// ⭐通过构造器进行依赖传递
    		OpenAndClose openAndClose = new OpenAndClose(changHong);
    		openAndClose.open();
    	}
    }
    ```
3. **方式 3：****`setter()`** **方式传递**
    ```Java
    /**
     * 电视接口
     */
    interface ITV {
    	public void play();
    }
    /**
     * 开关接口
     */
    interface IOpenAndClose {
    	public void open();
    	public void setTv(ITV tv);
    }
    class ChangHong implements ITV {
    	@Override
    	public void play() {
    		System.out.println("长虹电视机，打开");
    	}
    }
    class OpenAndClose implements IOpenAndClose {
    	private ITV tv;
    	public void setTv(ITV tv) {
    		this.tv = tv;
    	}
    	public void open() {
    		this.tv.play();
    	}
    }
    public class DependencyPass {
    	public static void main(String[] args) {
    		ChangHong changHong = new ChangHong();
    		// ⭐通过setter方法进行依赖传递
    		OpenAndClose openAndClose = new OpenAndClose();
    		openAndClose.setTv(changHong);
    		openAndClose.open();
    	}
    }
    ```
# 四、接口隔离原则（Interface Segregation Principle）
## 1. 介绍
1. **接口隔离原则**原则的意思是：使用多个专门的隔离接口，比使用单个总接口要好
2. ==**也就是说接口应该尽可能的小，只包含客户端需要的方法**==
3. 它还有另外一个意思是：降低类之间的耦合度。由此可见，其实设计模式就是从大型软件架构出发、便于升级和维护的软件设计思想，它强调降低依赖，降低耦合

# 五、迪米特法则 / 最少知道原则（Demeter Principle）
## 1. 介绍
1. 迪米特法则（Demeter Principle）
    1. 又叫最少知道原则，即==**一个类对自己依赖的类知道的越少越好**==
    2. 也就是说，**==对于被依赖的类不管多么复杂，都尽量将逻辑封装在类的内部==**。对外除了提供的 `public` 方法，不对外泄露任何信息
2. 迪米特法则还有个更简单的定义：**只与直接的朋友通信**
    - 直接的朋友：每个对象都会与其他对象有耦合关系，只要两个对象之间有耦合关系，我们就说这两个对象之间是朋友关系。耦合的方式很多，依赖，关联，组合，聚合等
    - 其中，称出现成员变量，方法参数，方法返回值中的类为直接的朋友
    - 而出现在局部变量中的类不是直接的朋友。也就是说，陌生的类最好不要以局部变量的形式出现在类的内部
## 2. 代码示例
1. 原代码：
    ```Java
    //客户端
    public class Dp {
        public static void main(String[] args) {
            //创建了一个 SchoolManager 对象
            SchoolManager schoolManager = new SchoolManager();
            //输出学院的员工id 和 学校总部的员工信息
            schoolManager.printAllEmployee(new CollegeManager());
        }
    }
    //学校总部员工类
    @Data
    class Employee {
        private String id;
    }
    //学院的员工类
    @Data
    class CollegeEmployee {
        private String id;
    }
    //管理学院员工的管理类
    class CollegeManager {
        //返回学院的所有员工
        public List<CollegeEmployee> getAllEmployee() {
            List<CollegeEmployee> list = new ArrayList<CollegeEmployee>();
            for (int i = 0; i < 10; i++) { //这里模拟增加了10个员工到 list
                CollegeEmployee emp = new CollegeEmployee();
                emp.setId("学院员工id= " + i);
                list.add(emp);
            }
            return list;
        }
    }
    /**
     * 学校管理类
     *
     * 分析：
     *      SchoolManager类的直接朋友类： Employee、CollegeManager
     *      CollegeEmployee 不是 直接朋友 而是一个陌生类，这样违背了 迪米特法则
     */
    class SchoolManager {
        //返回学校总部的员工
        public List<Employee> getAllEmployee() {
            List<Employee> list = new ArrayList<Employee>();
            for (int i = 0; i < 5; i++) { //这里模拟增加了5个员工到 list
                Employee emp = new Employee();
                emp.setId("学校总部员工id= " + i);
                list.add(emp);
            }
            return list;
        }
        //该方法完成输出学校总部和学院员工信息(id)
        void printAllEmployee(CollegeManager sub) {
            //⭐分析问题
            //1. 这里的 CollegeEmployee 不是  SchoolManager的直接朋友
            //2. CollegeEmployee 是以局部变量方式出现在 SchoolManager
            //3. 违反了 迪米特法则
            //获取到学院员工
            List<CollegeEmployee> list1 = sub.getAllEmployee();
            System.out.println("------------学院员工------------");
            for (CollegeEmployee e : list1) {
                System.out.println(e.getId());
            }
            //获取到学校总部员工
            List<Employee> list2 = this.getAllEmployee();
            System.out.println("------------学校总部员工------------");
            for (Employee e : list2) {
                System.out.println(e.getId());
            }
        }
    }
    ```
2. 存在的问题：
    - `printAllEmployee(CollegeManager sub)` 中出现了`CollegeEmployee` 类型的局部变量
    - 违反了迪米特法则
    - 🚨 **==但不强求，如果要以来下游查出来的东西再解析，那陌生类就不可避免要出现在类的局部变量中==**

# 六、合成复用原则（Composite Reuse Principle）
## 1. 介绍
1. 合成复用原则是指：==**尽量使用合成/聚合的方式，而不是使用继承**==
2. **类 B 要使用 类 A 的功能，尽量避免使用继承，而是**==**将 类A 的对象 组合或聚合在 类 B 中**==**，再调用 A 对象的方法**
