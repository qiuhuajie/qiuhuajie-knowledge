### 1.1. 实体
1. 看起来比较重要的对象
2. **这些对象通常会随着业务进展产生状态和属性的变更**，但是他们代表的业务对象不变
3. 注意到出行计划、共乘等实体里有个东西叫状态
4. 状态并不是说这个状态本身很重要，而是说很多事情都是围绕着出行计划在做的。比如说，我先创建了一个出行计划，如果撮合成功了，这个出行计划就变成了撮合已成功
5. 如果在共乘里面，可能就是匹配已经完成，或者所有乘客都已经上车、到达下车点等
6. 也就是说这些对象之所以重要，是因为**它们往往是跟我们的业务是紧密联系起来的**
7. 另外如果一个东西会变化，我们就必须得有东西去跟踪它
8. **使用唯一的标识符跟踪那些具有重要业务意义的对象**
9. 例如，系统中有很多出行计划，你怎么找到那个计划呢？需要一个唯一的标识来标识实体，以便在实体发生变化时仍能找到它

需要注意的是，这里的实体不同于以前我们讲的ORM和持久化框架中的Entity实体（只要能够存储到数据库中的都可以叫Entity）

### 1.2. 值对象
1. **值对象仅仅用来描述实体的特征**，所以我们只关心值对象的属性，不关心它有没有唯一标识符
2. **区分实体对象和值对象有助于减少系统的复杂性**，可以加速思考过程，例如帮助我们决定一个对象需不需要有ID

### 1.3. 领域服务
1. 自身不持有数据，而是表达了某种业务计算逻辑或者业务策略
2. 一些业务逻辑并不是和领域对象相关，它本身**代表了一种商业策略或者业务处理过程**
3. **服务自身是无状态**的

### 1.4. 领域时间
1. 领域事件**表达了“系统中发生了什么”**
2. 领域事件是“领域专家关系的事件”，源自业务活动结果
3. 领域事件还**可增强系统的可回溯性（记录发生了什么），解耦业务之间的复杂关系**
4. 领域事件是一种特殊的“值对象”

本地的事件不一定需要ID，但在远程交互中的事件就需要用ID来标识唯一性了

## 2. 聚合✨
- 将实体和值对象划分为聚合，并围绕聚合定义边界
- 将聚合作为一个整体来看待，定义聚合的属性和不变量，并把其执行责任赋予聚合根或指定的框架机制
- 选择一个实体作为每个聚合的根，并仅允许外部对象持有聚合根的引用，而不持有聚合根里面的局部实体，这样就没法越过聚合根去更改里面的东西
- 聚合提升了对象系统的粒度，保证了业务逻辑的完整性，减少了错误产生的概率

➕🩸充血模型（Rich Domain Model）

1. 业务逻辑和数据结合：充血模型强调将业务逻辑和数据封装在同一个领域对象中。这意味着对象不仅仅存储数据，还包含操作数据的业务方法
2. 强封装性：对象自身具备对其状态和行为的完整封装，这提高了系统的内聚性。业务逻辑和数据统一管理，改动时更集中，不易遗漏
3. 对象责任：每个对象对其自己的行为负责，符合面向对象设计原则（如单一责任原则）

➖🩸贫血模型（Anemic Domain Model）

1. 数据与逻辑分离：领域对象仅仅作为数据结构存在，不包含业务逻辑。业务逻辑通常存在于服务层中
2. 更简单的领域对象：对象仅包含getter和setter方法，用于数据的存取
3. 集中业务逻辑：业务逻辑在更上层的服务或管理者中实现
4. 缺点
- 内聚性差：逻辑与数据分离可能导致内聚性不足
- 偏离领域驱动设计初衷：如果模型过于贫血，可能不符合领域驱动设计希望实现的对业务领域的贴近和表达
4. 另外【贫血模型】中，为了支持服务层管理数据，那么 coTrip 中的 vehicle 就可以暴露给外界进行修改，都不知道在什么情况下会改变座位数

## 3. 聚合的划分
1. 如果一个人取消行程了，那么两个人的共乘就取消了，但此时另一个人的出行计划还是在的，他可以用这个出行计划加入另一个共乘
2. 划分聚合的**生命周期一致性原则**
- 生命周期一致性是指聚合边界内的对象，和聚合根之间存在“人身依附”关系。即：如果聚合根小时，聚合内的其他元素都应该同时消失

- 在不破坏业务逻辑完整性的基础上，小聚合带来更大的灵活性

# 五、限界上下文
- 上面的域（领域模型）可以称之为**问题域**
- 限界上下文也可以称之为**方案域**
1. 限界上下是一个显示边界
2. 限界上下文形成了一个自治系统，有自己的规则和明确的边界
3. 就和细胞膜一样，规定了哪些物质可以进出，哪些物质不能进出，有清晰的定义，有真正的守护

![[IMG-20260405035607840.png]]

3. 边界是区分，更是集成

4. **如果在重构全新的系统，请让限界上下的边界和子域定义保持一致**

![[IMG-20260405035607978.png]]

3. **自治**是限界上下文的基本特征

4. 一个好的自治系统，依赖下面各个子系统的稳定自治，做好各自分内的事，且各自边界清晰

5. 一个好的自治系统，各个子系统要拥有清晰的、公共认可的领域模型

6. 一个好的自治系统，能够独立承担本领域的核心职责

7. 一个好的自治系统，不能和其他领域共性核心数据，边界不清晰

8. **用领域模型和统一语言抵御破坏边界的冲动**

# 六、限界上下文映射
1. **边界是问题出现的高发区域**

![[IMG-20260405035608735.png]]

2. 限界上下文映射的模式

![[IMG-20260405035609183.png]]

- **提供者****模式 1：****对外提供 API（公开发布语言）**
- 特点：有很多使用方，提供的标准的服务，不会为使用者定制功能，服务以契约形式清晰描述
- 例如高德地图 API
- 领域对象是服务接口定义的基础，但是外部不应该直接感知领域对象的变化

![[IMG-20260405035609381.png]]

- **提供者****模式 2：****客户/供应商**
- 依赖方（可以有多个，但是很少）是被依赖方的客户
- 我提供什么功能，由客户提出，由客户决定，双方是一个合作关系

![[IMG-20260405035609524.png]]

- 依赖方和被依赖方可以通过共同编写自动化验收测试，来定义双方的边界
- 后面在演进的过程中，我越来越知道自己干什么了，或者我的用户越来越多了，就会向发展对外提供API的模式发展
- **使用者****模式 1：****防腐层 ACL**

[是你理解的防腐层吗？](https://ata.atatech.org/articles/11000243455?spm=ata.23639746.0.0.cba12e50HqP3ZQ)[善用DP和防腐层](https://ata.atatech.org/articles/11000211004?spm=ata.23639746.0.0.cba12e50AKdrn2#eFBbvlWD)

- 被依赖方没有对依赖方做出承诺
- 被依赖方的领域模型质量不好，或者不符合依赖方的需要，或者依赖方不想和被依赖方捆绑在一起的时候使用
- 依赖方可以自己写防腐层进行隔离，ACL的本质就是请求方接口和依赖倒置原则

![[IMG-20260405035609722.png]]

- **使用者****模式 2：****追随者**
- 被依赖方没有对依赖方做出承诺，且依赖方很多
- 被依赖方的领域模型质量很好，且比较稳定
- 例如日常业务系统中，对 ic 和 tp 模型的依赖
- 此时，使用者就不必再转换，直接使用被依赖方的模型即可
- **耦合****模式：****共享模型**
- 共享必然存在
- 共享的数据越多，耦合越严重，就越不能自治
- 大多时候，共享带来的是一种低成本的诱惑，是利率很高的贷款
- （共享的东西可以有，但不能太多）可能在当下发现，被依赖方有一个东西概念和你的很接近，就直接拿来用了，但是他们是属于不同的业务方，发展方向是不一样的，因此这个依赖会变得越来越不可控
