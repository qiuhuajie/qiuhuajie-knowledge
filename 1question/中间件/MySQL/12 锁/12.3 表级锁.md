# 一、概述
1. **每次操作锁住整张表**，MySQL 自身就提供了**表锁**的能力
2. 锁定粒度大，发生锁冲突的概率最高，并发度最低
3. 应用在 MyISAM、InnoDB 等存储引擎中
4. 对于表级锁，主要分为以下三类：
    - 表锁
    - 元数据锁（meta data lock，MDL）
    - 意向锁
# 二、️⃣表锁
## 1. 介绍
1. 对于表锁，分为两类：
    - 读锁：`LOCK TABLE table_name READ` 用读锁锁表，会阻塞其他事务的写操作
    - 写锁：`LOCK TABLE table_name WRITE` 用写锁锁表，会阻塞其他事务的读和写操作
2. ==**只让 读-读 操作并发，读-写、写-写 都同步互斥，在多读环境下，提高查询效率**==
3. 释放锁：`unlock tables`
4. ==InnoDB 的====**行锁是针对索引字段**====加的锁，====**表级锁是针对非索引字段**====加的锁==

# 三、️⃣元数据锁
1. 元数据锁（meta data lock），简写MDL
2. **元数据：可以简单理解为就是**==**一张表的表结构**==
3. MDL 锁主要作用：
    1. ==**维护表元数据的数据一致性，在表上有活动事务的时候，不可以对元数据进行写入操作**==
    2. 也就是说，**某一张表涉及到未提交的事务时，是不能够修改这张表的表结构的**
    3. 以避免 DML 与 DDL 冲突，保证读写的正确性
4. MDL加锁过程是**系统自动控制**，无需显式使用，**==在访问一张表的时候会自动加上==**
5. **常见的SQL操作时，所添加的元数据锁**

|   |   |   |
|---|---|---|
|**对应SQL**|**锁类型**|**说明**|
|lock tables xxx read / write|SHARED_READ_ONLY / SHARED_NO_READ_WRITE||
|select 、select ... lock in share mode|**SHARED_READ**|与SHARED_READ、SHARED_WRITE兼容，与 EXCLUSIVE互斥|
|insert 、update、 delete、select ... for update|**SHARED_WRITE**|与SHARED_READ、SHARED_WRITE兼容，与 EXCLUSIVE互斥|
|`alter table` ...|**EXCLUSIVE**|与其他的MDL都互斥|
> 💡 兼容/互斥：即两个锁能否同时加在一张表上
6. 示例：
    1. 事务 A 给表加了 **`SHARED_READ`** 锁，此时另一个事务 B 可以给该表加 **`SHARED_READ`** 锁或 **`SHARED_WRITE`** 锁
    ![[IMG-20260620224156748.png|800]]
    2. 事务 A 给表加了 **`SHARED_READ`** 锁、**`SHARED_WRITE`** 锁，则其它事务无法给该表加 **`EXCLUSIVE`** 锁，其它事务中会给表加 **`EXCLUSIVE`** 锁的操作会阻塞
    ![[IMG-20260620224156835.png|800]]
    3. 这样就实现了：==**在表上有活动事务的时候，不可以对元数据进行写入操作（因为修改表结构会加**== **`EXCLUSIVE`**==**锁，修改表结构的操作会阻塞）的目的**==
7. 查看元数据锁的加锁情况：
    ```SQL
     select object_type,object_schema,object_name,lock_type,lock_duration from performance_schema.metadata_locks ;
    ```
# 四、️⃣意向锁
## 1. 意向锁的作用
1. 假如事务 A 加了**行级读锁**，锁住了表中的一行，让这一行只能读，不能写。之后，事务 B 尝试申请**整个表的写锁**
    ![[IMG-20260620224203544.png|800]]
    1. ==如果事务 B 申请成功，那么理论上它就能修改表中的任意一行，这与 A 持有的行级读锁是==**==冲突==**==的==
    2. 数据库需要避免这种冲突，就势必要让 B 的申请被阻塞，直到 A 释放行级读锁
    3. 那数据库要怎么判断这个冲突呢？
        - 步骤 1：判断表是否已被其他事务用表级锁锁住了整张表
        - 步骤 2：==判断表中的每一行是否已被行级锁锁住，需要遍历逐行检查！==
2. ==**而引入意向锁后**==
    1. 事务 A 加**行级读锁之前，会先在表上加一个表级意向锁**
    2. 因此，事务 B ==**可以直接根据意向锁来判断是否可以成功加表锁，从而减少了表锁的逐行检查**==
    ![[IMG-20260620224203632.png|800]]
## 2. 意向锁的分类
1. **意向共享锁(IS)**: 由语句**`select ... lock in share mode`**添加
    1. 与表锁共享锁(read)兼容
    2. 与表锁排他锁(write)互斥
2. **意向排他锁(IX)**: 由**`insert、update、delete、select...for update`**添加
    1. 与表锁共享锁(read)及排他锁(write)都互斥
    2. 意向锁之间不会互斥
> 一旦事务提交了，意向共享锁、意向排他锁，都会自动释放
3. 查看意向锁及行锁的加锁情况
    ```SQL
     select object_schema,object_name,index_name,lock_type,lock_mode,lock_data from performance_schema.data_locks;
    ```
