# 一、行级锁介绍
1. 每次操作锁住对应的行数据
2. 锁定粒度最小，发生锁冲突的概率最低，并发度最高
3. 应用在 InnoDB 存储引擎中，而 InnoDB 的数据是基于索引组织的，==**行级锁**====是通过====**对索引上的索引项加锁**====来实现的==，而不是对记录加的锁
## 1. 行锁（Record Lock）
- ==锁定单个行记录的锁==
- 防止其他事务对此行进行 **`update`** 和 **`delete`**，**避免产生幻读**
- **在** **`RC`**、**`RR`** 隔离级别下都支持
    ![[Attachment/1question/中间件/MySQL/12 锁/IMG-20260620224137488.png|800]]
- InnoDB 实现了以下两种类型的行锁：
    - **共享锁（行级读锁）**：允许其他事务去读本行，阻止其他事务写
    - **排他锁（行级写锁）**：其他事务对本行的读写都会阻止
    ![[IMG-20260620224140589.png|800]]
- **==哪些语句上面会加行锁==**❓
    1. 对于常见的 DML 语句（如 **`UPDATE`**、**`DELETE`** 和 **`INSERT`**），InnoDB 会自动给相应的记录行加写锁
    2. `SELECT * FROM table_name WHERE ...` **`FOR UPDATE`**，加**行级写锁**
    3. `SELECT * FROM table_name WHERE ...` **`LOCK IN SHARE MODE`**，加**行级读锁**
    4. 默认情况下对于普通 `SELECT` 语句（快照读），InnoDB 不会加任何锁，但是在 **`Serializable`** 隔离级别下会加行级读锁
## 2. 间隙锁（Gap Lock）
- ==锁定索引记录间隙（不含该记录）==
- 确保索引记录间隙不变，防止其他事务在这个间隙进行 **`insert`**，**避免产生幻读**
- 间隙锁唯一目的是防止其他事务插入间隙。间隙锁可以共存，一个事务采用的间隙锁不会阻止另一个事务在同一间隙上采用间隙锁
- **只在** **`RR`** **隔离级别下支持**
    ![[IMG-20260620224146686.png|800]]
## 3. 临键锁（Next-Key Lock）
- ==行锁和间隙锁组合==
- 给本行数据加行锁的同时，还要将本行与上一行之间的间隙用间隙锁锁住
- **只在** **`RR`** **隔离级别下支持**
    ![[IMG-20260620224156919.png|800]]
# 二、写个 SQL 语句然后问加了哪些锁
## 1. 加锁规则的两条核心⭐
1. **==查找过程中访问到的对象才会加锁==**
    1. 比如有主键 id 为 1 2 3 4 5 ... 10 的10 条记录，我们要找到 id = 7 的记录
    2. 注意，查找并不是从第一行开始一行一行地进行遍历，而是根据 B+ 树的特性进行二分查找
    3. 所以一般存储引擎只会访问到要找的记录行（id = 7）的相邻区间
2. **==加锁的基本单位是邻键锁 Next-key Lock==**
    1. 默认情况下，InnoDB 在 `REPEATABLE READ` （可重复读）事务隔离级别运行（存在幻读）
    2. 因此，[[6.5 事务的隔离级别]]

### 2.1 案例 1：唯一索引等值查询
1. 当我们用唯一索引进行等值查询的时候，根据查询的记录是否存在，加锁的规则会有所不同
2. ==当查询的记录是====**存在**====的，Next-key Lock 会退化成====**行锁**==
    ![[IMG-20260620224203751.png|800]]
    1. 结合加锁的两条核心：查找过程中访问到的对象才会加锁 + 加锁的基本单位是 Next-key Lock（左开右闭），我们可以分析出，这条语句的加锁范围是 `(20, 25]`
    2. 不过，由于这个唯一索引等值查询的记录 `id = 25` 是存在的，因此，Next-key Lock 会退化成**行锁**，因此最终的加锁范围是 `id = 25` 这一行
3. ==当查询的记录是====**不存在**====的，Next-key Lock 会退化成====**间隙锁**==
    ![[IMG-20260620224209589.png|800]]
    1. 结合加锁的两条核心：查找过程中访问到的对象才会加锁 + 加锁的基本单位是 Next-key Lock（左开右闭），我们可以分析出，这条语句的加锁范围是 `(20, 25]`
    ![[IMG-20260620224214229.png|800]]
    2. 由于这个唯一索引等值查询的记录 `id = 22` 是不存在的，因此，Next-key Lock 会退化成**间隙锁**，因此最终在主键 id 上的加锁范围是 Gap Lock `(20, 25)`
### 2.2 案例 2：唯一索引范围查询
1. 唯一索引范围查询的规则和等值查询的规则一样
2. 只有一个区别，就是==唯一索引的==**==范围查询==**==需要==**==一直向右遍历到第一个不满足条件的记录==**
    ![[IMG-20260620224214312.png|800]]
    1. 先来看语句查询条件的前半部分 `id >= 20`
        1. 因此，这条语句最开始要找的第一行是 id = 20，结合加锁的两个核心，需要加上 Next-key Lock `(15,20]`
        2. 又==由于 id 是唯一索引，且 id = 20 的这行记录是存在的，因此会退化成====**记录锁**==，也就是只会对 `id = 20` 这一行加锁
    2. 再来看语句查询条件的后半部分 `id < 22`
        1. 由于是范围查找，就会继续往后找第一个不满足条件的记录，也就是会找到 `id = 25` 这一行停下来，然后加 Next-key Lock `(20, 25]`
        2. 但==由于== ==`id = 25`== ==不满足== ==`id < 22`====，因此会退化成间隙锁==，加锁范围变为 `(20, 25)`。
    3. 所以，上述语句在主键 id 上的最终的加锁范围是 Record Lock `id = 20` 以及  Gap Lock `(20, 25)`
### 2.3 案例 3：非唯一索引等值查询
1. 当我们用非唯一索引进行等值查询的时候，根据查询的记录是否存在，加锁的规则会有所不同：
2. ==当查询的记录是====**存在**====的，除了会==**==加 Next-key Lock 外，还会额外加间隙锁==**==（规则是向下遍历到第一个不符合条件的值才能停止，因为是非唯一，右面可能还有符合的记录），也就是会加两把锁==
    ![[IMG-20260620224217190.png|800]]
    1. 结合加锁的两条核心，这条语句首先会对普通索引 a 加上 Next-key Lock，范围是 `(8,16]`
    2. 又因为是非唯一索引等值查询，且查询的记录 `a= 16` 是存在的，所以还会加上间隙锁，规则是向下遍历到第一个不符合条件的值才能停止，因此间隙锁的范围是 `(16,32)`
    3. 所以，上述语句在普通索引 a 上的最终加锁范围是 Next-key Lock `(8,16]` 以及  Gap Lock `(16,32)`
    > 很好记忆，就是要查找记录的左区间加 Next-key Lock，右区间加 Gap lock
3. ==当查询的记录是====**不存在**====的，Next-key Lock 会退化成====**间隙锁**====（这个规则和唯一索引的等值查询是一样的）==
    ![[IMG-20260620224219065.png|800]]
    1. 结合加锁的两条核心，这条语句首先会对普通索引 a 加上 Next-key Lock，范围是 `(16,32]`
    2. 但是由于查询的记录 `a = 18` 是不存在的，因此 Next-key Lock 会退化为间隙锁，即最终在普通索引 a 上的加锁范围是 `(16,32)`
### 2.4 案例 4：非唯一索引范围查询
1. 范围查询和等值查询的区别在上面唯一索引章节已经介绍过了，就是范围查询需要一直向右遍历到第一个不满足条件的记录
2. 和唯一索引范围查询不同的是，==非唯一索引的范围查询并==**==不会退化成 Record Lock 或者 Gap Lock==**
    ![[IMG-20260620224219157.png|800]]
3. 先来看语句查询条件的前半部分 `a >= 16`
    1. 因此，这条语句最开始要找的第一行是 a = 16，结合加锁的两个核心，需要加上 Next-key Lock `(8,16]`
    2. ==虽然非唯一索引== ==`a = 16`== ==的这行记录是存在的，但此时====**并不会**====像唯一索引那样退化成记录锁==
4. 再来看语句查询条件的后半部分 `a < 18`
    1. 由于是范围查找，就会继续往后找第一个不满足条件的记录，也就是会找到 `id = 32` 这一行停下来，然后加 Next-key Lock `(16, 32]`
    2. ==虽然== ==`id = 32`== ==不满足== ==`id < 18`====，但此时====**并不会**====向唯一索引那样退化成间隙锁==
5. 所以，上述语句在普通索引 a 上的最终的加锁范围是 Next-key Lock `(8, 16]` 和  `(16, 32]`，也就是 `(8, 32]`
### 2.5 案例 5：更新条件使用无索引字段时，行锁会升级为表锁
1. 事务 A 执行如下 SQL 时
    ```Java
    update user set a = 40
    where b = Bob
    ```
2. 其他事务是不能当前表的任何记录的
3. why❓
    1. 因为事务A是根据 a 字段进行更新的，而 a 字段是没有索引的
    2. ==如果没有索引，此时行锁会升级为表锁（⭐因为==**==行锁是对索引项加的锁，而 a 没有索引==**==）==
4. 即证明了，InnoDB 的行锁是针对索引字段加的锁，表级锁是针对非索引字段加的锁
# 三、MySQL 死锁
## 1. 案例
1. 死锁案例：转账
2. A 账户给 B 账户转账 50 元的同时，B 账户也给 A 账户转账 30 元
3. 正常情况下，如果只有一个操作，A 给 B 转账 50 元，可以在一个事务内完成，先获取 A 用户的余额和 B 用户的余额，因为之后需要修改这两条数据，所以需要通过写锁（for UPDATE）锁住他们，防止其他事务更改导致我们的更改丢失而引起脏数据
    ![[IMG-20260620224220369.png|800]]
4. **但如果 A 给 B 转账和 B 给 A 转账同时发生，那就是两个事务，可能发生死锁：**
    1. A 用户给 B 用户转账 50 元，需在程序中开启事务 1 来执行 SQL，获取 A 的余额同时锁住 A 这条数据
    2. B 用户给 A 用户转账 30 元，需在程序中开启事务 2 来执行 SQL，并获取 B 的余额同时锁住 B 这条数据
    3. 在事务 1 中执行剩下的 SQL，此时事务 1 是获取不到 B 的锁的，也即 select for update 就会被阻塞住
    4. 同理，事务 2 继续执行剩下的 SQL，请求 A 的锁，也是获取不到的
5. 事务 1 和事务 2 存在相互等待获取锁的过程，导致两个事务都挂起阻塞，最终抛出获取锁超时的异常
    ![[IMG-20260620224221273.png|800]]
## 2. 解决 MySQL 死锁
1. 由于**互斥**和**不剥夺**是锁本质的功能体现，无法修改，所以从另外两个条件尝试去解决
2. **破坏 “请求和保持” 条件**：
    1. A 和 B 之间的操作用==同一个锁锁住（比如用 Redis 分布式锁做==，A 和 B 之间的锁的 key 表示为 `A:B`，可以让 id 小的用户排在前面，id 大的用户排在后面，这样来设计 key。如果存在分库分表的情况，用 hashcode 来做比较也行）
    2. 保证 A 向 B 转账和 B 向 A 转账这两个事务同一时刻只能有一个事务能成功获取锁
    ![[IMG-20260620224221790.png|800]]
3. **破坏 “循环等待” 条件**：
    1. ==先获取更小的锁，获取到了小的锁才能获取大锁==（所谓小锁还是大锁，也可以简单的根据用户的 id 来进行区分，先请求用户 id 较小的，再请求用户 id 较大的）
    2. 比如 A.id < B.id，那么 A 和 B 之间的操作，都是要先获取 A 锁，再获取 B 锁
    ![[IMG-20260620224221836.png|800]]
