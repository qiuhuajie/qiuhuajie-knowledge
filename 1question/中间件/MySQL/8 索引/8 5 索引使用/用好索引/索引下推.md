# 1. **索引下推介绍**
1. **索引下推：**
    
    1. 应该叫做**索引条件下推**（Index Condition Pushdown，**ICP**）
    
    1. 就是索引失效时，失效部分的==过滤的动作==**==由下层的存储引擎层通过使用索引来完成==**==，而==**==不需要上推到 Server 层进行处理==**
    
    1. ==可以有效减少回表操作的次数，提高性能==
    
# 2. 不开启索引下推存在的问题
1. 假设 `name` 和 `phone` 上有联合索引
1. sql 语句
    
    ```SQL
    SELECT * FROM user_innodb WHERE name = "蝉沐风" AND phone LIKE "%6606";
    ```
    
1. **由于 `phone` 前加了 % ，导致 `phone` 字段的索引失效**
1. 此时查询过程是
    
    1. InnoDB使用联合索引查出所有`name`为蝉沐风的二级索引数据，得到3个主键值：3485，78921，423476
    
    1. 拿到主键索引进行回表，到聚簇索引中拿到这三条完整的用户记录
        
        ![[IMG-20260405035438380.png|Untitled 559.png]]
        
    
    1. ==**InnoDB 会把这 3 条完整的用户记录返回给 MySQL 的 Server 层，在 Server 层过滤出**====`phone LIKE "%6606"`====**的用户**==
        
        > 索引的使用是在存储引擎中进行的，而数据记录的比较是在Server层中进行的
        
        ![[IMG-20260405035455355.png|Untitled 1 417.png]]
        
    
1. 现在把上述搜索考虑地极端一点，**假如数据表中10万条记录都符合`name='蝉沐风'`的条件，而只有1条符合`phone LIKE "%6606"`条件**
1. 这就意味着🚨
    
    1. 首先，**==InnoDB需要将99999条无效的记录传输给Server层让其自己筛选==**
    
    1. **更严重的是，**==**这99999条数据都是通过回表搜索出来的（回表是有代价的）**==
    
# 3. 开启索引下推的好处
1. 上面的例子中，**开启索引下推后：**
    
    1. 第一步已经通过 `name = "蝉沐风"` 在联合索引的叶子节点中找到了符合条件的3条记录，而且 `phone` 字段也恰好在联合索引的叶子节点的记录中
    
    1. 这个时候==可以直接在联合索引的叶子节点中进行遍历，筛选出尾号为6606的记录==，后面**==只需一次回表==**，找到主键值为 78921 的记录，返回给Server层
        
        ![[IMG-20260405035508956.png|Untitled 2 338.png]]
        
    
    1. 很明显，使用 ICP 的方式能==有效减少回表的次数==
    
1. 另外，**ICP是默认开启的**，对于二级索引，只要能把条件甩给下面的存储引擎，存储引擎就会进行过滤，不需要我们干预
1. 查看当前ICP的状态：
    
    ```SQL
    SHOW VARIABLES LIKE 'optimizer_switch';
    ```
