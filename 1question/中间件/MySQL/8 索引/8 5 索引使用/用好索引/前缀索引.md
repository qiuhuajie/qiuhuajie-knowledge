# 一、前缀索引
1. 字段类型为字符串（varchar，text，longtext等）时，有时候**需要索引很长的字符串，这会让**==**索引变得很大**==，查询时，浪费大量的磁盘IO， 影响查询效率
2. 此时可以==**只将字符串的一部分前缀，建立索引**==，这样可以**大大节约索引空间，从而提高索引效率**
3. 语法：
    ```SQL
     create index idx_xxxx on table_name(column(n)) ;
     create index idx_email_5 on tb_user(email(5));
    ```

5. 前缀索引的弊端：
    1. **使用前缀索引可能会增加记录扫描次数与回表次数，影响性能**
        1. 可能存在多条前缀一样的记录，当回表第一次发现不是想找的记录；就需要再去索引里找符合前缀的记录主键，再回表
        2. 所以如果前缀的长度选取不当，会导致单个前缀对应多个记录，进而增加回表次数
    2. **使用前缀索引就用不上覆盖索引对查询性能的优化了**
        1. 如果使用 email 整个字符串的索引结构的话，==可以利用上覆盖索引==，从索引树上查到结果后就可以返回了（id，email 都包含），==不需要进行回表==
    ![[IMG-20260620224147778.png|800]]
        2. 而如果使用 email(6) 前缀索引结构的话，就==不得不再次根据主键值去回表==，判断 email 字段的值是否真的是 'zhangs2001'
