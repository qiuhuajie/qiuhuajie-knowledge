# 一、概述
1. MySQL 的**==索引 是在 存储引擎层 实现==**的
2. 不同的存储引擎有不同的索引结构，主要包含以下几种

|   |   |
|---|---|
|**索引结构**|**描述**|
|B+Tree索引|最常见的索引类型，**大部分引擎都支持 B+ 树索引**|
|Hash索引|底层数据结构是用哈希表实现的, 只有精确匹配索引列的查询才有效, 不支持范围查询（因为hash中数据无序）|
|R-tree(空间索 引）|空间索引是MyISAM引擎的一个特殊索引类型，主要用于地理空间数据类型，通常使用较少|
|Full-text(全文 索引)|是一种通过建立倒排索引,快速匹配文档的方式。类似于 Lucene,Solr,ES|
3. 上述是MySQL中所支持的所有的索引结构，接下来，下面是不同的存储引擎对于索引结构的支持情况

|   |   |   |   |
|---|---|---|---|
|**索引**|**InnoDB**|**MyISAM**|**Memory**|
|B+tree索引|支持|支持|支持|
|Hash 索引|不支持|不支持|支持|
|R-tree 索引|不支持|支持|不支持|
|Full-text|5.6版本之后支持|支持|不支持|
    - 在 MySQL 中，无论是 Innodb 还是 MyIsam，都使用了 B+树作为索引结构
# 二、Hash
1. MySQL中除了支持B+Tree索引，还支持一种索引类型---Hash索引
2. 结构
    1. 哈希索引就是采用一定的hash算法，将键值换算成新的hash值，映射到对应的槽位上，然后存储在hash表中
    ![[IMG-20260620224137519.png|800]]
    2. 如果两个(或多个)键值，映射到一个相同的槽位上，他们就产生了**hash冲突（也称为hash碰撞）**，可以通过**==链地址法==来解决冲突**
    ![[IMG-20260620224141048.png|800]]
3. 特点：
    1. **效率通常要高于B+tree 索引**，通常（不存在hash冲突的情况）只需要**一次检索**就可以了
    2. 但是‼️，==**Hash 索引不支持顺序和范围查询**==
        1. **==因为 hash 表中的数据是无序的==，所以 Hash 索引只能用于对等比较(`=`，`in`)，不支持范围查询（`between`，`>`，`<` ，...）**
        2. **故 MySQL 没有使用其作为索引的数据结构，而是使用 B+tree**

## 5. MySQL 的 B+ Tree
1. 上述的结构是标准的B+Tree的数据结构，**MySQL中的 B+Tree 是经过优化之后的 B+Tree**
2. MySQL在原 B+Tre e的基础上：
    - 增加一个指向相邻叶子节点的链表指针==**（叶子节点组成了一个双向循环有序链表）**==，**提高区间访问的性能，利于排序**
    ![[IMG-20260620224223217.png|800]]
3. ==**Mysql B+ 树高度的计算**== ==🙋‍♂️==
    > 💡 **MySQL 的 B+ Tree 中的一个节点的大小是如何决定的**
    > ❓
    > - 在实际设计中，我们把**一个结点设为一个页**，因为磁盘预读是以页为单位的，读取一页就代表访问一次磁盘，也就是代表一次I/O操作。
    > - ==一个节点用一个页（16 KB）来存储==
    > - 页是 InnoDB 存储引擎==磁盘管理的最小单元==，每个页的大小默认为 16KB
    > - 那么一个节点的大小设为一页或页的倍数最为合适。因为如果一个节点的大小 < 1 页，那么==读取这个节点的·时候==其实读取的还是一页，这样就造成了==资源的浪费==
    1. **计算公式：假设中间结点的扇出系数为** **`k`** **，则叶子结点数为** **`k^(h-1)`** **；每个叶子结点的数据记录数为** `n`，则总记录数为 `k^(h-1)*n`
    2. 举例：
        1. 在 InnoDB里，每个页面默认**`16KB`**，假设主键是 `4B` 的 `int` 类型。对于中间节点，每个主键值后有个页号 `4B`，还有 `6B` 的其他数据
        2. 那么扇出系数 `k = 16KB/(4B+4B+6B) ≈ 1170`
        3. 我们再假设每行记录大小为 `1KB`，则每个叶子结点可以容纳的记录数 `n = 16KB/1KB = 16`
        4. 在高度 **`h=3`** 时，`叶子结点数 = 1170^2 ≈ 137W`，`总记录数 = 1170^2*16 =` **`2190W`**
        5. 也就是说，InnoDB通过三次索引页面的I/O，即可索引2190W行记录
        6. 同理，在高度 **`h=4`** 时，`总行数 = 1170^3*16 ≈` **`256亿`**
    3. 这只是我们的理论分析，如何查看索引的真实高度
        1. InnoDB 每个页都包含一个 PAGE_LEVEL 的信息，用于表示当前页所在索引中的高度
        2. 默认叶子节点的高度为 0，那么 **Root 页的 `PAGE_LEVEL + 1`** 就是这棵索引的高度
        3. 具体语句省略
## 6. ⭐🙋‍♂️为什么 InnoDB 存储引擎选择使用 B+tree 索引结构❓
1. ==**相对于 BST 二叉树，二叉平衡树**==
    1. 维护树的代价是非常大，在进行插入或更新时，经常会**需要多次左旋或右旋来维持平衡，带来性能消耗**
    2. 而且**数据量多的时候，树会很高**，需要多次I/O操作，效率低
2. ==**相对于 B 树**==
    1. **相同树高，存储更多数据：**
        1. B+ 树**只有叶子节点存储数据，非叶子节点仅仅起到索引数据作用**，在层高相同的情况下，比 B 树的存储数据量更多
        2. 也意味着，存储相同数据量，树的高度更低，IO次数也更低，查询效率更高
    2. **范围查询和全表扫描更高效：**
        1. MySql B+ 树的所有存储在叶子节点的数据使用了**双向链**表来关联，所以 B+ 树在范围查询的时候只需查两个节点进行遍历就行，而 B 树要遍历所有节点
        2. 同时，B+ 树的**叶子节点存储所有数据**，所以B+树的全局扫描能力更强一些，因为它只需要扫描叶子节点，而 B 树要遍历所有节点
    3. **B+ 树查询效率更稳定**
        - B 树的检索的过程相当于对范围内的每个节点的关键字做二分查找，可能还没有到达叶子节点，检索就结束了
        - 而 B+ 树的检索效率就很稳定了，**任何查找都是从根节点到叶子节点的过程**
3. ==**相对 Hash 索引**==
    1. **在等值查询(`=`，`in`)时，效率要高于B+tree 索引**，通常（不存在hash冲突的情况）只需要一次检索就可以了
    2. 但是‼️，**因为 hash 表中的数据是无序的，所以 Hash 索引不支持范围查询（`between`，`>`，`<` ，...）**
    3. 而 B+ 树天生就是一个 搜索树，元素插入时会维护其有序性
4. 综上，MySQL 没有使用其它数据结构作为索引的数据结构，而是使用 B+tree
