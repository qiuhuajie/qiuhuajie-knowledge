# 1. 使用**联合索引时**
## 1.1 联合索引的**最左前缀法则**
1. **索引了多列（联合索引），要遵守最左前缀法则**
1. 最左前缀法则：==**查询条件要从索引的最左列开始，并且不跳过索引中的列。如果跳跃某一列，右面的索引将会失效**==
    
    - 举例说明
        
        1. 在 tb_user 表中，有一个联合索引，这个联合索引涉及到三个字段，索引顺序分别为：**`profession，age，status`**
        
        1. 对于最左前缀法则指的是：查询时
            
            - 索引**最左边的列，也就是** `**profession**` **必须在条件中（不分顺序），否则索引全部失效**
            
            - **而且中间不能跳过索引某一列，否则条件中该列后面的字段索引将失效**
            
        
        1. 索引**生效**的示例：
            
            ```SQL
            explain select * from tb_user where profession = '软件工程' and age = 31 and status = '0';
            
            explain select * from tb_user where profession = '软件工程' and age = 31;
            
            explain select * from tb_user where profession = '软件工程';
            ```
            
        
        1. 索引**全部失效**的示例：
            
            ```SQL
             explain select * from tb_user where age = 31 and status = '0';
            ```
            
        
        1. 索引**部分失效**的示例：
            
            ```SQL
             explain select * from tb_user where profession = '软件工程' and status = '0';
            ```
            
        
    
1. 为啥❓
    
    1. 如果建立了一个三个字段的联合索引：（a，b，c）
    
    1. **==实际是建立了三个索引：（a），（a，b），（a，b，c）==**
    
    1. 如果查询条件跳过任何一个字段（a，c）（b，c），都会导致找不到完全匹配的联合索引，因此索引只能左边的部分生效
        
        > ==b 和 c 列的有序性都是依托于 a 列的存在的==
        
    
    1. 但查询（a，b）可以用到这个联合索引
    
## 1.2 联合索引的任意一个索引列使用**范围查询 > 或 <，则右侧的失效**
1. 联合索引中，出现==**范围查询（**==**`>,<`==），范围查询右侧的列索引失效==**
    
    - 举例说明
        
        ```SQL
         explain select * from tb_user where profession = '软件工程' and age > 30 and status = '0';
        ```
        
        当范围查询使用> 或 < 时，走联合索引了，但是索引的长度为49，就说明**范围查询右边的status字段是没有走索引的**
        
        ![[IMG-20260405035438321.png|Untitled 239.png]]
        
    
1. 在业务允许的情况下，**尽可能的使用类似于** `**>=**` **或 `<=` 这类的范围查询**，而避免使用 `>` 或 `<`
    
    - 举例说明
        
        ```SQL
         explain select * from tb_user where profession = '软件工程' and age >= 30 and status = '0';
        ```
        
        当范围查询使用>= 或 <= 时，走联合索引了，但是**索引的长度为54，就说明所有的字段都是走索引的**
        
        ![[IMG-20260405035447150.png|Untitled 1 172.png]]
        
    
# 2. 使用**单列索引时**
## 2.1 对**索引列运算**
- **==在索引列上进行运算操作， 索引将失效==**
    
    - 举例说明
        
        1. 在tb_user表中，phone 字段是一个单列索引
        
        1. 当根据phone字段进行函数运算操作之后，索引失效
            
            ```SQL
             explain select * from tb_user where substring(phone,10,2) = '15';
            ```
            
            ![[IMG-20260405035505826.png|Untitled 2 144.png]]
            
        
    
## 2.2 **字符串不加引号**
- **==字符串类型字段使用时，不加引号，索引将失效==**
    
    - 举例说明
        
        ```SQL
         explain select * from tb_user where phone = '17799990015';  （生效）
         explain select * from tb_user where phone = 17799990015;  （失效）
        ```
        
        ![[IMG-20260405035512905.png|Untitled 3 110.png]]
        
        - 经过上面两组示例，会明显的发现，如果字符串不加单引号，对于查询结果，没什么影响，但是**数据库存在隐式类型转换，索引将失效**
        
    
## 2.3 头部**模糊查询：**%xxx
1. 如果仅仅是尾部模糊匹配，索引不会失效
1. **==如果是头部模糊匹配，索引失效==**
    
    - 举例说明
        
        示例：模糊查询时，`%`加在关键字之前，和加在关键字之后的影响
        
        ```SQL
         explain select * from tb_user where profession like '软件%';  （生效）
         
         explain select * from tb_user where profession like '%工程';  （失效）
         
         explain select * from tb_user where profession like '%工%';  （失效）
        ```
        
    
## 2.4 **or 的左右出现没索引的字段**
- **==当==** `**==or==**` **==连接的条件==**
    
    1. 左右两侧字段都有索引时，索引才会生效
    
    1. **==只要有一个字段没有索引，则索引全部失效==**
    
    - 举例说明
        
        1. age没有索引，所以即使id、phone有索引，索引也会失效
            
            ```SQL
             explain select * from tb_user where id = 10 or age = 23;
             
             explain select * from tb_user where phone = '17799990017' or age = 23;
            ```
            
        
        1. 对 age 字段建立索引
            
            ```SQL
             create index idx_user_age on tb_user(age);
            ```
            
        
        1. 再次执行上述的SQL语句，看看前后执行计划的变化

## 2.4 评估到走全表扫描更快
1. MySQL在查询时，**会根据数据分布，评估==使用索引的效率==与==走全表扫描的效率==，如果走全表扫描更快，则放弃索引，走全表扫描**
1. 示例：
    
    ```SQL
     select * from tb_user where phone >= '17799990005';  （生效）
     
     select * from tb_user where phone >= '17799990015';  （失效）
    ```
    
    1. 相同的SQL语句，只是传入的字段值不同，最终的执行计划也完全不一样❓
        
        1. 因为==**符合条件的数据占大多数时，MySQL 评估：通过索引查询返回大批量的数据**==
        
        1. **因此使用全表扫描的效率比走索引快，则放弃索引，走全表扫描**
