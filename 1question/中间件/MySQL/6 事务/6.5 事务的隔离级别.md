# 1. 隔离级别介绍
1. A教室和B教室中间有一道墙，这道墙可以很厚，也可以很薄。这就是事务的隔离级别
1. 这道墙越厚，表示隔离级别就越高
1. **查看隔离级别**
    
    ```SQL
     SELECT @@tx_isolation
     +-----------------+
     | @@tx_isolation  |
     +-----------------+
     | REPEATABLE-READ |    \#mysql默认的隔离级别
     +-----------------+
    ```
    
1. **设置事务隔离级别**
    
    ```SQL
     SET [ SESSION | GLOBAL ] TRANSACTION ISOLATION LEVEL { READ UNCOMMITTED | READ COMMITTED | REPEATABLE READ | SERIALIZABLE }
    ```
    
1. 事务和事务之间的**隔离级别有4个级别**
    
    ![[IMG-20260405035437138.png|Untitled 233.png]]
    
    > [!important] 注意：事务隔离
    > 
    > **级别越高**，**数据越安全**，但是**性能越低**
    
# 2. 五个隔离级别
## 2.1 **读未提交**read uncommitted
1. 最低的隔离级别
1. 事务A可以读取到事务B未提交的数据
    
    ![[IMG-20260405035446833.png|Untitled 1 165.png]]
    
1. **存在的问题：**
    
    1. 脏读现象！(Dirty Read)
    
    1. ==**这里脏并不单单是错，而是毫无意义的数据修改（因为这些修改被其他事务回滚了）**==
    
1. 这种隔离级别一般都是理论上的，大多数的数据库隔离级别都是二档起步
1. 验证：read uncommited
    
    ```SQL
     set global transaction isolation level read uncommitted;  # 设置事务隔离级别为 read uncommitted
     事务A                                             事务B
     --------------------------------------------------------------------------------
     use bjpowernode;
                                                         use bjpowernode;
     start transaction;
     select * from t_user;
                                                         start transaction;
                                                         insert into t_user values('zhangsan');
     select * from t_user;#======>事务B不需要提交，事务A就可以查到zhangsan
    ```
    
## 2.2 **读已提交** read committed
1. **oracle 数据库默认的隔离级别**
1. 事务A只能读取到事务B提交之后的数据
    
    ![[IMG-20260405035446843.png|Untitled 2 138.png]]
    
1. **优点：**解决了脏读的现象，每一次读到的数据是绝对的真实
1. **存在的问题：**不可重复读取数据
1. 验证：read commited
    
    ```SQL
     set global transaction isolation level read committed;      # 设置事务隔离级别为 read committed
     事务A                                             事务B
     --------------------------------------------------------------------------------
     use bjpowernode;
                                                         use bjpowernode;
     start transaction;
                                                         start transaction;
     select * from t_user;
                                                         insert into t_user values('zhangsan');
     select * from t_user;
                                                         commit;
     select * from t_user;#=====>事务B提交之后，事务A才可查到zhangsan
    ```
    
## 2.3 **可重复读** repeatable read
1. **mysql 中默认的事务隔离级别**
1. 事务A开启之后，不管是多久，**每一次在事务A中读取到的数据都是一致（不变）的**
    
    1. 即使事务B将数据已经修改，并且提交了，事务A读取到的数据还是没有发生改变，这就是可重复读
    
    1. 也就是忽略本次事务开启期间，其他事务对数据的修改
        
        ![[IMG-20260405035446899.png|Untitled 1 166.png]]
        
    
1. **优点：**解决了不可重复读取数据
1. **存在问题：**幻读——前后查询的结果集不一致
    
    ![[IMG-20260405035505644.png|Untitled 3 105.png]]
    
    > [!important] ⭐
    > 
    > **可重复读级别中会存在幻读，但如果将隔离级别提升到序列化，根本不可能，**==**那么问题来了：如何保证在 REPEATABLE READ 级别下避免幻读？**==
    > 
    > - 如果执行的是**普通** `**select**`**，此时会以** `**MVCC**` **快照读的方式读取数据**[[MVCC]]
    >     
    >     - 在快照读的情况下，RR 隔离级别只会在事务开启后的第一次查询生成 Read View，并使用至事务提交
    >     
    >     - 所以在生成 Read View 之后，其它事务所做的更新、插入记录版本对当前事务并不可见
    >     
    >     - 可见，快照读天然避免了 “幻读”
    >     
    > 
    > - **所以，幻读只会在当前读下出现，那么是如何避免的❓**
    > 
    > - 当**使用 `select ... lock in share mode` 、**`**select...for update/delete/insert**` **时，此时会以** `**MVCC**` **当前读的方式读取数据**
    >     
    >     - 当前读会读到其他事务新提交的修改数据，也就**产生了幻读**。这种情况下，InnoDB 会使用 [[12.4 行级锁]][[12.4 行级锁]]来防止这种情况
    >     
    >     - **当执行当前读时，会锁定读取到的记录的同时，锁定它们的间隙，防止其它事务在查询范围内插入数据**（只要我不让你插入，就不会发生幻读）
    >     
    >     - 即当执行当前读的时候，就不止是给数据库中已有的 n 个记录加上了行锁，还同时加了 n + 1 个间隙锁（这两个合起来也成为 Next-Key Lock 临键锁）
    >     
    
    > [!important] ==**可重复读级别可以减少幻读的发生，但并不能完全解决幻读问题**==
    > 
    > - 特例：
    > 
    > - 但是如果事务B**一开始使用快照读**，然后**在事务A又插入了一条数据，接着事务B才使用 update 开启了当前读**，这种情况下，还是存在幻读的情况
    >     
    >     ![[IMG-20260405035512530.png|Untitled 2 139.png]]
    >     
    
1. 验证：repeatable read
    
    ```SQL
     set global transaction isolation level repeatable read;     # 设置事务隔离级别为 repeatable read
     事务A                                             事务B
     --------------------------------------------------------------------------------
     use bjpowernode;
                                                         use bjpowernode;
     start transaction;
                                                         start transaction;
     select * from t_user;
                                                         insert into t_user values('lisi');
                                                         insert into t_user values('wangwu');
                                                         commit;
     select * from t_user;#=====>读不到数据，事务A结束之前，只能查到事务A开启那个时刻的数据
    ```
    
## 2.4 **串行化** serializable
1. 最高的隔离级别
1. 这是最高隔离级别，解决了所有的问题，每一次读取到的数据都是最真实的
1. 这种隔离级别下，**==事务需要排队，不能并发==**：synchronized，线程同步（事务同步）
1. **读取每条数据都会加锁，会导致大量的超时和锁争用问题**
    
    ![[IMG-20260405035512591.png|Untitled 3 106.png]]
    
1. **缺点：**只能串行处理事务效率是最低的
1. 验证：serializable
    
    ```SQL
     set global transaction isolation level serializable;
     事务A                                             事务B
     --------------------------------------------------------------------------------
     use bjpowernode;
                                                         use bjpowernode;
     start transaction;
                                                         start transaction;
     select * from t_user;
     insert into t_user values('abc');
                                                         select * from t_user;
                                                         _  #=====>光标卡在这里了，等到事务A结束才能继续执行
    ```
