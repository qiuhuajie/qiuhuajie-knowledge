> 💡 MVCC的具体实现，还需要依赖于数据库记录中的三个隐式字段、undo log日志、readView
# 一、三个隐式字段
1. 当我们创建了下面的这张表，在查看表结构的时候，就可以显式的看到三个字段
    ![[IMG-20260620224137567.png|161]]
2. 实际上除了这三个字段以外，**InnoDB 还会自动添加三个隐藏字段**
3. ⭐**其含义分别是：**

|   |   |
|---|---|
|**隐藏字段**|**含义**|
|**`DB_TRX_ID`**|**==最近修改事务ID==**，记录插入这条记录或最后一次修改该记录的事务ID，是自增的|
|**`DB_ROLL_PTR`**|**==回滚指针==**，指向这条记录的**上一个版本**，用于配合 undo log，**来指定如果发生回滚，回滚到哪一个版本**|
|**`DB_ROW_ID`**|**==隐藏主键==**，如果表结构没有指定主键，将会生成该隐藏字段（**如果有主键，则不会添加该隐藏字段**）|

# 二、undo Log 中的版本链
1. **版本链的生成：**
    1. 一张表原始数据为
    ![[IMG-20260620224141511.png|460]]
        - **`DB_TRX_ID`** : 代表**最近修改事务ID**，记录插入这条记录或最后一次修改该记录的事务ID，是自增的
        - **`DB_ROLL_PTR`** ： **回滚指针（指向上一个 undolog 记录）**，由于这条数据是才插入的，没有被更新过，所以该字段值为null
    2. 然后，有四个并发事务同时在访问这张表
        1. 第一步：
    ![[IMG-20260620224148263.png|633]]
            1. 当事务2执行第一条修改语句时，会记录 undo log 日志，记录数据变更之前的样子
            2. 然后更新记录的隐藏字段，并且**记录本次操作的 事务ID、回滚指针**
    ![[IMG-20260620224158206.png|800]]
        2. 第二步：
    ![[IMG-20260620224158293.png|796]]
            1. 当事务3执行第一条修改语句时，也会记录 undo log 日志，记录数据变更之前的样子
            2. 然后更新记录的隐藏字段，并且**记录本次操作的 事务ID、回滚指针**
    ![[IMG-20260620224204465.png|800]]
        3. 第三步：
    ![[IMG-20260620224210559.png|800]]
            1. 当事务4执行第一条修改语句时，也会记录 undo log 日志，记录数据变更之前的样子;
            2. 然后更新记录的隐藏字段，并且**记录本次操作的 事务ID、回滚指针**
    ![[IMG-20260620224210653.png|800]]
        4. ==**最终，不同事务或相同事务对同一条记录进行修改，会导致该记录的 undolog 生成一条记录版本链表**==
            - **链表的头部是最新的旧记录**
            - **链表尾部是最早的旧记录**
# 三、readView
1. 那么问题又来了，一个记录会被一堆事务进行修改，一个记录上会存在许许多多的 undo log
    1. **那么对于其中某一个事务来说，它能看见哪些 undo log**？
    2. 这就需要 ReadView 读视图来实现
2. 在 ReadView 中规定了**==版本链数据的访问规则：==**
    - ReadView 中包含了四个核心字段：

|   |   |
|---|---|
|**字段**|**含义**|
|creator_trx_id|ReadView **创建者**的事务ID|
|min_trx_id|**最小**活跃事务ID|
|max_trx_id|预分配事务ID，当前**最大**事务ID+1（因为事务ID是自增的）|
|m_ids|**当前活跃的事务ID集合（活动：未提交）**|

    - 假设 `trx_id` 是在 undo log 版本链上查询过程中，取出的当前链节点记录的事务 id

|   |   |   |
|---|---|---|
|**条件**|**说明**|**是否可以访问**|
|`trx_id == creator_trx_id`|说明当前链中遍历到的数据**是当前这个事务更改的**|**可以访问**该版本|
|`trx_id < min_trx_id`|说明**数据已经提交了**|**可以访问**该版本|
|`trx_id > max_trx_id`|说明该事务**是在 ReadView 生成后才开启**|**不可以访问**该版本|
|`min_trx_id <= trx_id <= max_trx_id`|数据**可能已经提交，有可能没提交**，还需用 **`m_ids`** 再筛选一下|**如果 `trx_id` 不在 `m_ids` 中**， 是**可以访问**该版本的|
![[IMG-20260620224214760.png|800]]

3. **⭐不同的隔离级别，**==**生成 ReadView 的时机**==**不同：**

    1. **`READ COMMITTED`** ：在事务中**每一次**执行快照读时生成ReadView

    2. **`REPEATABLE READ`**：仅在事务中**第一次**执行快照读时生成ReadView，后续复用该ReadView
