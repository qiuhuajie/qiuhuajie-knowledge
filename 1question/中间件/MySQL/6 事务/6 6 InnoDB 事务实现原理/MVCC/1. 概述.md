# 一、MVCC 介绍
1. 多版本并发控制，全称 Multi-Version Concurrency Contro，维护一个数据的多个版本
2. ⭐主要是为了==提高数据库====**并发性能**====，用更好的方式去处理====**读-写**====冲突==
    1. ==即使有读写冲突时，也能做到====**不加锁**====，====**非阻塞并发读（**====这个读指的就是====**快照读，**====而非====**当前读）**==
    2. MVCC 比单纯的加锁更高效
3. **MVCC 只在 `Read Committed` 和 `Repeatable Read` 两个隔离级别下工作**
# 二、快照读 与 当前读
## 1. 快照读
1. 快照读 ==**读取的是记录数据的可见版本**==，有可能是历史数据，不加锁，是非阻塞读
    1. 快照读是==**基于 MVCC 实现的**==，提高了并发的性能，降低开销
    2. 简单的 **`select`**（不加锁）就是快照读，因此大部分业务代码中的读取都属于快照读
2. **在各个隔离级别下：**
    - `Read Committed`：**每次 select**，都生成一个快照读（对应 readview 的生成时机）
    - `Repeatable Read`：开启事务后**第一个 select 语句**才是快照读的地方（对应 readview 的生成时机）
    - `Serializable`：快照读会**退化为当前读**
3. 测试：
    1. 在测试中，即使事务B提交了数据，事务A中也查询不到（符合默认的 RR 隔离级别）
    2. 原因就是因为普通的 select 是快照读
        1. 而**在当前默认的 RR 隔离级别下，开启事务后第一个 select 语句才是快照读的地方**
        2. 后面执行相同的 select 语句**都是从快照中获取数据**，可能不是当前的最新数据，这样也就保证了可重复读

## 2. 当前读
1. **==读取的是记录的最新版本==**，读取时**==还要保证其他并发事务不能修改当前记录，会对读取的记录进行加行和间隙锁==**
    1. 行锁：锁住当前行，不让修改
    2. 间隙锁：锁住间隙，不让插入新行
2. 如：
    1. **`select ... lock in share mode`**(共享锁)
    2. **`select ... for update`**、**`update`**、**`insert`**、**`delete`**（排他锁）（`update`、`delete`、`insert` 语句虽然没有 `select`，但是它们也会先进行读取，而且只能读取最新版本）
    3. 都是一种当前读
3. 示例：
    1. 在测试中可以看到，即使是在默认的 RR（可重复读）隔离级别下，事务A中依然可以读取到事务B最新提交的内容
    2. 不符合 RR（可重复读）隔离级别的特性呀
    3. 因为在**查询语句后面加上了** **`lock in share mode`** **共享锁**，此时是**当前读操作**。当然，当加排他锁的时候，也是当前读操作
