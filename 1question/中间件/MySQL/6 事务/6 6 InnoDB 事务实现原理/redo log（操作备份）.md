# 一、Redo Log 简介
1. **为什么需要 redo log❓**
    1. 如果在从缓冲池（[[7.2 InnoDB 体系架构]]）将页的新版本刷新到磁盘时发生了宕机，那么这个数据就不能恢复了
    2. 所以，为了==避免发生数据丢失==的问题，当前事务数据库系统普遍都采用了 **`WAL`****（`Write Ahead Log`，预写日志**）策略：
        1. 即==当事务提交时，先写重做日志（redo log），再修改页（先修改缓冲池，再刷新到磁盘）==
        2. ==当由于发生宕机而导致数据丢失时，在重启时，通过 redo log 来完成数据的恢复==
        3. ==在事务提交时，会将 redo log buffer 中的数据刷新到 redo log file 磁盘文件中==
    3. 这也是事务 ACID 中 D（Durability ==持久性==）的要求
2. **记录的是事务提交时数据页的**==**物理修改日志**==，记录的是 “==在某个数据页上做了什么修改==”
3. 该日志文件由两部分组成：
    1. **重做日志缓冲**（**`redo log buffer`**）：内存中
    2. **重做日志文件**（**`redo log file`**）：磁盘中
4. ==InnoDB 中 redolog 并不是一次性写完的，而是分==**==两阶段==**
    1. 第一阶段：prepare 阶段，将事务操作写入 redo log，将 redo log 设置为 `prepare` 状态
    2. 第二阶段：commit 阶段：将事务操作写入 bin log，将 redo log 改成 `commit` 状态
    > 💡 **知识点串联：[[面试题汇总]]**
# 二、redo Log 是如何保证事务的持久性的❓
1. 在InnoDB引擎中的内存结构中，主要的内存区域就是**缓冲池（**[[7.2 InnoDB 体系架构]]**）**，在缓冲池中缓存了很多的**数据页**
2. 当在一个事务中，执行多个增删改的操作时
    1. InnoDB引 擎会先操作缓冲池中的数据
    2. 如果缓冲区没有对应的数据，会通过后台线程将磁盘中的数据加载出来，存放在缓冲区中
    3. 然后将缓冲池中的数据修改，修改后的数据页称为==**脏页**==**（暂时没被持久化的数据页）**
3. 而==缓冲区的====**脏页数据并不是实时刷新**====的，而是====**一段时间之后**====将缓冲区的数据刷新到磁盘中==，从而保证缓冲区与磁盘的数据一致
    ![[IMG-20260620224137540.png|800]]
4. **假如刷新到磁盘的过程出错了**🙀**，而提示给用户事务提交成功，而数据却没有持久化下来**‼️，这就出现问题了，没有保证事务的持久性
5. ⭐**引入 redolog 日志后，可以解决这个问题：**
    1. 有了 redolog 之后，当对缓冲区的数据进行增删改之后
        1. ==首先====**将操作的数据页的变化，记录在**== **`redo log buffer`** ==**中**==
        2. **==在事务提交时==**，会将 **`redo log buffer`** 中的数据刷新到 **`redo log file`** 磁盘文件中
    ![[IMG-20260620224146930.png|800]]
    2. 过一段时间之后，即使**刷新缓冲区的脏页到磁盘时发生错误** ==🙀==
        1. 此时就**==可以借助于 redo log 进行数据恢复，这样就保证了事务的持久性==** 🆗
        2. **redolog 日志 相当于一个修改操作的备份记录**
# 三、Redo Log File 日志文件
1. `redo log` 在磁盘上的存储形式就是 `redo log file`
2. 刷盘的三种方式
    1. 1️⃣ 后台线程轮询写入，调用 write 将 redolog buffer 中的日志写到文件系统的 page cache，然后调用 fsync 持久化到磁盘
    2. 2️⃣ 每次事务提交的时候，都执行 fsync 将 redolog 直接持久化到磁盘
    3. 3️⃣ 当 `redo log` 使用空间达到一半时，后台线程会主动执行 fsync 写盘
    > 💡 由方式 1、2 可以看出，可能会有
    > **未提交的事务日志**，随着后台线程触发 fsync 指令刷到磁盘中
3. ==**❓疑问：**==**在事务提交时，会刷新** **`redo log buffer`** **到磁盘中，既然都要刷新数据到磁盘，为什么每一次提交事务，不直接将** **`buffer pool`** **中的脏页刷新到磁盘呢**❓
    1. 在业务操作中，==数据页刷盘是随机写==，因为一个数据页对应的位置可能在硬盘文件的随机位置
    2. 而 `redo log buffer` 在往磁盘文件中写入数据，由于是==日志文件，一个是数据量本来就小，另一个是以顺序追加方式写入的==
    3. ==顺序写的效率，要远大于随机写==。 Kafka 等消息中间件性能高的原因也是顺序写
4. **存储形式**
    1. 硬盘上存储的 `redo log file` 日志文件不只一个，而是以一个**日志文件组**的形式出现的，每个的 `redo log file` 日志文件大小都是一样的
    2. 比如可以配置为一组 `4` 个文件，每个文件的大小是 `1GB`，整个 `redo log file` 日志文件组可以记录`4G`的内容
    3. 它采用的是**==环形数组形式，从头开始写，写到末尾又回到头循环写==**
    ![[IMG-20260620224147593.png|678]]
5. `redo log file` 的大小设置对于 InnoDB 存储引擎的性能有着非常大的影响：
    1. 如果设置得很大，在恢复时可能需要很长的时间
    2. 如果设置得很小，可能导致一个事务的日志需要多次切换重做日志文件
# 四、CheckPoint 技术
1. CheckPoint 技术就是在 `redo log file` 中找到一个位置，将这个位置前的页都刷新到磁盘中去，这个位置就称为 CheckPoint（检查点）
2. **CheckPoint 技术的作用：**
    1. ==缓冲池不够用时，将脏页刷新到磁盘：==
        1. 缓冲池**（**[[7.2 InnoDB 体系架构]]**）的大小不是无限大的，**当缓冲池的空间无法存放新读取到的页时，InnoDB 引擎会采用 LRU 算法释放 LRU 列表中尾端的页
        2. 如果这个被释放出来（溢出）的页是脏页，那么就需要强制执行 CheckPoint，将脏页刷新到磁盘中去
    2. ==redo log 不可用时，将脏页刷新到磁盘：==
        1. redo log file 是循环使用而不是无限大的，会存在写满的时候，导致 redo log 不可用
        2. 但其实 redo log 中的数据不是总是有用的，存在 ”可以被重用的部分“，当数据库发生宕机时，数据库恢复操作不需要这部分的 redo log
        3. 使用 write pos 和 CheckPoint 来标识这部分可重用的部分
            1. `write pos` 是当前 redo log 记录的位置，随着不断地写入磁盘，write pos 也不断地往后移
            2. `CheckPoint` 是当前要擦除的位置（将 Checkpoint 之前的页刷新回磁盘），也是往后推移并且循环的
            3. 如果 write pos 追上 CheckPoint，就表示 redo log file 满了，需要 CheckPoint 后移将一部分 redo log 页刷到 redo log file
            4. write pos 和 CheckPoint 之间的就是 redo log file 上的”可以被重用的部分“，可以用来记录新的操作
    ![[IMG-20260620224147683.png|800]]
    3. ==缩短数据库宕机后，利用 redo log 恢复的时间：==
        1. 当数据库发生宕机时，数据库不需要重做所有的日志，因为 Checkpoint 之前的页都已经刷新回磁盘
        2. 故数据库只需对 Checkpoint 后的 redo log 进行恢复就行了。这显然大大缩短了恢复的时间
# 五、Redo Log 两阶段提交🙋‍♂️
1. 两阶段提交流程（[[面试题汇总]]）
    1. 写 redo log
        1. 引擎将这行数据更新到内存中后，还会**将这个更新操作记录到 redo log 里面**
        2. **并将 redo log 设置为 `prepare` 状态**
        3. 然后告知执行器执行完成了，随时可以提交事务
    2. 写 bin log
        1. **执行器生成这个操作的 bin log，并把 bin log 写入磁盘**
        2. 执行器调用存储引擎的提交事务接口
        3. 存储引擎把刚刚写入的 **redo log 状态改成提交（`commit`）状态，事务提交成功，更新完成**
    > 💡 **一个事务是否执行成功的判断依据是是否在 bin log 中写入成功**
2. **问什么 redo log 需要两阶段提交❓**
    1. ==redo log 采用两阶段提交，为了==**==确保 redolog 和 binlog 写入的数据一致性==**==，进而保证数据库在使用 bin log 主从复制时，主库和分库的数据一致==
    2. 根据两阶段提交，**==崩溃恢复时的判断规则==**是这样的：
        1. 如果 redo log 里面的事务是完整的，也就是已经有了 `commit` 标识，则直接提交
        2. 如果 redo log 里面的事务处于 `prepare` 状态，则判断对应的事务 bin log 是否存在并完整
            1. 如果 binlog 存在并完整，则提交事务
            2. 否则，回滚事务
    > 💡 ==redo log 在准备阶段表示当前sql已经在主库上执行过了，但 bin log 是不完整的，如果从库用这个 bin log 从主库复制数据，无法执行当前这个sql操作，会导致主从不一致==
