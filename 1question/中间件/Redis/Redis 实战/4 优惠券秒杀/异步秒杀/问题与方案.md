# 一、当前存在的问题
1. 当前所有的数据库操作都由一个线程串行执行，前面的查询优惠券等写操作执行速度还算快，不会对业务性能造成影响

    ![[IMG-20260620224127741.png|800]]

2. 但是后面的减库存、创建保存订单等数据库写操作会耗费大量时间，而当前是由一个线程串行执行的
3. 因此用户必须等待读写数据库的操作全部完成后，才能得到响应，故在性能上还有很大的提升空间
# 二、优化方案：异步秒杀
> 💡 ==**重点理解这个解决方案，其中阻塞队列可以使用 MQ 代替**==
![[IMG-20260620224132202.png|800]]
1. **优化 1：添加Redis缓存**
    1. 在创建秒杀优惠券的业务中，也将秒杀券的信息写到redis缓存中，而且并不做超时处理，活动结束后人工删除
    2. 这样对于库存的判断，以及校验一人一单的业务就可以直接操作redis缓存来完成，提升性能
    3. redis缓存中需要保存两个数据
        - **库存**：使用String类型

    ![[IMG-20260620224137852.png|382]]

        - **已下单的用户集合**：使用Set类型，保证集合元素不重复

    ![[IMG-20260620224142206.png|800]]

    4. 同时，对redis缓存的操作都是通过编写`Lua`脚本完成的，保证了操作的原子性
2. **优化 2：使用阻塞队列**
    1. 通过前面的业务判断后，如果当前的用户是可以创建订单的，则直接创建一个订单对象，不需要等待减库存、保存订单等操作完成就可以将订单信息返回给用户，因此用户无需等待数据库写操作完成后再得到订单信息，完成支付等操作
    2. 将订单信息保存在一个阻塞队列中，可以让异步线程从中读取订单数据完成数据库写操作
3. **优化 3：使用异步线程**
    1. 将对数据库的读写操作分开，将减库存、创建保存订单等数据库写操作交给一个异步的线程去做
    2. 异步线程从阻塞队列中读取订单数据完成数据库写操作
