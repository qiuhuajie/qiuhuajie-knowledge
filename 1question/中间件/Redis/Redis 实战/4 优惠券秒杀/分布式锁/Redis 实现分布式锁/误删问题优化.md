# 1. 当前存在的问题：**（考虑极端情况）**
- 当线程1在执行时被阻塞，==在阻塞期间线程1创建的锁，由于超时被删除了==
- 此时线程2，尝试获取锁成功，随后线程2开始自己的业务
- ==然后，线程1阻塞结束被唤醒，执行完业务后就去释放锁，由于释放锁的逻辑时删除，所以==**==线程1会将线程2的锁删除==**🚨
- 此时线程3，尝试获取锁成功，随后线程3开始自己的业务
- **就**==**会出现线程2和线程3同时获得锁的情况**====！==

# 2. 优化的方案
- 在**获取锁**的时候，==添加线程的标识，==**==标识此锁是哪个线程创建的==**
- 在**释放锁**的时候，==先要==**==判断是否是自己的锁==**==，只有是自己的锁才能释放==

# 3. 示例：分布式锁优化 误删锁的问题
- 在**获取锁**时存入线程标识
    
    - 由于不同JVM可能出现相同线程ID，因此再拼接一个 UUID 可以标识不同的 JVM
    
    - 所以，可以⭐**用 `UUID + 线程ID` 作为线程标识**
    
- 在**释放锁**时先获取锁中的线程标识，⭐**判断是否与当前线程标识一致**
    
    - 如果一致则释放锁
    
    - 如果不一致则不释放锁
        
        ![[IMG-20260405035500754.png|Untitled 2 343.png]]
        
    
```Java
 public class SimpleRedisLock implements ILock{
 
     private static final String KEY_PREFIX =  "lock:";
     //拼接的 UUID
     private static final String ID_PREFIX = UUID.randomUUID().toString(true) + "-";
 
     private String name;
     private StringRedisTemplate stringRedisTemplate;
 
     public SimpleRedisLock(String name, StringRedisTemplate stringRedisTemplate) {
         this.name = name;
         this.stringRedisTemplate = stringRedisTemplate;
     }
 
     @Override
     public boolean tryLock(long timeoutSec) {
         // ⭐ 获取线程标识
         String threadId = ID_PREFIX + Thread.currentThread().getId();
         // 获取锁：用 UUID + 线程ID 作为线程标识
         Boolean success = stringRedisTemplate.opsForValue().setIfAbsent(KEY_PREFIX + name, threadId, timeoutSec, TimeUnit.SECONDS);
 
         return BooleanUtil.isTrue(success);
     }
 
     @Override
     public void unLock() {
         //获取线程标识
         String threadId = ID_PREFIX + Thread.currentThread().getId();
         //获取锁中的标识
         String lockId = stringRedisTemplate.opsForValue().get(KEY_PREFIX + name);
         // ⭐ 判断标识是否一致
         if (threadId.equals(lockId)) {
             //释放锁
             stringRedisTemplate.delete(KEY_PREFIX + name);
         }
     }
 }
```
