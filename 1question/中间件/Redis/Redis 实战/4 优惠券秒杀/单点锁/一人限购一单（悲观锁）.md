# 一、业务需求
修改秒杀业务，要求同一个优惠券，一个用户只能下一单
![[IMG-20260620224127746.png|800]]
# 二、存在的问题
- 如何保证同一个优惠券，一个用户只能下一单？
- 在扣减库存之前，新加一个判断操作：**判断当前用户ID和优惠券ID是否已经存在订单**

    ```Java
     //4. 判断库存是否充足
     if (seckillVoucher.getStock() < 1) {
         return Result.fail("库存不足！");
     }
     //5. ⭐一人限购一单
     Long userId = UserHolder.getUser().getId();
     //5.1 查询订单
     Integer count = query().eq("user_id", userId).eq("voucher_id", voucherId).count();
     //5.2 判断是否存在
     if (count > 0) {
         return Result.fail("不能重复购买！");
     }
     //6. 扣减库存
     boolean success = SeckillVoucherService.update()
         .setSql("stock = stock - 1")
         .eq("voucher_id", voucherId)
         .gt("stock",0)
         .update();
     if (!success) {
         return Result.fail("库存不足！");
     }
    ```
- 存在的问题
    - 又是**先查询共享资源，再操作共享资源的流程 ➡️ 一定会存在多线程并发操作的安全问题**
    - 例如：当前优惠券一个订单都没有，此时有100个线程查询，得到的都是 count = 0，进而都会下单

# 三、使用悲观锁实现一人限购一单
- 继续使用加锁的方式对共享资源进行锁定
    - **如果使用乐观锁，由于**==**当前操作是**====**插入**====**订单操作，并不是修改，故无法使用版本号法或CAS来判断共享资源是否被修改**==
    - 因此**使用悲观锁：使用 `synchronized` 同步锁，对共享资源加锁**

    ![[IMG-20260620224132500.png|800]]
    ```Java
     @Service
     public class VoucherOrderServiceImpl extends ServiceImpl<VoucherOrderMapper, VoucherOrder> implements IVoucherOrderService {
         @Resource
         private ISeckillVoucherService SeckillVoucherService;
         @Resource
         private RedisWorker redisWorker;
         @Override
         public Result seckillVoucher(Long voucherId) {
             //1. 查询优惠卷
             SeckillVoucher seckillVoucher = SeckillVoucherService.getById(voucherId);
             //2. 判断秒杀是否开始
             if (seckillVoucher.getBeginTime().isAfter(LocalDateTime.now())) {
                 return Result.fail("秒杀尚未开始！");
             }
             //3. 判断秒杀是否已经结束
             if (seckillVoucher.getEndTime().isBefore(LocalDateTime.now())) {
                 return Result.fail("秒杀已经结束！");
             }
             //4. 判断库存是否充足
             if (seckillVoucher.getStock() < 1) {
                 return Result.fail("库存不足！");
             }
             return getResult(voucherId);    //将需要添加悲观锁的代码做一个封装
         }
         @Transactional  //上面方法的都是查询操作，并不需要添加事务
         public Result getResult(Long voucherId) {
             //5. 一人限购一单
             Long userId = UserHolder.getUser().getId();
             //⭐ 使用 synchronized 同步代码块，对共享资源加锁
             synchronized (userId.toString().intern()){
                 //5.1 查询订单
                 Integer count = query().eq("user_id", userId).eq("voucher_id", voucherId).count();
                 //5.2 判断是否存在
                 if (count > 0) {
                     return Result.fail("不能重复购买！");
                 }
                 //6. 扣减库存
                 boolean success = SeckillVoucherService.update()
                         .setSql("stock = stock - 1")
                         .eq("voucher_id", voucherId)
                         .gt("stock",0)
                         .update();
                 if (!success) {
                     return Result.fail("库存不足！");
                 }
                 //7. 创建订单
                 VoucherOrder voucherOrder = new VoucherOrder();
                 //7.1 设置订单Id、
                 long voucherOrderId = redisWorker.nextID("voucher_order");
                 voucherOrder.setId(voucherOrderId);
                 //7.2 设置用户Id
                 voucherOrder.setUserId(userId);
                 //7.3 设置代金券Id
                 voucherOrder.setVoucherId(voucherId);
                 //8. 订单写入数据库
                 save(voucherOrder);
                 //9. 返回订单Id
                 return Result.ok(voucherOrderId);
             }
         }
     }
    ```
    > 💡 `synchronized` 同步代码块 [详见 JUC 线程的同步](https://www.notion.so/37cd901487d948119921b241d518ac26?pvs=21)
    > 1. 代码：
    >     ```Java
    >     synchronized(同步监视器){
    >     	需要被同步的代码（临界区：操作共享数据）
    >     }
    >     ```
    > 2. 同步监视器：
    >     1. 锁的原理：在JVM内部维护了一个锁的监视器对象，监视器对象用的 UserId， UserId 是在常量池里
    >     2. 同一个 JVM 维护同一个常量池，保证池子里只有唯一的同步监视器对象
    >     3. 监视器对象定义要求：
    >         1. 多个线程必须要共用一把锁（唯一的对象），即对象声明在执行同步代码块的方法之外
    >         2. 这里锁是 userId.toString().intern() ，每个线程都会去数据库中查出userId，故写在执行方法内也可以保证是唯一对象
    > [!important]
    > [[StringTable（实现在堆中）]] [[StringTable（实现在堆中）]]
    > 3. 由于 `toString()`，底层实现的代码中，每调用一次都会 `new` 一个新的 String 对象
    >     ```Java
    >     public static String toString(long i) {
    >     	...
    >     	return new String(buf, true);   //⭐
    >     }
    >     ```
    > 4. 这样就无法保证锁是唯一的对象
    > 5. 故使用 `intern()`
    >     1. 该方法返回字符串对象的标准表示法
    >     2. 如何保证每次返回同一个字符串对象？
    >         1. 由 String 类私下维护一个字符串池，最初是空的
    >         2. 当调用 intern 方法时，如果池子里已经包含了一个与这个String对象相等的字符串，正如equals(Object)方法所确定的，那么池子里的字符串会被返回
    >         3. 否则，这个String对象被添加到池中，并返回这个String对象的引用
    > 6. 这样就可以⭐==**保证锁是唯一的对象**==
