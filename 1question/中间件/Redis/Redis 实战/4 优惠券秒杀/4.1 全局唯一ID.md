# 2. **全局ID生成器**
1. 是一种在分布式系统下用来生成全局唯一 ID 的工具，一般要满足下列特性：
    
    - 唯一性
    
    - 高可用
    
    - 高性能
    
    - 安全性
    
    - 递增性
    
1. **全局唯一ID生成策略的比较：**
    
    1. ==**数据库自增：**==
        
        - id 的规律性太明显，受单表数据量的限制，数据库**分表**时，自增 ID 每张表各自计算，**出现重复 ID**
        
        - 如果两台数据库分别设置**不同步长**，生成不重复 ID 的策略来实现高可用，这种方式生成的 id 有序，但是需要独立部署数据库实例，**成本高，还会有性能瓶颈**
        
    
    1. ==**UUID：**==
        
        - **不适合作为主键**，因为太长了，并且**无序不可读**，查询效率低
        
        - 比较适合用于生成唯一的名字的标示比如文件的名字
        
    
    1. ==**snowflake 算法：**==
        
        - 雪花算法是由Twitter公布的分布式主键生成算法，它能够保证不同表的主键的不重复性，以及相同表的主键的有序性
        
        - 长度共64bit（一个long型）
            
            ![[Attachment/1question/中间件/Redis/Redis 实战/4 优惠券秒杀/IMG-20260405035438355.png|Untitled 8 50.png]]
            
            - 首先是一个符号位，1bit：**标识**，由于long基本类型在Java中是带符号的，最高位是符号位，正数是0，负数是1，所以id一般是正数，最高位是0
            
            - 41bit：**时间戳**(毫秒级)，存储的是时间截的差值（当前时间截 - 开始时间截)，结果约等于69.73年
            
            - 10bit：**作为机器的ID**（5个bit是数据中心，5个bit的机器ID，可以部署在1024个节点）
            
            - 12bit：作为**毫秒内的流水号**（意味着每个节点在每毫秒可以产生 4096 个 ID）
            
        
        - 优点：整体上按**照时间自增排序**，并且**整个分布式系统内不会产生 ID 碰撞，并且效率较高**
        
    
    1. ==**Redis 自增：**==✅
        
        - 组成：
            
            ![[IMG-20260405035447787.png|Untitled 425.png]]
            
            - **符号位**：1bit，永远为0
            
            - **时间戳**：31bit，以秒为单位，可以使用69年
            
            - **序列号**：32bit，秒内的计数器，支持每秒产生 $2^{32}$ 个不同ID
                
                - **可以==使用 Redis String类型的== `INCR` ==命令生成序列号==**
                    
                    ```Java
                     //每次让序列号自增1
                     //序列号的 key 由业务标识和当天日期拼接成
                     stringRedisTemplate.opsForValue().increment("incr:" + keyPrefix + ":" + date);
                    ```
                    
                
            
        
        - 优点：
            
            - 灵活方便
            
            - ID 构造是 时间戳 + 序列号，安全且有一定的可读性
            
            - 每天在 redis 中会有一个序列号的 key，可以方便统计订单量
            
        
        - 缺点：
            
            - 引入了新的组件造成系统更加复杂，可用性降低，编码更加复杂，增加了系统成本
            
        
    
# 3. 全局ID生成器代码示例
1. RedisWorker
    
    ```Java
     @Component
     public class RedisWorker {
     
         public static void main(String[] args) {
             LocalDateTime time = LocalDateTime.of(2022, 1, 1, 0, 0, 0);
             long second = time.toEpochSecond(ZoneOffset.UTC);
             System.out.println(second);
         }
     
         //自定义时间戳的起始时间（有上面main生成，从2000/1/1/0/0/0算起）
         private static final long BEGIN_TIMESTAMP = 1640995200L;
     
         //序列号的位数
         private static final long COUNT_BITS = 32;
     
         @Resource
         private StringRedisTemplate stringRedisTemplate;
     
         public long nextID(String keyPrefix){   //keyPrefix 传入的业务标识：相同的业务共用一个Id key
             //1. 生成时间戳
             LocalDateTime now = LocalDateTime.now();
             long nowSecond = now.toEpochSecond(ZoneOffset.UTC);
             long timestamp = nowSecond - BEGIN_TIMESTAMP;
     
             //2. 生成序列号
             //2.1 获取当天日期，精确到天（用作在 redis 中保存自增序列号（value）的 key）
             String date = now.format(DateTimeFormatter.ofPattern("yyyy:MM:dd"));
             //2.2 自增长，每获取一次序列号，redis 中的序列号就会自增 1（⭐使用redis string类型的INCR命令，每次让Id key自增1，作为生成的序列号）
             long increment = stringRedisTemplate.opsForValue().increment("incr:" + keyPrefix + ":" + date);
     
             //3. 将时间戳与序列号拼接
             //这里并不是字符串的拼接，拼接后需要得到long类型的数字
             //⭐先让时间戳做位运算，向左移动32位，此时后32都是0；再与序列号做或运算，将序列号拼接在后32位上
             return timestamp << COUNT_BITS | increment;
         }
     }
    ```
    
    > [!important] 时间戳与序列号的拼接，使用位运算
    > 
    > `<<`
    
1. 测试ID生成器
    
    ```Java
     @Resource
     private RedisWorker redisWorker;
     
     //创建500线程的线程池
     private final ExecutorService executorService = Executors.newFixedThreadPool(500);
     
     @Test
     void testRedisWorker() throws InterruptedException {
     
         //CountDownLatch接收一个int型参数，表示要等待的工作线程的个数（latch：闩）
         CountDownLatch latch = new CountDownLatch(300);
     
         //定义线程要执行的任务
         Runnable task = () -> {
             for (int i = 0; i < 100; i++) {
                 long id = redisWorker.nextID("order");
                 System.out.println("id = " + id);
             }
             //使latch的值减1，如果减到了0，则会唤醒所有等待在这个latch上的线程
             latch.countDown();
         };
     
         //将任务提交300次
         for (int i = 0; i < 300; i++) {
             executorService.submit(task);
         }
         //使当前线程进入同步队列进行等待，直到latch的值被减到0或者当前线程被中断，当前线程就会被唤醒
         latch.await();
     }
    ```
    
    - `CountDownLatch`：==允许一个或者多个线程去等待其他线程完成操作==
        
        接收一个`int`型参数，表示要等待的工作线程的个数。当然也不一定是多线程，在单线程中可以用这个`int`型参数表示多个操作步骤
        
    
    - `CountDownLatch` 提供了一些方法：
        
        - `await()` 使当前线程进入同步队列进行等待，直到latch的值被减到0或者当前线程被中断，当前线程就会被唤醒
        
        - `await(long timeout, TimeUnit unit)` 带超时时间的await()
        
        - `countDown()` 使latch的值减1，如果减到了0，则会唤醒所有等待在这个latch上的线程
        
        - `getCount()` 获得latch的数值
