# 一、缓存和数据源的数据一致性问题
- **`Cache Aside`** **中**==**更新缓存**==**时，**==**为了保证数据一致性**==**有三个问题需要考虑：**
## 1. ️⃣ 删除缓存还是更新缓存❓
- 更新缓存：每次更新数据库都更新缓存，无效写操作较多
- 删除缓存：更新数据库时让缓存失效，查询时再更新缓存 ✅
## 2. ️⃣ 如何保证缓存与数据库的操作的同时成功或失败❓
- 单体系统，将缓存与数据库操作放在一个事务 `@Transactional`
- 分布式系统，利用 TCC 等分布式事务方案
## 3. ️⃣ 操作顺序：先操作缓存还是先操作数据库❓
### 3.1 A. 删除缓存，再写据库
1. 正常情况：

    ![[Attachment/1question/中间件/Redis/Redis 实战/3 商户查询缓存/3 2 缓存一致性/IMG-20260620224127743.png|800]]
2. 异常情况：在删完缓存还没来得及更新数据库时，第二个线程到达，出现缓存中的数据和数据库数据不一致的情况

    ![[IMG-20260620224132315.png|621]]
3. 这种不一致的情况可以使用==**延时双删**==来避免：**（这种方案延迟等待的时间无法准确估计，风险太大）**
    1. 即上图 线程1 在更新数据库之后，延迟一段时间再次删除缓存，这样会把 线程2 写入的缓存删掉
    2. 这个时间略大于 线程2 查询 DB + 写入缓存的时间，但是这个时间值只是一个估计值，这是延时双删最大的缺点
    3. 延迟的实现可以在代码中 `sleep` 或采用延迟队列

### 3.2 B. 先写数据库，再删除缓存（实时一致性方案）
1. **（耗时长的先执行，耗时短的后执行）**
2. 正常情况：

    ![[IMG-20260620224138535.png|611]]
3. 异常情况：出现缓存中的数据和数据库数据不一致的情况，但是这种情况出现的概率很低

    ![[IMG-20260620224138564.png|615]]
    > 💡 **这种异常情况出现的条件很严苛：**
    > - **对缓存的操作是很快速的，期间其他线程的并发几乎不需要考虑**
    > - 而上面的异常情况（数据不一致）就是考虑的极端情况：在线程1写入缓存微秒级时间内，出现了并发操作数据库的线程2，这种情况概率很低
    > - **所以是满足实时性条件下，上述方案（先写数据库，再删除缓存）是能尽量满足一致性的最优解**

### 3.3 C. 先写数据库，通过 Binlog，异步更新缓存（最终一致性方案）
1. 这种方案，主要是监听 MySQL 的 Binlog，然后通过异步的方式，将数据更新到 Redis
2. 这种方案有个前提：查询的请求中不能有写缓存的操作

    ![[IMG-20260620224142271.png|800]]
3. 这个方案，会保证 MySQL 和 Redis 的最终一致性，但是如果中途请求 B 需要查询数据，如果缓存无数据，就直接查 DB；如果缓存有数据，查询的数据也会存在不一致的情况
4. **所以这个方案，是实现最终一致性的终极解决方案，但是不能保证实时性**
    - 对于**==异地容灾、数据汇总==**==等，建议会用这种方式==，比如 binlog + kafka，数据的一致性也可以达到秒级
    - ==纯粹的高并发场景，不建议用这种方案==，比如抢购、秒杀等

# 二、代码示例：给查询商铺的缓存添加 超时剔除 和 主动更新 的策略
1. 修改ShopController中的业务逻辑，满足下面的需求：
    - 根据id查询店铺时，如果缓存未命中，则查询数据库，将数据库结果写入缓存，并设置超时时间（查询上面已经实现，这里只需要加一个超时时间即可，以满足超时剔除的兜底方案）
    - 根据id修改店铺时，先修改数据库，再删除缓存
2. ShopController

    ```Java
     /**
      * 更新商铺信息
      * @param shop 商铺数据
      * @return 无
      */
     @PutMapping
     public Result updateShop(@RequestBody Shop shop) {
         // 写入数据库
         shopService.update(shop);
         return Result.ok();
     }
    ```
3. ShopServiceImpl

    ```Java
     @Service
     public class ShopServiceImpl extends ServiceImpl<ShopMapper, Shop> implements IShopService {
         @Resource
         StringRedisTemplate stringRedisTemplate;
         @Override
         public Result queryById(Long id) {
             ...
             //6. 存在，写入redis缓存，新加超时时间
             stringRedisTemplate.opsForValue().set(RedisConstants.CACHE_SHOP_KEY + id,JSONUtil.toJsonStr(shop),30L, TimeUnit.MINUTES);
             ...
         }
         @Override
         @Transactional  // ⭐如果出现异常，则回滚
         public Result update(Shop shop) {
             Long id = shop.getId();
             if (id == null){
                 return Result.fail("店铺id不能为空！");
             }
             //1. 更新数据库
             updateById(shop);
             // ⭐2. 删除缓存
             stringRedisTemplate.delete(RedisConstants.CACHE_SHOP_KEY + id);
             return Result.ok();
         }
     }
    ```
    - 这里 `Long 30L == int 30` 恒为 `true`，L只是表示是Long类型（int 4字节，Long 8字节，实在不够用时才用Long）
    - 这里Controller 中是 Rest 请求风格：@PutMapping（详见 SpringBoot 2.3.1 请求映射Rest 请求风格）
    - `@Transactional` ： 在service层的类或方法上标注，表示开启事务，执行出现异常会回滚事务
