# 一、存在的问题
1. **缓存单值击穿问题也叫热点 Key 问题：**
    1. 就是一个==**被高并发访问**==并且==**缓存重建业务较复杂**==的 `key` 突然失效了
    2. 多值雪崩 🆚 单值击穿
        - 多值雪崩是很多个 `key` 都同时失效，造成每个 `key` 的请求都穿透了缓存
        - 单值击穿是一个并发访问量很高的 `key` 失效，造成访问这一个 `key` 的请求都穿透了缓存
2. ==⭐====**后果：**==
    - 在某个线程执行缓存重建的这一段时间内，由于缓存中还没有数据，所以无数的**请求访问会在瞬间全打到数据库**，给数据库带来巨大的冲击
    - 且每来一个请求都不会命中缓存，进而**所有请求都会开启缓存重建**

    ![[IMG-20260405035413816.png|800]]

# 二、解决方案
## 1. 方案 1：互斥锁

![[IMG-20260404031818551.png|800]]

- 线程争夺互斥锁
    - 拿到锁的才可以查数据库对缓存进行重建
    - 没拿到锁的，一直等待重试，不断尝试命中缓存

    ![[IMG-20260621000619195.png|737]]

## 2. 方案 2：逻辑过期 + 锁 + 线程池异步

![[IMG-20260620224127751.png|800]]

- 并**不对缓存的数据设置 TTL**，意味着缓存**永远不会物理失效（故也不会出现缓存未命中的情况）**，而是一种逻辑上的失效
    - 逻辑失效的实现：在缓存数据中多维护一个字段 `expire`，由 **当前时间 + 期望的TTL** 得到
- 如果缓存失效，由当前线程开启一个新线程，进行缓存重建，并返回过期数据

    ![[IMG-20260620224132957.png|800]]

## 3. 两种方案如何选择⭐
> 各有优缺点，根据不同场景需求，选择是合格的解决方案
- **==互斥锁：==**
    - 优点：
        1. 没有额外的内存消耗
        2. ==保证====**一致性 C**==
        3. 实现简单
    - 缺点：
        1. 当一个线程获得互斥锁时，其他所有的线程都需要等待，==性能受影响==
        2. 可能有死锁（两个业务互相拿着对方的锁）风险
- **==逻辑过期：==**
    - 优点：
        1. ==线程无需等待，性能较好==
        2. ==保证====**可用性 A**==
    - 缺点：
        1. 不保证一致性，如图中：
            1. 线程1得到互斥锁，并开启新线程2重建缓存时，线程1并不等重建完成，而是直接返回一个旧值
            2. 线程3发现得不到互斥锁，说明此时有人在做缓存重建，自己么必要管了，自己也不等也返回一个旧值
        2. 有额外内存消耗
        3. 实现复杂
