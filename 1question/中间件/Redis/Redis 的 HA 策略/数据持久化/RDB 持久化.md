# 1. 介绍
1. RDB 全称 Redis Database Backup file （**Redis 数据备份文件**）也被叫做Redis数据快照
    
    1. 简单来说就是把内存中的所有数据都记录到磁盘中
    
    1. 当Redis实例故障重启后，从磁盘读取快照文件，恢复数据
    
    1. 快照文件称为 `.rdb` 文件，默认是保存在当前运行目录
    
1. Redis 是**默认开启** RDB 持久化的，在停机时会执行一次RDB
# 2. 执行 RDB 的命令
1. 有两个命令可以执行 RDB
    
    - `**save**`：
        
        - ==由 Redis== ==**主进程来执行 RDB**====，会阻塞所有命令==
        
        - 持久化是写入磁盘，这个 IO 是很慢的，使用 save 可能会影响主进程执行其他命令
        
        - 适合主动停机前的持久化操作
            
            ![[IMG-20260405035438370.png|Untitled 439.png]]
            
        
    
    - **`bgsave`**：
        
        - ==开启====**子进程执行RDB**====，避免主进程受到影响==
        
        - 推荐使用，由额外的子进程在后台异步执行持久化，主进程不受影响
        
        - 适合在 Redis 运行过程中使用
        
    
1. ==**周期性执行 RDB 操作**==**（执行的是 `bgsave`），以**==**防止 Redis 突然宕机带来的数据丢失**==

# 3. **bgsave** 执行过程
1. 执行过程
    
    1. **`bgsave` 开始时会 fork 主进程得到子进程，子进程共享主进程的内存数据。完成 fork 后读取内存数据并写入 RDB 文件**
    
    1. **fork 采用的是 `copy-on-write` 技术**
        
        1. **读操作：**
            
            1. fork 操做会将主进程的**页表**（虚拟内存技术：页表是内存与磁盘空间的映射关系）复制一份给子进程
            
            1. 这样主进程和子进程都对页表进行读操作（**共享读**），同时磁盘中的数据会被标记为**只读**
            
        
        1. **写操作：**
            
            1. 如果在子进程持久化的过程中，有新的数据要写入
            
            1. 为了防止子进程读数据出现幻读，**主进程会将磁盘中，要修改的数据拷贝一份再写入（图中数据B），之后的读操作会在新的副本上进行**
            
        
    
    1. **持久化：**
        
        1. 子进程使用和主进程共享读的方式拿到数据后
        
        1. 将数据备份在磁盘的 `.rdb` 文件中，并**替换**掉旧的 `.rdb` 文件

# 4. **RDB 还是有缺陷的** 🚨
- **fork 子进程、压缩、写出RDB文件都比较耗时**
- **RDB执行间隔时间长，两次RDB之间写入数据有丢失的风险**
    
    1. 如果周期时间设置为 30 s
    
    1. 那么刚执行完持久化后，不到 3 s Redis 宕机了，那么这 3 s 内的数据还是会丢失，存在数据安全问题
    
    1. 所以 RDB 还需要配合其他持久化方案来做数据的持久化
