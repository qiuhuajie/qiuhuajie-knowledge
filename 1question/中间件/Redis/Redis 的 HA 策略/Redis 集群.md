# 一、Redis 集群介绍
1. Redis 集群是一种通过将**多个 Redis 节点**连接在一起以实现**高可用性、数据分片和负载均衡**的技术
2. 它允许 Redis 在不同节点上同时提供服务，提高整体性能和可靠性。根据搭建的方式和集群的特性
3. Redis集群主要有三种模式：
    - 主从复制模式（Master-Slave）
    - 哨兵模式（Sentinel）
    - Cluster 模式
4. **Redis 集群的作用和优势**
    1. **容灾恢复**：通过主从复制或哨兵模式，Redis 集群可以在主节点出现故障时，快速切换到从节点，**保证服务的高可用**
    2. **负载均衡**：Redis 集群可以将客户端请求分发到不同的节点上，有效地分摊节点的压力，**提高系统的并发能力**
    3. **数据分片**：Redis Cluster 模式集群可以将数据分散在不同的节点上，从而**突破单节点内存限制，利用插槽机制实现动态扩容**
    4. **易于扩展**：Redis 集群可以根据业务需求和系统负载，动态地添加或移除节点，实现水平扩展

## 1. 介绍
1. 单节点 Redis 的并发能力是有上限的，要进一步提高 Redis 的并发能力，就需要**搭建**==**主从集群**==**，实现**==**读写分离**==
2. 为什么不搭建负载均衡的多节点，而要使用主从结构，读写分离
    1. 因为 Redis 在大部分都是**读的操作居多**
    2. 使用多节点大多是为了缓解读请求的压力
3. 搭建主从集群：
    - 假设有A、B两个Redis实例，执行以下命令就可以让 B 作为 A 的 slave 节点
    - 在 B 节点执行命令：**`slaveof A的IP A的port`**
4. ==主节点故障时需要==**==手动切换==**
## 2. 数据同步原理
### 2.1 情况一：第一次同步请求——全量同步
- 主从节点同步的流程

    ![[IMG-20260620224129261.png|800]]

    - 从节点像主节点请求同步数据，从节点拿到主节点的版本信息，版本信息里有两个很重要的概念
        1. **`Replication ld`**：**（判断是不是第一次同步）**
            1. 简称replid，是数据集的标记，id一致则说明是同一数据集，每一个master都有唯一的replid，slave则会继承master节点的replid
            2. 将从节点发来的 replid 与 主节点的 replid 比较
                1. 与主节点不一致：表示是第一次同步请求，会进行**全量同步，也即要执行 RDB**
                2. 与主节点一致：表示不是第一次同步请求，会进行**增量同步，不需要执行 RDB，使用 repl_backlog 即可**
        2. **`offset`**：**（确定同步的进度）**
            1. 偏移量，随着记录在repl baklog中的数据增多而逐渐增大
            2. slave完成同步时也会记录当前同步的offset
            3. 如果slave的offset小于master的offset，说明slave数据落后于master，需要更新
            4. 因此slave做数据同步，必须向master声明自己的replication id 和offset，master才可以判断到底需要同步哪些数据
    - **主节点会使用** **`RDB`** **的方式将整个内存的快照发给从节点**
    - 而 RDB 的方式是不完整的，**在执行 RDB 的过程中可能有新的数据修改**，为了解决这个问题，**主节点会将这些命令记录在一个** **`repl_backlog`** **中**
        - 之后将这个 repl_backlog 也发送给从节点
        - 从节点执行接收到的命令，保持与主节点之间的同步
    - **之后的增量同步过程，都会基于 repl_backlog ，在从节点的现有数据上不断执行命令日志文件 repl_backlog**

# 三、Sentinel 模式
> 💡 slave 节点宕机恢复后可以找 master 节点同步数据，
> **那 master 节点宕机怎么办❓**
## 1. Redis 哨兵的作用
1. **Redis 提供了哨兵（Sentinel）机制来，**==**在主从复制的基础上实现自动故障转移**==**，提高高可用性，适用于高可用性要求较高的场景**
2. 哨兵机制的作用
    1. ==**集群监控**==：Sentinel 会不断检查您的 master 和 slave 是否按预期工作
    2. ==**自动故障恢复**==：如果 master 故障，Sentinel 会将一个 slave 提升为 master。当故障实例恢复后也以新的 master 为主
    3. ==**服务发现**==：Sentinel 充当 Redis 客户端的服务发现来源，当集群发生故障转移时，会将最新信息推送给 Redis 的客户端

    ![[IMG-20260620224155513.png|800]]

## 2. Sentinel 如何判断一个 Redis 实例是否健康❓
1. Sentinel 自身就是一个多节点的
2. ==Sentinel 基于====**心跳机制**====监测服务状态，每隔 1 秒向集群的每个实例发送 ping 命令==
    - **主观下线：**如果某 sentinel 节点发现某实例未在规定时间响应，则认为该实例主观下线
    - **客观下线：**若超过指定数量（quorum）的 sentinel 都认为该实例主观下线，则该实例客观下线
    - quorum 值最好超过 Sentinel 实例数量的一半

    ![[IMG-20260620224155531.png|800]]

    ![[IMG-20260620224202735.png|800]]

## 3. 选举新的 Master
1. 旦发现 master 故障，sentinel 需要在 salve 中选择一个作为新的 master
2. 各个 sentinel 使用 [[Raft 算法]] 投票，选出新的 master
3. 选择依据是这样的
    - 首先会**判断 slave 节点与 master 节点断开时间长短**，如果超过指定值 (down-after-milliseconds* 10) 则会排除该slave节点
    - 然后判断 slave 节点的slave-priority值，越小**优先级越**高，如果是0则永不参与选举
    - 如果slave-prority一样，则**判断 slave 节点的 offset 值**，越大说明数据越新，优先级越高
    - 最后是**判断 slave 节点的运行 id 大小**，越小优先级越高

## 4. 如何实现故障转移
1. 当选中了 slave1 为新的 master后，故障的转移的步骤如下
    1. sentinel **给备选的 slave1 节点发送`slaveof no one`命令，让该节点成为master**
    2. sentinel **给所有其它 slave 发送`slaveof 192.168.150.101 7002` 命令，让这些 slave 成为新 master 的从节点**，开始从新的master上同步数据
    3. 最后，sentinel **将故障节点标记为 slave**，当故障节点恢复后会自动成为新的 master 的 slave 节点

# 四、Cluster 模式
1. 主从和哨兵可以解决高可用、高并发读的问题。但是依然有两个问题没有解决
    - **海量数据存储问题**
    - **高并发写的问题**
2. Cluster 模式是 Redis的一种高级集群模式，**它通过==数据分片==和==分布式存储==实现了突破单节点的内存限制，实现更大规模的数据存储，适用于**==**大规模数据存储和高性能要求场景**==
3. ==**Cluster 模式的工作原理：**==
    1. 集群中有多个 master，master 之间通过 ping 监测彼此健康状态，每个 master 都可以有多个 slave 节点

    ![[IMG-20260620224208687.png|724]]

    2. 每个 master 保存不同数据

    3. ==将所有的键值对==**==数据分散在多个节点上==**==。每个节点负责一部分数据，称为槽位==

    4. Redis Cluster 将数据分为 16384 个槽位，每个节点负责管理一部分槽位

    5. ==当客户端向 Redis Cluster 发送请求时，Cluster 会根据==**==键的哈希值==**==将请求==**==路由==**==到相应的节点==
