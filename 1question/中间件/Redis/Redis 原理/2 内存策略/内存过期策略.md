>     ![[IMG-20260620224127709.png|708]]
> 2. `TTL` 到期以后，再次访问name返回的是`nil`，说明这个key已经不存在了，对应的内存也得到释放。从而起到内存回收的目的
# 一、如何知道一个 key 是否过期❓ 🙋‍♂️
1. **使用 key 的 TTL 来判断是否过期，那么 TTL 在哪里存储的 ？**
2. **使用有两个`Dict` 来存储==一个 key 的 TTL==**
    - **`dict dict`**：用来记录 `key-value` **（键 值）**
    - **`dict expires`**：用来记录 `key-TTL` **（键 过期时间），**只有设置的 TTL 的 key 才会同时存在 dict expires 中，即 **dict expires 是 dict dict 的真子集**

    ![[IMG-20260620224129627.png|800]]
3. 因此**只需要去查询 `dict expires` 即可得到想要查询 `key` 的 `TTL`**
4. 过期字典中 **`TTL`** 使用一个 `long long` 类型的整数存储
# 二、何时删除已过期的 key❓ 🙋‍♂️
> 💡 **==惰性删除 + 周期删除== 结合**
## 1. ️⃣ 惰性删除
1. 原理：
    1. 并不是在 `TTL` 到期后就立刻删除，而是**在访问一个 `key` 的时候，检查该 `key` 的存活时间，如果已经过期才执行删除**
    2. 这个和 缓存更新时使用删除缓存的策略 很像[[更新缓存]]
2. 源码：

    ![[IMG-20260620224135427.png|800]]
3. **SLOW**模式规则：
    1. 执行频率受`server.hz`影响，默认为`10`，即每秒执行`10`次，每个执行周期`100ms`
    2. 执行清理耗时不超过一次执行周期的`25%`，默认`slow`模式耗时不超过2`5ms`
    3. 逐个遍历`db`，逐个遍历`db`中的`bucket`，抽取`20`个`key`判断是否过期
    4. 如果没达到时间上限（`25ms`）并且过期`key`比例大于`10%`，再进行一次抽样，否则结束
4. **但是惰性删除存在一个问题**🚨**：如果一个 `key` 过期之后，就没有人再访问了，那么这个`key` 就永远不会被删除了**
5. 因此还需要同时使用 周期删除的策略
## 2. ️⃣ 周期删除
1. 原理
    - 是**通过一个定时任务，周期性的抽样部分过期的 `key` ，然后执行删除**
2. 执行周期有两种：
    1. `SLOW` 模式：
        1. Redis服务初始化函数`initServer()`中设置定时任务，按照`server.hz`的频率来执行过期`key`清理
        2. `SLOW`模式执行频率默认为`10`，每次不超过`25ms`
    2. `FAST` 模式：
        1. Redis的每个事件循环前会调用`beforeSleep()`函数，执行过期`key`清理
        2. `FAST`模式执行频率不固定，但两次间隔不低于`2ms`，每次耗时不超过`1ms`
3. 源码：

    ![[IMG-20260620224139722.png|800]]
4. **FAST**模式规则（过期key比例小于10%不执行 ）：
    1. 执行频率受`beforeSleep()`调用频率影响，但两次`FAST`模式间隔不低于`2ms`
    2. 执行清理耗时不超过`1ms`
    3. 逐个遍历`db`，逐个遍历`db`中的`bucket`，抽取`20`个`key`判断是否过期
    4. 如果没达到时间上限（`1ms`）并且过期`key`比例大于`10%`，再进行一次抽样，否则结束

# 三、仅有过期策略还不够
1. 但是，仅仅通过给 key 设置过期时间还是有问题的。因为**还是可能存在定期删除和惰性删除漏掉了很多过期 key 的情况**
2. 这样就导致大量过期 key 堆积在内存里，然后就 Out of memory 了
3. 还需要使用 ==**Redis 内存淘汰机制**==
