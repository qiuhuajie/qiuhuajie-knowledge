# 一、种策略
1. **内存淘汰**：**就是当Redis**==**内存使用达到设置的上限时**==**，**==**主动挑选**==**部分`key`**==**删除**==**，以释放更多内存的流程**
2. 淘汰的时机：
    1. Redis会在处理客户端命令的方法`processCommand()`中尝试做内存淘汰
    2. 源码：

    ![[Attachment/1question/中间件/Redis/Redis 原理/2 内存策略/IMG-20260620224127708.png|800]]

3. ⭐Redis支持**8种不同策略**来选择要删除的`key`：
    1. `noeviction`：
        - 不淘汰任何key，但是内存满时**不允许写入新数据**，默认就是这种策略
    2. `volatile-ttl`：
        - 对设置了TTL的key，比较key的剩余TTL值，TTL越小越先被淘汰**（快过期了，提前送你一程）**
    3. `allkeys-random`：（随机挑选的策略太随意了）
        - 对全体key ，**随机**进行淘汰。也就是直接从db->dict中随机挑选
    4. `volatile-random`：
        - 对设置了TTL的key ，**随机**进行淘汰。也就是从db->expires中随机挑选
    5. `allkeys-lru`：
        - 对全体key，基于**LRU算法**进行淘汰
    6. `volatile-lru`：
        - 对设置了TTL的key，基于**LRU算法**进行淘汰
    7. `allkeys-lfu`：
        - 对全体key，基于**LFU算法**进行淘汰
    8. `volatile-lfu`：
        - 对设置了TTL的key，**基于LFU**算法进行淘汰

# 二、LRU 和 LFU
1. `LRU` 和 `LFU`
    - **LRU**（**L**east **R**ecently **U**sed）
        - 最少最近使用
        - 用==当前时间减去最后一次访问时间==，这个值越大则淘汰优先级越高
    - **LFU**（**L**east **F**requently **U**sed）
        - 最少频率使用
        - 会==统计每个key的访问频率==，值越小淘汰优先级越高
            > 注意：
            >
            > LFU的访问频率叫做**逻辑访问次数**，并不是每次key被访问都计数，而是通过运算得到：
            >
            > 1. 逻辑访问次数的增长过程：计数器增长的速率越来越小
            >
            >     2. 生成`0~1`之间的随机数`R`
            >
            >     3. 计算 (`旧次数 * lfu_log_factor + 1`)，记录为`P`
            >
            >     4. 如果 `R < P` ，则计数器 + 1，且最大不超过255
            >
            >
            > 5. 逻辑访问次数的减小过程：在没有访问时，计数器也会衰减
            >
            >     6. 访问次数会随时间衰减
            >
            >     7. 距离上一次访问时间每隔 `lfu_decay_time` 分钟，计数器 -1
            >
2. `redisObject` 结构体中使用`lru` 字段来记录上面两种策略所使用的指标

# 三、内存淘汰的流程

![[IMG-20260620224135375.png|800]]

- Redis 中有数百万的`key`不能将所有`key`的 `LRU | LFU | TTL` 都做比较，因此在 Redis 中使用一个==`eviction_pool`==
- `eviction_pool` 开始是空的
- 从每个数据库`DB`中**挑选一些样本**放在 `eviction_pool` 中
    - 挑选的样本数由 `maxmemory_samples` 决定
    - 根据不同的策略计算出`key` 的 `idleTime`
    - 如果 `eviction_pool` 已经满了，如果此时要加入池中的`key`的 `idleTime` 要比池中最小 `idleTime` 大，才能挤进池中（比烂淘汰）
- `eviction_pool` 中的 `key` 是按照 `idleTime` 升序排序
- 删除时，倒序从`eviction_pool`中获取一个`key`删除
