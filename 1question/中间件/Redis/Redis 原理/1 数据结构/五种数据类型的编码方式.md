- 知识点速览：每种数据类型的使用的编码方式如下 🙋‍♂️
    1. **String**
        1. 两种编码方式都基于简单动态字符串（**`SDS`**）实现
        2. 数据量不同时，采用不同的编码方式：
            1. 数据量较大时，采用 **`RAW`** 编码：`object head` 中的指针指向一片独立的 `SDS` 空间
            2. 数据量较小时，采用 **`EMBSTR`** 编码：此时 `object head` 与 `SDS` 是一段连续空间
    2. **List**
        1. 在3.2版本之后，Redis 统一采用 **`QuickList`** 来实现 List
        2. 在3.2版本之前，当元素数量小于 512 并且元素大小小于 64 字节时采用 `ZipList` 编码，超过则采用 `LinkedList` 编码
    3. **Hash**
        1. Hash 结构默认采用 **`ZipList`** 编码，用以节省内存
        2. 当数据量较大时，Hash 结构会转为 **`HT`** 编码，也就是 `Dict`
    4. **Set**
        1. Set 默认采用 **`HT`** 编码（`Dict`）
        2. 其中Dict中的 key 用来存储元素，value 统一为 null
            > 类似的：Java中 HashSet 底层实现就是 HashMap，同样使用 HashMap 的 Key 来存储元素，value 统一为 null （[[Set 接口]] ）
    5. **ZSet**
        1. ZSet 使用 **`HT + SkipList`** 的编码方式来实现，`HT` 用来存储元素和 score 的键值对，`SkipList` 保证排序
        2. 但是，当元素数量不多时，`HT + SkipList`的优势不明显，而且更耗内存，所以采用 **`ZipList`** 结构来节省内存

# 二、编码方式
1. Redis 中有五种数据类型：**String、List、Set、ZSet、Hash**
2. Redis 中会根据存储的数据类型不同，选择不同的编码方式

    ![[IMG-20260620224129511.png|800]]

## 1. String
1. String是Redis中最常见的数据存储类型
2. 其底层编码方式由三种：
    1. **RAW**
        1. 基本编码方式
        2. 基于简单动态字符串（SDS）实现，存储上限为512mb
        3. SDS 是一片独立的内存空间，每次在创建新的String时，要先申请RedisObject的地址空间，再申请SDS的地址空间，需要两次内存 分配函数

    ![[IMG-20260620224135328.png|800]]

    2. **EMBSTR**
        1. 如果存储的 SDS 长度小于`44`字节，则会采用EMBSTR编码
        2. **此时`object head`与`SDS`是一段连续空间**，申请内存时只需要调用一次内存分配函数，效率更高

    ![[IMG-20260620224139639.png|800]]

    > 💡 为什么是44字节？
    > 1. SDS中：len、alloc 分别是两个 uint8_t类型的字段，各占1个字节，flag 字段是一个字符，占1个字节，结束符再占1个字节，所以整个SDS占$1 + 1 + 1 + 1 + 44 = 48$ 字节
    > 2. 而整个RedisObject的头占 16 字节
    > 3. 故一共是64 字节，恰好是 $2^n$
    > 4. redis 底层采用的内存分配方法是jemalloc，在分配内存时，采用 $2^n$ 来分配内存，64 恰好是一个分片大小，因此不会产生页内碎片
    > 5. 建议：==redis存储字符串时，大小最好不要超过44字节，否则会转为使用RAW编码方式，内存分配效率就不如EMBSTR编码==
    3. **INT**
        1. 如果存储的字符串是整数值，并且大小在 `LONG_MAX` 范围内，则会采用 INT 编码
        2. 直接将数据保存在RedisObject的 `ptr` 指针位置（刚好8字节），而要存的数值，不管是在c还是Java中，最大的数值数据类`Long` 占用的大小刚好是 `8` 字节
        3. 因此，不再需要额外的SDS了，直接把整数值的二进制存在指针里即可

    ![[IMG-20260620224145184.png|629]]

3. 示例：

    ```Bash
     127.0.0.1:6379> set name qhj
     OK
     127.0.0.1:6379> object encoding name
     "embstr"
     127.0.0.1:6379> set name aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa   \#44个字母：44个字节
     OK
     127.0.0.1:6379> object encoding name
     "embstr"
     127.0.0.1:6379> set name aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa  \#45个字母：45个字节
     OK
     127.0.0.1:6379> object encoding name
     "raw"
     127.0.0.1:6379> set age 123
     OK
     127.0.0.1:6379> object encoding age
     "int"
    ```
## 2. List
1. Redis的 List 类型类似一个双端链表，可以从首、尾操作列表中的元素

    ![[IMG-20260620224145273.png|476]]

2. List 底层编码方式：
    1. 在3.2版本之前，Redis采用 `ZipList + LinkedList` 来实现List，当元素数量小于512并且元素大小小于64字节时采用ZipList编码，超过则采用LinkedList编码
    2. 在3.2版本之后，**Redis统一采用 `QuickList` 来实现 List**

    ```C
         int j;
    ```

    ```C
     robj *createQuicklistObject(void) {
         // 申请内存并初始化QuickList
         quicklist *l = quicklistCreate();
         // 创建RedisObject，type为OBJ_LIST
         // ptr指向 QuickList
         robj *o = createObject(OBJ_LIST,l);
         // 设置编码为 QuickList
         o->encoding = OBJ_ENCODING_QUICKLIST;
         return o;
     }
    ```
## 3. Set
1. Set是Redis中的单列集合，满足下列特点：
    - 不保证有序性
    - 保证元素唯一（可以判断元素是否存在）
    - 求交集、并集、差集

    ![[IMG-20260620224202905.png|656]]

2. 可以看出，**Set对查询元素的效率要求非常高，基本上所有操作都涉及对元素是否存在的判断**
3. Set是Redis中的集合，不一定确保元素有序，可以满足元素唯一、查询效率要求极高
4. 因此底层使用**两种编码**实现 Set：
    1. `HT` 编码（`Dict`）默认
        1. 为了查询效率和唯一性，set采用`HT`编码（`Dict` ，不过`Dict`是双列集合,可以存键值对）
        2. ==其中Dict中的====`key`====用来存储元素，====`value`====统一为====`null`==
            > Java中 HashSet 底层实现就是 HashMap，同样使用 HashMap 的 Key 来存储元素，value统一为null
        3. 但是 Dict 中使用的大量指针，会占用和大的内存空间
    2. `IntSet`编码
        1. 当存储的所有数据都是整数，并且元素数量不超过`set-max-intset-entries`时，Set会采用`IntSet`编码，以节省内存
        2. `set-max-intset-entries`的默认值是`512`：

## 4. ZSet
1. ZSet也就是SortedSet，其中每一个元素都需要指定一个score值和member值：

    ![[IMG-20260620224220077.png|598]]

    1. 可以根据score值排序后
    2. member必须唯一
    3. 可以根据member查询score
2. 因此，zset底层数据结构必须满足**键值存储、键必须唯一、可排序**这几个需求
    - **SkipList**
        - 可以排序，并且可以同时存储score和ele值（member）
        - 但是要根据member查询score，需要遍历整个链表
    - **HT**（Dict）
        - 可以键值存储，并且可以根据 key 找 value
        - 但是无法满足排序和 member 必须唯一的要求
3. **==故将两者结合起来，使用== `HT + SkipList` ==的编码方式来实现 ZSet==**

    ![[IMG-20260620224221111.png|300]]

    ![[IMG-20260620224221764.png|584]]

4. ZSet 内存分析：

    ![[IMG-20260620224222427.png|800]]

5. 但是 `HT + SkipList` 的编码方式，虽然各种需求的性能效率很高，但是**需要存储大量的指针**，故内存消耗很大
    1. ==当元素数量不多时，====`HT + SkipList`====的优势不明显，而且更耗内存==
    2. 因此zset ==**默认**====采用====`ZipList`====结构来节省内存==，不过需要同时满足两个条件：
        1. 元素数量小于`zset_max_ziplist_entries`，默认值`128`
        2. 每个元素都小于`zset_max_ziplist_value`字节，默认值`64`
    3. 但`ZipList`结构并不自带排序的功能，也没有键值对的概念，因此需要zset通过编码实现：
        1. ZipList是连续内存，因此`score`和`element`是紧挨在一起的`两个entry`， element在前，score在后
        2. score越小越接近队首，score越大；越接近队尾，按照score值升序排列

    ![[IMG-20260620224222486.png|800]]

6. 源码：
    1. 插入元素前，判断ZSet是否存在，根据插入的第一个元素的大小，来创建相应编码的 `RedisObject`

    ![[IMG-20260620224223111.png|800]]

    2. 真正的插入元素的操作，如果当前是 `ZipList` ，且元素不存在，需要新增，则要判断 `ZipList` 长度有没有超`zset_max_ziplist_entries`、元素的大小有没有超过`zset_max_ziplist_value`，如果超过了则需要做编码转换：

        `ZipList -> Hash + SkipList`

## 5. Hash
1. Hash结构与Redis中的Zset非常类似：

    ![[IMG-20260620224223348.png|549]]

    1. 都是键值存储
    2. 都需求根据键获取值
    3. 键必须唯一
2. 区别如下：
    - zset的键是member，值是score；hash的键和值都是任意值
    - zset要根据score排序；hash则无需排序
3. 因此，==Hash底层采用的编码与Zset也基本一致，只需要把排序有关的SkipList去掉即可：==
    1. `ZipList`（默认）
        1. Hash结构默认采用ZipList编码，用以节省内存。 ZipList中相邻的两个entry 分别保存field和value
        2. 内存分析：

    ![[IMG-20260620224223677.png|800]]

    2. `HT`
        1. 当数据量较大时，Hash结构会转为HT编码，也就是Dict，触发条件有两个：
            - ZipList中的元素数量超过了`hash-max-ziplist-entries`（默认`512`）
            - ZipList中的任意entry大小超过了`hash-max-ziplist-value`（默认`64`字节）
