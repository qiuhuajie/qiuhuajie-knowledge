2. ==**SDS 的优点**==
    1. **动态扩容 + 预分配**
        - C 语言的字符串是静态分配的，容易造成内存碎片，且修改字符串时，一旦没有分配足够长度的内存空间，就会造成缓冲区溢出
        - ==Redis 的动态字符串可以====**动态扩容**====：根据 len 属性检查空间大小是否满足要求，如果不满足，则先扩展至所需大小再进行修改操作==
        - ==并且在扩容后，还会根据特定的算法多分配一些内存空间，称为内存的====**预分配**==，因为内存分配需要在内核态进行，当用户需要分配内存时，需要从用户态切换到内核态，这个过程需要消耗大量资源
    2. **字符串长度的获取**
        - C 语言中的字符串的长度通常是经过遍历计数来实现的，时间复杂度为 O(n)
        - SDS 在创建时就==缓存了字符串的长度信息== `len` ==，可以直接获取==，效率更高
    3. **二进制安全，估可以保存图片、视频等二进制文件**
        - C 语言中的字符串以空字符 `\0` 作为字符串结束的标识，这存在一些问题，像一些二进制文件（比如图片、视频、音频）就可能包括空字符，C 字符串无法正确保存
        - SDS ==使用== `len` ==属性判断字符串是否结束==，不存在这个问题
3. ==**SDS 的动态扩容 + 预分配**==
    1. 假如我们要给一个内容为`“hi”` 的SDS追加一段字符串 `“,Amy”`，这里首先会**申请新内存空间：**
        1. **动态扩容：**
            - 先算出扩展后的字符串长度：**`扩展后字符串长度`**
        2. **内存预分配：**
            - 如果新字符串小于 `1M`，则新空间为 `扩展后字符串长度 * 2 + 1`
            - 如果新字符串大于 `1M`，则新空间为 `扩展后字符串长度 + 1M + 1`
    > 💡 为什么要预分配，即多分配一些内存空间？
    > - 内存分配需要在内核态进行，当用户需要分配内存时，需要从用户态切换到内核态，这个过程需要消耗大量资源
    2. 再将新加的字符和原来的字符存到新分配的内存空间中：`len = 2 + 4`、`alloc =（6 * 2）+ 1 = 13` （包括结束符）

# 二、Dict（HT * 2）
1. **数据结构：**由两个 HashTable ，以及一些辅助字段构成

    ![[IMG-20260620224155018.png|800]]

    ![[IMG-20260620224144123.png|800]]

2. **Dict 插入：**类似 java 的 HashTable，==底层是数组 + 链表来解决哈希冲突，链表采用头插法==
3. **Dict 扩容：**在==每次新增键值对时==，都会==检查负载因子==（`LoadFactor = used/size`） ，满足以下两种情况时会触发哈希表扩容：
    - 情况一：哈希表的 `LoadFactor >= 1`，并且服务器没有执行 BGSAVE 或者 BGREWRITEAOF 等后台进程
    - 情况二：哈希表的 `LoadFactor > 5`
4. **Dict 缩容：**==每次删除元素时==，也会==检查负载因子==，当`LoadFactor < 0.1` 时，会做哈希表收缩
5. ==**Dict 的 rehash**==
    1. 解决的问题：
        1. ==由于扩容和收缩导致 HashTable 大小的变化==，由于原来旧元素的位置是使用旧 size 计算出来的位置，当再次查询时，使用的是新 size，会导致找不到元素
        2. 因此，==必须对哈希表中的每一个 key 重新计算索引，插入新的哈希表，这个过程称为 rehash==
    2. rehash 的过程：
        1. 计算新 hash 表的 `realeSize`，按照新的 `realeSize` 申请内存空间，创建 `dictht` ，并赋值给 `dict.ht[1]`
        2. 设置 `dict.rehashidx = 0`，表示开始 rehash
        3. 将 `dict.ht[0]` 中的每一个 `dictEntry` 都 rehash 到 `dict.ht[1]`
        4. 将 `dict.ht[1]` 交换 `dict.ht[0]` ，再将`dict.ht[1]`初始化为空哈希表，释放内存
        5. 设置 `dict.rehashidx = -1`，表示结束rehash
6. ==**渐进 rehash**== 🙋‍♂️
    > 💡 **可以理解为每次只 rehash 迁移访问到的那个拉链链表，而之前是将 hashtable 上的所有链表都做迁移**
    1. 解决的问题：
        1. 如果Dict中包含数百万的 entry，要在一次 rehash 完成，极有可能导致主线程阻塞
        2. 因此 Dict 的 ⭐ ==渐进式 rehash 并不是一次性完成的，是分多次、渐进式的完成，==**==每次只 rehash 一个== ==`dictEntry`==**
    2. 渐进式 rehash 的改进点：
        1. 普通 rehash：~~将`dict.ht[0]`中的每一个 `dictEntry` 都rehash到`dict.ht[1]`~~
        2. 渐进式 rehash：
            1. 每次执行新增、查询、修改、删除操作时，都检查一下 `dict.rehashidx` 是否大于 -1，如果是，则表示需要 rehash
            2. 且==只将== ==`dict.ht[0].table[rehashidx]`== ==的== ==`entry`== ==链表rehash到== ==`dict.ht[1]`==，即每次只 rehash 一个`entry`
            3. 并且将 `rehashidx++` （代表当前渐进 rehash 的进度）
            4. 直至多次执行新增、查询、修改、删除操作后，将`dict.ht[0]`的所有数据都rehash到`dict.ht[1]`
    3. 渐进式 rehash 过程中
        1. ==新增操作，直接写入== ==`dict.ht[1]`== （确保`dict.ht[0]`的数据只减不增，随着rehash最终为空）
        2. ==查询、修改和删除 ，会在====`dict.ht[0]`====和====`dict.ht[1]`====中依次查找并执行==（`dict.ht[0]`和`dict.ht[1]`中元素的并集才是当前所有元素）

# 三、ZipList
> 💡 在 Redis 7.0 中，压缩列表数据结构已经废弃了，交由 listpack 数据结构来实现了
1. ZipList 的结构：

    ![[IMG-20260620224156689.png|800]]

2. 可以看做一种**==连续内存空间==**==的"==**==双向链表==**=="==
3. **为什么可以做到空间压缩**❓
    - ==ZipList 的 entry== **==节点之间不是通过指针连接==**==，而是==**==通过记录上一节点和本节点长度来寻址==**==，因此内存占用较低==
        - 正向：根据 `encoding` 记录 `content` 的数据类型（字符串还是整数）以及长度，即可得到下一个 entry 的首地址
        - 反向：根据 `previous_entry_length` 记录上一个entry的长度，即可得到上一个entry的首地址
        - 这样 ZipList 就可以实现双端链表那样的双向遍历
4. 缺点：
    1. 如果列表数据过多，导致**链表过长，可能影响查询性能**
    2. 增或删较大数据时有可能发生**连续更新**问题

1. QuickList 的结构：`LinkList + ZipList`

    ![[IMG-20260620224127753.png|800]]

    2. 创建==多个 ZipList 来分片存储数据，控制了单个 ZipList 大小==，解决连续内存空间申请效率问题（ZipList 需要再连续空间上分配）

# 五、SkipList 🙋‍♂️
1. **介绍**
    1. 对于一个有序链表，在进行插入、删除、查找时，需要从头逐个遍历整个链表。其时间复杂度为O(N)
    2. 如何提高查找效率❓
        1. ==把链表中的一些节点提取出来，作为索引，类似于二叉搜索树==，查询的时间复杂度可以降到 $O(logn)$

    ![[IMG-20260620224203514.png|800]]

        2. Skip List 可以看做是一个二叉树的变种，它在性能上与红黑树、AVL树很相近；但是实现相比前两者要简单很多
    3. ==广泛应用于 Redis（ zset 实现）、Kafka、LevelDB 中==
2. **跳表的特性**
    1. 每个节点都可以包含==多级指针==，不同层指针到下一个节点的跨度不同，层级越高，跨度越大
    2. 每一层都是一个==有序==的列表
    3. 一般==上层索引个数是下层索引个数的 1/2==
    4. ==最底层包含所有的元素==
    5. 每个节点元素由节点 key、节点 value 和指向当前节点所在level的指针数组组成
3. **跳表的查询**
    1. 假设初始跳跃列表中已经存在这些元素，他们分布的结构如下所示：

        1. 从Skip List跳跃列表**最顶层 level3 开始**，往后查询到10 < 88 && 后续节点值为 null && 存在下层 level2
        2. level2 10 往后遍历，27 < 88 && 后续节点值为null && 存在下层 level1

    3. ==**查询的时间复杂度：**==
        1. 一般上层索引个数，是下层索引个数的 1/2，那么第 k 层元素个数为 $\frac{n}{2^k}$
        2. 当最高层只有两个节点时，就不再增加层数了，也就是最高层 h 的元素个数为 2，即 ==$\frac{n}{2^h} = 2$== ，得到==**跳表的层高**== ==$h = (log_2n) - 1$==
        3. 由于查找时，是从最高层逐层向下查找，假设每层遍历的元素个数为 x
        4. 那么查询的时间复杂度为：$O(xlog_2n) \approx O(logn)$

    2. 接着==构造一个新的节点== Node(90)，同时，为==插入的节点计算一个随机 level==
        1. 这个新节点的层数，从 1 开始递增，至于递增到几，由一个随机数决定，且不能超过层数的最大限制
