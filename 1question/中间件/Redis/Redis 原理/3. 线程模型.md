# 一、Redis 是单/多线程❓
## 1. Redis 到底是单线程还是多线程❓
1. ==对于 Redis 的==**==核心网络模型==（命令处理）**，在Redis 确实都是单线程：==是利用== `epoll`（Linux系统）这样的 ==IO 多路复用技术==在事件循环中不断处理客户端情况
2. 但是，**Redis 程序**并不是单线程的，Redis 在启动的时候，是会启动==后台线程==的：
    - Redis 在 2.6 版本，会启动 2 个后台线程，分别处理关闭文件、AOF 刷盘这两个任务
    - Redis 在 4.0 版本之后，新增了一个新的后台线程，用来异步释放 Redis 内存，也就是 lazyfree 线程

## 2. 为什么 Redis 要选择单线程❓
1. 如果使用多线程
    1. **性能损耗**：增加线程上下文切换的开销
    2. **线程安全问题**：单线程意味着 ==Redis 一定是线程安全的==，不存在并行执行的情况（例如，多个 Lua 脚本同时执行）
    3. **没用**：Redis 是纯内存操作，执行速度非常快，它的==性能瓶颈是网络延迟，而不是 CPU 执行速度==，因此多线程并不会带来巨大的性能提升
2. 但单线程也意味着 ==Redis 在高并发环境下有性能瓶颈：可以使用== ==[[Redis 的 HA 策略]]====来解决==
# 二、Redis 为什么这么快❓
- Redis 内部做了非常多的性能优化，比较重要的主要有下面 3 点：
    - Redis ==基于内存==，内存的访问速度是磁盘的上千倍
    - Redis ==内置了多种优化过后的数据结构==，如动态字符串、快表、跳表等，性能非常高[[1. 数据结构]]
    - Redis 的线程模型：I/O 多路复用 + 单线程
        1. ==单线程读写==：单线程不需要进行线程上下文切换，避免了额外开销
        2. ==IO 多路复用==：是指一个线程处理多个 IO 流（即 `select`/`epoll` 机制）[[IO 模型]]
            1. 在 Redis 只运行单线程的情况下，该机制允许内核中，同时存在多个监听 Socket 和已连接 Socket
            2. 并一直监听这些 Socket 上的连接请求或数据请求，一旦有请求到达，就会交给 Redis 线程处理
