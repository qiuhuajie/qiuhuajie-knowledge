> 💡 **防止消费者丢失消息**
# 一、存在的问题
1. RabbitMQ是**阅后即焚**机制，RabbitMQ确认消息被消费者消费后会立刻删除
2. 因此，设想这样的场景：
    1. RabbitMQ投递消息给消费者
    2. 消费者获取消息后，返回 ACK 给RabbitMQ
    3. RabbitMQ删除消息
    4. 此时，==消费者====**宕机**====或处理消息的====**代码出现异常**==，消息尚未处理
    5. 这样消息就丢失了：**队列中的被 RabbitMQ 删了，而消费者拿到的又由于异常也丢失了** ，因此消费者**==返回ACK的时机==**非常重要

# 二、解决方式
1. 而 **SpringAMQP** 则允许配置三种**确认模式**：
    1. **`auto`**：==**自动 ack，由 spring 监测 listener 代码是否出现异常，没有异常则返回 ack；抛出异常则返回 nack，此时消息不会从队列中删除**==
    > 💡 此模式下可能出现重复消费问题，详见
    > [[MQ 面试问题]]
    2. `none`：关闭 ack，MQ假定消费者获取消息后会成功处理，因此消息投递后立即被删除
    3. `manual`：手动 ack，需要在业务代码结束后，调用 api 发送ack
2. RabbitMQ **ACK 确认机制**：
    1. 消费者**处理消息后**可以向 MQ 发送 **`ack` 回执**（表明自己已经处理消息）
    2. **MQ 收到 `ack` 回执后才会删除该消息**

## 1. None 模式（❌）

    ```YAML

         host: 192.168.10.120 # rabbitMQ的ip地址

         password: 123321
    ```

## 2. Auto 模式（🤔）
1. 上面的示例中显示，none 模式是存在问题的！
2. 配置 auto 模式

    ```YAML
     spring:
       rabbitmq:
         host: 192.168.10.120 # rabbitMQ的ip地址
         port: 5672 # 端口
         username: itcast
         password: 123321
         virtual-host: /
         listener:
           simple:
             prefetch: 1
             acknowledge-mode: auto  \#⭐
    ```
3. 手动向队列中添加一条消息

    ![[IMG-20260404035838764.png|800]]
4. 在消费者模块处理消息的代码中 写一个 异常，并打断点，debug 启动

    ![[IMG-20260404035838947.png|800]]
5. 启动后，监听队列的代码会立刻收到投递的消息 `msg`

    ![[IMG-20260404035839192.png|800]]
6. 此时消息者正在处理消息，再返回浏览器查看，此时队列中的消息：
    1. **==消息被消费者取到后，RabbitMQ 并不会将消息删除==**
    2. **而是将消息的状态由 `Ready` 变为 `Unacked`**

    ![[IMG-20260620224127211.png|800]]
7. **放行断点后，**==**由于处理逻辑中有异常，会触发消息重新投递**==**，消息被再次投递到队列，并被消费者监听到，因此，程序再次运行，并卡在了断点处**

    ![[IMG-20260620224128228.png|800]]
8. 就**这样不断循环**，**==只要消息处理的代码中有异常，就会不会给 mq 返回==** **`==ACK==`****==，因此一直重新投递，并再次监听到，但继续执行又会有异常，再次触发重新投递==**
9. 如果不打断点，直接执行，控制台会不断报异常
    - 当消费者出现异常后，消息会不断 `requeue`（重新入队）到队列
    - 再重新发送给消费者，然后再次异常，再次requeue，无限循环，==**导致 mq 的消息处理飙升，带来不必要的压力**==

    ![[IMG-20260620224134619.png|800]]
## 3. 开启消费失败重试机制 Retry 解决 Auto 模式下的无限重试问题（✅）
### 3.1 本地重试
1. 上面的消费者确认机制，都存在问题：
    1. `none`：RabbitMQ会在消息被消费者拿到后，删除队列中的消息，但如果消费者代码有异常，会导致消息丢失，消息就永远没了
    2. `auto`：虽然RabbitMQ不会删除消息，但如果消费者代码有异常，会导致不断重新投递，进而导致不断循环投递，这样处理并不是很友好
2. 为了找到一个完美的解决方案：
    - ⭐**可以利用 Spring 的 `retry` 机制**
    - **在消费者出现异常时利用**==**本地重试，并设置一个重试次数上限**==**，而不是无限制的 `requeue` 到 mq 队列**
    - 在开启重试模式后，==**重试次数耗尽后，如果消息依然失败**==**，则需要有 `MessageRecoverer` 接口来处理**，它包含三种不同的实现

    ![[IMG-20260620224139348.png|700]]
        1. `RejectAndDontRequeueRecoverer`：重试耗尽后，直接 reject，丢弃消息。默认就是这种方式
        2. `ImmediateRequeueMessageRecoverer`：重试耗尽后，返回 nack，==消息重新入队（类似 auto 确认模式下的 requeue）==
        3. **`RepublishMessageRecoverer`：重试耗尽后，==将失败消息投递到指定的交换机（error.queue）==**✅

### 3.2 实现思路
- 示例：
    1. 开启 `retry`机制
    2. 消费者监听的代码中还是故意写上异常
    3. 手动向 `simple.queue` 队列中发送一条消息
    4. 消费者监听到消息后，开始执行信息处理，由于有异常，会触发重试
    5. 当重试次数耗尽后，本例选择的是 `RepublishMessageRecoverer` 作为失败策略
    6. 该条消息会被重新发到 `error.queue` 队列中

### 3.3 配置文件中开启 Retry 机制
```YAML
spring:
  rabbitmq:
    host: 192.168.10.120 # rabbitMQ的ip地址
    port: 5672 # 端口
    username: itcast
    password: 123321
    virtual-host: /
    listener:
      simple:
        prefetch: 1
        retry:
          enabled: true           # 开启消费者失败重试
          initial-interval: 1000  # 初始的失败等待时长为 1 秒
          multiplier: 1           # 下次失败的等待时长的倍数：下次等待时长 = multiplier * last-interval
          max-attempts: 3         # 最大重试次数
          stateless: true         # true：无状态，false：有状态（如果业务中包含事务，则需要设置为 false）
```
### 3.4 声明 交换机、队列、失败策略
- 声明要将失败消息投递到的交换机和队列，以及重试耗尽后的失败策略
    ```Java
    @Configuration
    public class ErrorMessageConfig {
        //失败后重发到哪个交换机
        @Bean
        public DirectExchange errorMessageExchange(){
            return new DirectExchange("error.direct");
        }
        //失败后重发到哪个队列
        @Bean
        public Queue errorQueue(){
            return new Queue("error.queue");
        }
        //绑定对象
        @Bean
        public Binding errorMessageBinding(){
            return BindingBuilder.bind(errorQueue()).to(errorMessageExchange()).with("error");
        }
        //⭐失败策略（失败后，将失败消息发送到指定队列）
        @Bean
        public MessageRecoverer RepublishMessageRecoverer(RabbitTemplate rabbitTemplate){
            return new RepublishMessageRecoverer(rabbitTemplate,"error.direct","error");
        }
    }
    ```
