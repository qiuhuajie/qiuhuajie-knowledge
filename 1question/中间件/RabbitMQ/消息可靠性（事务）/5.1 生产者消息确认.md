> 💡 **防止生产者丢失消息**
# 一、生产者消息确认机制
1. 消息发送到 MQ 以后，会返回一个结果给发送者，表示消息是否处理成功
2. 返回的确认结果可能有两种（两段确认机制）

    ![[IMG-20260405035438331.png|800]]
    1. `publisher-confirm`，发送者确认
        - 消息**成功**投递到交换机，返回 `ack`
        - 消息投递到交换机**失败**，返回 `nack`
        - 回执返回过程中出现**异常**，则发送者拿不到任何信息
    2. `publisher-return`，发送者回执
        - 消息**成功**从交换机路由到队列了，不返回任何信息
        - 消息从交换机路由到队列**失败**了。返回 `ACK`，及路由失败原因
3. 两段确认机制都需要在配置文件中开启，且都基于 callback 机制
4. **消息发送过程中的确认机制实现方式：使用回调函数实现**
    1. **消息由发送者到 `exchange` 的过程中的消息确认：**
        1. `new` 一个 **`CorrelationData`** 对象，并设置 消息ID：`CorrelationData correlationData = new CorrelationData(UUID.randomUUID().toString())`

    ![[IMG-20260404031930712.png|630]]
        2. 为 `CorrelationData` 添加两个回调函数：
            1. **`SuccessCallback`**（收到回执）：取出返回的 `confirm` 字段
                1. 如果为 `ack` ，表示消息成功发送到交换机
                2. 如果为 `nack` 表示消息没有到达交换机
            2. **`FailureCallback`**（未收到回执）：消息发送过程中出现异常，连回执都没有收到，如果有需要，在这里可以写重新发送消息的代码
    2. **消息成功发送到 `exchange`，再路由到 `queue` 的过程中的消息确认：**
        1. 在系统启动时，为 `RabbitTemplate` 配置一个 **`ReturnCallback`** 回调函数
        2. 只要消息未到达 `queue` 就会调用定义好的 `ReturnCallback`，并返回响应码、失败原因、交换机、路由键等信息
        3. 成功到达 `queue` 不会调用 `ReturnCallback`

# 二、代码实现
- 示例：SpringAMQP实现生产者确认
## 1. 配置文件
在 publisher 的配置文件`application.yml`中==**开启**====消息确认机制==
```YAML
 spring:
   rabbitmq:
     host: 192.168.10.120 # rabbitMQ的ip地址
     port: 5672 # 端口
     username: itcast
     password: 123321
     virtual-host: /
     publisher-confirm-type: correlated  # 开启异步回调
     publisher-returns: true   # 开启publish-return功能
     template:
       mandatory: true   # 定义消息路由失败时的策略
```
**配置说明：**
- `publish-confirm-type`：==开启 publisher-confirm==，这里支持两种类型：
    - `simple`：同步等待confirm结果，直到超时
    - `correlated`：异步回调，定义 ConfirmCallback，MQ返回结果时会回调这个ConfirmCallback
- `publish-returns`：==开启 publish-return 功能==，同样是基于callback机制，不过是定义ReturnCallback
- `template.mandatory`：定义消息路由失败时的策略
    - `true`：则调用 ReturnCallback
    - `false`：则直接丢弃消息

## 2. 定义 ReturnCallback
**在 publisher 中写的 配置类：**
```Java
 @Slf4j
 @Configuration
 public class CommonConfig implements ApplicationContextAware {
     @Override
     public void setApplicationContext(ApplicationContext applicationContext) throws BeansException {
         //从spring容器中，获取 RabbitTemplate 对象
         RabbitTemplate rabbitTemplate = applicationContext.getBean(RabbitTemplate.class);
         //给 RabbitTemplate 对象设置 ReturnCallback
         rabbitTemplate.setReturnCallback(new RabbitTemplate.ReturnCallback() {
             @Override
             public void returnedMessage(Message message, int replyCode, String replyText, String exchange, String routingKey) {
                 //记录日志
                 log.error("消息发送到队列失败，响应码：{}，失败原因：{}，交换机：{}，路由键：{}，消息：{}",
                         replyCode,replyText,exchange,routingKey,message.toString());
                 //（失败兜底）如果有需要，在这里可以写重新发送消息的代码
             }
         });
     }
 }
```
1. **Tip1：每个`RabbitTemplate`只能配置一个`ReturnCallback`，因此需要在项目启动过程中配置**
2. **Tip2：**让配置类**实现 `ApplicationContextAware` 接口**
    1. 实现 `ApplicationContextAware` 接口后，这个类就可以**方便获得 `ApplicationContext` 中 的所有 `bean`**
    2. 换句话说，就是这个类可以直接获取spring 配置文件中，所有引用到的 bean 对象
    3. 项目启动时，`BeanFactory` 准备好后，就会调用重写的 `setApplicationContext()` 方法，可以在此方法中`applicationContext.getBean();`
3. **Tip3：**
    1. `rabbitTemplate.setReturnCallback()` 需要传入一个 `RabbitTemplate.ReturnCallback` 对象，而 `RabbitTemplate.ReturnCallback` 是一个接口

    ![[IMG-20260405035452338.png|800]]
    2. 因此**如何在不实现这个接口的情况下，得到一个接口的实现类？**
        1. `new` 一个 `RabbitTemplate.ReturnCallback` 的匿名实现类（详见 匿名实现类）
            - 红色部分是接口的名称
            - 蓝色部分是接口的方法，将接口的方法重写后，相当于得到了一个接口的实现类（匿名内部类）
            - 因此 `new` 得到的就是这个接口的一个实现类的实例

    ![[IMG-20260405035507921.png|800]]
        2. 一个匿名内部类，也可以写成下面的`lamda`表达式的形式（`alt + enter`）

    ```Java
     rabbitTemplate.setReturnCallback((message, replyCode, replyText, exchange, routingKey) -> {
      //记录日志
      log.error("消息发送到队列失败，响应码：{}，失败原因：{}，交换机：{}，路由键：{}，消息：{}",
              replyCode,replyText,exchange,routingKey,message.toString());
      //如果有需要，在这里可以写重新发送消息的代码
     });
    ```
## 3. 定义 ConfirmCallback
1. `ConfirmCallback`可以在发送消息时指定，因为每个业务处理`confirm`成功或失败的逻辑不一定相同
2. 在`publisher`服务的`cn.itcast.mq.spring.SpringAmqpTest`类中，定义一个单元测试方法
    ```Java
     @Slf4j
     @RunWith(SpringRunner.class)
     @SpringBootTest
     public class SpringAmqpTest {
         @Autowired
         private RabbitTemplate rabbitTemplate;
         @Test
         public void testSendMessage2SimpleQueue() throws InterruptedException {
             String routingKey = "a.simple.test";
             // 1. 准备消息
             String message = "hello, spring amqp!";
             // 2. 准备 CorrelationData
             // 2.1 new 一个 CorrelationData 对象，并设置 消息ID
             CorrelationData correlationData = new CorrelationData(UUID.randomUUID().toString());
             // 2.2 为 CorrelationData 添加 ConfirmCallback
             // ⭐ addCallback() 需要两个参数：
                     //successCallback（收到回执）：收到的回执又分为 ACK 和 NACK 两种情况
                     //failureCallback（未收到回执）
             correlationData.getFuture().addCallback(new SuccessCallback<CorrelationData.Confirm>() {
                 @Override
                 public void onSuccess(CorrelationData.Confirm confirm) {
                     //判断返回的是否为 ACK
                     if (confirm.isAck()){
                         //1️⃣返回ACK：消息发送至交换机成功
                         log.debug("消息投递到交换机成功！消息ID：{}", correlationData.getId());
                     }else {
                         //2️⃣返回NACK：消息发送至交换机失败
                         log.error("消息投递到交换机失败！消息ID：{}", correlationData.getId());
                         //（失败兜底）如果有需要，在这里可以写重新发送消息的代码
                     }
                 }
             }, new FailureCallback() {
                 @Override
                 public void onFailure(Throwable throwable) {
                     //3️⃣消息发送异常，没有收到回执
                     log.error("消息发送异常！", throwable);
                     //（失败兜底）如果有需要，在这里可以写重新发送消息的代码
                 }
             });
             // 3. 发送消息，⭐发消息时需要将 correlationData 作为参数传入
             rabbitTemplate.convertAndSend("amq.topic", routingKey, message, correlationData);
         }
     }
    ```
