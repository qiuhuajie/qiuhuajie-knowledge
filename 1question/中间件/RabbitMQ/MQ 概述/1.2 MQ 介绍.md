# 一、介绍
1. MQ （MessageQueue），中文是消息队列，字面来看就是存放消息的队列，也**==就是事件驱动架构中的 Broker==**
2. 🙋‍♂️ **技术选型：**

    ![[IMG-20260404031925615.png|800]]

    - Apache Kafka
        - 特点：==高吞吐量（每秒 10w+）、低延迟、高可靠性==、支持分布式部署、支持多种语言、支持大规模数据存储和处理
        - 选型方案：==适合==**==大规模数据==**==流处理场景，==**==追求高吞吐量==**，如日志收集、实时数据分析、流式处理等。==对于对性能和可靠性有较高要求的场景==，可以考虑选择 Kafka
    - RabbitMQ
        - 特点：==简单易用==、支持多种消息协议、支持多种语言、支持持久化、支持消息路由和过滤
        - 选型方案：==适合中小规模的分布式应用场景==，如任务分发、异步处理、系统集成等。==对于需要高度灵活的消息路由和过滤的场景==，可以考虑选择 RabbitMQ
    > 💡 **选用 RabbitMQ 的原因**
    > 1. 灵活的路由：RabbitMQ 支持多种消息模型，如发布订阅模式中的 Fanout、Direct、Topic 三者模式，可以根据 **`BindingKey`** 灵活的路由和过滤
    > 2. 多种编程语言支持：RabbitMQ 支持多种编程语言，如 Java、Python、Ruby、.NET 等
    > 3. 持久化存储：RabbitMQ 支持消息的持久化存储，即使在消息队列重启或者发生故障时，也能保证消息的不丢失
    > 4. **项目规模和数据量不是特别大**，使用 Kafka 体量太大了，不好维护
    - RocketMQ
        - 特点：==高吞吐量、低延迟、高可靠性==、支持分布式部署、支持多种语言、支持扩展和定制化
        - 选型方案：适合大规模的异步消息处理场景，如电商订单处理、社交网络通知等。==对==**==可靠性==**==有较高要求的==**==金融场景==**，可以考虑选择 RocketMQ
    - ActiveMQ
        - 特点：支持多种消息协议、支持持久化、支持集群部署、支持消息转发、支持多种语言
        - 选型方案：适合中小规模的异步消息处理场景，如任务分发、事件通知、消息转发等。对于需要在消息队列中实现数据转换和数据处理的场景，可以考虑选择 ActiveMQ
3. 🙋‍♂️ **消息队列的优点：**[[1.1 同步调用与异步调用]]
    1. 业务解耦，提高业务的响应速度，增加吞吐量
    2. 流量削峰
    3. 故障隔离，服务没有强依赖关系，避免级联失败
4. 🙋‍♂️ **消息队列的缺点：**
    1. **业务没有明显的流程线，不好追踪管理**
    2. 使用消息队列会**增加系统的复杂性**，需要额外的维护工作
    3. **（单点故障）**系统可用性依赖于消息队列：使用消息队列的系统可用性会变得更加依赖于消息队列的稳定性和可靠性。如果消息队列出现故障，整个系统的可用性也会受到影响
    4. 数据不一致问题：由于消息队列是异步的，生产者和消费者之间的处理逻辑可能存在不一致的情况
    5. 性能问题：消息队列的使用可能会带来额外的性能开销，例如网络传输、序列化/反序列化、持久化等
