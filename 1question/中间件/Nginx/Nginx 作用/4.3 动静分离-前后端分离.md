# 一、介绍
1. 适合中小型项目架构，**目的：可以大幅增加并发量**
    - 动静分离之前
        - 请求由nginx代理打到目标服务的tomcat服务器上，返回首页index.html
        - index.html中内嵌了很多静态资源，浏览器再次发起请求，向nginx代理请求这些静态资源
        - nginx再去访问目标tomcat服务器获取静态资源，在返回给客户端
        - **当一个页面中有大量的静态资源时，一瞬间的资源请求并发量会达到很高，服务器压力会很大**
    - 动静分离之后
        - **静态资源（CSS、js、HTMLTemplate等等）放在性能高的专用静态资源服务器上，或者直接前置放在 nginx 代理服务器上**
        - **客户端的静态资源请求，就不再去 tomcat 上找了，而是直接由静态资源服务器提供**
2. **可以理解成**==**使用 Nginx 处理静态页面，使用 Tomcat 处理动态页面**==

    ![[IMG-20260405035438323.png|754]]

3. 动静分离从目前实现角度来讲大致分为两种
    1. 一种是纯粹把静态文件独立成单独的域名，放在独立的服务器上，也是目前主流推崇的方案
    2. 另外一种方法就是动态跟静态文件混合在一起发布，通过 nginx 来分开

# 二、示例
1. 示例描述：实现以下案例

    ![[IMG-20260404031924119.png|587]]

2. 在192.168.10.192 上跑着一个nginx，暂且将他看作一个 Tomcat 业务服务
3. 修改其主页 index.html

    ```Plain
     <!DOCTYPE html>
     <html>
     <head>
     <title>Error</title>
     </head>
     <body>
     <h1>dynamic and static separation</h1>
     <img src="/img/test.png">
     </body>
     </html>
    ```
4. 使用192.168.10.191 上的 nginx 作为192 的代理服务器

    ```Plain
     # 192.168.10.191 的nginx 配置文件
     worker_processes  1;
     pid        logs/nginx.pid;
     events {
         worker_connections  1024;
     }
     http {
         include       mime.types;
         default_type  application/octet-stream;
         sendfile        on;
         keepalive_timeout  65;
         server {
             listen       80;
             server_name  localhost;
             location / {
                 proxy_pass http://192.168.10.192:80;    # ⭐
             }
             error_page   500 502 503 504  /50x.html;
             location = /50x.html {
                 root   html;
             }
         }
     }
    ```
5. 此时访问：[http://192.168.10.191:80/](http://192.168.10.191/)
    - 虽然确实 191 代理了 192，但是因为 192 上此时根本没有静态图片，故页面中图片肯定加载不出来

    ![[IMG-20260405035451631.png|610]]

6. 将静态文件创建在 191 代理服务器上

    ```Plain
     [root@nginx ~]# tree /usr/local/nginx/static/
     /usr/local/nginx/static/
     └── img
         └── test.png
     1 directory, 1 file
    ```
7. 修改 191 代理服务器的配置文件

    ```Plain
     worker_processes  1;
     pid        logs/nginx.pid;
     events {
         worker_connections  1024;
     }
     http {
         include       mime.types;
         default_type  application/octet-stream;
         sendfile        on;
         keepalive_timeout  65;
         server {
             listen       80;
             server_name  localhost;
             location / {
                 proxy_pass http://192.168.10.192:80;
             }
             # ⭐ 再新配置一个名为 /img 的 location
             location /img {
                 root   /usr/local/nginx/static;
                 index  index.html index.htm;
             }
             error_page   500 502 503 504  /50x.html;
             location = /50x.html {
                 root   html;
             }
         }
     }
    ```
    - 也可以使用正则表达式 来写location，再需要配置的location较多时，比较方便

    ```Plain
     location ~*/(css|img|js){
         root  /usr/local/nginx/static;
         index  index.html index.htm;
     }
    ```
    - 正则 location 前缀：
        - / 通用匹配，任何请求都会匹配到
        - = 精准匹配，不是以指定模式开头
        - ~ 正则匹配，区分大小写
        - ~* 正则匹配，不区分大小写
        - ^~ 非正则匹配，匹配以指定模式开头的location
    - 正则 **location 匹配顺序：**
        - 多个正则location直接按书写顺序匹配，成功后就不会继续往后面匹配
        - 普通（非正则）location会一直往下，直到找到匹配度最高的（最大前缀匹配）
        - 当普通location与正则location同时存在，如果正则匹配成功,则不会再执行普通匹配
        - 所有类型location存在时，“=”匹配 > “^~”匹配 > 正则匹配 > 普通（最大前缀匹配）
8. 重新加载配置
9. 测试访问 [http://192.168.10.191:80/](http://192.168.10.191/)
    - 图片加载成功

    ![[IMG-20260405035507704.png|674]]

    - 且图片确实是从 nginx 191 代理服务器上响应返回的

    ![[IMG-20260405035515127.png|662]]
