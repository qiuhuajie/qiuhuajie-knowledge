# 一、数据库模式
1. ==三级模式==：外模式、概念模式、内模式
2. 外模式对应用户视图、概念模式对应基本表、内模式对应数据库物理存储文件
3. ==两层映射关系==：外模式和概念模式之间的映射、概念模式和内模式之间的映射
    1. 外模式和概念模式之间的映射可以保证==逻辑独立性==：外部的应用程序和逻辑结构的基本表是相互独立的，在其中一方有变化时，只需要改动映射关系即可
    2. 概念模式和内模式之间的映射可以保证==物理独立性==：外部的应用程序和物理存储结构文件是相互独立的，在其中一方有变化时，只需要改动映射关系即可

    ![[IMG-20260404032059637.png|Untitled 1 90.png]]

# 二、关系表类型
1. 关系的三种类型：
    1. 基本关系（通常又称为基本表或基表）：实际存在的表，实际存储数据的逻辑表示
    2. 查询表：查询结果对应的表
    3. 视图表：由基表或其他视图导出的表，本身不独立存储，数据库只存放它的定义，常称为虚表
2. 数据库视图：
    1. 它一个虛拟表（逻辑上的表），其内容由查询定义 （仅保存SQL查询语向）
    2. 同真实的表一样，视图包含一系列带有名称的列和行数据。但是，视图并没有真正存储这些数据，而是通过查询原始表动态生成所需要的数据
    3. 视图的优点：
        1. 视图能简化用户操作
        2. 视图使用户能以多种角度看待同一数据
        3. 视图对重构数据库提供了一定程度的逻辑独立性
        4. 视图可以对机密数据提供安全保护
    4. 物化视图：
        1. 视图的性能问题：每次展示前都要先查，所以存储下来会提升性能
        2. 物化视图就是视图的持久化：它不是传统意义上虚拟视图，是实体化视图，其本身会存储数据。同时当原始表中的数据更新时，物化视图也会更新
        3. 视图持久化后，涉及到与基本表中原数据的一致性问题，所以物化视图适用于多查询，少更新的场景

# 三、分布式数据库
## 1. 特点
1. ==数据独立性==。除了数据的逻辑独立性与物理独立性外，还有数据分布独立性（分布透明性）
2. ==集中与自治共享结合的控制结构==
    1. 各局部的DBMS可以独立地管理局部数据库，具有自治的功能
    2. 同时，系统又设有集中控制机制，协调各局部DBMS的工作，执行全局应用
3. ==适当增加数据冗余度==
    1. 在不同的场地存储同一数据的多个副本，可以提高系统的可靠性和可用性，同时也能提高系统性能
    2. 提高系统的可用性，即当系统中某个节点发生故障时，因为数据有其他副本在非故障场地上，对其他所有场地来说，数据仍然是可用的，从而保证数据的完备性
4. ==全局的一致性、可串行性和可恢复性==

## 3. 分布透明性

🔥

1. **分片透明性**
    - 分为：水平分片、垂直分片、混合分片
    - 是指用户不必关心数据是**如何**==**分片**==**的**，它们对数据的操作在全局关系上进行，即如何分片对用户是透明的
2. **位置透明性**
    - 是指用户不必知道所操作的数据==**放在何处**==，即数据分配到哪个或哪些站点存储对用户是透明的
3. **复制透明性**
    - 用户不用关心数据库在网络中各个节点的==**复制**==**情况**，被复制的数据的更新都由系统自动完成
4. **局部数据模型透明性（逻辑透明）**
    - 是最低层次的透明性，该透明性提供数据到局部数据库的映像，即用户**不必关心**==**局部DBMS**==**支持哪种数据模型**、使用哪种数据操纵语言，数据模型和操纵语言的转换是由系统完成的
    - 因此，局部映像透明性对异构型和同构异质的分布式数据库系统是非常重要的

## 1. 过程介绍🔥

![[IMG-20260404032100201.png|Untitled 6 31.png]]

# 五、关系代数🔥

![[IMG-20260404032101607.png|Untitled 17 9.png]]

![[IMG-20260404032101707.png|Untitled 18 8.png]]

1. 笛卡尔积：
    1. 实质上是 R 与 S 的无条件连接，使得任意两个关系的信息可以组合在一起
    2. 在多表联查的 SQL 语句中，`form` 后面跟的就是笛卡尔积的结果
2. 选择：
    1. $\sigma _{F(R)} = \{ t | t \in R, F(t) = true \}$简单地理解就是从 R 中选出**满足F为真**的元组组成新元组

        ![[IMG-20260404032101834.png|Untitled 19 7.png]]

    2. 在 SQL 语句中，`where` 语句后跟的就是选择结果
3. 投影：
    1. $\Pi_A(R) = \{ t [A] | t \in R \} , A \subseteq R$ 是从关系 R 中**取**若干列组成新的关系（集合）

        ![[IMG-20260404032101936.png|Untitled 20 6.png]]

    2. 在 SQL 语句中，`select` 语句后跟的就是投影结果
4. 🔥自然链接：
    1. 从两个关系的**广义笛卡儿积中选取**在**相同属性列** B 上**取值相等**的元组，并**去掉重复**的行，表达式为：

        $$R \Join S = \{ \mathop{rs}\limits^{\frown}[B] | r\in R \wedge s \in S \wedge r[B] = S[B] \}$$

    2. 与等值连接的区别在于：属性列相同，去重
    3. 举例：

        ![[IMG-20260404032102046.png|Untitled 21 6.png]]

        r与s的B、D两个属性重复，具体而言有3个子元组(BD)重复，为(1a,1a,2b)，2个1a内部做笛卡尔积，得到4个新元组，1个2b内部做笛卡尔积，得到1个新元组，组成自然连接的结果

    4. 性能上：**自然链接优于笛卡尔积**

1. 设R(U)是属性U上的一个关系模式，X和Y是U的子集，r 为 R 的任一关系，如果对于 r 中的任意两个元组u，v，只要有u[X]=v[X]，就有u[Y]=v[Y]，则称 X 函数决定V，或称Y函数依赖于X，记为X-Y

## 3. Armstrong 公理
- 从已知的函数依赖，可以推导出另外一些函数依赖，这就需要一系列的推理规则，这些规则常被称为”**Armstrong 公理**“
![[IMG-20260404032102803.png|Untitled 28 4.png]]
![[IMG-20260404032102932.png|Untitled 29 4.png]]
> [!important] **通俗的讲：🔥**
> 
> 1. 自反律：Y是X的子集，则X → Y
> 
> 2. 增广律：X → Y，则XZ → YZ
> 
> 3. 传递律：X → Y，Y → Z，则X → Z
> 
> 4. 合并律 = 增广 + 传递
> 
> 5. 伪传递律 = 增广 + 传递（另一种用法）
> 
> 6. 分解律 = 自反 + 传递（也约等于废话）
> [!important] 例题：C
> 
> - AA → AB，等价于 A → AB，又 BA → BC，所以 A → BC（注AA、BA都是”并“的关系，所以分别等同于A、AB）

## 4. 范式⭐
1. 关系数据库设计的方法之一就是满足适当范式模式，通常可以通过判断分解后的模式达到几范式来评价模式的规范化程度
2. 范式包括：1NF、2NF、3NF、BCNF、4NF、5NF

    ![[IMG-20260404032103150.png|Untitled 31 4.png]]

    1. **==第一范式（1NF）==**
        1. 在关系模式R中，当且仅当所有域==只包含原子值==，既每个属性都是==不可再分==的数据项，则关系模式R属于第一范式
        2. 例如：以下不满足1NF，高级职称人数可以再分教授和副教授

            ![[IMG-20260404032103305.png|Untitled 32 2.png]]

    2. **==第二范式（2NF）==**
        1. 若关系模式R $\in$ 1NF，且每个非主属性完全依赖主码时（不存在部分依赖），则关系模式R属于第二范式
        2. 如果主键为单属性，则一定满足第二范式
        3. 例如：以下不满足2NF，课程号可以包含学分（==学分属性部分依赖主键，只依赖了课程号==）

            ![[IMG-20260404032103400.png|Untitled 33 2.png]]

    3. ==**第三范式（3NF）**==
        1. 若关系模式R $\in$ 2NF，且没有非主属性对主码的传递函数依赖（==传递的中间属性要是非主属性==）。则关系模式R属于第三范式
        2. 如果没有非主属性，则一定满足第三范式
        3. 例如：以下不满足3NF，学号决定系号，系号决定系位置，存在传递依赖（且中间属性系号是非主属性）

        ![[IMG-20260404032103495.png|Untitled 34 2.png]]

    4. ==**BC范式（BCNF）**==
        1. 设R是一个关系模式，F是它的依赖集，R属于BCNF当且仅当其F中每个依赖的决定因素必定包含R的某个候选码
