## 1. 架构风格
1. 使用独立构件风格中的隐式调用
    1. 由于本平台使用的是微服务架构，资源管理模块、业务支撑模块、业务资源监控模块等应用采用了分布式部署。另外，业务流程中涉及到大量的与外部第三方应用的数据交互，如运营平台需要与下游的运营商 BSS 接口对接，调用 BSS 的资源核查服务来实现 MEC 边缘云服务的自动计费和自动结算能力。BSS 对账接口被调用的时涉及到许多数据密集型和计算密集型的任务，如果采用同步调用，业务接口的调用出现大面积调用超时的情况。
    2. 因此应用之间的解耦尤为关键，我们最终采取了通过异步消息传递进行通信，而不是依赖直接的接口调用，对账接口中用户的账户信息与所购产品信息将被发送至消息队列，由 BSS 根据其负载异步消费，进而实现了应用模块之间的解耦。经过多轮技术选型，最终采用了开源的消息中间件 RabbitMQ 作为具体的实现方式，RabbitMQ 支持多种消息模型，如发布订阅模式中的 广播、路由、Topic 三者模式，可以根据 `BindingKey` 灵活的路由和过滤消息；RabbitMQ 还支持消息的持久化存储，即使在消息队列重启或者发生故障时，也能保证消息的不丢失。
    3. 使用消息队列的另外一个好处就是流量削峰，无论调用方的流量波动多大，都由消息代理接收，业务订阅者可以按照自身的性能负荷来采取对应的处理速度。
    4. 消息队列可以通过发布订阅模式，在智能运维中页发挥了关键作用。服务组件可以在遇到异常或监测到潜在问题时将告警消息发布到消息队列中，而监控服务则订阅这些告警，实时处理警报，快速响应可能的系统故障。
2. 使用虚拟机风格中的解释器
    1. 由于边缘业务开通秉持业务流程可编排、业务开通透明化、流程监控以及开通效率精细分析的原则，需要实现从客户在线订购到自动出账计费的业务全流程自动开通。因此业务中涉及到大量的工作流处理

## 2. 可靠性
1. 使用 Kubernetes 进行应用容器化部署
    1. 我们使用了 Kubernetes 进行==应用容器化部署==。Kubernetes 是一个开源的容器编排平台，以下简称 K8s，我们的平台应用运行在封装好的容器中，使得系统的部署、扩展和管理变得自动化，同时提高了系统的可用性。
    2. 平台设计之初便考虑了应用部署的高可用方案，来保证最终平台运行的可靠性，经过架构组的多轮技术选项，我们最终借助了 K8s 的==跨节点部署==和==自我修复能力==来保证平台应用的高可用性。K8s 支持跨多个主机节点部署容器化的应用程序，即使某个节点发生故障，服务也不会中断，因为 K8s 会自动在其他健康的节点上重新启动容器，确保服务持续可用。同时，K8s 会==监测和维护应用的状态==，以确保其符合用户定义的预期状态，若某个 pod 发生故障或无响应，K8s 会自动重新部署该 pod 到健康节点上，从而实现系统自我修复。
    3. 另外，MEC边缘云计算平台具有良好的弹性扩展能力，这都归功于K8s的==自动伸缩功能==。K8s 可以根据系统的负载自动伸缩，即在负载增大时增加实例数量，负载减小时回收资源。这种弹性能够处理访问流量的波动，避免系统过载。
    4. 最后，K8s 支持服务发现和负载均衡，通过自动分配IP地址和单一的DNS名给容器，由后端自动进行负载均衡，这保证了用户请求均匀地分布在所有可用的实例上，提高了系统处理高并发请求的能力。
2. 通过主备形式的数据库部署进行软件容错
    1. 本平台的核心数据如MEC 边缘计算节点数据、 EdgeUPF/EdgePOD 相关的软硬件 ICT 平台资源信息以及基站（BSS）的用户计费数据等都存储在服务端的数据库中，因此数据库的稳定性对于上层的业务模块的正常运行极为关键。因此我们通过主备形式的数据库部署以及告警和备份机制来实现数据库的容错。
    2. 主备数据库部署是一种常见数据库层面的软件容错策略，可以确保数据的持续可用性和完整性，同时考虑到我们的平台数据的读场景远对于写场景，所以我们采用==一主多备的数据库部署方式==，其中主库负责写请求，负责处理读请求的多个备库拥有与主数据库相同的数据。
    3. 主备数据库还可以实现==自动故障转移==，如果主服务器因为硬件故障、网络问题或其他原因挂掉，自动故障转移机制可以确保从主服务器切换到备服务器，从而保持服务的连续性不受影响。
    4. 由于我们业务对数据实时性的要求不高，所以采用了==基于 binlog 的异步主从复制方案==，在主数据库在处理写操作之后并不立即同步备数据库，而由备数据库定期或在数据库负载较少时复制主数据库的数据，在保证数据的一致性的同时，提高数据库的吞吐量与资源利用率。
    5. 另外，我们引入了==数据库告警机制==，对运行的数据库连接数和实例数量设置预定的告警阈值和告警群组，可以及时发现数据相关性能指标的异常变动，以确保数据库集群运行在合理的容量和负载上，防止意外的实例骤减导致的服务中断或性能下降等问题。
    6. 最后，我们设置了数据的==增量备份和定期全量备份==，参考我们业务数据的体量，定期全量备份的频率我们设定为一周一次，而使用一小时一次作为增量备份的周期。两种备份策略互为补充，对数据库进行快照，以防出现极端情况下的数据丢失。
    7. 通过上述一系列方法，我们提高了数据库的可靠性，也同时保证了整个系统的高可用。
3. 通过防卫式程序设计方面进行软件的容错和避错
    1. 我们在程序架构设计以及系统编码的过程中，使用了大量的防卫式程序设计，以及一些容错和避错措施来减少故障的发生。
    2. 首先，对于==异常处理==我们设计了健壮的异常处理逻辑可以在遇到错误时防止程序崩溃。例如通过 try-catch-finally 块捕获和处理异常，使程序能够在出现问题时优雅地恢复或退出。
    3. 对所有业务接口的==外部输入进行验证==，确保输入符合预期的格式和范围，从而防止恶意输入或无效数据导致的程序错误和安全风险。
    4. 在与==外部系统==通信时，本平台利用==超时重试和失败重试机制==避免了因为长时间等待响应或下游应用的错误结果返回而使本系统整个程序阻塞。例如，本运营平台需要与下游的运营商 BSS 接口对接，调用 BSS 的资源核查服务来实现 MEC 边缘云服务的自动计费和自动结算能力。如果在 BSS 对账接口被调用的时候，出现了网络抖动和丢包的情况，那么业务支撑模块如果没有相应的容错设计，那么就会为用户返回结算失败的提示，这种体验对使用者来说很不友好，而引入了失败重试与超时重试机制后，可以很好的解决上述由偶发性网络故障所导致的结算失败问题。
    5. 针对客户端重复连续多次点击的情况，例如客户在点击开通业务按钮时，服务端业务受理的接口就可以通过 Redis 存储请求所携带的特定 Token，在业务开通处理程序之前进行 Token 验证来保证==业务幂等操作==，确保即使重试多次也不会产生不一致的结果，这对网络调用和分布式系统尤其重要。
