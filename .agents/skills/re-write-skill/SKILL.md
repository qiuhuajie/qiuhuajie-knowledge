# re-write-skill

适用场景：当用户说“重写”“整理成笔记”“归档这篇文章/网页/资料”“按我的风格整理”时使用。

目标：把原始材料转成适合 Obsidian 长期积累的知识笔记，而不是只做表面清洗。默认输出应是“可归档成品”，一次尽量做完整。

## 核心原则

1. 先归档，再润色
- 首先保证信息被完整、安全地落到正确目录与文件名中。
- 再进行结构化整理、摘要提炼、链接增强与格式统一。

2. 不是简单改写，而是知识化重组
- 删除网页噪声、导航、按钮、广告、作者勋章、重复页脚等非内容信息。
- 保留作者、来源、日期、原文链接、关键图示、关键代码、关键论点。
- 按“元信息 → 摘要 → 我的整理 → 关联笔记 → 正文整理”重组。

3. 一次归档默认应包含“增强项”
- 不要等用户二次提醒才补：
  - frontmatter 规范化
  - aliases
  - tags 分层
  - 原文信息块
  - 核心摘要
  - 我的整理/一句话结论/全文主线/值得记住的判断
  - 关联笔记
  - 文内相关 wikilink
  - 标题层级统一
- 只有在信息缺失或存在明显歧义时才询问。

4. 优先融入现有知识网络
- 如果库里已有相关主题、同系列文章、同作者/同项目笔记，应主动互链。
- 至少检查：同主题、同系列、同项目、同方法论的已有笔记。
- 若能自然加入“延伸阅读/关联笔记/对照阅读”，默认加入。

5. 必须严格遵守 my-markdown 的编号与段落规则
- 在 qiuhuajie 的 Obsidian 库中改写 `.md` 时，必须同时遵守 `obsidian-markdown` 与 `my-markdown`。
- 格式冲突时，`my-markdown` 优先。
- 一级标题使用 `# 一、`、`# 二、` 这类中文编号。
- 二级标题使用 `## 1.`、`## 2.` 这类阿拉伯编号；如果是“我的整理 / 正文整理 / 关联笔记”这类前置区块，可维持无编号二级标题，但正文主体章节不得混乱。
- 三级标题应继承二级标题编号，如 `### 3.1`、`### 3.2`。
- 每个标题下的正文段落都必须使用 `1.` `2.` 或 `*` 组织，不能留下普通裸段落。
- 不要把 `# 1. 背景` 和 `# 一、...` 混在同一层级。
- 不要为了补结构而破坏已有编号系统；若原文已形成稳定编号，优先在原系统上微调而不是重排。

6. 保持“我的风格”统一
- 标题清晰，层级稳定。
- 句子尽量简洁，少废话，强调结论先行。
- 中文排版统一：中英文之间适度留空，专有名词尽量规范。
- 不把笔记写成聊天口吻，不留下“我之后还可以再补”这种未完成措辞。

## 标准工作流

### 第 1 步：识别归档对象
确认以下信息：
- 输入来源是什么：本地 md、网页、PDF、摘录文本。
- 应放入哪个目录。
- 是否已有同名或已归档版本。

若用户只说“归档这篇笔记”，默认理解为：
- 生成可直接保存的最终 Obsidian 笔记；
- 顺手完成必要的结构化增强，而不是只移动文件。

### 第 2 步：检查上下文与关联资料
在动手前尽量检查：
- 同目录下是否已有同系列笔记；
- 全库是否已有同主题笔记；
- 是否有可互链的总纲/方法论/作者专题笔记。

至少要考虑补这些：
- 同系列文章互链
- 方法论总纲互链
- 项目专题页互链

### 第 3 步：重写为归档成品
默认结构建议：

```md
---
title: 标题
tags:
  - 一级
  - 一级/二级
updated: YYYY-MM-DD
author: 作者
date: YYYY-MM-DD
aliases:
  - 原始标题
  - 其他常见叫法
source_name: 来源名
source_url: 原文链接
---

> [!info] 原文信息
> 作者：
> 来源：
> 发表：
> 更新：
> 链接：
> 说明：

> [!abstract] 核心摘要
> 2~4 句讲清文章核心价值。

## 我的整理

### 一句话结论

### 全文主线

### 最值得记住的判断

---

## 关联笔记

* [[相关笔记 A]]
* [[相关笔记 B]]

## 正文整理
```

说明：
- “我的整理”是默认项，不要省略。
- “关联笔记”也是默认项，哪怕只有 2~4 条。
- 若正文特别长，可把一级结构改成“一、二、三、四”。

### 第 4 步：增强正文可读性
处理时默认做这些事：
- 统一标题层级；
- 清理异常空格、残缺列表、网页抓取符号；
- 将裸链替换为更适合阅读的链接表达；
- 识别可链接的专有名词/系列文章并加上 wikilink；
- 尽量保留关键图片引用和代码块；
- 避免把正文整理成过度摘要版，原文主线仍应保留。

### 第 5 步：自检后再结束
结束前检查：
- 文件是否已写入正确位置；
- frontmatter 是否完整；
- 是否有“关联笔记”；
- 是否有至少若干文内 wikilink；
- 是否保留来源信息；
- 是否已统一标题层级；
- 是否仍存在明显网页噪声；
- 是否已达到“第一次交付就可归档”的标准。

## 归档任务的默认完成标准

当用户说“归档这篇笔记/文章”时，默认完成标准不是“文件存在了”，而是：

1. 已放到合理目录；
2. 文件名可读、可检索；
3. frontmatter 完整；
4. 内容已去噪并结构化；
5. 有摘要与我的整理；
6. 有关联笔记；
7. 有必要的文内互链；
8. 标题层级与格式已统一；
9. 可直接进入长期知识库，无需用户再提醒补全基础项。

## 常见补强动作

### 1. 同系列互链
如果文章提到“这是系列第 N 篇”，默认去找其余篇目并加入：
- 文首推荐阅读
- 关联笔记区

### 2. 方法论互链
如果内容本质上属于某一方法论主题，例如：
- AI Agent 系统设计
- Prompt / Context / Harness
- Memory / Skills / Tool / Guardrail
则优先链接到总纲笔记或主题索引笔记。

### 3. 专题入口互链
如果库中已有某个项目专题页，例如 OpenClaw、Hermes Agent、Claude Code，应把单篇文章挂回专题入口。

### 4. 标题重整
当原始文章层级混乱时：
- 总论级标题可升为 `#`
- 分节标题统一为 `##`
- 子节统一为 `###`
- 不要同时混用中文编号和 Markdown 层级到不可读的程度

## 不该做的事

- 不要只做机械搬运。
- 不要只补一个 frontmatter 就算完成。
- 不要把“还可以补充关联阅读”留到下次；默认这次就补。
- 不要在未检查知识库关联的情况下就直接结束。
- 不要为追求简洁而删掉关键论证、示意图、代码或案例。
- 不要输出“建议你之后再补标签/互链/摘要”，除非当前确实缺少必要信息。

## 完成后的汇报风格

向用户汇报时要直接说明已完成的具体增强项，例如：
- 已补关联笔记
- 已加入文内互链
- 已统一标题层级
- 已增强 frontmatter/aliases/tags

如果发现这次任务暴露出流程缺项，应及时更新本 skill，避免下次重复漏做。
