# 商家签证 Agent 技术方案落地分析

> 基于《商家签证售前 Agent 技术方案 v3》，从**平台**和**行业开发**两个视角分析落地路径。

---

## 一、方案核心回顾

技术方案采用 **ReAct 循环 + 动态决策树** 架构，通过 MCP + Prompt 混合模式实现签证售前咨询的智能化和精准化。核心链路：

```java
用户输入 → Observe（字段提取）→ Reason（决策树）→ Act（追问/匹配/推荐）
```

关键依赖：

- **签证标准库 MCP**：提供国家、签证类型、领区等结构化数据
- **FAI 知识库 MCP**：商家知识混合召回（结构化 + RAG）
- **商品/套餐 API**：商家在售签证套餐查询
- **Prompt 层**：意图理解、话术生成、口语映射

---

## 二、平台视角：平台需要做什么

### 2.1 平台基础设施

| 模块 | 负责内容 | 优先级 |
|---|---|---|
| **MCP Server（签证标准库）** | 统一封装国家列表、签证类型、领区匹配等结构化查询能力，对外暴露标准 MCP 接口 | P0 |
| **FAI 知识库 MCP** | 封装 FAI 平台知识库召回能力，提供 `mcp__fai__retrieve_knowledge` 接口，支持结构化 + RAG 双层召回 | P0 |
| **MCP 注册与分发** | 将签证标准库 MCP 和 FAI MCP 注册到 MCP 路由/注册中心，供 Agent 按需调用 | P0 |
| **会话上下文协议** | 定义会话初始化时的上下文注入规范（sellerId、sellerName、sellerConfig），确保行业侧能正确接入 | P1 |
| **商品/套餐接口** | 标准化 `get_product` 和 `get_packages_by_product` 接口，提供确定性商品查询 | P1 |
| **Agent 运行时** | 提供 ReAct Agent 的执行框架（循环控制、状态管理、工具调用编排） | P1 |
| **监控与日志** | 记录每个 ReAct 循环的决策路径、工具调用耗时、召回命中率等，供平台和行业侧分析 | P2 |

### 2.2 平台需提供的标准能力

1. **签证标准库 MCP Server**
   - 数据源：签证数据中心（MySQL/PostgreSQL）
   - 部署方式：FastMCP 或其他 MCP SDK
   - 接口协议：MCP JSON-RPC 2.0
   - 访问控制：内网 API Key 鉴权
   - **关键接口**：
     - `mcp__visa__get_countries` — 国家列表查询（支持大洲筛选 + 语义召回）
     - `mcp__visa__get_visa_types` — 某国签证类型列表
     - `mcp__visa__get_consulates` — 某国领区列表
     - `mcp__visa__match_consulate` — 常住城市 → 送签领区匹配

2. **FAI 知识库 MCP Server**
   - 封装 FAI 平台知识库
   - 提供 `mcp__fai__retrieve_knowledge` 接口
   - 支持按 seller_id、country_code、visa_type_code、occupation 等维度过滤
   - 返回结构化结果优先，RAG 结果兜底

3. **商品维度的标准化定义**
   - 商品 = sellerId + countryCode + visaTypeCode + consulateCode
   - 每个商品下挂多个套餐（package_id, name, price, duration_days, features, tags）

### 2.3 平台落地里程碑建议

| 阶段 | 内容 | 负责方 |
|---|---|---|
| **Phase 1** | MCP Server（签证标准库）开发、联调、部署 | 平台 |
| **Phase 1** | FAI 知识库 MCP 开发、商家知识接入 | 平台 |
| **Phase 2** | Agent 运行时框架 + ReAct 循环实现 | 平台 |
| **Phase 2** | 商品/套餐接口标准化 | 平台 |
| **Phase 3** | 监控、日志、AB 实验能力建设 | 平台 |

### 2.4 平台需要注意的风险

- **MCP 工具返回质量**：签证标准库的数据准确性和完整性直接影响行业侧体验，需建立数据校验机制
- **召回策略的一致性**：结构化召回 vs 语义召回的边界需要明确，避免 RAG 结果覆盖结构化结果
- **商家知识接入成本**：FAI 知识库需要商家主动上传文档或配置结构化数据，平台需提供便捷的商家入驻工具
- **时区与日期处理**：出行日期、办理余量计算涉及时区，需统一处理规范

## 三、行业开发视角：行业侧需要做什么

### 3.1 行业侧核心工作

作为行业开发，核心工作是**接入平台能力 + 业务定制**。

#### 3.1.1 接入 MCP 工具

行业侧需要对接的 MCP 工具：

| MCP 工具 | 调用方式 | 行业侧工作 |
|---|---|---|
| `mcp__visa__get_countries` | 直接调用 | 传入 `continent` 筛选参数，处理返回的国家列表展示 |
| `mcp__visa__get_visa_types` | 直接调用 | 传入 `country_code`，展示签证类型供用户选择 |
| `mcp__visa__match_consulate` | 间接调用 | Skill 协调层自动调用，行业侧无需感知 |
| `mcp__fai__retrieve_knowledge` | Skill 协调层调用 | **关键**：行业侧需将商家知识（PDF/Word/结构化配置）录入 FAI 知识库 |
| `get_product` | 直接调用 | 传入 sellerId + countryCode + visaTypeCode + consulateCode，获取套餐列表 |
| `get_packages_by_product` | 直接调用 | 传入 productId，获取详细套餐信息 |

#### 3.1.2 Skill 协调层实现

行业侧需要实现 Skill 协调层，负责 **Prompt 层 + MCP 层** 的路由和组合：

```java
用户输入
  │
  ▼
┌─────────────────────────────────┐
│     Skill 协调层（行业侧实现）     │
│                                 │
│  1. 判断走哪条召回路径            │
│  2. 调用对应 MCP 工具            │
│  3. 聚合 MCP 返回结果            │
│  4. 传递给 Prompt 层做决策        │
└─────────────────────────────────┘
```

关键 Skill 及行业侧职责：

**Skill 1：`country_match`**

- Prompt 层（行业侧实现）：判断用户输入是洲名/概念/具体国家
- MCP 层（平台提供）：`mcp__visa__get_countries` 做结构化筛选或语义召回
- 行业侧可扩展：根据商家主营业务国家做优先排序

**Skill 2：`visa_type_match`**

- Prompt 层（行业侧实现）：口语 → 签证类型候选映射（参考文档中的映射表）
- MCP 层（平台提供）：`mcp__visa__get_visa_types` 查询该国支持的签证类型
- 行业侧可扩展：商家自定义签证类型别名

**Skill 3：`occupation_match`**

- 纯 Prompt 层实现，参考文档中的口语→职业类型映射表
- 行业侧可扩展：添加商家特有的职业分类或豁免职业处理

**Skill 4：`ask_field`**

- 纯 Prompt 层实现，生成追问话术
- 行业侧可扩展：注入商家个性化话术（通过 sellerConfig）

#### 3.1.3 商家知识录入

**这是行业侧最重要的工作之一。** FAI 知识库的质量直接决定了材料清单召回的准确性。

| 知识来源 | 录入方式 | 优先级 |
|---|---|---|
| 商家结构化材料配置（表格录入） | 商家后台配置界面 → FAI 结构化知识库 | 最高 |
| 商家上传文档（PDF/Word/TXT） | 商家后台上传 → FAI RAG 向量库 | 高 |
| 商家特殊流程/注意事项 | 商家配置 → 注入 System Prompt | 中 |
| 平台通用材料清单 | 平台预置，兜底使用 | 保底 |

**行业侧需重点投入**：

- 引导商家完成结构化材料配置（而非仅上传文档）
- 审核商家上传文档的质量，确保 FAI 召回结果准确
- 定期更新商家知识库（领馆政策变化、节假日闭馆等）

#### 3.1.4 Agent 输出定制

行业侧可通过 `sellerConfig` 注入个性化配置：

```json
{
  "seller_id": "xxx",
  "seller_name": "XX签证专营店",
  "custom_scripts": {
    "greeting": "欢迎来到XX签证！...",
    "closing": "感谢选择XX签证..."
  },
  "supported_countries": ["JP", "KR", "US", "GB"],
  "special_notes": "日本签证目前广州领区办理较慢..."
}
```

这些配置会注入到 System Prompt 中，影响 Agent 的对话风格和推荐策略。

### 3.2 行业侧落地检查清单

| 事项 | 说明 | 优先级 |
|---|---|---|
| [ ] 接入平台 MCP 路由 | 确保 Agent 能正确调用平台提供的 MCP 工具 | P0 |
| [ ] 实现 Skill 协调层 | Prompt 层 + MCP 层的路由组合逻辑 | P0 |
| [ ] 商家知识录入 FAI | 引导商家配置结构化材料 + 上传文档 | P0 |
| [ ] 接入商品/套餐 API | 确保能查询商家在售签证套餐 | P0 |
| [ ] 会话初始化上下文注入 | 正确传入 sellerId、sellerName 等 | P1 |
| [ ] 商家话术定制 | 通过 sellerConfig 注入个性化话术 | P1 |
| [ ] 异常处理对接 | 对接平台定义的异常响应策略 | P1 |
| [ ] 监控埋点 | 记录关键决策节点和工具调用情况 | P2 |
| [ ] 持续优化知识库 | 根据召回效果持续更新商家知识 | P2 |

### 3.3 行业侧需要注意的风险

- **知识库召回质量**：行业侧若商家知识录入不完整，FAI 召回结果会降级到平台通用清单，准确性下降。**建议优先确保结构化材料配置的完整性。**
- **话术与实际业务不符**：商家话术定制需审核，避免 Agent 给出与实际办理流程不符的承诺
- **跨商家比价场景**：当前方案是单商家视角，若未来支持跨商家推荐，行业侧需做好商品数据质量管控
- **领区匹配失败**：当用户常住城市不在某国领区管辖范围时，需引导用户选择其他领区，涉及送签地变更的业务逻辑

## 四、平台与行业的协作边界

```java
┌─────────────────────────────────────────────────────────────┐
│                        平台（Platform）                      │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  MCP Server: 签证标准库 + FAI 知识库                  │  │
│  │  Agent 运行时: ReAct 循环框架 + 状态管理              │  │
│  │  标准化接口: 商品/套餐 API + 会话上下文协议           │  │
│  └──────────────────────────────────────────────────────┘  │
└────────────────────────┬────────────────────────────────────┘
                         │ 调用 MCP 工具 / API / 上下文注入
┌────────────────────────▼────────────────────────────────────┐
│                      行业侧（Industry）                      │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  Skill 协调层: Prompt + MCP 路由                      │  │
│  │  商家知识录入: FAI 结构化配置 + 文档上传              │  │
│  │  业务定制: 话术 + 特殊流程 + 商家配置                 │  │
│  │  输出聚合: 推荐卡片格式化 + 引导下单                  │  │
│  └──────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

**清晰的协作原则**：

- **平台提供标准能力**：MCP 工具、Agent 运行时、数据模型
- **行业负责业务适配**：Skill 协调层、商家知识质量、话术定制
- **行业不碰底层数据**：签证标准库由平台维护，行业侧只调用不写入

## 五、后续迭代对行业侧的影响

技术方案中提到的后续迭代方向，对行业侧有直接影响：

| 迭代方向 | 行业侧影响 |
|---|---|
| **材料预审 Skill** | 行业侧需接入 OCR 能力，或对接平台提供的材料预审 API |
| **进度查询 Phase** | 行业侧需提供订单状态查询接口，供 Agent 调用 |
| **多语言 Skill** | 行业侧需针对支持的语言准备对应的材料清单知识 |
| **Agent 自学习** | 行业侧需埋点采集推荐结果反馈（是否下单、满意度），对接到平台反馈系统 |
| **跨商家比价** | 当前方案为单商家，行业侧可提前规划商品数据标准化，为未来跨商家场景做准备 |

## 六、总结

| 视角 | 核心关注点 | 落地优先级 |
|---|---|---|
| **平台** | MCP Server 建设、Agent 运行时、标准化接口、数据质量 | P0 |
| **行业开发** | Skill 协调层实现、商家知识录入、话术定制、商品数据接入 | P0 |

**关键成功因素**：

1. **平台侧**：MCP 工具的返回质量和稳定性是整个链路的基础
2. **行业侧**：商家知识的完整性和准确性决定了最终用户体验的上限
3. **双方**：清晰的协作边界和标准化的接口协议是高效对接的前提
