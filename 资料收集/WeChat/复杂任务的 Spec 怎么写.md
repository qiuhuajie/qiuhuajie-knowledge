# 一、复杂任务 Spec 的组织方式

* 来源：古法程序员 cc-codex 实践，2026 年 5 月 25 日 16:34。
* 复杂任务的 Spec，不再只是“一段提示词”，而是一套能支撑多角色协作、分阶段推进、全程可追溯的工程资产。

## 1. 多 Agent：一个编排者 + 一队专职执行者

1. 复杂任务的第一个决定，是不要让一个 Agent 从头干到尾。
    ![[Attachment/24a0856cbb4878c55a002a93643adfba_MD5.webp|419]]
2. 如果一个 Agent 同时做需求分析、写代码、做评审，上下文会很快膨胀，职责也会越来越模糊。更稳妥的做法，是由一个编排者带一队专职执行者。
3. 编排者，也就是入口 Agent，只做编排和收口：判断当前阶段、检查门禁、决定派给谁、判断是否继续。它不亲自写业务代码。具体执行工作交给专职子 Agent，每个子 Agent 在一个全新上下文里启动，只加载这次需要的知识。
4. 执行者可以按职责拆成以下几类角色。
    * 需求分析师：把需求文档和范围拆成结构化输入。
    * 架构师：做技术设计、确定跨端统一契约、编写 ADR，但不写实现代码。
    * 各端实现者：按平台拆分，每端一个，只写自己负责的那一端。
    * 代码评审者：从安全、性能、规范、参考实现对齐等维度做核验。
    * 测试工程师：补测试口径，先写失败的验收测试。
    * 联调工程师：跑端到端构建、安装、运行、日志和跨端一致性检查。
    * 效率工程师：把反复出现的模式沉淀成可复用资产。
5. 角色划分还有一个经常被低估的作用：**选错角色，不只是分工不对，更会绕过这个角色绑定的规则约束。** 例如各端实现者会自动加载对应平台的硬约束清单，而架构师不会。如果把本该交给实现者的活错派给架构师，那套实现层检查就会整体失效。
6. 因此，派发不能凭感觉。更稳妥的规则是：先按阶段选角色，再用“改动文件路径 -> 角色”的映射表校验写入范围。两者冲突时就停下来澄清，绝不擅自扩大某个 Agent 的写入范围。跨端改动也不应由一个 Agent 包办，而应先由架构师产出统一契约，再让各端实现者在隔离的写入范围里并行。
7. 并行也需要节制。同一条主线最多挂几个子 Agent、每个子 Agent 要交付什么产物、交付格式是什么、验收条件是什么，都应该在派发时声明清楚。完成后要先核验产物是否存在、是否合格，再关闭该 Agent 释放并发名额。没有这套配额和产物核验，多 Agent 很容易变成一群失控的并发进程。

## 2. 编排者入口（CLAUDE.md / AGENTS.md）写什么

1. 编排者的行为，很大程度上由它的入口文件决定，也就是 `CLAUDE.md` 或 `AGENTS.md`。这个文件写得好坏，直接决定整个协作是有序还是混乱。
2. 第一条原则是：**入口只放启动索引和硬边界，不放具体步骤。**
3. 很多人会把所有规则、流程都塞进 `CLAUDE.md`，最后它长到几千行，模型每次启动都要读一遍，既慢又抓不住重点。更合适的做法，是让入口保持很薄：它只告诉模型“任务来了先判断类型，然后去读哪个文件”。具体步骤放到 skill，硬规则放到 rules，角色契约放到 agents，背景和决策记录放到 docs。入口本身只是一张地图。
4. 第二条原则是把编排者的契约写死。入口文件里至少应明确以下约束。
    * 它是编排者和收口者，不是执行者。
    * 它只负责判断阶段、门禁、交接、是否继续，以及派给谁。
    * 小型机械修订、索引更新、跑校验可以自己做，其余默认派给子 Agent。
    * 它不直接改生产代码，不绕过门禁、评审和校验。
    * 遇到非致命问题，先记录 owner、route、证据、下一步，而不是立刻停下来提问。
    * 只有致命错误，或当前门禁明确卡住时，才允许停。
5. 这套契约的价值在于减少“十分钟问八次”的频繁打断。复杂工作流最怕 Agent 一遇到小问题就等待人工拍板。把“什么情况记录后继续，什么情况必须停”提前写进契约，它才能一口气跑完整条链路。
6. 还有一个常见坑需要明确规避：**不要直接编辑生成态的入口文件。** 更稳的做法，是把真正的权威源放在模板文件里，`CLAUDE.md` 和 `AGENTS.md` 都只作为从模板投影出来的产物。否则你以为改了规则，实际上只是改了一个会被下次同步覆盖的临时文件。

## 3. Rules / Docs / Skills 怎么组织

1. Spec 不只是流程，还包括“知识、约束、步骤分别放在哪里，怎么互相引用”。一个常见且好维护的分法，是用三类目录分别回答三个问题。
    ![[Attachment/926744671278393fd5671b984deeec94_MD5.jpg|438]]
    * `skills`：怎么做，也就是具体步骤和流程。
    * `rules`：必须遵守什么，也就是硬约束和检查清单。
    * `docs`：为什么现在这样设计，也就是架构背景和长期决策记录。
2. 这套分法的核心好处是，同一条知识只有一个权威位置。流程变了改 skill，约束变了改 rule，要追溯某个设计为什么这么定就翻 docs，不会到处都是重复又相互矛盾的说明。

### 3.1 一个 Skill 的目录结构

1. 每个 skill 可以组织成一个独立目录，标准结构不需要复杂。

```text
某个 skill/
  SKILL.md       入口：路由 + 职责 + 边界 + 输出契约
  references/    按需加载的长内容：方法论、模板、检查段
  scripts/       确定性脚本（只有 script-backed skill 才有）
```

2. 测试样例不建议直接放在 skill 目录里，而是放到平行的 `evals` 目录下，并按 skill 名一一对应。
3. `SKILL.md` 本身也应该保持固定骨架，但每一段都要尽量短。
    * `frontmatter`：`name`、用于路由的 `description`、层级、风险等级、是否需要人工复核。
    * 职责边界：这个 skill 管什么。
    * 适用与不适用：什么时候召唤，什么时候不要召唤。
    * 工作方式：最小流程，以及什么时候该去读哪个 reference。
    * 按需加载的 references：列清每个场景对应的 reference 文件。
    * 输出与验证：交付什么，怎么算通过。
4. 关键原则只有一条：**`SKILL.md` 要薄。** 它是给模型快速判断“要不要用我、怎么用我”的路由卡，而不是知识库本体。把平台细节、反模式、长模板全堆进去，只会让每次召唤都像先读一篇论文，反而削弱路由判断。

### 3.2 Skill 怎么引用 rules、docs 和知识库

1. `SKILL.md` 保持轻量后，真正的长知识需要通过引用关系被拉进来，而不是直接堆在正文里。
2. `references` 是 skill 自己的长内容，应该按需点名加载。方法论、模板、检查段都拆成独立文件，在“工作方式”里写清楚“这一步去读哪个 reference”，这样模型只在真正走到那一步时才把内容读进上下文。
3. rule 不应该在 skill 正文里抄一份，而应通过路径和角色自动挂载。一种方式是目录级规则：规则声明自己管哪些路径，Agent 一进入这些路径，规则就自动加载。另一种方式是角色绑定：某个角色被派出去时，它该遵守的检查清单自动随之生效。最硬的那几条约束则可以常驻。
4. docs 是被消费的长期知识库，它的引用关系必须显式。docs 负责存放背景和决策记录，而 skill 在需要上下文时去读它。这里有一条非常重要的纪律：**docs 中的每条长期知识，都应该有明确的消费方，例如某个 skill、rule 或角色。** 没有消费方的知识，最后往往会变成没人读、没人维护的孤儿知识。
5. 这三类内容连在一起之后，会形成一张清晰的引用网：skill 正文负责路由，references 承担该 skill 的长内容，rules 负责自动挂载的硬约束，docs 负责长期背景知识，而且每类信息都只有一个权威位置。

### 3.3 Script 怎么组织

1. `scripts` 目录放的是确定性工具，比如校验器、命令包装器、流程辅助脚本。它存在的核心意义，是**缩小 Agent 自由发挥的空间**。
2. 凡是存在唯一正确做法的事情，就不应该让模型每次现编，而应尽量写成脚本固化下来。
3. 这类脚本通常至少要满足几条硬要求。
    * 优先使用标准库和 POSIX shell，减少外部依赖。
    * 写状态文件时采用原子写入，并在并发场景下加锁。
    * 命令包装器只调用配置中声明过的入口，完整日志落盘，主会话只回传摘要。
    * 新脚本必须有测试，流程脚本修改后要跑回归，而不能只过一遍语法检查。
4. 脚本的价值在于兜住确定性。如果脚本本身不可靠，那它不仅不能减少 Agent 的不确定性，反而会把问题埋得更深。

## 4. Skill 怎么分层，为什么分层

1. Skill 的组织还可以继续往里拆成三层，而且三层职责不应重叠。
    ![[Attachment/5eb1e1a28e5dce12de3f7c2ac3333eee_MD5.jpg|434]]
    * 编排层（orchestrator）：通常只有一个，负责维护状态机和派发控制权，不干具体活。
    * 阶段层（phase）：对应交付链路的每一段，例如需求、设计、开发、评审、测试、发布，每段管理自己的输入输出和门禁。
    * 原子层（atom）：最底下的单一确定能力，每个只做一件事，不编排别的 skill，也不发明命令。
2. 阶段层里最关键的设计之一是 gate。每个阶段结束时，不应该只给出“做完了”这种模糊结论，而应明确返回四态之一：`pass`、`blocked`、`not_required`、`risk_accepted`。这样一来，后续编排逻辑才知道该继续、回退、跳过还是带风险放行。
3. 仅仅在概念上分层还不够，最好把整张图落实成一份单一事实来源的清单文件。每个节点至少要声明：属于哪一层、怎么执行、用于路由的描述、输入输出、可用工具、风险等级，以及是否需要人工复核。
4. 节点之间的边通常可以分成三类。
    * `handoff`：阶段之间的流转，由编排者驱动。
    * `dynamic_load`：按当前任务类型，在子 Agent 的新上下文里动态加载对应能力。
    * `semantic`：证据依赖，例如“发现候选”只产出候选地图，真正的定点证据必须由下游证据 skill 收敛。
    ![[Attachment/da2bb12205cd4da83d11e52fef52b6ba_MD5.jpg|517]]
5. 之所以坚持分层，是因为编排逻辑、阶段契约和原子能力的变化频率、失败模式完全不同。编排逻辑变动少，但一改就是全局的；阶段契约中等频率变，改的是某一段的输入输出；原子能力变化最频繁，但每次只应该动一个点。分层之后，每个能力都能被独立路由、独立测试、独立修改，而不会悄悄拖动别的部分。

## 5. 记住一句话

> **复杂任务的 Spec，核心是三件资产：工作流怎么编排、skill 怎么组织、知识库怎么建连。把这三件分别说清楚，比写一段更长的提示词更有用。**

1. 回过头看，这篇内容其实就是围绕这三件资产展开的。
2. 工作流编排对应前两节：多 Agent 决定任务交给谁、各自守什么边界；编排者入口决定从哪进、谁说了算、遇到问题是继续推进还是停下来等待。
3. skill 组织贯穿第三、四节：单个 skill 的文件怎么摆，整套 skill 怎么分成编排、阶段、原子三层，才能在能力增多时仍然保持清晰。
4. 知识库建连则主要落在第三节：rule 靠路径和角色自动挂上，docs 的每条知识都应当有消费方，skill 只在需要时点名读取自己的 references。这样一来，一条知识只有一个权威位置，改一处就能全局生效。
5. 这三件事合在一起，才能让“AI 到底在做什么”真正变得可追踪：每一步该谁做、该守什么、该产出什么证据、卡住往哪回退，都写在 Spec 里，而不是藏在某次对话的临场发挥中。
6. 这背后延续的其实还是同一个主题：克制。不要让能力随意生长，每加一个能力，都要明确它的位置、职责和契约。
7. 但写好 Spec 只解决了“组织是否清楚”，还没有回答“每个 skill 到底好不好用”。下一步真正要解决的，是怎么做对照测试、怎么沉淀结果，以及为什么不能随便用另一个大模型来当裁判。
