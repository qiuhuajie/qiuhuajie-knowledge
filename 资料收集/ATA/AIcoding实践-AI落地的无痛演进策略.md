从抗拒到依赖：AI落地的无痛演进策略 很多人对 AI 的真实情绪，并不是兴奋，而是复杂。 一方面，大家已经看到 AI 能写代码、拆任务、补测试、做文档；另一方面，真正把它放进研发流程后，最常见的感受却是：它不是不会干，而是很容易“看起来干完了”，但实际上干偏了、漏了，甚至在错误方向上越跑越远。 结合ATA上的一些优秀的案例和想法，探索了一套适合我们的模式，去解决【怎么让 AI 做得对、做得准、做得越来越懂我们】 一、我们到底碰到了什么问题 真正碰到的问题，其实并不抽象，反而非常一致。 1. 需求扔给 AI，由于一些信息缺失，AI 边猜边做，最后“假装完成”了 这是今天很多 AI coding 实践里最典型的问题。你把一个任务扔给 AI，它开始执行，过程看起来也很热闹，日志也不少，等了很久回头一看——它做错了。 错得最危险的地方不在于它失败了，而在于它经常看起来像完成了。因为 AI 缺少的不是行动力，而是对缺失信息的敏感度。信息不够时，人会本能地停下来问；AI 往往会自动补全，边猜边做，最后输出一个形式上完整、实际上偏离目标的结果。 很多人第一次把任务交给一些 harness 工程时，都经历过类似场景：任务扔进去，等了很久，日志看起来很热闹，过程也显得很努力，结果回头一看——做错了。 这类问题最伤信任，因为它消耗的不只是时间，还有注意力。人本来是想省脑力，最后却变成要花更多精力去复盘 AI 到底错在哪一层：是理解错需求了，还是任务拆错了，还是实现和验收标准根本没对齐。 所以，AI 落地的第一步，不是追求“让 AI 更自主”，而是先解决一个更现实的问题：如何避免 AI 虚假完成。 2. 上下文窗口导致长任务执行效果变差，做着做着就遗漏关键信息 第二个问题是长任务执行问题。 只要任务足够长，或者跨的阶段足够多，AI 的表现就会明显下滑。不是因为它不会做某一步，而是因为它在连续执行过程中，开始逐步丢失前面最关键的约束：哪些 AC 最重要、哪些接口是锚点、哪些边界条件必须守住、哪些内容是用户刚刚明确过的。 这是长任务缺少中间锚点的问题。只要任务过程中没有明确的回读、校验、阻断和重定位机制，AI 就很容易越做越偏。 3. 想自建垂直领域知识基座，但不知道从哪开始，做到什么粒度合适 很多人已经意识到，AI 如果只靠模型通用能力，是很难稳定理解本团队的业务上下文的。所以大家会自然想到：要不要建一个团队自己的领域知识库？ 问题在于，一旦真要开始做，马上就会卡住： 先做哪些领域？ 按项目建，还是按业务域建？ 一开始要写多细？ 是一次性梳理完，还是边做边补？ 于是这件事常常停在“有必要”这一层，但很难进入“怎么落地”。 4. 想建立代码和流程约束，但一时又想不起来具体规则 第四个问题更隐蔽。很多团队其实不是没有规范，而是规范散落在资深工程师脑子里、代码 review 评论里、群聊里、临时提醒里。 你真要问： 哪些命名习惯是团队默认要求？ 哪些评审点每次都要检查？ 哪些需求分析动作必须先做？ 大家往往能说出一些，但很难一次性完整说清楚。结果就是：规则明明存在，却无法被 AI 稳定执行。 5. OpenSpec 太重，而且有些诉求覆盖不到 最后一个问题是很多团队会参考 OpenSpec 一类方案，但真用起来又会觉得重。 OpenSpec 的优点很明显：流程正式、治理严谨、适合做规格驱动开发。但落到 AIcoding 场景里，很多团队会遇到两个痛点： 一是它更像一套治理流程，而不是一套适合 AI 长任务执行的“轻量锚点系统”； 二是很多团队真正想解决的不是“如何再加一层完整规范”，而是“怎么让 AI 在真实研发流里少猜、少丢、少跑偏”。 所以问题不在于 OpenSpec 不好，而在于它不一定适合每个团队当前的落地阶段。 二、我们的目标与解法 如果把这些问题收敛一下，本质上团队想要的目标其实很清晰： 目标 1：让 AI 做得对、做得准，而不是做得快但偏 对应的解法，不是单纯加大 prompt，也不是盲目增加规则，而是通过一套自建 spec 流，增强盲点识别和执行锚点。 在这个项目里，request-analysis 负责把原始需求加工成 spec.md、.feature、待确认清单和 AC。这里的重点不是“再写一份文档”，而是把模糊需求转成后续可以稳定引用的结构化锚点。 目标 2：把长任务执行交给 Harness 做得更好的 IDE / Agent 环境 长任务本来就不应该只靠一次性对话推进。更合理的方式，是把长任务的 orchestration 交给 harness 工程做得更好的环境，比如 opencode+omo、Claude Code 这类工具，让它们负责： 任务分波次执行； 中间状态记录； 任务失败后的熔断与继续； 文件级别的追踪和回写。 而 skills 要解决的不是“替代 harness”，而是为 harness 提供正确的轨道。 目标 3：通过归档环节，让 AI 随需求迭代不断总结领域知识 团队不可能一开始就写出一套完整领域知识库。更现实的做法，是让 AI 每做完一个需求，就把本次需求中已经被验证过的业务流程、规则、接口契约、数据模型补充进领域文档。 这就是 archival 的作用：不是把文档收尾，而是把一次交付变成团队长期知识资产。 目标 4：通过 skill 自进化，在使用中逐步沉淀规则和约束 团队规则不是一开始就设计完的，而是在使用中被不断发现的。所以与其一开始闭门梳理全部规则，不如在真实协作里，让系统把那些反复被强调、被纠正、被确认的内容记录下来，再在归档时统一确认、统一合并。 这就是 coding-skill、code-review 与 archival 联动的意义：让团队约束从“说过很多次”变成“以后 AI 也会照做”。 三、核心亮点一：盲点检测增强——避免 AI 虚假完成 这是整套方法里最先解决、也最能立刻带来体感改善的一点。 因为很多团队当前最痛的，不是 AI 什么都不会，而是它在信息不完整的时候继续执行。 在 request-analysis/SKILL.md 中，需求分析阶段不是直接产出 spec 然后往下推进，而是明确设计了几道关键闸门。 第一道闸门，是 .feature 生成后的 HARD STOP。AI 必须把结果展示给用户确认，当前响应必须结束，不能顺手继续生成 spec.md 或 tasks.md。这一步的本质，是强制 AI 在“自认为理解完成”之前接受一次外部校准。 第二道闸门，是 spec.md 初稿生成后的 盲区检测 和 交互式问答澄清循环。盲区检测会把缺失数据来源、边界条件不明、依赖关系不清的内容标记出来；交互式循环则要求这些问题全部被确认后，才能进入下一步。 第三道闸门，是进入 tasks.md 前的 验收标准完整性校验。这里要求： 每个功能点必须有 AC； AC 必须使用 Given-When-Then； AC 编号必须唯一； 所有待确认项必须清零。 这套机制解决的不是“让 AI 更慢”，而是让 AI 不要在错误前提上越做越多。 如果进一步进入 code-review/SKILL.md 和 code-implementation-review-spec.md，还能看到另外一层控制： AC 覆盖率不足自动升级为 P0； 图谱工具用于识别高风险改动和影响半径； 编码阶段的失败修复最多尝试 3 次，超过即熔断。 所以，盲点检测增强的真正价值，不只是“让 AI 多问几句”，而是把“该停下来的时机”制度化。这样 AI 才不容易“假装完成”。 四、核心亮点二：执行期间的spec - 一份人和 AI 都可以看懂的规范 很多人听到 spec，会本能联想到“重文档”。但在 AIcoding 场景里，真正有效的 spec，不是更厚，而是更像一种人机共享的执行接口。 传统 OpenSpec 的长处在于治理：提案、设计、任务、归档都很完整，适合严肃协同和正式审计。但它的问题也很明显：对于 AI 来说，很多内容仍然是“人类可以脑补理解”的自由文本。 而在这个项目里，spec 的作用不是停留在“定义清楚”，而是要继续参与后续执行。 体现在几个地方： 每个功能点必须有 AC 编号； AC 必须严格 Given-When-Then； 每个任务必须带 Spec锚点、关联 AC、目标文件、目标方法、伪代码、预期测试数量； 如果存在 .feature 文件，还要通过 @AC-XXX 把 Gherkin 场景和 spec 绑定起来。 这意味着 spec 不再只是“需求方和开发方沟通的文档”，而是后续 task 拆解、编码、测试、评审都要持续引用的结构化控制面。 request-analysis/REFERENCE/task-breakdown-spec.md 就很能说明这一点。它不只是说“把任务拆出来”，而是要求任务拆解遵守依赖顺序，并用 Wave 来组织可并行任务，同时固定单元测试任务和验收验证任务的位置。 所以，与其说这是在对比 OpenSpec，不如说它提供了一个更适合 AIcoding 的 spec 形态： 传统 OpenSpec 更偏治理； 这里的 spec 更偏执行锚点。 前者强调流程严谨，后者强调 AI 不跑偏。对很多团队来说，后者恰恰是当前更急需补上的能力。 五、核心亮点三：归档机制——分领域归档，逐步沉淀团队知识库 团队想建设团队 spec 或领域知识库，但又不知道从哪下手，这几乎是一个普遍问题。 这个项目给出的答案非常务实：不要先想着一次性建完，而是让知识随着需求交付不断长出来。 在 archival/SKILL.md 和 archival/REFERENCE/archival-spec.md 里，归档被放在 SDD 流程最后一环。它不是可有可无的总结动作，而是一个正式阶段： spec.md → tasks.md → 代码实现 → review/ → domain-spec → 归档 这里最关键的设计，是归档输出物不是某个需求总结，而是 specs/{领域名}-domain-spec.md。 这意味着知识不是按“需求编号”堆起来的，而是按“业务领域”沉淀的。这样后面无论是新人还是 AI，都更容易查： 某个领域有哪些核心业务规则； 某个领域有哪些关键流程； 哪些接口是对外契约，哪些只是内部实现； 哪些数据模型变化已经被正式沉淀。 domain-spec-template.md 把这件事做得很具体：领域概述、业务流程、业务规则、接口契约、数据模型、变更历史，全部按领域维度组织。 更重要的是，它强调的是增量更新。这点非常关键，因为团队知识库建设最怕的不是开始，而是维护。只要每次归档都只更新本次需求涉及的章节，这件事就能持续。 多仓库场景下，archival-spec.md 还进一步做了分层： 跨仓库内容写 workspace 级别 specs/； 单仓库内容写 repo-local 的 specs/。 这就避免了两种典型混乱： 全都堆在一起，最后没人知道边界； 每个仓库各写各的，最后权威版本失真。 所以这套归档机制真正解决的不是“怎么整理文档”，而是“怎么让团队从今天开始，低成本地把领域知识真正积累起来”。 六、核心亮点四：自进化 skill——团队约束在使用中自然形成 最后一个亮点，其实是最接近团队现实的一点。 因为团队规则往往不是预先被设计完整的，而是在使用过程中逐步浮现出来的。 在 coding-skill/SKILL.md 和 code-review/SKILL.md 中，都设计了“实时规则捕获”。当用户明确说“记住这条规则”，或者同一类问题被重复纠正两次以上，系统就会把这些内容记录到 .changes/${变更名称}/learned-rules-pending.md。 这件事的价值在于：它允许规则先在使用中被发现，而不是要求团队一开始就把所有规范想清楚。 然后到了 archival 阶段，再统一做几件事： 识别这些规则属于 request-analysis、coding-skill、code-review 还是 archival； 列给用户确认； 确认后再合并进对应 skill 的原始文档； 合并前先去重，避免重复累积。 这让“自进化”不再是一个失控的自动学习概念，而是一个可观察、可确认、可沉淀的过程。 对于团队来说，这非常重要。因为很多代码风格、流程约束、评审重点，并不是没人知道，而是没人能一次性整理完整。现在有了这套机制，团队完全可以在真实使用中逐渐把这些规则长出来。 也就是说： 规则不必一开始就完备； 规则可以在使用中被捕获； 规则会在归档时被真正固化； 下次 AI 执行时，就能自动继承这些团队经验。 这才是团队约束形成最自然、也最可持续的方式。 七、最终我更想强调的，不是把 skill 做重，而是找到“减负、约束、信任”的平衡 写到最后，我反而越来越相信一件事：我们不应该把 skill 做得非常笨重。 随着模型能力升级、底层 harness 功能升级，我们真正需要的，应该是不断“减负”。 这里的减负，不是把约束都去掉，而是把真正有价值的约束留下，把不必要的流程负担拿掉。也就是说： 通过约束，让 AI 走在我们希望的方向上； 同时，也要给 AI 足够的信任，让它承担真正的执行责任。 这个平衡可以用下面这张图来理解： 真正更好的状态，其实是在中间： 用 spec、AC、盲区检测、归档这些机制守住关键锚点； 把长任务 orchestration 交给 opencode+omo、Claude Code 这类 harness 能力更强的环境； 让 skill 负责方向、边界和团队记忆，而不是把每一步都写死。 所以如果把全文最后压缩成一句话，我会更愿意这样说： AI 落地，不是不断给 AI 增加负担，而是找到“减负、约束、信任”之间的平衡。 约束解决跑偏，spec 解决对齐，归档解决遗忘，自进化 skill 解决团队经验沉淀，而信任决定 AI 能不能真正成为生产力。 参考文档：
