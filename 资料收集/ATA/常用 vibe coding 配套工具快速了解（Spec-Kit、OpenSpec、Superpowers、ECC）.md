1. 功能对比 各种工具目的是一致的，让 vibe coding 「规范化」。对于这些工具，一种区分方式是将其分为「决策层」和「执行层」，决策层管写代码之前的事，执行层管写代码本身这件事。 本文提到的工具都适配 Claude Code、Cursor、CodeX、Gemini CLI 等。 工具 定位 核心能力 常用命令 优缺 Spec-Kit 规格驱动开发 SDD（决策层） 写代码前，生成结构化的规格文档，定义「做什么」 /speckit.constitution 建立项目规则（宪法） /speckit.specify 描述功能需求 /speckit.clarify 进行需求澄清 /speckit.plan 制定技术方案 /speckit.tasks 拆解方案生成可执行任务 /speckit.implement 基于之前所有规划，开发代码 优： 流程结构化，可控性拉满。避免氛围很对代码稀碎。 把规范文档作为主角，代码只是落地版本。 缺： 流程偏重，小需求如明确的bugfix完全没必要用。 对文档质量敏感，小偏差到后期可能无限放大。 OpenSpec 轻量级变更管理（决策层） 用三件套管理功能迭代，增量演进规格 /openspec:propose 提出变更提案，生成需求规范、技术方案、任务清单 /openspec:apply 基于文档，开发代码 /openspec:archive 变更记录归档 优： 流程结构化，但给人体感没那么重。 存在规范生命周期，增量规范 → 合并主规范 → 归档。 缺： 自动化程度高的同时又蕴含着耦合程度高的缺点，在需求未确定反复更改时比较浪费token Superpowers 工作流标准化（执行层） 通过skills强制执行工作流 skills会在开发过程中自动触发，无需手动调用 优： 让ai写代码的流程规范化，增强记录性 缺： 强制要走流程，比较烧token Everything Claude Code (ECC) agents、skills、hooks、rules 大全（执行层） 工具大全，按需安装使用 对自动触发的强制性没有那么强，更多需要手动调用 优： 作者宣称这些配置在多个生产应用中经过实战测试 缺： 工具太多了，需人工筛选实践出自己真正所需，全装太耗token了 2. 工具 以下工具的安装请参考官方文档，基本上都一两行命令就行。 2.1. Spec-Kit https://github.com/github/spec-kit Spec‑Kit 可以看做是 「文档规范化引擎」 ——它让 ai 能够自动生成、分析、评审、同步 PRD/SPEC 文档，确保团队的需求与设计文档 结构统一、内容完整、可直接驱动代码实现。 工作流程 项目初始化 执行 specify init <项目名> --ai <AI名称，如cursor>。目录摆好，AI “入职”。 立规矩（项目宪法） 执行 /speckit.constitution：把代码质量、测试要求、体验一致性、性能指标先写清楚——先讲家法，再动家伙。 写需求规范（Spec） 执行 /speckit.specify：只说“要做什么、为啥做”，别急着聊技术。AI 会开分支、生成 spec.md。 消除歧义（澄清完善） 执行 /speckit.clarify：AI 负责追问，你负责补齐；顺便把验收清单更新好，少走回头路。 出技术方案（Plan） 执行 /speckit.plan：这时再定技术栈、架构、API、数据模型。AI 还会写 research.md，你负责审一审：别过度设计，也别漏关键点。 拆任务（Tasks） 执行 /speckit.tasks：生成 tasks.md，按模块拆到“能直接干”的小任务，标依赖、可并行 [P]、文件路径，并按 TDD 顺序来。 开发（Implement） 执行 /speckit.implement：AI 按 tasks.md 写代码、跑命令、跑测试一路推进。你最后手测抓虫，扔回去让它改到服。 适用场景 适合启动一个全新的项目。它强制团队从一开始就采用结构化的规范，确保大型项目在AI协作下的可控性。对于小功能或Bug修复，它可能显得“过重”。 作为「决策层」，可以与「执行层」的 Superpowers / Everything Claude Code 结合使用。 2.2. OpenSpec https://github.com/Fission-AI/OpenSpec OpenSpec 是一个规范驱动开发的工具，它的核心理念非常简单：先定规矩，再写代码。 可以对比 spec-kit 发现，OpenSpec 指令少了很多，轻量化了很多。 工作流程 项目初始化 openspec init --tools <AI名称，如cursor>，项目目录执行 提出变更： /openspec:propose <变更需求描述> AI 会自动帮你创建一个变更文件夹，并且生成 4 个核心文档：proposal.md-为什么要做这个功能，要改什么、specs/-详细的功能需求规范、design.md-技术设计方案、tasks.md-拆分好的实现任务清单 可以发现，OpenSpec 一步做了 Spec-Kit 几步的事情 开发代码： /openspec:apply ，ai 就会按照第一步生成的规范文档，开始编写代码 归档 /openspec:archive 将本次变更归档，方便后续查询 适用场景 非常适合已有的项目。当需要在一个现存的代码库中添加新功能或进行重构时，OpenSpec 可以帮助你快速创建和管理“变更集”的规范，而无需重构整个项目。 作为「决策层」，可以与「执行层」的 Superpowers / Everything Claude Code 结合使用。 2.3. SuperPowers https://github.com/obra/superpowers 通过一系列可组合的“技能”（skills）强制 AI 编码代理遵循严格、可靠的软件开发方法论，而不是随意写代码。强调 TDD（测试驱动开发）、系统化调试、详细规划、子代理协作等，避免 AI 的常见问题（如跳过测试、过度复杂化）。 工作流程 Superpowers 中的技能会根据场景自动触发。 触发场景 自动激活的技能 开始写代码前 brainstorming - 澄清需求 需求确认后 writing-plans - 拆解任务 开始实现后 subagent-driven-development - 子代理执行 编写代码时 test-driven-development - TDD 流程 完成任务后 requesting-code-review - 代码审查 遇到 Bug 时 systematic-debugging - 系统化调试 适用场景 作为「执行层」，可以与「决策层」的 Spec-Kit / OpenSpec 结合使用。 2.4. Everything Claude Code（ECC） https://github.com/affaan-m/everything-claude-code Anthropic 黑客松冠军公开自己的Claude code 的配置和使用技巧，包括 skills、hooks、subAgents、rules、mcp 等全面的内容以及各种技巧。 可以看做是一个很会 vibe coding 的人的工具包大全。 工作流程 因为 ECC 是个工具包大全，所以最好按需选择安装，不然会装上很多不需要的工具。 支持的所有工具如下： everything-claude-code/\n|-- .claude-plugin/   #
