# 一、背景

## 1. 为什么我会重新思考个人知识库工具

1. 这篇文章想回答的，不是“Obsidian 有哪些功能”，而是另一个更实际的问题：如果今天要认真搭建一个可长期积累、可与大模型协同、可沉淀方法论的个人知识库，为什么我会更倾向于选择 Obsidian。
2. 过去几年，很多人第一次认真做知识管理，往往都是从云端笔记工具开始，Notion 当然是其中最有代表性的一个。它足够优雅，也足够通用，尤其适合做协作式文档、项目信息收集和轻量数据库。
3. 但当知识库从“记一点东西”进入“长期沉淀、持续迭代、不断复用”的阶段之后，工具的评判标准就会发生变化。这个时候，决定体验的往往不再是页面是否好看，而是底层数据是不是属于自己、结构能不能长期演化、是否方便和 AI 工具形成高频联动。
4. 这也是我从 Notion 逐步转向 Obsidian 的真正原因。

## 2. Karpathy 使用 Obsidian 的案例给我的启发

1. 我最早被 Obsidian 打动，并不是因为某个单独功能，而是因为看到一些真正长期使用者的案例之后，意识到它不是一个“笔记软件”，而更像一个个人知识操作系统。
2. 以 Andrej Karpathy（社区里也常被称作 kapaqi）分享的实践为例，Obsidian 的价值并不只是写笔记本身，而是把知识拆成可连接、可复用、可演进的最小单元，再通过目录、链接、标签、白板、插件和工作流把这些单元组织成自己的知识网络。
3. 2026 年 4 月，Karpathy 在 X 上分享了一套很有代表性的 LLM Wiki 工作流：他把论文、文章、图片、聊天记录等原始素材先放进 raw 目录，再让大模型把这些材料“编译”成 wiki 目录中的一组 Markdown 页面，包含摘要、概念条目、关系链接和可继续扩展的主题页面。
4. 在这套工作流中，Obsidian 更像知识系统的 IDE，LLM 更像持续维护 wiki 的程序员，而 wiki 本身就是那份不断演化的“代码库”。查询不再只是一次性问答，而是基于已有 wiki 持续吸收新信息、生成新见解，并把结果重新写回知识网络。
5. 这个案例对我最大的启发，是知识库的重点不再是“我收藏了多少篇文章”，而是“这些内容能不能被持续调用，最终变成自己的知识结构”。当 LLM 可以参与编译、更新和互链之后，知识积累开始呈现明显的复利效应。

## 3. 为什么我会从 Notion 转向 Obsidian

1. Notion 的优点很明显：上手快、界面整洁、数据库能力强、协作体验好，所以它非常适合做团队协同和轻量项目管理。
2. 但如果目标是搭建个人知识库，Notion 的一些优点会慢慢变成新的限制。
3. 第一，数据控制感偏弱。虽然 Notion 使用起来很方便，但它本质上仍然是一个平台化产品，你对内容的持有方式、导出形态、组织自由度和后续加工能力都受平台约束。
4. 第二，知识连接的摩擦仍然偏高。Notion 可以做页面引用、数据库关联，但它更擅长的是“结构化管理”，不一定是“低成本连接与漫游式探索”。而个人知识库非常依赖这种随手建立连接、逐步长出结构的能力。
5. 第三，与本地 AI 工作流结合时不够自然。很多大模型工具、脚本工具、命令行 Agent、自动化流程，都更适合直接操作本地目录、Markdown 文件和可版本化的文本资产。Obsidian 恰好天然兼容这一点。

## 4. Notion AI 的弊端为什么会成为转向点

1. 很多人一开始会觉得，既然 Notion 已经有 Notion AI，是不是意味着它更适合作为 AI 时代的知识工具？
2. 我的感受恰恰相反。Notion AI 的问题不在于它不能生成内容，而在于它更像是“附着在文档上的能力”，而不是“深度进入知识系统的能力”。
3. 第一，它对知识资产的调用边界依然受平台限制。你可以让它在页面里总结、改写、续写，但它很难像本地 Agent 一样，自由地跨目录检索、批量清洗、重组文件结构、自动归档、持续更新已有知识网络。
4. 第二，它更偏向“单次内容增强”，而不是“长期工作流自动化”。例如你想做定时抓取、自动归档、批量整理、跨来源汇总、脚本化更新，这些都是个人知识库里非常高频的需求，但 Notion AI 并不擅长承接这样的流程。
5. 第三，Notion AI 在真实使用中还有一些很明显的体验短板，例如生成内容未必会稳定沉淀为长期资产、上下文理解更偏页面级、对外部脚本和定时任务的集成能力有限，而复杂页面或数据库一旦变重，整体性能与调用成本也会同步上升。
6. 第四，当你开始高频使用 AI 时，你会越来越在意模型选择权、调用成本、自动化边界以及与外部工具的组合能力。Notion AI 在这些方面相对封闭，而 Obsidian + 本地文件 + Agent 工具的组合则更开放。
7. 所以，从 Notion 转向 Obsidian，不只是从一个笔记工具切换到另一个笔记工具，而是从“平台功能消费”转向“知识系统自定义”。

# 二、Obsidian 是什么，以及它为什么适合做个人知识库

## 1. Obsidian 的基础介绍

1. Obsidian 本质上是一个基于本地 Markdown 文件的知识管理工具。
2. 这一点非常关键，因为它意味着你的内容不是被锁在某个私有平台的数据结构里，而是以普通文本文件的形式存在于自己的目录中。
3. 你可以把这些 Markdown 文件当作最基础的知识资产：它们可以被 Obsidian 阅读，也可以被任何文本编辑器、脚本程序、Git 仓库、搜索工具和 AI Agent 直接处理。
4. 从产品形态上看，Obsidian 提供了几个核心能力：目录化组织、双向链接、标签系统、图谱视图、插件体系、白板能力，以及对 PDF、图片、音频、网页裁剪等内容的承接。
5. 如果只把它当成“写 Markdown 的软件”，其实低估了它。它真正强的地方，是把这些看似零散的能力组合成了一个足够开放的知识工作台。

## 2. Obsidian 适合做个人知识库的几个核心原因

### 2.1 本地文件是最稳定的知识底座

1. 个人知识库首先需要的是可控性，而不是花哨功能。
2. 基于本地 Markdown 的方案有一个天然优势：内容长期可读、可迁移、可备份、可版本化。
3. 这会带来非常强的安全感，因为即使未来不用 Obsidian，这些知识本身仍然是标准文本，不会随着平台变化一起失效。

### 2.2 链接比层级更重要

1. 文件夹当然重要，但一个真正能成长的知识库，不能只靠目录层级来组织。
2. Obsidian 的双向链接和块级引用，让“知识之间的关系”可以显式表达出来。
3. 你写一篇工具笔记时，可以顺手链接到方法论、案例、项目、专题入口；你整理一篇文章时，也可以很自然地把它挂回已有知识框架中。
4. 这会让知识库从“仓库”变成“网络”，而网络结构比树状结构更适合思考和复用。

### 2.3 插件生态让它更像可编程知识平台

1. Obsidian 很大的魅力在于它不是一个封闭产品，而是一个可以不断长能力的平台。
2. 你可以根据自己的工作方式去接入不同插件，例如网页裁剪、任务管理、日历、看板、Canvas、Dataview、模板系统、同步方案等。
3. 这意味着你的知识库不会被某一种固定交互方式限制，而是可以随着你的工作流逐步演化。

### 2.4 它天然适合和 AI 工具协同

1. 这一点是我现在最看重的原因。
2. 因为当知识以 Markdown 文件存在时，大模型不只是“帮你写一段文字”，而是可以真正参与知识库的建设：读取、搜索、总结、补全、归档、互链、改写、定时更新、自动生成专题综述。
3. AI 与知识库的关系，开始从“问答助手”变成“知识系统协作者”。
4. 这正是 Obsidian 在大模型时代最有价值的地方。

# 三、Obsidian 结合大模型工具的典型使用场景

## 1. 使用 Codex 快速做日程规划

1. 很多人对 AI 工具的第一反应，是拿它来写代码或总结文章，但实际上，日常知识工作的第一层价值常常来自“帮我整理今天该做什么”。
2. 如果把 Obsidian 作为个人知识工作台，那么日程安排、本周重点、任务拆解、行动优先级这些内容，本来就应该沉淀在同一个系统里。
3. 这时候 Codex 这类 Agent 工具的作用，不是替代思考，而是把原本零散的任务背景快速整理成一个可执行计划。
4. 一个很实用的工作流是：先把当前项目上下文、待办事项、会议纪要、截止日期等信息交给 Codex，让它输出一份结构清晰的日计划或周计划，然后再把结果整理进 Obsidian 的日记或待办笔记中。
5. 这样做的好处是，规划动作不再停留在聊天窗口，而是被固化进知识系统。之后无论是复盘、追踪任务演进，还是总结阶段产出，都能沿着同一个脉络继续往下走。

## 2. 使用 Hermes Agent 搭建个人 wiki，探索未知课题

1. 对我来说，Hermes Agent 与 Obsidian 的组合，是当前最接近“知识搭建助手”的方案之一。
2. 它不仅能回答问题，更重要的是能直接操作文件、沉淀内容、维护结构、管理记忆、保存技能，并把一次性的探索过程转成可复用的知识资产。
3. 这个工作流和 Karpathy 提到的 LLM Wiki 理念高度一致：先把原始资料沉淀到一个可持续扩展的素材层，再让 Agent 基于这些材料生成摘要、概念页、专题入口、关联页面和待补充问题，把原本零散的信息编译成可以继续生长的 wiki。
4. 一个非常适合的场景，是当我想理解一个陌生主题时，不再满足于只要一段答案，而是希望同步生成一个可以继续扩展的 wiki 入口。
5. 例如当我要研究一个新的 Agent 框架、一个陌生的工程方法论、一个全新的技术方向时，我会让 Hermes Agent 做几件事：先检索资料，再生成结构化综述，然后拆出主题分支，最后写回到 Obsidian 的 wiki 或专题目录里。
6. 这个过程的价值在于，它把“获取答案”升级成了“搭建知识框架”。下一次我继续研究这个主题时，面对的不是一段已经滚走的聊天记录，而是一个已经长出目录、关系和上下文的知识入口。
7. 如果说 Obsidian 负责承接知识资产，那么 Hermes Agent 负责加速知识架构的形成。两者组合后，探索未知课题就不再只是做信息查询，而是在不断扩建自己的认知地图。

## 3. 使用 Obsidian Web Clipper 快速抓取外部内容并归档进已有笔记系统

1. 个人知识库真正难的，不是写，而是把外部信息稳定、高质量地吸收进来。
2. 平时看到一篇 ATA 文章、一篇微信公众号文章、一个 Grok Chat 记录，如果每次都复制粘贴再手工清洗，成本会非常高，最后知识库就会逐渐断流。
3. Obsidian Web Clipper 在这里解决的是“低摩擦采集”问题：先把网页、文章、聊天记录快速抓进来，作为原始素材落到知识库中，而且它能够自动保留标题、来源、时间、摘要、图片等元信息，便于后续继续加工。
4. 但真正让这件事闭环的，不只是抓取本身，而是后续的归档工作流。
5. 我会把 Web Clipper 抓下来的内容统一落进资料收集区，再结合 Obsidian 官方能力、现有的归档规则，以及我自己定制化的写作/重写 skill，对内容进行去噪、重构、加标题层级、补 frontmatter、补关联笔记和文内链接，最后归档到正式知识目录。
6. 对 ATA 文章来说，这种流程特别有价值，因为很多 ATA 内容本身质量很高，适合沉淀为方法论或专题笔记；对微信公众号文章来说，它更适合做行业观察、产品分析、趋势判断；对 Grok Chat 来说，它更像是一次性对话材料，需要进一步抽取结论、上下文和后续可复用的信息。
7. 从实际效率看，这条链路的核心价值是把“抓取—整理—归档—互链”压缩成一个低成本动作：先快速采集，再由 Agent 和 skill 帮助完成结构化整理，最后并入已有知识网络，而不是让素材长期堆积在收藏夹或临时目录里。
8. 所以，Web Clipper 只是入口，真正重要的是：采集之后如何自动并入我已有的知识网络。只有当“抓取—整理—归档—互链”形成闭环，个人知识库才会越用越厚，而不是越堆越乱。

# 四、用 Hermes Agent 构建定时任务，让知识库持续生长

## 1. 定时任务是个人知识库从静态仓库走向动态系统的关键一步

1. 很多人的知识库停留在“有空再记”，所以它更像一个被动容器。
2. 但如果引入定时任务，知识库就会开始具备“主动摄取信息、定期更新专题、自动生成日报”的能力。
3. 这也是我非常看重 Hermes Agent 的原因之一，因为它不只是一个对话式 Agent，还内置了 Cron 调度能力，可以把很多原本需要人工重复执行的知识工作流自动化。

## 2. 利用 Ata-all Skill 定时获取 ATA 高质量文章并做每日汇总

1. ATA 上有不少值得长期追踪的高质量技术文章，但人工每天去筛选、打开、收藏、整理，动作会很碎。
2. 如果把这件事交给 Hermes Agent，就可以把“发现优质文章”变成一个定时执行的流程。
3. 一个可行的方案是：通过 ata-all skill 定时拉取 ATA 的目标文章列表，筛选出当天值得关注的内容，再自动生成一份“ATA 每日精选”摘要。
4. 这份摘要可以重点保留几个部分：文章标题、核心观点、为什么值得读、与我现有知识库的关联方向，以及建议归档到哪个专题目录。
5. 这样做的意义不只是省时间，更重要的是把“高质量输入”从偶发行为变成稳定机制。长期来看，它会显著提升知识库的信息密度。

## 3. 定时抓取 TechCrunch AI 栏目，并生成中文 AI 资讯报告

1. 如果说 ATA 更适合沉淀方法论和技术实践，那么 TechCrunch 的 AI 栏目更适合做行业动态跟踪。
2. 我希望把 https://techcrunch.com/category/artificial-intelligence/ 这一类持续更新的信息源，也纳入个人知识库的自动输入体系。
3. 一个很实用的自动化方式是：让 Hermes Agent 定时抓取该栏目最近更新的文章，提取重点事件、产品发布、融资动态、模型进展、行业趋势，再统一整理成一份中文 AI 资讯报告。
4. 这份报告不需要追求面面俱到，重点是帮助我快速判断：今天有哪些值得关注的 AI 新闻，它们分别属于技术、产品、资本还是生态层面的变化，这些变化是否会影响我已有的观察框架。
5. 如果再进一步，这份中文报告还可以和我知识库中的既有专题互链，例如 Agent、开源模型、AI Coding、行业应用等，从而让新闻流进入长期知识体系，而不是停留在一次性的阅读消费上。

## 4. 为什么说“定时任务 + 知识库”会改变使用方式

1. 当知识库开始自动接收高质量文章、行业新闻和专题摘要时，你使用它的方式会发生变化。
2. 它不再只是你有想法时才打开的地方，而是一个会持续长出新内容、持续提醒你哪里值得继续研究的知识环境。
3. 从这个角度看，Obsidian 提供的是知识承载层，Hermes Agent 提供的是知识自动化层，而 skill、记忆、归档规则和定时任务则共同构成了这套系统的运行机制。
4. 这也是我认为“Obsidian + 大模型 Agent”值得认真实践的原因：它真正改变的不是记笔记效率，而是个人知识系统的生长方式。

# 五、结语

## 1. Obsidian 的价值，不只是替代 Notion

1. 如果只是把 Obsidian 理解为“一个更自由的 Notion 替代品”，其实还是低估了它。
2. 它真正的意义在于：让个人知识库重新回到用户自己手里，并且为 AI 时代的知识工作流提供了一个足够开放、足够稳定、足够可编排的底座。
3. 对我来说，选择 Obsidian 并不是为了追求功能更多，而是为了让知识资产更可控，让知识结构更容易生长，让 AI 工具真正参与到知识建设过程里。

## 2. 真正值得投入的，是“知识系统”而不是“单个工具”

1. 工具本身当然重要，但更重要的是你是否围绕它建立了一套可持续的工作流。
2. 从 Notion 转向 Obsidian，只是第一步；真正关键的是后面这条链路有没有建立起来：采集、整理、归档、互链、复盘、更新、自动汇总、定时追踪。
3. 当 Codex、Hermes Agent、Web Clipper、定时任务、写作 skill、归档规则都逐步接入之后，Obsidian 才真正从一个笔记工具，变成了个人知识库的基础设施。
4. 这也是本文想表达的核心观点：在大模型时代，最值得搭建的不是一个“会记笔记的软件使用习惯”，而是一套“可以不断吸收信息、组织认知、辅助决策、持续进化”的个人知识系统。

## 3. 后续可以继续扩展的方向

1. 如果这篇文章后续要继续完善，我建议至少还可以补三类内容。
2. 第一类是截图和真实案例，例如 Obsidian 的目录结构、Web Clipper 抓取示例、Hermes Agent 定时任务示意、Codex 生成日程规划的示例输入输出。
3. 第二类是更具体的操作说明，例如如何设计资料收集区、如何定义归档命名规则、如何规划 wiki 目录、如何设计日报/周报模板。
4. 第三类是失败经验和踩坑总结，例如从 Notion 迁移时最容易出现的问题、知识库目录过深带来的维护成本、AI 生成内容如何避免污染正式知识层。
5. 这些内容补进去之后，这篇文章就会从“观点型初稿”进一步升级成“既有方法论、又有实操路径”的完整分享稿。

# 六、Obsidian 插件推荐：基于我当前正在使用的插件配置

## 1. 为什么插件推荐要基于“正在使用”而不是“网上榜单”

1. 写 Obsidian 插件推荐时，我更倾向于从自己当前真实装着、正在使用的插件出发，而不是简单罗列一份热门榜单。
2. 因为插件这件事很容易陷入“装很多、用很少”的误区，真正有价值的不是插件数量，而是它们是否真的嵌进了你的知识工作流。
3. 我当前这套 Vault 里，核心插件大致可以分成几个方向：模板与自动化、知识组织与检索、任务与规划、可视化表达、写作编辑增强、附件与 PDF 管理，以及 AI 协同。
4. 下面这份推荐，就是基于我当前已经安装的插件清单依次整理出来的，更适合作为一套可落地的 Obsidian 插件组合参考。

## 2. 模板、结构化与知识组织类插件

### 2.1 Templater

1. Templater 是我最推荐的基础插件之一，它解决的是“如何把重复性的笔记结构标准化”的问题。
2. 不管是日记、周报、项目笔记、专题页、归档页还是 wiki 页面，只要你有固定结构，就可以交给 Templater 自动生成。
3. 对个人知识库来说，它的价值并不只是省时间，而是让你的笔记从一开始就具备一致的骨架，后续无论是 Dataview 检索，还是 Agent 自动写入，都会顺畅很多。

### 2.2 Dataview

1. Dataview 是 Obsidian 从“笔记软件”升级成“知识数据库”的关键插件之一。
2. 它可以把散落在不同 Markdown 文件中的 frontmatter、标签、任务和字段重新查询、聚合和展示出来。
3. 如果你想做专题导航、项目总览、待读清单、日报周报汇总，甚至想把 wiki 页面自动组织成内容索引，Dataview 几乎都是绕不过去的基础设施。

### 2.3 Number Headings

1. Number Headings 非常适合中文长文写作和结构化知识整理。
2. 当一篇文章层级很多时，自动编号可以显著提升结构感，尤其适合方法论、教程、综述和归档笔记。
3. 对我这种经常写长篇知识型文章的人来说，它最大的价值是降低排版维护成本，让层级结构始终保持稳定。

### 2.4 Quiet Outline

1. Quiet Outline 是我非常喜欢的目录增强插件。
2. 它比原生 Outline 更安静、更聚焦，也更适合在长文编辑时快速定位章节。
3. 当文章已经进入多章节、多层级状态时，一个好用的侧边目录能显著提升写作流畅度。

### 2.5 Outliner

1. Outliner 会把列表编辑体验拉近 Workflowy 或 Roam 那种大纲式操作方式。
2. 如果你经常用列表来做观点拆解、会议记录、思维发散、任务嵌套，它会明显提升编辑效率。
3. 对于“先列提纲、再扩正文”的写作方式来说，这类插件会比普通 Markdown 编辑体验更顺手。

### 2.6 File Explorer Note Count

1. 这个插件的价值很朴素，但非常实用：它能直接在文件夹旁边显示笔记数量。
2. 当你的 Vault 目录越来越大时，这种可见性会帮助你快速判断哪些目录在持续生长，哪些目录还是空壳。
3. 对长期维护知识库的人来说，这其实是一种很轻量的“结构体感”。

### 2.7 Manual Sorting

1. Manual Sorting 适合那些不想完全依赖文件名排序的人。
2. 有些目录本质上是内容入口、专题入口或阅读路线，手动调整顺序会比字母排序更符合真实阅读路径。
3. 如果你会认真经营某些入口目录，这个插件会很有帮助。

### 2.8 File Color

1. File Color 让文件树从“纯文本列表”变成更具辨识度的视觉导航区。
2. 你可以用颜色区分资料收集区、正式知识区、项目区、日程区等不同目录。
3. 这种视觉分层在内容很多时会非常省认知成本，尤其适合有稳定分类体系的人。

### 2.9 Banners

1. Banners 让笔记首页更有“页面感”，适合专题页、MOC、项目首页和长文封面。
2. 它不是必须插件，但对于希望知识库既可用又好看的用户来说，能明显提升阅读和展示体验。
3. 如果你准备把 Obsidian 笔记当成半成品文章或长期专题入口，Banner 的视觉价值会很明显。

## 3. 日程规划、项目推进与可视化类插件

### 3.1 Calendar

1. Calendar 是 Daily Notes 工作流的经典插件。
2. 它让日记、日志、复盘与日期之间建立起非常自然的入口关系，适合每天都在 Obsidian 中工作的人。
3. 对我这种会把日程、记录、阶段复盘都沉淀在同一个系统里的人来说，Calendar 基本属于长期必备。

### 3.2 Day Planner

1. Day Planner 更进一步，把“有日期的笔记”推进成“有时间块的日程管理”。
2. 如果你希望把 AI 生成的日计划、会议安排、深度工作时间块直接放进 Obsidian，这个插件会非常实用。
3. 它也很适合和 Codex / Agent 生成的计划结合，把规划结果落成真正可执行的日程。

### 3.3 Kanban

1. Kanban 适合那些不想切到其他任务软件、而希望继续在 Markdown 世界里管理项目状态的人。
2. 它让待办事项、阶段任务、项目推进过程拥有更直观的看板视图。
3. 如果你的项目管理偏轻量，Kanban 往往比再开一个外部工具更顺手。

### 3.4 Excalidraw

1. Excalidraw 是 Obsidian 生态里非常强的可视化表达插件之一。
2. 它特别适合画知识结构图、流程图、系统关系图、课程框架图，以及各种手绘风格的思维整理图。
3. 对知识型写作来说，很多时候图不是装饰，而是帮助你厘清结构本身。Excalidraw 在这类场景下非常强。

### 3.5 Advanced Canvas

1. 如果说 Excalidraw 更像自由手绘白板，那么 Advanced Canvas 更适合做结构化拼接与内容编排。
2. 它适合搭建研究地图、专题资料墙、项目结构白板、演示型知识页面。
3. 对于需要把很多笔记、卡片、截图、链接放在一个工作台里进行关系整理的人来说，这个插件很值得装。

## 4. 写作编辑增强与阅读体验类插件

### 4.1 Editing Toolbar

1. Editing Toolbar 会把很多常用编辑动作收拢到一个更直观的工具栏里。
2. 对习惯可视化按钮操作，或者经常在移动端写作的人来说，它能显著降低 Markdown 输入门槛。
3. 如果你既写知识笔记，又写偏发布型文章，这类编辑增强插件会让流程更顺。

### 4.2 Commander

1. Commander 的核心价值是把命令做成更顺手的入口，例如按钮、宏命令和快捷操作。
2. 当你的 Obsidian 工作流越来越复杂时，很多动作其实不是“不会做”，而是“入口太深”。
3. Commander 很适合把高频动作提到前台，尤其适合和模板、归档、打开特定视图等操作结合。

### 4.3 Codeblock Customizer

1. 如果你在笔记里经常写代码、命令、配置片段，这个插件会让代码块展示更舒服。
2. 它既能改善阅读模式，也能改善编辑模式下的代码块体验。
3. 对技术写作、教程整理和 AI 生成代码片段归档来说，这类细节增强会明显提升可读性。

### 4.4 Easy Typing / Highlightr 等细节增强插件

1. 这类插件解决的不是“大功能”，而是长期写作中非常高频的小摩擦。
2. 比如中文输入体验、文本高亮、局部格式调整，这些看起来小，但日积月累会明显影响写作顺手程度。
3. 如果你每天都在 Obsidian 里写很多内容，这类“细节打磨型插件”其实很值得保留。

## 5. 附件、图片与 PDF 管理类插件

### 5.1 Attachment Management

1. 对个人知识库来说，附件管理是否清晰，会直接影响后续可维护性。
2. Attachment Management 可以按规则管理附件路径，并在文件变化时自动重命名或调整路径。
3. 如果你的 Vault 不只是纯文本，而是包含截图、文章配图、PDF 附件、流程图，这个插件非常有必要。

### 5.2 Local Images Plus

1. 这个插件非常适合网页资料归档场景。
2. 它可以把笔记里的外部媒体链接下载到本地，并把链接替换成 Vault 内部引用。
3. 对使用 Web Clipper、大量抓取网页资料的人来说，这能避免“几年后图片失效”的问题，是非常关键的资料固化能力。

### 5.3 Imagine

1. Imagine 是一个更偏图片管理和处理的插件，适合图片较多的知识库。
2. 它能帮助你更方便地插入、查看、调整和批量处理图片。
3. 如果你的文章会加入大量示意图、截图、封面图，它会让图像工作流更完整。

### 5.4 PDF++

1. PDF++ 是阅读型知识库里很值得保留的插件。
2. 它让 PDF 标注、跳转、引用和阅读体验更贴近 Obsidian 原生工作流。
3. 对于经常读论文、白皮书、技术文档的人来说，这意味着 PDF 不再是知识库外围材料，而是可以更紧密地纳入笔记系统内部。

## 6. AI 协同与自动化增强类插件

### 6.1 Claudian

1. Claudian 这类插件的意义，不只是把 AI 放进 Obsidian，而是把 Agent 工作流直接嵌进 Vault。
2. 它能把 Claude Code 这类能力带进 Obsidian，使 AI 可以更直接地围绕当前 Vault 做文件读写、搜索、命令执行和多步骤处理。
3. 对于希望把 Obsidian 变成 AI 协作工作台的人来说，这一类插件代表的是非常明确的下一阶段方向。

### 6.2 Execute Code

1. Execute Code 很适合技术型知识库用户。
2. 它允许你直接在笔记里执行代码片段，把一些“可运行说明”变成真正可以验证的内容。
3. 如果你在 Obsidian 里写教程、实验记录、数据分析笔记或技术方案，这类能力会很有价值。

### 6.3 其他 AI/自动化方向插件

1. 从我当前的插件结构也能看出来，我对 Obsidian 的期待并不只是“写笔记”，而是“让它逐步拥有自动整理、自动执行、自动生成和自动协作的能力”。
2. 这也是为什么我会把 Web Clipper、模板、Dataview、Agent、代码执行和附件本地化这些能力放在同一套系统里理解。
3. 在大模型时代，插件不再只是功能补丁，而是在不断把 Obsidian 从记录工具推进成知识操作系统。

## 7. 我对 Obsidian 插件选择的几个原则

1. 第一，不要只追求“插件多”，而要优先选择能真正嵌进日常工作流的插件。
2. 第二，优先补齐结构化、自动化和资料固化能力，因为这些能力最能决定知识库是否可持续。
3. 第三，视觉和编辑体验类插件可以有，但最好建立在知识结构已经稳定之后。
4. 第四，一套好的插件组合，最终目的不是让 Obsidian 更花哨，而是让你更低成本地采集信息、组织结构、生成内容、管理项目和沉淀长期知识。
5. 所以如果让我给插件推荐做一个总结，我会说：先搭骨架，再补体验；先保证知识能沉淀，再追求界面好看；先服务工作流，再考虑收藏插件本身。
